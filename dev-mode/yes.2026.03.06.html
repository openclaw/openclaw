SEC-27 Untrusted Channel Metadata Blocked from System Prompts 2026.2.3
What it does
Channel metadata from Slack and Discord (channel name, topic, description) is stripped from the AI's system prompt. The agent can't "see" what channel it's operating in, losing useful context about conversation purpose and scope.
Source files
  • System prompt construction modules (released in 2026.2.3)
Dev-mode plan
Re-include channel metadata (name, topic, description) in the system prompt so the agent knows where it's talking and can adapt its behavior to the channel's purpose.
SEC-59 Default Tool Profile Restricted to "messaging" 2026.3.2
What it does
New installs default to the messaging tool profile, which only allows 5 tools: sessions_list, sessions_history, sessions_send, session_status, message. All powerful tools (file ops, shell, web, memory, scheduling, subagents) are hidden unless you manually switch to coding or full.
Source files
  • src/commands/onboard-config.ts line 6
  • src/agents/tool-catalog.ts lines 248-259
  • src/agents/tool-policy-shared.ts lines 45-47
Dev-mode plan
Skip setting tools.profile during onboarding. When unset, resolveCoreToolProfilePolicy() returns undefined, the pipeline step is skipped, and all tools are available. Nothing breaks — the field is optional everywhere.
SEC-67 Compaction Safeguard Mode (Default) 2026.2.1
What it does
Compaction mode defaults to "safeguard" which applies stricter guardrails during session history compression. Requires structured summary headings, cancels compaction if no real conversation exists or if the content ratio is too aggressive. Slower and more conservative than "default" mode.
Source files
  • src/agents/pi-extensions/compaction-safeguard.ts (lines 1-690)
  • src/agents/pi-extensions/compaction-safeguard-runtime.ts
  • src/agents/pi-embedded-runner/extensions.ts (lines 60-88)
  • src/config/defaults.ts (line 526)
Dev-mode plan
Set default compaction mode to "default" instead of "safeguard". Faster compression with less overhead, at the cost of potentially losing more context during long sessions.

SEC-96 Host Environment Variable Sanitization for Child Processes 2026.2.19
What it does
Blocks dangerous environment variables (AWS secrets, API keys, tokens, passwords, private keys) from being passed to child processes when the AI runs commands. Broader than SEC-11 (which only blocks LD/DYLD vars). In dev, your scripts can't access env vars that OpenClaw considers sensitive.
Source files
  • src/infra/host-env-security.ts (lines 1-100+)
  • src/infra/host-env-security-policy.json (blocked variable list)
Dev-mode plan
Pass all env vars through to child processes without filtering.

When a new agent is created, the workspace bootstrap creates AGENTS.md, SOUL.md, TOOLS.md, IDENTITY.md, USER.md, HEARTBEAT.md, and BOOTSTRAP.md — but never creates MEMORY.md. New agents end up with no memory or accidentally read the main agent's memory file.
Source files

  • src/agents/workspace.ts lines 32-33 (constants exist but unused)
  • src/gateway/server-methods/agents.ts lines 476-546
  • src/commands/agents.commands.add.ts lines 51-368
Fix plan
Add an empty MEMORY.md template to the templates directory and include it in the bootstrap file list in ensureAgentWorkspace so each new agent gets its own isolated memory file on creation.
FIX-02 Integrate YourJarvisHub as Built-in Notification Hub feature
What it does
External apps, cron jobs, and scripts have no first-class way to send focused notifications to the agent. Messages either get lost in the heartbeat stew (enqueueSystemEvent) or require a separate hub server. This integrates the YourJarvisHub concept directly into OpenClaw.
Planned endpoints
  • POST /notify — submit notification with priority
  • POST /done/{id} — confirmation callback
  • GET /pending — list unhandled notifications
  • GET /history — notification audit trail
Integration plan
Add a built-in notification hub at src/dev-mode/notification-hub/. Each notification triggers a dedicated agent turn via the chat completions path (not heartbeat). Cron isolated jobs get a new delivery.mode: "notify". Config: gateway.http.endpoints.notificationHub.enabled=true.
