# Task: Fix raw API errors leaking to chat surface (Issue #11038)

## Problem

When a session transcript becomes corrupted (orphaned tool_result without matching
tool_use), the raw Anthropic API error is delivered directly to Telegram/Discord:

```
LLM request rejected: messages.144.content.1: unexpected tool_use_id found in
tool_result blocks: toolu_01HjX9c7NLJaBLDzyBasSkKw. Each tool_result block must
have a corresponding tool_use block in the previous message.
```

This raw API error:

- Exposes internal message indices and tool_use IDs
- Spams the channel on every subsequent message
- Contains model implementation details users should not see

## Expected behavior

1. Raw API errors like this should NOT reach the channel
2. Channel should receive a generic safe message
3. Full error stays in internal logs only

## Key files to investigate

- `src/agents/pi-embedded-helpers/errors.ts` — `formatAssistantErrorText()`, has patterns for various error types
- `src/auto-reply/reply/agent-runner-execution.ts` — where errors become channel payloads
- `src/agents/pi-embedded-helpers/turns.ts` — tool_use_id validation logic already exists
- Look for where "unexpected tool_use_id" or "tool_result" API errors reach the reply payload

## What the fix should do

1. Detect the "unexpected tool_use_id" / orphaned tool_result error pattern
2. Add it to the error classification in `errors.ts` (like `isCompactionFailureError`)
3. In `agent-runner-execution.ts`, intercept this error class and replace with:
   `"⚠️ Session context error. Use /new to start a fresh session."`
4. Keep full error in internal logs
