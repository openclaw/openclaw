## 1. Paperclip: ИИ-компания из 8 агентов

| Параметр   | Значение                              |
| ---------- | ------------------------------------- |
| **URL**    | https://habr.com/ru/articles/1015470/ |
| **Автор**  | @DVZakusilo                           |
| **Чтение** | ~9 мин                                |
| **Охват**  | 5.6K                                  |

### Суть

**Paperclip** — open-source платформа (30.6k GitHub stars за 12 дней), моделирующая AI-компанию: CEO, CTO, инженеры, дизайнеры — каждый «сотрудник» — LLM-агент. Поддерживает BYO Agent (OpenClaw, Claude Code, Codex, Cursor, Bash, HTTP).

### Ключевые механизмы

- **Оргструктура как код**: иерархия ролей, бюджетирование на уровне атомарных операций
- **Heartbeat**: агенты просыпаются по расписанию, берут задачи, засыпают (не жрут токены 24/7)
- **Goal Alignment**: каждая задача знает родословную от миссии компании до конкретного тикета
- **Бюджеты**: 80% лимита → предупреждение, 100% → агент отключается

### Результат эксперимента

- 8 агентов за 2 дня: **0 строк полезного кода**, 79 000 токенов на болтовню
- 90% времени ушло на координацию, 10% на работу
- Один агент (Cursor) за **1.5 часа** сделал всю задачу

### Вывод автора

> «Paperclip имитирует не работу, а бюрократию.» Правильная модель: человек = Board of Directors (утверждает стратегию, устанавливает бюджеты, вмешивается только при отклонениях). Но текущие LLM недостаточно умны для такой автономии.

## 2. Пайплайны, bounce-back и cron для ИИ-агентов (agent-pool-mcp)

| Параметр   | Значение                              |
| ---------- | ------------------------------------- |
| **URL**    | https://habr.com/ru/articles/1014502/ |
| **Автор**  | @Mr_FatCat                            |
| **Чтение** | ~4 мин                                |
| **Охват**  | 7.6K                                  |

### Суть

**agent-pool-mcp** — MCP-сервер для мультиагентной оркестрации поверх Gemini CLI. Фрактальная архитектура: воркер может сам делегировать задачи дальше.

### Ключевые механизмы

- **Pipeline**: `create_pipeline` собирает шаги с триггерами (`on_complete`, `on_complete_all`, `on_file`), демон detached
- **Bounce-back**: возврат задачи на предыдущий этап с фидбеком (аналог «Request Changes»), защита `maxBounces`
- **Cron scheduler**: `schedule_task` с cron-синтаксисом, атомарные файловые блокировки (10 IDE → 1 выполнение)
- **SSH runners**: задачи на удалённых серверах, ночная работа без ноутбука
- **Сессии**: передача `session_id` между агентами для сохранения контекста
- **Политики**: `read-only`, `safe-edit`, кастомный YAML с точным списком инструментов
- **Группы**: `create_group` с наследованием раннера/скилла/политики, `max_agents` для защиты от размножения

### Архитектурная диаграмма

```
          ┌─ frontend ─┐
research ─┤            ├── deploy
          └─ backend  ─┘
```

### Релевантность для OpenClaw

Модель pipeline + bounce-back + политики близка к нашему `PipelineExecutor`. Стоит рассмотреть: bounce-back механизм для reflexion_fallback, cron для scheduled tasks, политики ограничения инструментов.

## 3. Агентная разработка в VS Code: Настройка RooCode (мультиагентная система)

| **Автор**  | @VedgeKonn                            |
| **Чтение** | ~24 мин                               |
| **Охват**  | 7.3K                                  |

### Суть

Глубокий туториал по настройке RooCode — расширения VS Code с мультиагентной системой из 6 **Модов** (агентов), каждый со своей LLM и температурой.

### Архитектура MAS из 6 агентов

| Мод                  | Роль                     | Температура | Логика                                   |
| -------------------- | ------------------------ | ----------- | ---------------------------------------- |
| **Orchestrator**     | Главный, управляет всеми | 0.8         | Творческое планирование, НЕ пишет код    |
| **Ask**              | Советник                 | 0.5         | Отвечает на вопросы                      |
| **Project Research** | Ищейка                   | 0.3         | Точный поиск участков кода               |
| **Architect**        | Архитектор               | 0.8         | Самая умная модель, ключевые решения     |
| **Code**             | Кодер                    | 0.3         | Легковесная модель, следует инструкциям  |
| **Debug**            | Дебаггер                 | 0.8         | Умная модель, творческое решение проблем |

### Ключевые паттерны

- **Оркестратору запрещено исполнять код** → вынужден делегировать → оптимизация токенов
- **Скиллы** (SKILL.md): MCP-скиллы (`mcp-*`), CLI-скиллы (`cli-*`), пользовательские (`usr-*`)
- **MCP Marketplace**: готовые серверы (git, поиск и др.)
- **Auto-Approve**: гранулярные разрешения для операций, белый/чёрный список команд
- **Enhance Prompt**: LLM перефразирует запрос перед выполнением (Documentation-First + TDD workflow)

### Релевантность для OpenClaw

- Модель «каждый агент = своя температура + своя модель» идеально ложится на нашу `openclaw_agents.json` конфигурацию
- Паттерн «Оркестратор не пишет код» = наш planner vs tool_execution агенты
- Скиллы как SKILL.md = наша `data/local_skills/` система

## Выводы и применимость к OpenClaw Bot

1. **Bounce-back паттерн** (agent-pool) стоит интегрировать в `reflexion_fallback` — возврат с конкретным фидбеком вместо слепого retry
2. **Разделение температур по ролям** (RooCode) уже реализовано в нашем `openclaw_agents.json`, но стоит сделать более гранулярным
3. **Оркестратору запрещено писать код** (RooCode) — хороший паттерн для нашего planner-агента
4. **Heartbeat** (Paperclip) — потенциально полезен для Brigade/cron задач
5. **Иерархия «компании»** (Paperclip) — **анти-паттерн** при текущем уровне LLM; плоская структура с pipeline эффективнее
