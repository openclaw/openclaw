Purpose: Prevent the agent from leaking internal state (file paths, system info, API keys) or external URLs in responses to channels with restrictive output policies. This is the output-side complement to Phase 2's input detection.

Output: `src/security/output-policy.ts` with rule definitions and filter function, extended `SecurityConfig` with `outputPolicy` section, CSP filter integrated into `normalizeReplyPayload()`, tests.
</objective>

- `OutputCspRule` type: `{ id: OutputCspRuleId; detect: (text: string) => { matched: boolean; matches: string[] }; redact: (text: string) => string }`

- Implement 6 rules in a `RULES` Map:
  - `no-external-urls`: Detect `https?://` URLs excluding loopback (127.0.0.1, localhost, ::1) and RFC 1918 (10.x, 172.16-31.x, 192.168.x). Redact to `[URL redacted]`.
  - `no-file-paths`: Detect Unix paths starting with `/home/`, `/Users/`, `/tmp/`, `/var/`, `/etc/`, `/opt/`, `/usr/`, `/root/`, `/mnt/`, `/media/`, `/srv/` and Windows paths `C:\...`. Redact to `[path redacted]`.
  - `no-code-blocks`: Detect triple-backtick fenced code blocks. Redact to `[code block redacted]`.
  - `no-system-info`: Detect common system info patterns: hostname patterns, kernel versions, OS release strings, environment variable dumps (`KEY=value` lines). Redact to `[system info redacted]`.
  - `no-api-keys`: Detect common API key patterns: `sk-`, `pk_`, `api_key`, Bearer tokens, AWS keys (`AKIA`). Redact to `[key redacted]`.
  - `no-internal-ips`: Detect RFC 1918 IP addresses (10.x, 172.16-31.x, 192.168.x) appearing standalone (not in URLs already caught by no-external-urls). Redact to `[IP redacted]`.

- `applyOutputCsp(text: string, rules: OutputCspRuleId[]): OutputCspResult` — iterate rules in order, apply detect+redact for each match, collect stripped rules.

- `resolveChannelOutputRules(channel: string, config: OutputCspConfig): OutputCspRuleId[]` — follow the same pattern as `resolveChannelSensitivity` from `input-screening.ts`: case-insensitive channel lookup in `config.channels`, fall back to `config.defaultRules`, fall back to empty array.

  ```typescript
  /** Channel identifier for CSP security event emission. */

  ```

- In `normalizeReplyPayload()`, AFTER the `sanitizeUserFacingText()` call (line ~65) and BEFORE the LINE directives check, add CSP filtering:
  ```typescript
  if (text && opts.outputCspRules && opts.outputCspRules.length > 0) {
    const cspResult = applyOutputCsp(text, opts.outputCspRules);
    if (cspResult.strippedRules.length > 0) {
      for (const stripped of cspResult.strippedRules) {
        emitSecurityEvent({
          eventType: "output.csp.stripped",
          timestamp: new Date().toISOString(),
          severity: "warn",
          channel: opts.outputCspChannel,
          sessionKey: opts.outputCspSessionKey,
          action: "redacted",
          detail: `Rule ${stripped.ruleId} matched ${stripped.matches.length} pattern(s)`,
          meta: {
            ruleId: stripped.ruleId,
            matchCount: stripped.matches.length,
            originalMatches: stripped.matches.slice(0, 5),
          },
        });
      }
      text = cspResult.text;
    }
  }
  ```

- This placement ensures CSP runs BEFORE chunking (normalizeReplyPayload runs before BlockReplyPipeline splits text) and BEFORE LINE directives parsing.

2. The caller (reply-dispatcher.ts `normalizeReplyPayloadInternal`) does NOT need changes in this plan — the new options are optional. Wiring the config resolution into the reply dispatcher is a natural follow-up but the filter is functional and testable through `normalizeReplyPayload` directly.

Note: Do NOT modify reply-dispatcher.ts. The optional fields on NormalizeReplyOptions allow callers to opt-in to CSP without breaking existing code paths.
