## Service Lifecycle

## RPC Method Design

### Should RPC methods read from .index/ JSON or parse markdown live?

| Option                               | Description                                                                          | Selected |
| ------------------------------------ | ------------------------------------------------------------------------------------ | -------- |
| Read .index/ JSON                    | Fast, already generated by ProjectSyncService. Consistent with architecture pattern. | ✓        |
| Parse markdown live                  | Always fresh but slower, duplicates parsing logic, contradicts architecture.         |          |
| Hybrid — JSON with markdown fallback | .index/ preferred, markdown fallback. More resilient but complex.                    |          |

**User's choice:** Read .index/ JSON (Recommended)
**Notes:** Consistent with established architecture: markdown = source of truth, JSON = read cache.

### What should projects.list return?

| Option               | Description                                                          | Selected |
| -------------------- | -------------------------------------------------------------------- | -------- |
| Summary per project  | Name, status, description, task counts, owner. Enough for list view. | ✓        |
| Full project index   | Entire ProjectIndex per project. More data than needed.              |          |
| Name and status only | Minimal. More round trips needed.                                    |          |

**User's choice:** Summary per project (Recommended)

### Should projects.get include board and queue data?

| Option             | Description                                                                   | Selected |
| ------------------ | ----------------------------------------------------------------------------- | -------- |
| Separate calls     | projects.get returns ProjectIndex only. board.get and queue.get are distinct. | ✓        |
| Composite response | Everything in one response. Fewer round trips but larger payloads.            |          |

**User's choice:** Separate calls (Recommended)
**Notes:** Matches method names in GATE-02.

## Event Broadcasting

### How should SyncEvents map to WebSocket broadcast events?

| Option                       | Description                                                                                                               | Selected |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------- | -------- |
| Direct mapping               | project:changed → projects.changed, task:changed/deleted → projects.board.changed, queue:changed → projects.queue.changed | ✓        |
| Single event with type field | Always broadcast 'projects.changed' with a type field.                                                                    |          |

**User's choice:** Direct mapping (Recommended)

### What should event payloads contain?

| Option                     | Description                                                     | Selected |
| -------------------------- | --------------------------------------------------------------- | -------- |
| Project name only          | { project: string }. Lightweight notification, UI refetches.    | ✓        |
| Full updated data          | Includes full index data. Larger payloads, potential staleness. |          |
| Project name + change hint | { project: string, taskId?: string }. Slightly more info.       |          |

**User's choice:** Project name only (Recommended)
**Notes:** Avoids stale data, keeps events small.

## Method Registration

### What scope/role should project methods require?

| Option                | Description                                            | Selected |
| --------------------- | ------------------------------------------------------ | -------- |
| operator.read for all | All four methods are read-only. Same as sessions.list. | ✓        |
| No scope restriction  | Any authenticated connection. Simpler but less secure. |          |
| Custom projects scope | New operator.projects scope. Overkill for read-only.   |          |

**User's choice:** operator.read for all (Recommended)

### Should project handlers go in a single file or split per method?

| Option             | Description                                                 | Selected |
| ------------------ | ----------------------------------------------------------- | -------- |
| Single file        | src/gateway/server-methods/projects.ts with all 4 handlers. | ✓        |
| Split per resource | Separate files per resource. More files than needed.        |          |

**User's choice:** Single file (Recommended)
**Notes:** Small surface area — each handler reads .index/ JSON and responds.
