# Phase 7: Gateway Service — Research

## Research Summary

Phase 7 connects the project data layer (Phases 1-3) to the gateway's WebSocket infrastructure so the UI can read project state in real time. This research covers the gateway architecture patterns, integration points, and implementation approach.

## Gateway Architecture Analysis

### Service Lifecycle Pattern

The gateway follows a consistent service lifecycle:

1. **Startup**: Services are created in `startGatewaySidecars()` (`src/gateway/server-startup.ts`), which is called from `startGatewayServer()` in `src/gateway/server.impl.ts` (line ~1263).
2. **Shutdown**: Services are stopped in `createGatewayCloseHandler()` (`src/gateway/server-close.ts`), which receives all stoppable services as params.

Reference implementations:
- `buildGatewayCronService` in `src/gateway/server-cron.ts` — creates a service object with start/stop, returns it for gateway to manage
- `startChannelHealthMonitor` in `src/gateway/channel-health-monitor.ts` — returns a stoppable handle
- `startHeartbeatRunner` in `src/infra/heartbeat-runner.ts` — starts and returns a `HeartbeatRunner` with `.stop()`

The pattern: create the service, start it, pass the stop handle to `createGatewayCloseHandler`.

### Method Handler Pattern

Gateway RPC methods are organized as:

1. **Handler module** at `src/gateway/server-methods/<domain>.ts` — exports a `GatewayRequestHandlers` record
2. **Registration** in `src/gateway/server-methods.ts` — handlers spread into `coreGatewayHandlers` object
3. **Method list** in `src/gateway/server-methods-list.ts` — method names added to `BASE_METHODS` array
4. **Scope classification** in `src/gateway/method-scopes.ts` — determines authorization level

Handler function signature (from `types.ts`):
```typescript
type GatewayRequestHandler = (opts: GatewayRequestHandlerOptions) => Promise<void> | void;

type GatewayRequestHandlerOptions = {
  req: RequestFrame;
  params: Record<string, unknown>;
  client: GatewayClient | null;
  isWebchatConnect: (params: ConnectParams | null | undefined) => boolean;
  respond: RespondFn;
  context: GatewayRequestContext;
};

type RespondFn = (ok: boolean, payload?: unknown, error?: ErrorShape, meta?: Record<string, unknown>) => void;
```

Handlers call `respond(true, payload)` for success, `respond(false, undefined, errorShape(...))` for errors.

### Event Broadcasting Pattern

Events use the `broadcast` function available on `GatewayRequestContext`:

```typescript
type GatewayBroadcastFn = (event: string, payload: unknown, opts?: GatewayBroadcastOpts) => void;
```

Events are registered in `GATEWAY_EVENTS` array in `server-methods-list.ts`. Event scope guards (for authorization) are in `server-broadcast.ts` `EVENT_SCOPE_GUARDS` record.

### Method Scopes

From `method-scopes.ts`, methods are classified into scope tiers:
- `operator.read` — read-only methods (sessions.list, config.schema.lookup, models.list)
- `operator.write` — mutation methods (sessions.send, send, config.set)
- `operator.admin` — administrative methods (config.patch, wizard.start, update.run)
- `operator.approvals` — exec approval methods

Project methods are all read-only → `operator.read` scope.

### Existing method-scopes.test.ts

The test file at `src/gateway/method-scopes.test.ts` verifies:
- Every method in `listGatewayMethods()` is classified in method-scopes
- Every handler in `coreGatewayHandlers` has a matching method in BASE_METHODS
- Scope resolution for specific methods

New project methods must be added to pass these invariant tests.

## ProjectSyncService Integration

`ProjectSyncService` (Phase 3, `src/projects/sync-service.ts`) already provides:
- `start()` — discovers projects, runs full reindex, starts chokidar watcher
- `stop()` — closes watcher, clears debounce timers
- Emits `"sync"` events typed as `SyncEvent`:
  - `project:changed` — PROJECT.md changed
  - `task:changed` — task file changed
  - `task:deleted` — task file deleted
  - `queue:changed` — queue.md changed
  - `reindex:complete` — full reindex done

`discoverProjects()` returns absolute paths to all project directories (including sub-projects).

## .index/ JSON Shapes

Already generated by Phase 3's index-generator.ts:

- **project.json** (`ProjectIndex`): ProjectFrontmatter + indexedAt
- **board.json** (`BoardIndex`): `{ columns: Array<{ name, tasks: BoardTaskEntry[] }>, indexedAt }`
- **queue.json** (`QueueIndex`): `{ available, claimed, blocked, done: QueueEntry[], indexedAt }`
- **tasks/TASK-NNN.json** (`TaskIndex`): TaskFrontmatter + indexedAt

## Implementation Plan

### File Structure

1. `src/gateway/server-projects.ts` — ProjectGatewayService class
   - Constructor takes `projectsRoot` and `broadcast` function
   - `start()`: creates and starts ProjectSyncService, subscribes to sync events
   - `stop()`: stops ProjectSyncService
   - `listProjects()`: discovers projects, reads .index/project.json for each
   - `getProject(name)`: reads .index/project.json for a specific project
   - `getBoard(name)`: reads .index/board.json
   - `getQueue(name)`: reads .index/queue.json

2. `src/gateway/server-methods/projects.ts` — RPC handlers
   - `projects.list` — calls service.listProjects(), responds with array
   - `projects.get` — validates params.project, calls service.getProject(), responds
   - `projects.board.get` — validates params.project, calls service.getBoard(), responds
   - `projects.queue.get` — validates params.project, calls service.getQueue(), responds

### Event Mapping

| SyncEvent type | WebSocket event | Payload |
|---|---|---|
| project:changed | projects.changed | { project: string } |
| task:changed | projects.board.changed | { project: string } |
| task:deleted | projects.board.changed | { project: string } |
| queue:changed | projects.queue.changed | { project: string } |
| reindex:complete | (no broadcast — internal) | — |

## Risks and Mitigations

1. **Large server.impl.ts**: This file is already very large (~1400+ lines). Changes should be minimal — just wiring the service through. The actual logic lives in server-projects.ts and server-methods/projects.ts.

2. **Projects root path**: Need to resolve `~/.openclaw/projects/` consistently. Use `resolveStateDir` from config/paths.ts and join with "projects".

3. **Missing .index/ directory**: If .index/ doesn't exist yet (no sync service has run), RPC methods should return empty results, not crash.
