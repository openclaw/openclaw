<objective>
Build the runtime enforcement engine for constitutional governance.

Purpose: convert governance doctrine into deterministic runtime behavior for tool execution so actions are evaluated as PERMIT/PROHIBIT/ESCALATE before mutation occurs.

Output: machine-readable governance runtime policy + bundled governance extension with mode-based enforcement.
</objective>

<context>
## Existing Runtime Capabilities

- Tool interception already exists in `src/agents/pi-tools.before-tool-call.ts` and plugin hook runner.
- Plugin hook contract supports blocking (`block: true`) and explicit block reasons.
- This plan intentionally uses extension hooks first to minimize core churn while establishing enforceable behavior.

## Design Constraints

1. Runtime policy must be deterministic and parseable (no prose-only logic).
2. Enforcement must support staged rollout (`off`, `shadow`, `enforce`).
3. Failure behavior in `enforce` mode must not silently allow unsafe operations.
4. Decisions must preserve reason codes for later telemetry (implemented in Plan 02-02).
</context>

  <action>
    Create a bundled plugin `frankos-governance` with config:
    - `mode`: `off | shadow | enforce`
    - `policyFile`: absolute path to runtime policy JSON
    - `mutatingTools`: optional override list
    - `emitDecisionLog`: boolean

    Implement hook behavior:
    - `before_tool_call`:
      - classify tool action (mutating vs non-mutating)
      - evaluate against policy by priority
      - `off`: no action
      - `shadow`: allow, but capture decision metadata in log
      - `enforce`:
        - `permit` -> allow
        - `prohibit` -> block with stable reason code prefix `GOVERNANCE_PROHIBITED:`
        - `escalate` -> block with stable reason code prefix `GOVERNANCE_ESCALATE_REQUIRED:`
      - on evaluation/policy read failure in enforce mode: block with `GOVERNANCE_POLICY_EVAL_FAILED`
    - optional `before_prompt_build`:
      - inject compact hierarchy reminder to keep model behavior aligned with runtime decision logic.
  </action>

    - Enforce mode blocks prohibited decisions
    - Shadow mode never blocks but records decision context
  </verify>
