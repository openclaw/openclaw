Create `src/test-utils/live-test-reporter.ts` implementing a Vitest custom reporter.

**Implementation:**

Import from `vitest/reporters` — extend `BaseReporter` or implement the reporter interface directly. The reporter hooks are:

- `onInit(ctx)` — called when Vitest starts
- `onTaskUpdate(packs)` — called as tests complete (provides task results)
- `onFinished(files)` — called when all tests complete
- `onWatcherRerun(files)` — for watch mode (not needed)

**Per-test output (in `onTaskUpdate`):**

For each completed test task, print one line:

```
  ✓ test name (1.2s)                    — pass (green)
  ✗ test name                           — fail (red)
  ○ test name — MINIMAX_API_KEY not set  — skip (yellow)
  ◌ test name — Browserless unavailable  — unavailable (dim/gray)
```

Use ANSI colors:

- Pass: `\x1b[32m` (green)
- Fail: `\x1b[31m` (red)
- Skip: `\x1b[33m` (yellow)
- Unavailable: `\x1b[90m` (gray)
- Reset: `\x1b[0m`

**Detecting skip reasons:**

The reporter needs to know WHY a test was skipped. Two sources:

1. Vitest's built-in skip (via `describe.skip`) — task state is `skip`. The reason may not be available directly. Solution: The `describeLive` helper from Plan 06-01 logs skip reasons to console. The reporter can also collect skip reasons from a shared registry.
2. Better approach: Use Vitest's `task.meta` or parse the test name/suite name.

**Pragmatic approach:** Since `describeLive` from Plan 06-01 logs `[live-skip] name: missing KEYS` to console, the reporter can collect these messages. But a cleaner approach is:

Create a module-level `Map<string, string[]>` (file path -> missing keys) in a shared file. The `describeLive` helper populates it, and the reporter reads it in `onFinished`. Export this map from `live-test-helpers.ts` as `liveSkipRegistry`.

However, since Plan 06-01 and 06-02 are independent (wave 1), **do not depend on Plan 06-01's helpers**. Instead:

1. Parse skip reasons from Vitest's task result. Vitest tasks with `mode: 'skip'` indicate skipped tests.

**Failed test output:**

For failed tests, after the fail line, print the error message (not the stack):

```
  ✗ test name
    Error: expected 200, got 401 (auth)
```

Extract error from `task.result?.errors?.[0]?.message`. Strip stack traces (everything after first `\n    at `). Classify using simple heuristics (same as `classifyLiveError` from Plan 06-01 but self-contained — match 401/403/429/502/503 in the message string).

**End-of-run summary (in `onFinished`):**

Print a separator, then:

```
─────────────────────────────────────
Live Test Summary

  Pass:        3
  Fail:        1
  Skip:        5
  Unavailable: 1
  Total:       10
  Duration:    42.3s
```

Only list keys that are actually missing from the environment. Deduplicate (if ZAI_API_KEY and Z_AI_API_KEY are both missing, show once). Color the summary counts: green for pass, red for fail (if >0), yellow for skip.
