# OpenClaw Architecture

What OpenClaw is, how it's built, what it depends on, and who consumes it.

## What it is

OpenClaw is a personal AI assistant gateway. It delivers responses across 25+ messaging channels (WhatsApp, Telegram, Slack, Discord, Signal, iMessage, Matrix, …) with native apps for macOS, iOS, and Android. The gateway is the control plane; extensions provide channel and model-provider integrations via a Plugin SDK.

Repo: https://github.com/openclaw/openclaw

## Stack

- **Runtime:** Node 22+ (Bun preferred for TS execution: `bun <file.ts>` / `bunx <tool>`).
- **Package manager:** pnpm workspaces (`bun install` also supported; keep `pnpm-lock.yaml` + Bun patching in sync when touching deps/patches).
- **Build:** tsdown → `dist/` (`pnpm build`).
- **Tests:** Vitest with V8 coverage (70% lines/branches/functions/statements). Threads pool default; hard `forks` exceptions for `gateway`, `agents`, `commands`.
- **Lint/format:** Oxlint (`pnpm check`), Oxfmt (`pnpm format` / `pnpm format:fix`).
- **Pre-commit:** `prek install` (runs `pnpm format` then `pnpm check`).
- **Docs:** Mintlify (docs.openclaw.ai). Foreign-language docs autogenerated in sibling `openclaw/docs`.

## Repo layout

Monorepo. pnpm workspace members are `.` + `ui` + `packages/*` + `extensions/*` (per `pnpm-workspace.yaml`). `apps/` lives in the repo for native client sources but is not a pnpm workspace member.

| Path | Owns |
|---|---|
| `src/cli/` | CLI wiring |
| `src/commands/` | command implementations |
| `src/web/` | shared web/model-provider runtime helpers |
| `src/infra/` | infra |
| `src/media/` | media pipeline |
| `src/plugin-sdk/*` | public plugin contract (only path extensions may import from core) |
| `src/plugins/*` | plugin discovery, manifest validation, loader, registry, contract enforcement |
| `src/channels/*` | core channel framework (routing, allowlists, pairing, command gating) — channel *implementations* live in `extensions/` |
| `src/gateway/protocol/*` | typed Gateway control-plane and node wire protocol |
| `extensions/` | bundled workspace plugin tree — ~130 packages for providers (openai, anthropic, google, …) and channels (telegram, discord, slack, signal, imessage, whatsapp, matrix, zalo, voice-call, …) |
| `ui/` | Control UI (web frontend) |
| `apps/macos/`, `apps/ios/`, `apps/android/` | native client apps (not pnpm workspaces) |
| `apps/shared/`, `apps/macos-mlx-tts/`, `apps/swabble/` | internal app helpers / experimental surfaces |
| `docs/` | English docs (Mintlify-rendered); also images, queue, Pi config |
| `dist/` | built output |

Tests are colocated as `*.test.ts`; e2e as `*.e2e.test.ts`.

For the contracts these layers expose, see `CHARTER.md`.

## Channels

When refactoring shared logic (routing, allowlists, pairing, command gating, onboarding, docs), consider ALL built-in + extension channels.

- **Core channel docs:** `docs/channels/`
- **Core channel framework:** `src/channels/` (routing, allowlists, pairing, command gating, onboarding shared by all channels)
- **Channel implementations (bundled plugins under `extensions/`):** Telegram, Discord, Slack, Signal, iMessage, WhatsApp, Matrix, Zalo, ZaloUser, Voice Call, Talk Voice, Webhooks, …

When adding channels/plugins/apps/docs: update `.github/labeler.yml` and create matching GitHub labels (use existing channel/plugin label colors).

## Consumers

- **macOS app** (`apps/macos/`) — menubar app plus a separate gateway LaunchAgent (label `ai.openclaw.gateway`, distinct from the menubar app's `ai.openclaw.mac`). See `apps/macos/Sources/OpenClaw/Constants.swift` and `GatewayLaunchAgentManager.swift`.
- **iOS app** (`apps/ios/`).
- **Android app** (`apps/android/`).
- **Control UI** (`ui/`) — web frontend.
- **CLI** — `openclaw` binary.
- **Mobile pairing peers** — see `SECURITY.md` for the trust boundary.
- **Third-party plugins** in the wild via Plugin SDK.

## Bundled plugin naming

For repo-owned workspace plugins, keep the canonical plugin id aligned across:
- `openclaw.plugin.json:id`
- the default workspace folder name
- package names (`@openclaw/<id>` or approved suffix forms: `-provider`, `-plugin`, `-speech`, `-sandbox`, `-media-understanding`)

Keep `openclaw.install.npmSpec` equal to the package name and `openclaw.channel.id` equal to the plugin id when present. Exceptions must be explicit and covered by the repo invariant test.

Plugin install runs `npm install --omit=dev` in the plugin dir; runtime deps must live in `dependencies`. Avoid `workspace:*` in `dependencies` (npm install breaks). Put `openclaw` in `devDependencies` or `peerDependencies` (runtime resolves `openclaw/plugin-sdk` via jiti alias).

Nomenclature: use "plugin" / "plugins" in docs, UI, changelogs, and contributor guidance. The bundled workspace plugin tree remains the internal package layout to avoid repo-wide churn from a rename.
