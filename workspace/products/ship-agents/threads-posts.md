## POST 1: 凌晨三點，我的 AI 自己修好了 bug

今天早上起床，拿起手機看 log。

3:17am — gateway crash detected
3:17am — auto-restart initiated
3:18am — health check passed
3:18am — incident logged, severity: low

我喝了口咖啡，翻完整份報告。從偵測到修復，47 秒。我在睡覺。

這不是什麼未來科技。這是一個 daemon，跑在我桌上那台 Mac Mini 裡面。它每幾個小時巡一次，發現異常就處理，處理完自己寫報告。

大部分人聽到「AI agent」想到的是聊天機器人。你打字，它回話。

但真正的 agent 不是這樣的。

真正的 agent 是你不在的時候，它還在。你睡了，它醒著。你忘了，它記著。不是因為它多聰明，而是因為你把規則寫清楚了，把邊界畫好了，把失敗的處理方式想透了。

「會聊天」跟「能值班」是兩件完全不同的事。

大部分人卡在第一件事。但第二件事才是真正值錢的。

## POST 2: 一個人，十個 agent

最近常被問：「你怎麼同時做這麼多事？」

答案不是我比較努力。答案是我不是一個人在做。

現在同時在跑的 agent：

- 客服應答（LINE + Telegram 雙通道）
- 數據分析（自動抓報表、算指標）
- 系統監控（異常偵測、自動重啟）
- 知識管理（每天消化對話，萃取重點）
- 排程巡邏（凌晨跑例行任務、產日報）
- 內容整理（跨 agent 交叉摘要）

還有幾個更小的，做特定用途。

重點是：它們之間沒有「中控台」。沒有一個 master agent 在指揮。

它們是平行的。靠一塊共享公告欄溝通。A 做完事，把結果貼上去。B 下次醒來，先看公告欄，再決定自己要幹嘛。

這個架構跑了快兩個月。穩定到我有時候忘記它們在跑。

不是科幻小說。是一台 Mac Mini、一個 config 檔、一堆寫得很囉唆的規則檔。

最難的部分從來不是技術。是想清楚每個 agent 的邊界在哪裡。

## POST 3: 你的 CLAUDE.md 大概是壞的

我看過很多人的 CLAUDE.md。認真講，九成都有同一個問題。

它們寫了 agent「要做什麼」。

但沒寫它「是誰」。
沒寫它「絕對不能做什麼」。
沒寫它「出錯的時候怎麼辦」。

這就像你請了一個新人，給他一張任務清單，然後就走了。他不知道自己的角色，不知道公司的底線，遇到意外只能愣在那邊。

我後來把一個檔案拆成三個：

**SOUL** — 身份。你是誰、你的個性、你跟其他 agent 的關係。
**CONSTITUTION** — 憲法。紅線在哪、什麼事絕對不能做、衝突時聽誰的。
**HEARTBEAT** — 心跳。怎麼自我檢查、怎麼回報狀態、失敗了怎麼降級。

三個檔案，各司其職。

我的正式版 CLAUDE.md 寫了 350 行。反覆改了三個月。每次 agent 出包，我就回去補一條規則。每次它做對了，我就確認那條規則有效。

這東西不是寫一次就好的。它是養出來的。
