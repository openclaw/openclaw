🐛 Issue #103190 — OpenClaw 破坏了 /tmp 目录权限

Issue github.com/openclaw/openclaw/issues/103190  ·  PR github.com/openclaw/openclaw/pull/103302

💥 出了什么问题？

一句话：OpenClaw 在 /tmp 目录上执行了 chmod 700，导致整个系统共享的临时目录变成 只有自己能用，其他程序全都打不开。

用生活场景理解

想象一栋公寓楼有一个公共快递架（这就是 /tmp）—— 所有住户都可以往上面放东西、取东西。

有一天 OpenClaw 想："我要在这个架子上给自己划一小块专属区域。" 于是它去设置了 "这个架子只能我自己碰" —— 结果整个快递架都被锁上了！其他住户全放不了也拿不了快递了。

这就是 OpenClaw 调用 chmod(0o700) 对 /tmp 做的事。

技术上说：

  • /tmp 的权限是 0o1777（任何用户都能读写，加 sticky bit🔒 防止互相删文件）
  • OpenClaw 的 ensurePrivateDirectory() 函数会执行 chmod(目录, 0o700)（改成仅自己可访问）
  • 如果这个目录就是 /tmp 本身 → 灾难：
# 修复前，/tmp 会从这样：
drwxrwxrwt   (0o1777, 任何人可读写)
# 变成这样：
drwx------   (0o0700, 只有 OpenClaw 能用)
# → sudo, apt, ssh 全部失效，因为都用 /tmp

🔍 根因：调用链分析

问题出在两处代码都直接传了系统临时目录给一个会改权限的函数：

① 临时下载目录

temp-download.ts

createTempDownloadTarget()
  → tempWorkspace({rootDir: os.tmpdir()})
    → ensurePrivateDirectory("/tmp")
      → chmod("/tmp", 0o700) ❌
② 安装临时目录

install-source-utils.ts

installPluginFromSource()
  → withTempDir()
    → tempWorkspace({rootDir: os.tmpdir()})
      → ensurePrivateDirectory("/tmp")
        → chmod("/tmp", 0o700) ❌

🛠️ 怎么修的？三层防御

就像一个安全系统有多道防线，我们的修复也是三层保护，每一层独立阻截问题：

🛡️ 第 1 层 — 避开雷区（temp-download.ts）

在创建临时目录之前，先检查目标是不是系统级共享目录。如果是，自动跳转到子目录：

// 修复前：直接使用 os.tmpdir() 作为工作区根目录
tempWorkspace({ rootDir: os.tmpdir(), ... })

// 修复后：如果是 /tmp 等系统目录，自动用 /tmp/openclaw 子目录
if (检测到 /tmp 等系统共享目录)
  rootDir = "/tmp/openclaw";  // 安全的子目录
tempWorkspace({ rootDir, ... })
// 这样 chmod(700) 只影响 /tmp/openclaw，不影响 /tmp 本身

🛡️ 第 2 层 — 看到"粘性位"就停手（tmp-openclaw-dir.ts）

即使在 ensurePrivateDirectory 内部增加一道检查：如果目录有 sticky bit（粘性位）—— 即别的程序也可以在里面写文件 —— 说明这是共享目录，绝对不能改权限：

// 修复前：不管三七二十一直接改
chmod(dir, 0o700);

// 修复后：先检查有没有 sticky bit
if (目录有 sticky bit) {  // 共享目录的标志
  return;  // 不动它！
}
chmod(dir, 0o700); // 只改非共享目录

🛡️ 第 3 层 — 不要乱传 os.tmpdir()（install-source-utils.ts）

安装插件时的临时目录函数，改为使用 OpenClaw 自己的安全临时目录：

// 修复前：直接传 os.tmpdir()
withTempDir(prefix, fn);

// 修复后：明确指定安全的根目录
withTempDir(prefix, fn, { rootDir: resolvePreferredOpenClawTmpDir() });

📐 为什么需要三层？

防御纵深（Defense in depth）—— 不依赖任何单一防线：

  • ✏️ 第 1 层：以后写新代码的人如果又传了 os.tmpdir()，会被自动纠正
  • ✏️ 第 2 层：即使第 1 层漏了，ensurePrivateDirectory 自己也会刹车
  • ✏️ 第 3 层：这个具体的调用点（安装插件）也被修正了

就像开车系安全带 + 安全气囊 + 防碰撞结构——任何一层失效，后面的还能兜底。

📊 总结

问题OpenClaw 把 /tmp 权限从 0o1777 改成 0o700，导致其他程序无法使用系统临时目录
根因直接传 os.tmpdir() 给 ensurePrivateDirectory()，后者无条件执行 chmod(dir, 0o700)
修复策略三层防御：调用前重定向 + 调用中检测 sticky bit + 修正具体调用点
