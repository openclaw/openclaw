# MAI Universe Project Registry Rules

_Last updated: 2026-02-26 (KST)_

## 1) 목적

`ops/universe/project-registry.yaml`의 `stage`와 4개 점수(`health/contrib/revenue/risk`)를
**과대평가 없이 보수적으로** 유지·갱신하기 위한 기준 문서.

---

## 2) 적용 범위

- 대상: MEMORY.md에 등록된 활성 프로젝트 16개
- 데이터 소스 우선순위:
  1. `MEMORY.md`
  2. `memory/<project>.md`
  3. 레포/배포 상태(릴리즈, 배포 URL, 테스트 결과 등)
- 충돌 시 원칙: **낮은 stage / 낮은 점수(리스크는 높은 값) 우선**

---

## 3) Stage 정의 및 승급 기준

Stage enum은 `run-ledger.schema.json`과 동일:
`COLLECT → DISCOVER → CREATE → BUILD → DEPLOY → REALIZE`

실무 운영에서는 최소 `DISCOVER`부터 사용.

### DISCOVER

- 아이디어/문서 중심, PRD/로드맵/시장조사 단계
- 코드/실행 결과가 없거나 매우 제한적

### CREATE

- 아키텍처/설계 확정, MVP 범위와 개발 계획이 문서화됨
- 초기 스캐폴딩/PoC 일부 존재 가능

### BUILD

- 실제 기능 구현 중, 핵심 모듈 동작 확인
- 테스트/로컬 실행/내부 데모 근거 존재

### DEPLOY

- 외부 접근 가능한 배포 환경 존재(앱스토어 심사 제출 포함)
- 운영/배포 루틴 또는 모니터링 근거 존재

### REALIZE

- 반복적 매출/유의미한 사용자/운영 안정화 확인
- 최소 2회 이상 기간(주/월) 지표로 재현됨

### Stage 변경 규칙

- 기본: 1회 업데이트당 **최대 1단계**만 변경
- 승급: 아래 3개 중 최소 2개 근거 필요
  - 문서상 마일스톤 완료 기록
  - 코드/테스트/릴리즈 근거
  - 배포/실사용 근거
- 강등: 30일 이상 정체, 배포 중단, 핵심 리스크 급증 시 1단계 강등 가능

---

## 4) 점수 체계 (0.00 ~ 1.00)

- `health_score`: 개발/운영 건강도 (진척, 안정성, 실행 가능성)
- `contrib_score`: 기여/재사용/시너지 기여도
- `revenue_score`: 수익화 성숙도 (실매출/유료화 준비)
- `risk_score`: 리스크 크기 (높을수록 위험)

### 보수적 초기값 가이드 (stage별 앵커)

| Stage    | health | contrib | revenue | risk |
| -------- | -----: | ------: | ------: | ---: |
| DISCOVER |   0.45 |    0.35 |    0.10 | 0.55 |
| CREATE   |   0.52 |    0.44 |    0.18 | 0.56 |
| BUILD    |   0.62 |    0.54 |    0.28 | 0.53 |
| DEPLOY   |   0.70 |    0.62 |    0.36 | 0.47 |
| REALIZE  |   0.78 |    0.66 |    0.55 | 0.42 |

### 점수 조정 규칙

- 기본 단위: 소수점 둘째 자리
- 1회 업데이트 변동폭 제한:
  - `health/contrib/revenue`: ±0.05 이내
  - `risk`: ±0.07 이내
- 근거가 약하면 **유지** (임의 상향 금지)
- 실증 부재 시 `revenue_score`는 0.40 초과 금지
- 금융/규제/개인정보/외부 의존도가 높으면 `risk_score`를 보수적으로 상향

---

## 5) 운영 주기

- 정기 업데이트: 주 1회(권장: 금요일)
- 수시 업데이트 트리거:
  - 대형 마일스톤 완료(출시, 배포, 승인, 유료화 개시)
  - 장애/차단/정책 리스크 발생
- 14일 이상 업데이트 근거 없음:
  - `health_score` -0.03
- 30일 이상 실질 진척 없음:
  - stage 재검토(강등 후보)

---

## 6) 기록/감사 규칙

업데이트 시 최소 다음을 남긴다.

1. 변경 일시 (`updated_at`)
2. 변경 대상 프로젝트 ID
3. 변경 항목(stage/score)
4. 근거 출처 (`MEMORY.md`, `memory/*.md`, 릴리즈/배포 링크)

근거가 불명확하면 변경하지 않고 `notes`에 "evidence insufficient"를 남긴다.

---

## 7) 예외 처리

- GitHub/배포 정보가 `TBD`이면 `repo: null` 유지 가능
- 근거 문서 인코딩/가독성 이슈가 있어도, 확인 가능한 사실만 반영
- 과장 가능성이 있는 서술은 점수 반영 시 1단계 낮춰 해석
