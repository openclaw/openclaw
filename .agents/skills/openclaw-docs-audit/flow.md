# Audit Viewer Flow

This document describes how `./scripts/audit-report-viewer.mjs` turns a detailed
rewrite-audit report into reusable JSON and a static HTML viewer, and how
`./assets/audit-viewer.html` renders that JSON.

## Artifacts

- `./scripts/audit-report-viewer.mjs`: CLI entrypoint. It parses a detailed
  audit table, resolves source and destination line ranges, writes JSON, and
  renders the HTML template.
- `./assets/audit-viewer.html`: static template. It embeds the generated JSON in
  a `<script type="application/json" id="audit-data">` block and renders the
  interactive UI in the browser.
- `*.json` output: reusable intermediate data. This is the canonical data source
  for the viewer and can also regenerate normalized detailed Markdown.
- `*.html` output: self-contained viewer. Do not hand-edit this file; patch the
  script or template and regenerate it.

## Command Flow

Typical report-to-viewer command:

```bash
node .agents/skills/openclaw-docs-audit/scripts/audit-report-viewer.mjs \
  --report .mem/main/specs/<spec-id>/reports/<detailed-report>.md \
  --out-dir .mem/main/specs/<spec-id>/report \
  --basename <short-audit-name> \
  --changed-pages docs/tools/plugin.md,docs/plugins/manage-plugins.md
```

The command writes:

```text
.mem/main/specs/<spec-id>/report/<short-audit-name>-audit-data.json
.mem/main/specs/<spec-id>/report/<short-audit-name>-audit-viewer.html
```

## Script Pipeline

`audit-report-viewer.mjs` runs these stages:

1. Parse CLI args.
2. Resolve paths relative to `--cwd` or the current working directory.
3. Read the detailed audit report passed through `--report`, or read an existing
   JSON document passed through `--data`.
4. Parse report frontmatter.
5. Resolve the source archive from `--source-ref` or frontmatter `source_ref`.
6. Parse each detailed audit table row into one unit.
7. Resolve source lines from the pre-rewrite source ref with `git show`.
8. Resolve destination and generator lines from the current checkout.
9. Mark each destination file as changed or unchanged relative to the source ref.
10. Build page selector views.
11. Validate required source coverage.
12. Write JSON, HTML, and optionally normalized detailed Markdown.

When using `--data`, the script reuses the existing JSON. It can rebuild
`pageViews` from `--changed-pages`, but it does not recompute source lines,
destination lines, or `changedSinceSource`. Use `--report` when the checkout or
source ref changed and the viewer needs fresh line or diff metadata.

## Report Parsing

The detailed report must use one table per source page. Each audit unit row must
have eight cells:

```text
ID | Source | Summary | Action | Reason | Destination | Status | Notes
```

The parser recognizes rows whose first cell starts with `P###:`. It preserves the
original Markdown cells under `unit.markdown` and writes normalized fields such
as `unit.id`, `unit.label`, `unit.summary`, `unit.action`, `unit.reason`,
`unit.status`, and `unit.notes`.

The source cell must use this shape:

```markdown
`docs/path.md:<start>-<end>` from `<SOURCE_REF_OR_ARCHIVE>`
```

Destination cells may contain multiple ranges separated by `<br />`:

```markdown
`docs/path-a.md:10-20`<br />`docs/path-b.md:30-40`
```

Generated destinations can cite a generator range:

```markdown
`docs/generated.md:10-20` (generated by `scripts/generator.mjs:30-44`)
```

External-only destinations are kept as notes with no repo line.

## JSON Shape

The generated JSON has this top-level shape:

```json
{
  "schemaVersion": 1,
  "sourceArchive": {
    "ref": "4e020c21a5^:docs/tools/plugin.md",
    "gitRef": "4e020c21a5^",
    "path": "docs/tools/plugin.md"
  },
  "sourceReport": ".mem/main/specs/.../reports/report.md",
  "changedPages": ["docs/tools/plugin.md"],
  "pageViews": [],
  "pages": []
}
```

Each page contains:

- `page`: source page path.
- `coverage`: page-level coverage summary from the report table.
- `counters`: parsed `source_units=... audited_units=... gaps=...` footer.
- `sourceDocument`: full pre-rewrite source Markdown from the source ref.
- `units`: parsed audit units for that source page.

Each unit contains:

- `source.ref`: source line range from the pre-rewrite ref.
- `source.lines`: resolved source lines for the source range.
- `destinations`: current destination ranges, generator ranges, notes, and line
  data.

Each destination contains:

- `ref`: current destination line range, when there is one.
- `lines`: current checkout lines for `ref`.
- `generator`: generator line range, when cited.
- `generatorLines`: current checkout lines for `generator`.
- `changedSinceSource`: file-level boolean from
  `git diff --quiet <source-ref> -- <destination-file>`.
- `external`: true when there is no repo destination range.
- `note` and `raw`: normalized and original destination-cell text.

`changedSinceSource` is intentionally file-level. A destination in an unchanged
file is related reference context, not a diff, even when the audit row is marked
`split` or `moved`.

## Page Views

`pageViews` drives the selector in the viewer header.

Source page view:

- `kind: "source"`
- Lists units whose original source page is the selected page.
- Doc mode renders `sourceDocument.lines` for that source page.

Destination page view:

- `kind: "destination"`
- Lists units whose destinations include the selected destination page.
- Filters the right-side destination rendering to that page.

When `--changed-pages` is passed, the selector order follows that CSV. Pages not
present in the audit data or destination ranges are skipped.

## Template Render Flow

`renderHtml(data, templatePath)` reads `./assets/audit-viewer.html`, serializes
the JSON, escapes `<` as `\u003c`, and replaces the `__AUDIT_DATA__` placeholder.

The resulting HTML is self-contained:

1. Browser parses embedded JSON from `#audit-data`.
2. Viewer builds `units` from `data.pages[*].units`.
3. Viewer initializes state:
   - `mode`: `block` or `doc`
   - `query`: sidebar search text
   - `selectedId`: active audit unit
   - `selectedViewId`: active page selector view
   - `wrapText`: text wrapping toggle
4. `render()` refreshes page selector, mode controls, summary metadata, legend,
   sidebar list, and right-side detail panes.

## Sidebar Modes

Block mode:

- Renders one selectable row per audit unit.
- Search matches unit IDs, labels, summaries, reasons, notes, and destinations.
- Rows are `div role="button"` so their text remains selectable while still
  supporting click, Enter, and Space activation.

Doc mode:

- Renders the full pre-rewrite Markdown document from `sourceDocument.lines`.
- Lines covered by an audit unit become clickable mapped lines.
- Clicking a mapped source line selects the associated audit unit.
- Search matches source line text and mapped unit metadata.

## Detail Panes

The left detail pane is `Mapped source ranges`.

- It shows only the selected audit unit's source block.
- It does not merge other units that happen to share a destination range.
- It resets scroll when the selected unit or page view changes.

The right detail pane is `Diff view`.

- It groups the selected unit's destinations in report order.
- Changed destination files render as diff sections.
- Unchanged destination files render as reference sections.
- External-only or removed destinations render notes instead of repo lines.

## Diff vs Reference Rules

The script computes `changedSinceSource` per destination file:

```bash
git diff --quiet <source-ref> -- <destination-file>
```

Viewer behavior:

- `changedSinceSource: true`: render a unified-style diff hunk.
- `changedSinceSource: false`: render a reference section with current lines.
- missing flag or external destination: treat as diffable/unknown unless the
  destination has no repo range.

This prevents unchanged reference pages from looking like migrated edits. For
example, if an audit unit maps to `docs/cli/plugins.md:74-238` but that file did
not change between the source ref and the current checkout, the viewer shows it
as related reference context instead of as a diff.

## Diff Construction

The viewer creates a lightweight line diff in the browser:

1. Source lines come from the selected audit unit's `source.lines`.
2. Destination lines come from the selected destination range's `lines`.
3. `diffLineOps()` computes a longest-common-subsequence table over line text.
4. The renderer emits context, remove, and add lines with source/destination line
   numbers.

This diff is a review aid, not a replacement for git's full file diff. It
compares one audit source block to one mapped destination range.

## Regeneration Discipline

Patch source artifacts, then regenerate:

1. Patch `./scripts/audit-report-viewer.mjs` for data, parsing, or output rules.
2. Patch `./assets/audit-viewer.html` for UI behavior or styling.
3. Rerun the script with `--report`.
4. Verify the generated JSON has the expected source and destination metadata.
5. Open or reload the generated HTML viewer.

Do not patch generated HTML under a spec `report/` directory directly. It will
be overwritten on the next regeneration.

## Validation Checklist

Use lightweight checks after script or template changes:

```bash
node_modules/.bin/oxfmt --check --threads=1 \
  .agents/skills/openclaw-docs-audit/SKILL.md \
  .agents/skills/openclaw-docs-audit/flow.md \
  .agents/skills/openclaw-docs-audit/scripts/audit-report-viewer.mjs \
  .agents/skills/openclaw-docs-audit/assets/audit-viewer.html

node --check .agents/skills/openclaw-docs-audit/scripts/audit-report-viewer.mjs

git diff --check -- \
  .agents/skills/openclaw-docs-audit/SKILL.md \
  .agents/skills/openclaw-docs-audit/flow.md \
  .agents/skills/openclaw-docs-audit/scripts/audit-report-viewer.mjs \
  .agents/skills/openclaw-docs-audit/assets/audit-viewer.html
```

For behavior changes, verify a generated viewer in the browser. Select a unit
with at least one changed destination and one unchanged related destination when
testing diff/reference behavior.
