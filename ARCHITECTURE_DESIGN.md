# OpenClaw-Tool: Dynamic CLI-Centric Agent Architecture

## 1. 痛点与根本认知

1. **痛点：** 我们就不该把工具一次性都发给Agent，Agent在很多场景下也不该使用死工具，这导致了严重的 Token 浪费和注意力分散。
2. **根本解决方案：动态工具化**
   - **天堑：** 本质上是高级语义解析。如果要用正统方式解决（一个 LLM 总结，另一个 LLM 使用），反而更浪费 token。
   - **换思路（塞尔达模式）：** 你玩塞尔达传说的时候有没有教程向你说明枯草可以点燃？利用模型已有训练、已有生态的惯性，将模型内部的“说明书”利用好。

## 2. 当前解决方案：统一入口 `openclaw-tool`

1. **理由：** CLI 本身带有动态、领域性、渐进性的特征。在精通 bash 的模型眼里，“`Openclaw --help` 就是展开帮助”是非常符合直觉的。同时兼容性很好，用法多样。
2. **额外特性：** CLI 非常方便与 function call、MCP、SKILL 协同或者转化。
3. **挑战与应对：** CLI 原本的痛点在于ni领域垂直时无法保持专注，容易遗忘。因此，**上下文管理是重中之重**。

## 3. 阶段性总结与执行原则

**核心：使用 CLI 格式，做好上下文处理，自动化动态处理，转换格式符合直觉，桥接已有生态。**

### 3.1. 感知方式

- 在 bash 命令的帮助里面增加指导。
- **待实验项目：** 它依旧被放在 function call 里面作为一个工具，但其实是一个独立的 bash 工具。
- Agent 可以通过 `bash` 输入 `openclaw-tool feishu update`。
- **突出新空间：** 这有助于集中 Agent 的注意力。

### 3.2. Bash 语法原则

- **语法：** `openclaw-tool <community> <function> ...`
- **铁律：符合 bash 最常见形态。** 参考 git, docker 等业界绝对主导的软件（模型被训练得最多的范式）。

### 3.3. 转换和聚合方式（中间件模式）

- **必须作为中间件：** 不能修改任何当前的权限、Agent 管理、工具管理相关的内容，只是修改使用形式。
- **已有本体 Tool：** 需要专门用参数和前缀聚合并精调使用形式，符合 bash 直觉。
- **插件和 MCP Tool：** `openclaw-tool <plugin-name/MCP-name> <tool-name> --params-name {params}`。找到通用映射方法以保持兼容。
- **生态体验：** 将工具说明、搜索，自动应用到错误提示、无参数调用、`--help` 等情况下。
- **可选性：** 现有的 agent 和权限应当可以配置某个或者某些 tool 是否要被转换为 bash-tool。尝试访问被屏蔽的 tool 应报错。

## 4. 实验特性规划

### 实验特性 1：Help 输出重定向（动态 Schema）

- `openclaw-tool` 的所有 help 输出**不再作为“返回”填充进对话历史**。
- **全部作为“额外上下文”**：应当和 function call 一起输出，以最大化集中程度。
- **机制：** 当模型使用 `openclaw-tool --help` 后，对话结果不是长篇的 stdout，而是将 guide 附加到 `bash` tool 的 Schema (Description) 中。

### 实验特性 2：动态工具（频次计算）

- 应当根据频率、远近给每个命令、命令下参数及其子空间算分，自动保持高频互动内容的全工具使用查询历史。

### 实验特性 3：动态工具（Embedding 语义拟合）

- 不是根据频率，而是根据“工具+描述”的语义向量和“thinking 或者最近请求”的语义向量进行本地拟合关联。
