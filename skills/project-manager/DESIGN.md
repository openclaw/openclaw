# Project Manager Skill — 设计哲学与架构总结

> **版本**: v2.1 | **日期**: 2026-04-23 | **作者**: nanobot 🐈

---

## 一、为什么需要这个 Skill？

### 1.1 根本矛盾：Agent 的无状态本质 vs 项目的连续性需求

AI Agent 有一个根本性的设计矛盾：

| Agent 的特性 | 项目的特性 |
|-------------|-----------|
| **无状态**：每次对话都是全新的开始 | **有状态**：项目有进度、有上下文、有历史决策 |
| **上下文有限**：窗口再大也会遗忘 | **信息累积**：项目越做信息越多 |
| **会话隔离**：重启后记忆清零 | **跨会话连续**：用户今天做一点，明天接着做 |

**没有这个 Skill 之前，会发生什么？**

1. **记忆爆炸**：Agent 把所有项目进度都塞进 `MEMORY.md`，文件膨胀到 1000+ 行，检索效率极低
2. **上下文污染**：项目 A 的进度写进了项目 B 的文件，或者混在全局记忆里
3. **断点丢失**：用户说"继续上次的项目"，Agent 根本不知道"上次"是哪个项目、做到哪了
4. **过度工程**：一个小任务（"帮我改个变量名"）也用了完整的项目模板，本末倒置

### 1.2 核心洞察：Agent 需要"外挂硬盘"

人类工作时，不会把所有项目细节都记在脑子里。我们会：
- 用笔记本记录每个项目的进度
- 用文件夹隔离不同项目的文件
- 用待办清单管理下一步行动

**Agent 也需要同样的机制**——只是它的"笔记本"是文件系统，它的"记忆"是 `STATUS.md`。

## 二、架构设计：三层记忆模型

```
┌─────────────────────────────────────────────────────────┐
│                    用户对话 (Session)                     │
│              短期记忆，会话结束即消失                        │
└────────────────────────┬────────────────────────────────┘
                         │ 写入/读取
┌────────────────────────▼────────────────────────────────┐
│              项目状态层 (STATUS.md)                       │
│         每个项目独立文件，宏观进度 + 下一步行动              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐      │
│  │ ec-platform │  │  nanobot    │  │  cmra-geo   │      │
│  │  STATUS.md  │  │  STATUS.md  │  │  STATUS.md  │      │
│  └─────────────┘  └─────────────┘  └─────────────┘      │
└────────────────────────┬────────────────────────────────┘
                         │ 沉淀/引用
┌────────────────────────▼────────────────────────────────┐
│              全局记忆层 (MEMORY.md)                       │
│         用户偏好 + 跨项目经验教训，长期不变                  │
│  • "不喜欢废话，只要结论"                                  │
│  • "上次用 X 方案踩了 Y 坑"                               │
│  • 多实例架构原则                                          │
└─────────────────────────────────────────────────────────┘
```

### 2.1 三层职责边界

| 层级 | 文件 | 存什么 | 不存什么 | 更新频率 |
|------|------|--------|---------|---------|
| **会话层** | 对话历史 | 当前讨论的细节 | 长期信息 | 每次对话 |
| **项目层** | `STATUS.md` | 进度、待办、关键决策 | 用户偏好、通用经验 | 任务完成/决策产生时 |
| **全局层** | `MEMORY.md` | 用户偏好、跨项目教训 | 具体项目进度 | 极少（新认知产生时） |

**关键原则**：项目进度从 `STATUS.md` 读，经验教训沉淀到 `MEMORY.md`，**两者绝不交叉**。

### 2.2 项目层内部结构

```
workspace/projects/
├── index.md              # 📇 项目索引（自动维护，支持模糊查找）
├── {project-name}/
│   ├── STATUS.md         # 🧠 核心记忆：进度、待办、关键决策
│   ├── docs/             # 方案、报告、调研数据
│   ├── src/              # 代码、产出物
│   └── status/           # auto-coding v3 微观工程文件（可选）
└── _archive/             # 📦 已归档项目（保留历史记录）
```

## 三、设计精髓

### 3.1 精髓一：物理隔离，而非逻辑隔离

**不是**在同一个文件里用标题区分项目，**而是**每个项目拥有独立的 `STATUS.md` 文件。

```
❌ 错误做法：MEMORY.md 里写
## 项目 A 进度
...
## 项目 B 进度
...

✅ 正确做法：
projects/A/STATUS.md
projects/B/STATUS.md
```

**为什么？**
- 读取时只加载当前项目上下文，不污染其他项目
- 文件天然隔离，不会误覆盖
- 支持并发（多 session 同时操作不同项目）

### 3.2 精髓二：分级模板，拒绝过度工程

不是所有任务都配得上完整的 STATUS.md。

| 场景 | 模板 | 模块数 | 示例 |
|------|------|--------|------|
| >3天多阶段项目 | A 完整模板 | 5 模块 | ec-platform 架构改造 |
| <3天单任务 | B 轻量模板 | 2-3 模块 | "帮我重构这个接口" |
| 调研/分析临时任务 | B 轻量模板 | 2 模块 | "查一下 XX 方案" |
| 不确定 | 先用 B | 可升级 | 后续发现复杂再升 A |

**设计哲学**：工具应该适配任务，而不是任务适配工具。

### 3.3 精髓三：意图优先，而非关键词匹配

**不依赖死板触发词**，根据对话上下文自动判断用户意图：

| 用户表达 | 识别意图 | 动作 |
|----------|---------|------|
| "新建项目 XX" | 明确新建 | 初始化 |
| "回到 XX" | 明确恢复 | 读取 STATUS.md |
| "上次那个方案" | 隐含恢复 | 根据 index.md 推断 |
| "接着来" | 隐含继续 | 找最近活跃项目 |
| "先停一下" | 保存状态 | 落盘 STATUS.md |
| "不管 XX 了，先搞 YY" | 切换项目 | 保存当前 + 恢复目标 |

**兜底策略**：无法确定时，展示最近 2-3 个活跃项目让用户选，**不盲目猜测**。

### 3.4 精髓四：状态显式化

> **Agent 没有记忆，所有"记忆"必须写入文件。没有记录在文件里的事，就是没发生。**

这意味着：
- 完成了一个子任务 → 更新 STATUS.md
- 做出了新的技术决策 → 更新 STATUS.md
- 发现了阻塞性 Bug → 更新 STATUS.md
- 只是改了个变量名 → **不更新**

**更新策略**：只记"结论"和"下一步"，不记过程流水账。

### 3.5 精髓五：并发安全

多 session 场景下（虾软 + 猫王同时运行），写 STATUS.md 前必须：
1. **先 read** 最新内容
2. **合并** 新信息
3. **再 write**

绝不基于过时的缓存直接覆盖。

## 四、解决了什么问题？

### 4.1 解决的问题清单

| 问题 | 之前 | 之后 |
|------|------|------|
| **跨会话记忆丢失** | Agent 重启后不知道之前做了什么 | 读 STATUS.md 即可恢复上下文 |
| **多项目并发混淆** | 项目 A 的进度写进项目 B 的文件 | 物理隔离，每个项目独立文件 |
| **上下文爆炸** | 所有历史塞进 Prompt，Agent 变慢变笨 | 只加载当前项目 STATUS.md |
| **断点续传困难** | 用户说"继续"，Agent 不知道继续什么 | index.md + 模糊匹配自动推断 |
| **过度工程化** | 小任务也用完整模板 | A/B 分级模板，按需选择 |
| **记忆边界模糊** | 项目进度和用户偏好混在一起 | 三层模型，职责清晰 |
| **多实例覆盖** | 虾软和猫王同时写同一文件 | 并发安全写（read-merge-write） |

### 4.2 量化效果

| 指标 | 之前 | 之后 |
|------|------|------|
| MEMORY.md 行数 | 1153 行 | 167 行（瘦身 85%） |
| 项目恢复时间 | 需要用户重新描述上下文 | 读 STATUS.md 即可，< 1 秒 |
| 上下文污染风险 | 高（所有项目混在一起） | 低（物理隔离） |
| 小任务开销 | 完整 5 模块模板 | 轻量 2-3 模块 |

## 六、设计原则总结

1. **Agent 是无状态的** → 必须依靠文件系统维持连续性
2. **物理隔离优于逻辑隔离** → 每个项目独立文件
3. **工具适配任务** → A/B 分级模板，拒绝过度工程
4. **意图优先** → 不依赖死板关键词，理解用户真实意图
5. **状态显式化** → 没写进文件的事，就是没发生
6. **极简主义** → 只记结论和下一步，不记流水账
7. **并发安全** → read-merge-write，防止多 session 覆盖
8. **自愈能力** → index.md 自动创建、自动补录
