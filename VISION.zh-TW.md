## OpenClaw 願景

OpenClaw 是真正能「動手做」的 AI。
它運行在你的設備上，存在於你的頻道中，遵循你的規則。

本文檔說明了專案的現狀與方向。
我們仍處於早期階段，因此迭代速度很快。
專案概覽與開發者文件：[`README.md`](README.md)
貢獻指南：[`CONTRIBUTING.md`](CONTRIBUTING.md)

OpenClaw 最初是作為學習 AI 的個人實驗場，旨在打造一個真正有用的工具：
一個能在真實電腦上執行真實任務的助手。
它經歷了多個名稱與外殼：Warelay -> Clawdbot -> Moltbot -> OpenClaw。

目標：一個易於使用、支援多種平台、且尊重隱私與安全的個人助手。

當前的重點是：

優先事項：

- 安全性與安全預設
- 錯誤修正與穩定性
- 設定可靠性與初次使用體驗 (UX)

次要優先事項：

- 支援所有主要的模型供應商
- 改進主要通訊頻道的支援 (並加入一些高需求的頻道)
- 效能與測試基礎設施
- 更好的電腦使用能力與 Agent 管理功能
- CLI 與 Web 前端的易用性
- macOS、iOS、Android、Windows 與 Linux 上的隨身應用程式

貢獻規則：

- 一個 PR = 一個議題/主題。不要捆綁多個不相關的修正/功能。
- 超過 ~5,000 行變動的 PR 僅在特殊情況下才會進行審核。
- 不要一次開啟大量的小型 PR；每個 PR 都有審核成本。
- 對於非常小的相關修正，鼓勵將其分組為一個集中的 PR。

## 安全性

OpenClaw 的安全性是一種深思熟慮後的權衡：強大的預設設定且不扼殺功能。
目標是在能執行真實工作的強大功能與明確的風險路徑（由操作者控制）之間取得平衡。

規範的安全政策與通報方式：

- [`SECURITY.md`](SECURITY.md)

我們優先考慮安全預設，但也為受信任的高權限工作流提供明確的控制開關。

## 插件與記憶 (Plugins & Memory)

OpenClaw 擁有豐富的插件 API。
核心保持精簡；選配功能通常應以插件形式提供。

偏好的插件路徑是 npm 套件分發，加上開發時的本地擴展載入。
如果你建立了一個插件，請在自己的倉庫中託管與維護。
將選配插件加入核心的門檻有意設得很高。
插件文件：[`docs/tools/plugin.md`](docs/tools/plugin.md)
社群插件列表：https://docs.openclaw.ai/plugins/community

記憶是一個特殊的插件插槽，同一時間只能有一個記憶插件處於活動狀態。
目前我們提供多種記憶選項；隨著時間推移，我們計畫收斂至一個推薦的預設路徑。

### Skills

我們仍會隨附一些捆綁的 Skills 以提供基礎體驗。
新的 Skills 應先發佈到 ClawHub (`clawhub.ai`)，而不是預設加入核心。
核心 Skill 的增加應該是罕見的，且需要強大的產品或安全理由。

### MCP 支援

OpenClaw 透過 `mcporter` 支援 MCP：https://github.com/steipete/mcporter

這保持了 MCP 整合的彈性，並與核心運行時解耦：

- 無需重啟 Gateway 即可新增或更改 MCP 伺服器
- 保持核心工具/上下文表面的精簡
- 減少 MCP 變動對核心穩定性與安全性的影響

目前，我們偏好這種橋接模式，而不是將第一類 MCP 運行時直接內建到核心中。

### 設定 (Setup)

OpenClaw 目前在設計上是以終端機 (Terminal) 為主的。
這能確保設定過程是明確的：使用者會預先看到文件、驗證、權限與安全性立場。

長期而言，隨著系統逐漸成熟，我們希望提供更簡易的上線流程。
但我們不希望透過便利的封裝層向使用者隱藏關鍵的安全決策。

### 為什麼選擇 TypeScript？

OpenClaw 主要是一個編排系統：Prompts、工具、協議與整合。
選擇 TypeScript 是為了保持 OpenClaw 的預設可開發性 (hackable)。
它廣為人知、迭代快速，且易於閱讀、修改與擴充。

## 我們暫時不會合併的內容

- 當 Skills 可以存在於 ClawHub 時的新核心 Skills
- 所有文件的完整翻譯集 (已推遲；我們計畫稍後使用 AI 生成翻譯)
- 不明顯屬於模型供應商類別的商業服務整合
- 在沒有明顯功能或安全差距的情況下，針對已支援頻道的封裝頻道
- 當 `mcporter` 已經提供整合路徑時，在核心中內建第一類 MCP 運行時
- 將 Agent 層級框架 (Manager-of-managers / 巢狀計畫樹) 作為預設架構
- 與現有 Agent 和工具基礎設施重複的重型編排層

這份清單是路線圖的護欄，而非物理定律。
強大的使用者需求與技術理由可以改變它。
