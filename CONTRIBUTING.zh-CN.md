## 如何贡献

1. **错误和小修复** → 打开 PR！
2. **新功能 / 架构** → 首先在 [GitHub Issue](https://github.com/openclaw/openclaw/issues/new/choose) 开始或在 Discord 中询问。大多数功能不被接受，应该使用我们的插件 SDK 作为第三方插件。
3. **仅重构 PR** → 不要打开 PR。我们不接受仅重构的更改，除非维护者明确要求它们作为具体修复的一部分。
4. **针对已知 `main` 失败的测试/CI 仅 PR** → 不要打开 PR。维护者团队已经在跟踪这些失败，并且只调整测试或 CI 以追逐它们的 PR 将被关闭，除非它们需要验证新的修复。
5. **问题** → Discord [#help](https://discord.com/channels/1456350064065904867/1459642797895319552) / [#users-helping-users](https://discord.com/channels/1456350064065904867/1459007081603403828)

## PR 限制

我们限制每个作者最多 **10 个开放的 PR**。如果您超过此限制，将添加 `r: too-many-prs` 标签，并且您的 PR 会自动关闭。这是一个硬性限制。

对于真正需要超过 10 个 PR 的协调变更集，请加入 Discord 中的 **#clawtributors** 频道并首先与维护者交谈。

## PR 之前

- 使用您的 OpenClaw 实例在本地测试
- 运行测试：`pnpm build && pnpm check && pnpm test`
- 对于迭代本地提交，`scripts/committer --fast "message" <files...>` 将 `FAST_COMMIT=1` 传递给预提交钩子，因此它会跳过整个仓库的 `pnpm check`。仅当您已经对触摸的表面运行了等效的目标验证时才使用它。
- 对于扩展/插件更改，首先运行快速本地通道：
  - `pnpm test:extension <extension-name>`
  - `pnpm test:extension --list` 查看有效的扩展 ID
  - 如果您更改了共享插件或通道表面，运行 `pnpm test:contracts`
  - 对于目标共享表面工作，使用 `pnpm test:contracts:channels` 或 `pnpm test:contracts:plugins`
  - 这些命令还覆盖了默认单元通道跳过的共享接缝/冒烟文件
  - 如果您更改了更广泛的运行时行为，在请求审查之前仍然运行相关的更广泛通道（`pnpm test:extensions`、`pnpm test:channels` 或 `pnpm test`）
- 如果您在共享代码中触及了捆绑插件边界，运行匹配的清单：
  - `node scripts/check-src-extension-import-boundary.mjs --json` 用于 `src/**`
  - `node scripts/check-sdk-package-extension-import-boundary.mjs --json` 用于 `src/plugin-sdk/**` 和 `packages/**`
  - `node scripts/check-test-helper-extension-import-boundary.mjs --json` 用于 `test/helpers/**`
- 共享测试助手必须使用 `src/test-utils/bundled-plugin-public-surface.ts` 而不是仓库相对的 `extensions/**` 导入。将插件本地深度模拟保留在拥有的捆绑插件包内。
- 如果您可以访问 Codex，请在打开或更新 PR 之前在本地运行 `codex review --base origin/main`。即使 GitHub Codex 审查也运行，将此视为当前 AI 审查的最高标准。
- 不要提交仅重构的 PR，除非维护者明确要求该重构用于活动修复或交付物。
- 不要提交针对已经在 `main` CI 上为红色的失败的测试或 CI 配置修复。如果在 [main 分支 CI 运行](https://github.com/openclaw/openclaw/actions) 中已经可见失败，这是维护者团队正在跟踪的已知问题，并且仅解决这些失败的 PR 将被自动关闭。如果您发现尚未在主 CI 中显示的 _新_ 回归，请首先将其报告为问题。
- 不要提交仅尝试使已知 `main` CI 失败通过的测试 PR。当测试更改需要验证同一 PR 中的新修复或覆盖新行为时，它们是可接受的。
- 确保 CI 检查通过
- 保持 PR 集中（每个 PR 一件事；不要混合无关的问题）
- 描述什么和为什么
- 在再次请求审查之前，回复或解决您解决的机器人审查对话
- **包含屏幕截图** — 一张显示问题/之前，一张显示修复/之后（对于 UI 或视觉更改）
- 在代码、注释、文档和 UI 字符串中使用美式英语拼写和语法
- 除非列出的所有者明确要求更改或已经在与您一起审查，否则不要编辑 `CODEOWNERS` 安全所有权覆盖的文件。将这些路径视为受限制的审查表面，而不是机会主义的清理目标。

## 审查对话由作者拥有

如果审查机器人在您的 PR 上留下审查对话，您应该处理后续工作：

- 一旦代码或解释完全解决了机器人的问题，自己解决对话
- 仅当您需要维护者或审查者判断时才回复并保持打开状态
- 不要留下"已修复"的机器人审查对话让维护者为您清理
- 如果 Codex 留下评论，请解决每一个相关的评论，或者在不适用时用简短的解释解决它
- 如果 GitHub Codex 审查由于某种原因没有触发，请无论如何在本地运行 `codex review --base origin/main` 并将该输出视为必需的审查工作

这适用于人工编写和 AI 辅助的 PR。

## 控制 UI 装饰器

控制 UI 使用 Lit 和 **遗留** 装饰器（当前的 Rollup 解析不支持标准装饰器所需的 `accessor` 字段）。添加反应性字段时，保持遗留风格：

```ts
@state() foo = "bar";
@property({ type: Number }) count = 0;
```

根 `tsconfig.json` 配置为遗留装饰器（`experimentalDecorators: true`）和 `useDefineForClassFields: false`。除非您也更新 UI 构建工具以支持标准装饰器，否则避免翻转这些。

## AI/氛围编码的 PR 欢迎！🤖

使用 Codex、Claude 或其他 AI 工具构建？**太棒了 - 只需标记它！**

请在您的 PR 中包含：

- [ ] 在 PR 标题或描述中标记为 AI 辅助
- [ ] 注明测试程度（未测试 / 轻度测试 / 完全测试）
- [ ] 如果可能，包含提示或会话日志（超级有帮助！）
- [ ] 确认您理解代码的作用
- [ ] 如果您可以访问 Codex，请在请求审查之前在本地运行 `codex review --base origin/main` 并解决发现
- [ ] 在解决机器人审查对话后解决或回复

AI PR 在这里是一等公民。我们只需要透明度，以便审查者知道要寻找什么。如果您使用 LLM 编码代理，请指示它解决它已经解决的机器人审查对话，而不是将它们留给维护者。

## 当前焦点和路线图 🗺

我们目前正在优先考虑：

- **稳定性**：修复频道连接（WhatsApp/Telegram）中的边缘情况。
- **用户体验**：改进入职向导和错误消息。
- **技能**：对于技能贡献，请前往 [ClawHub](https://clawhub.ai/) — OpenClaw 技能的社区中心。
- **性能**：优化令牌使用和压缩逻辑。

查看 [GitHub Issues](https://github.com/openclaw/openclaw/issues) 中的
["good first issue"](https://github.com/openclaw/openclaw/issues?q=is%3Aissue+is%3Aopen+label%3A%22good+first+issue%22)
标签。如果没有开放的，请选择一个小的文档或错误问题，并留下一个简短的评论说
您想处理它。

### 报告中必需的内容

1. **标题**
2. **严重性评估**
3. **影响**
4. **受影响的组件**
5. **技术重现**
6. **证明的影响**
7. **环境**
8. **补救建议**

没有重现步骤、证明的影响和补救建议的报告将被降级。鉴于 AI 生成的扫描器发现的数量，我们必须确保我们收到来自理解问题的研究人员的经过审查的报告。
