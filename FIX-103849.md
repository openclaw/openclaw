# FIX-103849 — Anthropic transport drops HTTP status & `Retry-After` on error

**Issue:** https://github.com/openclaw/openclaw/issues/103849
**Labels:** `good first issue`, `clawsweeper:queueable-fix`, `clawsweeper:fix-shape-clear`, `clawsweeper:current-main-repro`, `impact:auth-provider`
**Severity:** P2 — rate-limited `anthropic-messages` turns fail instead of waiting out the provider cooldown.

---

## The problem

Every `anthropic-messages` model uses the bespoke, non-SDK fetch client in
`src/agents/anthropic-transport-stream.ts`. When a request fails with a
non-2xx response (e.g. **HTTP 429 rate limit**), the client turned the
response into a _plain_ `Error`:

```ts
if (!response.ok) {
  const detail = await readAnthropicMessagesErrorBodySnippet(response);
  throw new Error(detail || `Anthropic Messages request failed with HTTP ${response.status}`);
}
```

The **HTTP status** and the **`Retry-After` header were discarded**. Only the
text survived.

Downstream, `AgentSession.prepareRetry`
(`src/agents/sessions/agent-session-execution.ts`) computes the backoff purely
from `retryCount`:

```ts
const delayMs = settings.baseDelayMs * 2 ** (this.retryCount - 1); // 2s / 4s / 8s
```

So a `429` carrying `Retry-After: 30` produced a fixed `2s / 4s / 8s` wait.
With `maxRetries: 3`, **all three retries landed inside the still-active
30-second window**, and the turn failed outright instead of succeeding after
the server's actual cooldown.

This only affected the `anthropic-messages` transport: the SDK-based path in
`src/agents/provider-transport-fetch.ts` already parses `Retry-After`
(`parseRetryAfterSeconds`), so the bug was an asymmetry between the two
transports.

---

## The fix

Thread the HTTP status and server `Retry-After` from the transport error all
the way into the `AssistantMessage`, then let `prepareRetry` honor it.

### 1. Capture status + `Retry-After` at the transport (`anthropic-transport-stream.ts`)

`parseRetryAfterSeconds` already existed in `provider-transport-fetch.ts`
(handles `retry-after-ms`, delta-seconds, and HTTP-date) — it is now **exported**
and reused instead of duplicating the logic.

```ts
if (!response.ok) {
  const detail = await readAnthropicMessagesErrorBodySnippet(response);
  const error = new Error(
    detail || `Anthropic Messages request failed with HTTP ${response.status}`,
  );
  const typedError = error as Error & { status?: number; retryAfterMs?: number };
  typedError.status = response.status;
  const retryAfterSeconds = parseRetryAfterSeconds(response.headers);
  if (retryAfterSeconds !== undefined && Number.isFinite(retryAfterSeconds)) {
    typedError.retryAfterMs = Math.round(retryAfterSeconds * 1000);
  }
  throw typedError;
}
```

### 2. Carry the values into `AssistantMessage`

`assignTransportErrorDetails` (`src/agents/transport-stream-shared.ts`) already
copied `errorCode` / `errorType` / `errorBody` from the error object onto the
output via `Object.assign`. It now also reads `status` and
`retryAfterMs` / `retryAfterSeconds`:

```ts
const errorStatus = readNumberLikeProperty(errorObject, "status");
const retryAfterMs =
  readNumberLikeProperty(errorObject, "retryAfterMs") ??
  (() => {
    const s = readNumberLikeProperty(errorObject, "retryAfterSeconds");
    return s !== undefined ? s * 1000 : undefined;
  })();
// ...returned and Object.assign'd onto the output
```

Two optional fields were added so the data survives the type boundary:

- `AssistantMessage.errorStatus?: number` — `packages/llm-core/src/types.ts`
- `AssistantMessage.errorRetryAfterMs?: number` — same file
- `TransportOutputShape.errorStatus?` / `errorRetryAfterMs?` —
  `src/agents/transport-stream-shared.ts`

(Adding optional fields is non-breaking for the published `@openclaw/ai` types.)

> Side benefit: SDK-based transports already attach `status` to their errors, so
> they now also populate `errorStatus` for free. `retryAfterMs` is set by the
> anthropic path today; other transports can opt in by attaching it.

### 3. Honor the server cooldown in `prepareRetry` (`agent-session-execution.ts`)

```ts
const exponentialDelayMs = settings.baseDelayMs * 2 ** (this.retryCount - 1);
const serverRetryAfterMs = message.errorRetryAfterMs;
const delayMs =
  serverRetryAfterMs && serverRetryAfterMs > 0
    ? Math.min(serverRetryAfterMs, MAX_RETRY_AFTER_WAIT_MS)
    : exponentialDelayMs;
```

`MAX_RETRY_AFTER_WAIT_MS = 60_000` mirrors the SDK retry cap
(`OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS`, default 60s) so a pathological header
can't stall a turn. When the server asks for 30s, the first retry now waits
~30s and the turn succeeds.

## How to verify

### Automated (one test file)

```bash
pnpm test src/agents/anthropic-transport-stream.test.ts src/agents/transport-stream-shared.test.ts
```

The two new tests assert the end-to-end data flow:

- Transport test: a stub `429` with `Retry-After: 30` yields an
  `AssistantMessage` with `errorStatus === 429` and `errorRetryAfterMs === 30000`.
- Shared test: `assignTransportErrorDetails` copies `errorStatus` /
  `errorRetryAfterMs` from a structured error.

> Note: the anthropic test mocks `provider-transport-fetch.js`. The mock was
> changed to `vi.importActual` spread so the real (now exported)
> `parseRetryAfterSeconds` is available.

### Manual

1. Point an `anthropic-messages` model's `baseUrl` at a stub that returns
   `HTTP 429` + `Retry-After: 30` with an Anthropic-shaped JSON body.
2. Drive one turn. With the fix, the session waits ~30s once and then
   succeeds (or exhausts retries _after_ the window) instead of failing inside
   the window with a `2s / 4s / 8s` burst.
