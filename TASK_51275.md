# Task: Fix sandbox security errors leaking PII to channels (Issue #51275)

## Problem

When an agent fails due to a sandbox security violation (e.g., a bind mount source outside
allowed roots), the full raw error message is posted to the configured channel (Discord, Telegram, etc.)
as if it were the agent's reply. This leaks:

- Full filesystem paths including OS username
- Sandbox configuration details
- Workspace directory structure
- Config file names and mount targets

Example of what leaks to the channel:

```
⚠️ Agent failed before reply: Sandbox security: bind mount "/home/user/.openclaw/sandbox-configs/config.json:/app/config.json:ro"
source "/home/user/.openclaw/sandbox-configs/config.json" is outside allowed roots
(/home/user/.openclaw/workspace-agent). Use a dangerous override only when you fully trust this runtime.
Logs: openclaw logs --follow
```

## Expected behavior

Error should be logged internally only. Channel receives either NO reply or a generic safe message:

```
⚠️ Agent failed to start. Check logs: openclaw logs --follow
```

## Key files to investigate

- `src/auto-reply/reply/agent-runner.ts` — main reply agent runner
- `src/auto-reply/reply/agent-runner-execution.ts` — where errors are caught, imports `sanitizeUserFacingText`
- `src/auto-reply/reply/agent-runner-payloads.ts` — payload building
- `src/agents/pi-embedded-helpers/errors.ts` — error classification, has `sanitizeUserFacingText`
- Look for where "Agent failed before reply" or "Sandbox security" error text gets into a ReplyPayload

## What the fix should do

1. Find where the raw sandbox error text gets converted to a channel message
2. Detect if error is a sandbox security error (look for "Sandbox security" or `FatalSandboxError`)
3. Replace the full raw error with a sanitized generic message before it reaches the channel
4. Keep the original error in internal logs only (not redact from logs, only from channel output)
5. The existing `sanitizeUserFacingText` function may already be useful here
