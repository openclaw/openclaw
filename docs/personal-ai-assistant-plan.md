# 个人 AI 助手方案

> 最后更新：2026-03-27
> 状态：需求确认阶段

## 核心价值

**AI 替你完成"看、筛、找、转"，你只做"想和决定"。**

你每天被微信、飞书、iMessage 的消息淹没，但真正需要亲自处理的不到 5%。
个人助手的价值不是"聊天机器人"，而是**主动过滤噪音、组织信息、在你需要时把关键内容送到面前**。

---

## 系统架构

### 设计原则

1. **OpenClaw 做中枢** — 连接所有消息渠道，控制系统操作，统一出入口
2. **双模型分层调度** — 日常任务走 DeepSeek（低成本），复杂决策走 Claude（高质量）
3. **尽量不在 OpenClaw 侧消耗 Claude token** — Claude 仅在需要深度推理时按需调用

### 模型分工策略

| 任务类型 | 模型 | 触发方式 | 示例 |
|----------|------|----------|------|
| 消息分类/摘要 | DeepSeek | 自动（所有入站消息） | 标注紧急/待回复/已知/忽略 |
| 模板化回复 | DeepSeek | 自动 | "好的收到"、日程确认 |
| 文件操作 | DeepSeek | 用户指令 | "找一下上周的报价单" |
| 深度研究 | Claude | 用户主动触发 `/deep-research` | 行业分析、技术调研 |
| 代码开发 | Claude | 用户主动触发 `/dev-proxy` | 修 bug、写功能 |
| 跨渠道协调 | Claude | 自动升级（复杂度检测） | 综合多渠道信息做判断 |

### 微信双通道说明

- **openclaw-weixin**：官方协议，能收发消息，但仅限1对1，无群聊，无历史搜索
- **MCP Server**：本地 DB 解密，能读取全量历史、搜索、解密图片，但不能发消息
- **组合使用**：MCP 提供深度查询能力，openclaw-weixin 负责发送 = 完整微信助手

## 核心场景

### 场景一：消息流转

**目标**：三大渠道消息统一处理，只把需要你关注的送到面前。

**用户故事**：
- 我在外面，微信来了 30 条消息，助手告诉我"闫绿梅问家里有没有电池，需要你回复；其余是公众号推送，无需关注"
- 飞书有人 @我，助手判断紧急程度，紧急的转发摘要到 iMessage 提醒我
- 我说"帮我找上周和闫绿梅讨论的报价单"，助手通过 MCP 搜索微信记录找到

**实现要点**：
1. 所有入站消息经 DeepSeek 预分类（紧急/待回复/已知/忽略）
2. 紧急消息跨渠道转发（微信 → iMessage 提醒）
3. 需要搜索历史时调用微信 MCP Server
4. 复杂的综合判断升级到 Claude

**已就绪**：渠道全通、inbox-triage skill 已安装、DeepSeek 已是主模型
**待开发**：消息预分类 hook、跨渠道转发逻辑、MCP 整合到 skill

### 场景二：设备协同

**目标**：手机发一条消息，就能操作电脑上的文件。

**用户故事**：
- "把桌面上的报价单发给我" → 找到文件，内容发到消息
- "帮我压缩 project 目录" → 执行打包
- "刚才复制了什么" → 读取剪贴板内容

**实现要点**：file-ops skill 已安装，DeepSeek 即可处理

**已就绪**：完整可用，不需要额外开发
**待优化**：剪贴板同步（需 MCP 工具）

### 场景三：远程开发

**目标**：手机上指挥电脑写代码。

**用户故事**：
- "帮我在 openclaw 项目里修复 #123 issue" → Claude 分析 + 改代码 + 跑测试 → 返回结果
- "跑一下测试" → DeepSeek 执行 `npm test` 返回结果
- "看看 main 分支最近改了什么" → DeepSeek 跑 git log 返回摘要

**实现要点**：
1. 简单指令（git/test/log）→ DeepSeek
2. 写代码/改架构 → 路由到 Claude
3. dev-proxy skill 已安装

**已就绪**：基本可用
**待开发**：模型路由规则（检测任务复杂度自动升级）

### 场景四：深度学习

**目标**：对任何主题，快速从入门到看透本质。

**用户故事**：
- "研究一下微信小程序的商业模式" → Claude 多步研究，结构化输出
- "对比 React 和 Vue 在大型项目中的取舍" → 深度技术分析

**实现要点**：deep-research skill 已安装，固定走 Claude

**已就绪**：完整可用
**待优化**：研究结果持久化存储

## 已知限制与风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| 模型路由判断不准 | 简单任务错送 Claude 浪费钱，复杂任务给 DeepSeek 质量差 | 先用规则路由（关键词+长度），后续用 DeepSeek 自身做分类器 |
| 微信 MCP 依赖 PC 在线 | 电脑关机时无法读取微信数据 | 短期接受；长期考虑服务器部署 |
| 微信仅支持1对1 | 无法处理群聊消息 | 平台限制，非技术问题，暂不解决 |
| 无定时任务 | 不能主动推送每日摘要 | Phase 2 用 Cron + CLI 触发 |
| 无持久记忆 | 跨会话丢失上下文 | Phase 3 接入文件存储或向量数据库 |

## 实施路线

### Phase 1：基础跑通（本周）

**目标**：Claude CLI 接入 OpenClaw，双模型路由验证通过

| 任务 | 说明 | 状态 |
|------|------|------|
| 创建 claude-cli skill | 通过 `claude -p` 调用本地 Claude Max，零额外成本 | ✅ 完成 |
| 配置模型路由规则 | CLAUDE.md 指导 DeepSeek 识别复杂任务并调用 Claude CLI | ✅ 完成 |
| 创建 wechat-search skill | 通过 Claude CLI 调用微信 MCP Server 搜索聊天记录 | ✅ 完成 |
| 端到端验证 | 手机发消息 → 分类 → 回复，全链路跑通 | 待测试 |

**关键决策**：
- 不用 Anthropic API key，利用 Claude Max 包月订阅零额外成本
- 不靠 DeepSeek 判断调用（不可靠），改用 OpenClaw 原生 CLI backend 机制
- 用户通过 `@model claude-cli/opus` 命令主动切换到 Claude 后端
- openclaw.json 中配置了 `cliBackends.claude-cli`，自动处理环境变量和命令参数

### Phase 2：消息流转核心（1-2 周）

**目标**：消息自动分类 + 跨渠道转发 + 微信历史搜索

| 任务 | 说明 | 依赖 |
|------|------|------|
| 消息预分类 hook | OpenClaw `before_model_resolve` hook，DeepSeek 预处理 | Phase 1 |
| 跨渠道转发 | 微信紧急消息 → iMessage 摘要推送 | 预分类 hook |
| wechat-search skill | 整合 MCP Server 到 OpenClaw skill | 无 |
| 每日摘要 MVP | Cron 触发，汇总当日三渠道消息 | 预分类 hook |

### Phase 3：智能升级（2-4 周）

**目标**：自动复杂度检测 + 持久记忆 + 定时任务

| 任务 | 说明 | 依赖 |
|------|------|------|
| 复杂度自动检测 | DeepSeek 判断任务复杂度，自动升级到 Claude | Phase 2 |
| 持久化记忆层 | 跨会话知识存储（文件/SQLite） | 无 |
| 定时任务系统 | 内置调度器替代外部 Cron | Phase 2 |
| 邮箱接入 | 调研 OpenClaw 邮箱支持或 MCP 接入 | 无 |
