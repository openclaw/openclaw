## Issue 1 — [HIGH] Custom summary method is dropped from the summary (report 4C) · app.havaya

**Evidence (prod container logs, test date):**

```
[drive/summarize] methodChars=134 transcriptChars=36230          ← custom method WAS read from DB + sent
[drive/summarize] replyChars=652 head="QA_MARKER_4C_20260626\nTITLE: …\nDATE: …\n\nבפגישה…"  ← model emitted the marker
```

Wiring is correct and the model obeyed. The marker is then **stripped after generation**.

**Root cause:** `parseTitleDate()` in `src/app/admin/api/drive/summarize/route.ts` (≈lines 28-46)
scans the first 15 lines for `TITLE`/`DATE` and returns `body = lines.slice(lastMeta + 1)` — i.e. it
**discards every line before the last meta line**. The marker sits on line 0 (above `TITLE`), so it is
thrown away. That preamble-stripping existed to tolerate the _old_ life-agent path leading with
sub-agent noise; since commit `87b1629` summarization is a **direct OpenAI call** with no such noise,
so the stripping now only destroys intended content.

**Fix:** Make the parser **extract** `TITLE`/`DATE` without deleting other content. Remove only the
matched meta lines (in place), preserve all remaining lines and their order, so a leading marker — and
any custom-method structure — survives into `body`. This also makes _real_ custom methods (extra
sections, different wording) reliably take effect.

- File: `src/app/admin/api/drive/summarize/route.ts` — rewrite `parseTitleDate`.
- **Test (the app has no runner):** extract `parseTitleDate` into a tiny sibling module and add a
  `node:test` self-check run via `node --test` (with `tsx`). Cases: (a) `QA_MARKER…\nTITLE:…\nDATE:…\nbody`
  → body keeps the marker; (b) plain `TITLE:…\nDATE:…\nbody` → title/date parsed, body intact;
  (c) no meta lines → whole reply is the body (existing fallback).

## Issue 2 — [HIGH] Durable memory not recalled across chats (report 4A) · openclaw/life

**Evidence:** graphiti-mcp logs show `Completed add_episode … group app_user_<id>` succeeding on the
test date → **saving works**. Graphiti tools (`add_memory`, `search_memory_facts`, …) are registered
and scoped per-user (`app_<userId>`) by the `before_tool_call` hook in
`ops/graphiti-life/extensions/life-memory-scope/index.js`.

**Root cause:** retrieval is **discretionary** — the agent must choose to call `search_memory_facts`,
and there is **no deterministic injection** at session start (unlike `app_profile`, which _is_ injected
every turn). Under the slim app prompt the agent often skips the search, so a new chat doesn't see
prior goals. Related prior work: `docs/experiments/plans/life-per-user-memory.md`.

**Fix (chosen: deterministic recall injection — mirrors the proven `app_profile` bootstrap):**

1. **New** `src/agents/graphiti-recall-client.ts` — a small server-side streamable-HTTP MCP client that
   calls Graphiti `search_memory_facts` directly (bypassing the agent tool loop), but keeps the same
   scope boundary as the proxy. Implement a tiny helper equivalent to
   `ops/graphiti-life/proxy/graphiti-proxy.js` `buildUpstreamCall` for this one tool: accept only a
   server-derived `groupId`, reject unsafe group ids, send `group_ids: [groupId]`, and never allow
   caller-supplied `group_id`, `group_ids`, or `center_node_uuid` through. Mirror the handshake/SSE-parse
   in the proxy (`initialize` → `notifications/initialized` → `tools/call`; headers incl.
   `Host: localhost:8000`; target `GRAPHITI_URL` default `http://graphiti-mcp:8000/mcp`). Use `fetch` +
   `AbortController`.
2. **New** `src/agents/memory-recall-context.ts` — `appendMemoryRecallBootstrapFile(...)` that:
   resolves the app userId via the existing `resolveAppUserId` / `appUserIdFromSessionKey`
   (`src/agents/app-profile-context.ts`); derives the group id with the **same** sanitize as the hook
   (`"app_" + id.replace(/[^A-Za-z0-9_]/g,"_")`, id already lowercased); fetches top ~8 facts; injects
   them as a synthetic **`MEMORY_RECALL.md`** (NOT `MEMORY.md` — that name is app-excluded).
   App-sessions only; **timebox ~1.5-2.5s + fail-open** (any error/empty/no-scope → return files
   unchanged, never block the turn). **Do not add a cross-turn TTL cache in v1**: immediate cross-chat
   recall after a successful memory write is more important than saving one Graphiti call. If latency
   proves too high, add cache later only with explicit invalidation on `add_memory` / upstream
   `add_episode` success, or with a per-run/in-flight dedupe that cannot serve stale facts to a new chat.
3. **Edit** `src/agents/bootstrap-files.ts` — chain `appendMemoryRecallBootstrapFile` onto the final
   return of `resolveBootstrapFilesForRun`, right after `appendAppProfileBootstrapFile` (same call site,
   reached by both `pi-embedded-runner/run/attempt.ts` and `auto-reply/.../commands-system-prompt.ts`).
4. **Prompt reinforcement** — one sentence in the life workspace `AGENTS.app.md` retrieval section (and
   its source `ops/graphiti-life/agents-md-memory-section.md`): the top durable facts are already
   provided in `MEMORY_RECALL.md`; lean on it first, call `search_memory_facts` only for more.
5. **Tests** (vitest) — mirror `src/agents/app-profile-context.test.ts`: builder + clamp + group-id
   parity; append no-ops for telegram / missing key / unresolved userId; injects for an app key with the
   right group id; **fail-open** when the client rejects; no stale cross-turn cache reuse after a memory
   write. Plus client wire/timebox/scope tests: unsafe group id fails closed, outbound arguments include
   exactly `group_ids: [groupId]`, and no caller-provided `group_id` / `group_ids` /
   `center_node_uuid` can be forwarded. Run
   `pnpm vitest run src/agents/memory-recall-context.test.ts src/agents/graphiti-recall-client.test.ts`.

**Risks:** (a) latency on first turn — bounded by timebox + fail-open; verify `search_memory_facts` p95
on host before shipping and only add caching with write invalidation if needed; (b) **scope enforcement**
must match the proxy's safety behavior even though the bootstrap client talks directly to Graphiti —
pin `group_ids: [groupId]`, unsafe-id rejection, and forbidden-argument stripping with unit tests;
(c) **group-id parity** is a cross-file contract with the hook's sanitize — pin it with the unit test;
(d) Graphiti unreachable → fail-open (degrades to today's behavior, logged `warn`); (e) keep the
synthetic name `MEMORY_RECALL.md` to avoid the app-excluded `MEMORY.md`.

## Issue 3 — [LOW/UX] Google Doc raw transcript not editable (report 5C) · app.havaya

**Root cause: by design.** `updateTextFile` in `src/lib/googleDrive.ts` (≈line 279-281) rejects Google
Docs (`415`) — you cannot media-upload over a Google Doc. Reading works (it exports to text). The edit
UI is offered and only fails on save, which reads as a bug.

**Fix (small UX, recommended):** `GET` in `src/app/admin/api/drive/file-text/route.ts` returns
`editable: entry.mimeType.startsWith("text/")`; `src/components/admin/DriveBrowser.tsx` hides/disables
the raw-edit **Save** for non-editable files and shows a short note ("Google Docs can't be edited
here — summaries still work"). Reading + summarizing are unaffected.

## Issue 4 — [DOCS/DATA] Home hub: owner note + latest videos missing (report E) · app.havaya

  (`page.tsx` / `HomeHub.tsx` never fetch or render them). **Decision: fix the guide** (Issue 5), no app
  code change.
- **Owner note — data, not a bug.** Home fetches the per-user `app_note` section via
  `getUserSection(userId, "app_note")` (2.5s timeout); `OwnerNote` renders `null` when empty. The test
  user had no note, so it was correctly hidden. **Action:** set an `app_note` for a test user to verify;
  optionally raise the 2.5s timeout for reliability (not required).
