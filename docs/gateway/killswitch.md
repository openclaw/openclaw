The killswitch pauses agent runs across the whole gateway without stopping
the gateway process itself. It exists for one scenario: the agent is doing
something wrong right now and you need it to stop immediately, before you've
had time to diagnose why.

There are two independent layers. Use whichever one you can reach.

## Layer 1: hard stop (CLI / SSH)

```bash
openclaw daemon service stop
```

This stops the whole gateway process, including every channel transport. It
is the only layer guaranteed to work regardless of what is broken — including
a compromised or wedged channel plugin itself, not just misbehaving agent
reasoning. Use it whenever you are unsure which layer applies, or when you
suspect the problem is in the transport/channel code rather than the agent's
decisions.

Resume with:

```bash
openclaw daemon service start
```

## Layer 2: soft pause (killswitch)

The killswitch blocks new agent runs and aborts in-flight ones, but keeps the
gateway process — and every channel transport, including Signal — running.
This only helps if you trust the channel transport itself is healthy; it
cannot rescue a wedged or compromised channel, since the pause depends on
that channel's own message pipeline still working to deliver the trigger.

### Via CLI / SSH

```bash
openclaw killswitch status
openclaw killswitch enable --reason "runaway tool calls"
openclaw killswitch disable
```

`enable` refuses new agent runs (interactive and cron-triggered) and aborts
every currently active run — both gateway RPC-tracked runs and
channel-triggered embedded runs.

### Via Signal (owner DM only)

Send an exact phrase in a 1:1 Signal DM from the configured owner number:

| Phrase              | Effect                                |
| ------------------- | ------------------------------------- |
| `KILLSWITCH-ENGAGE` | Pauses agent runs, aborts active runs |
| `KILLSWITCH-REVIVE` | Resumes agent runs                    |

This is recognized ahead of any LLM/agent involvement — it is a plain string
match against the owner's Signal allowlist entry, checked before the message
reaches the agent, so a misbehaving agent has no opportunity to reason around
it. Group messages and non-owner senders are ignored (the message falls
through to normal handling as if the phrase were never sent).

The confirmation reply is sent directly, not generated by the model.

## Persistence

The killswitch state survives a gateway restart or crash: if you engaged it
because the agent was misbehaving, and the gateway happens to restart while
paused, it stays paused. Nothing resumes it automatically — you must run
`openclaw killswitch disable` or send `KILLSWITCH-REVIVE` explicitly.

## What it does not cover

- It does not stop a channel plugin that is itself misbehaving (flooding,
  wedged, compromised) — only the agent-run path. Use Layer 1 for that.
- Signal is currently the only channel wired to the phrase trigger. The
  enforcement itself (state flag, agent-run refusal, cron gate) is
  channel-agnostic, so other channels can add the same trigger later.
