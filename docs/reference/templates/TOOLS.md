## 其他注意事项

_任何帮助你完成工作的特殊配置或注意事项_

### ⚠️ 项目归属判定规范（重要）

**判断"当前用户名下有哪些项目（topic）"，唯一正确的依据是系统注入的消息前缀。**

每条用户消息开头都带有后台管道注入的前缀，例如：

```
[userId:1749] [topicId:585 topicName:"广本监测专项" useSlaveTopic:false] 用户的实际消息…
```

- 该前缀是后台在消息送达前**实时查询 `entity_auth` + `feed_topic` 的权威结果**，直接信任并使用即可，无需再做任何确认
- `topicId` / `topicName` = 用户**最近授权**的主项目；`useSlaveTopic` = 是否从表专题
- 用户名下有多个项目时，前缀会追加 `[allTopics: 116:"专题A", 585:"专题B", …]` 列出**全部项目**；没有该块说明只有主项目一个
- 前缀中没有 `[topicId:...]` → 该用户名下当前没有项目映射，如实告知用户即可
- **你不需要、也无法直接连接 MySQL 数据库**——不要尝试自己执行 SQL 查询 entity_auth / feed_topic
- 报告生成任务的舆情数据同样由系统**预先查询**并完整附在任务提示词中（统计概览 + 条目明细），直接使用即可；不要在报告里声明"无法连接数据库"，**严禁编造或推测任何数据**

### 🔎 临时舆情查询：`feed_query` 工具

聊天中需要查舆情数据（如"排查昨天的高风险舆情"、"这周负面有多少条"）时，**调用 `feed_query` 工具**，不要只靠网络搜索，更不要说"无法接入数据库"。

- `mode:"search"` → 返回明细条目（标题、摘要、平台、风险等级、情感、链接等）
- `mode:"stats"` → 返回全量统计（总数 + 按 level/emotion/platform/day 等维度分组计数）
- 常用过滤：`startDate`/`endDate`（YYYY-MM-DD）、`level`（Red/Orange/Yellow/Blue）、`emotion`（Positive/Neutral/Negative）、`keyword`、`platform`
- 权限由系统自动限定在你所属用户的授权项目内；`topicId` 省略时默认查主项目，需要查其他项目时传 `[allTopics: ...]` 中列出的 id
- 工具返回 `success:false` 且带 `authorizedTopics` 时，说明传入的 topicId 不在授权范围，改用列表中的 id 重试
- 网络搜索仅用于补充**库外**的公开信息，结论应以 `feed_query` 的库内数据为准

**绝对禁止：**

- 向用户索要数据库连接信息（host、端口、账号、密码），或把任何数据库凭证写入本文件——凭证只存在于网关配置中，永远不进工作区
- 复用任何旧脚本里硬编码的 topic ID 列表（如 `id IN (270, 585, 586, ...)`）——那些是过去某次调试时手工挑选的样本，**不代表项目归属**，直接照抄会把别人的项目混进个人项目里

### 🛡️ 图文/视频侵权检测：`infringement_create_task` / `infringement_query`

观舆卫士的「图文/视频侵权检测」业务（涉企侵权举报：对违规图文链接、微信视频号等做研判、出文书）。聊天中用户要**立案研判**或**查侵权案件/账号/进度**时，用这两个工具，不要自己连库、不要编造。

**创建研判任务 `infringement_create_task`**（用户说"帮我把这几个链接立案研判"/"检测下这个账号的侵权内容"时）：

- `links`（必填）：要研判的链接，url 数组或多行文本；微信视频号没有网址，按"视频号：账号名 / 视频ID：xxx / 标题"整行文本传入
- `reporter?` / `target?` / `enterpriseType?`：举报人 / 被举报方 / 企业类型，知道就带上，不知道可省
- 返回 `caseId` 后**研判由后端异步完成**——告诉用户"已立案（案号 caseNo），研判进行中，稍后可查结果"，不要假装当场出了结论

**查询侵权数据 `infringement_query`**（4 种 mode）：

- `mode:"cases"` → 案件列表，可按 `stage`（draft/accepted/analyzing/analyzed/failed）、`accept_conclusion`、`archived`、`min_score`、`startDate`/`endDate`、`keyword`（举报人/被举报方/案号）筛选
- `mode:"case_detail"` + `caseId` → 单案全景：案件信息 + 所有链接的研判结果 + 整体报告
- `mode:"account"` + `account`（可选 `platform`）→ 某账号违规画像：总链接数、达标违规数、评分/平台分布、关联案件
- `mode:"kpi"` → 待处理 / 审核中 / 已办结 计数

**口径与权限：**

- 评分（0–10）**≥6 视为达标违规**；研判结果可能有人工覆盖评分（`manual_score`），以生效评分为准
- 权限由系统按 `entity_auth`（Legal 授权）自动限定；无侵权访问权限时工具返回 `success:false`，如实告知用户，不要追问凭证
- 保密案对普通账号不可见；举报人/被举报方电话邮箱已脱敏——不要尝试还原
- 同样**绝不**向用户索要数据库连接信息，也不把凭证写入本文件
