# Clawnet リファクタリング（プロトコル + 認証統一）

## はじめに

Peter へ — 素晴らしい方向性です。これにより、よりシンプルな UX とより強固なセキュリティが実現します。

## 目的

以下のための単一の厳格なドキュメントです:

- 現在の状態: プロトコル、フロー、トラスト境界。
- 課題: 承認、マルチホップルーティング、UI の重複。
- 提案する新しい状態: 1 つのプロトコル、スコープ付きロール、統一された認証/ペアリング、TLS ピニング。
- アイデンティティモデル: 安定した ID + かわいいスラッグ。
- 移行計画、リスク、未解決の質問。

## 目標（議論より）

- すべてのクライアント（Mac アプリ、CLI、iOS、Android、ヘッドレスノード）のための 1 つのプロトコル。
- すべてのネットワーク参加者が認証 + ペアリング済み。
- ロールの明確化: ノード vs オペレータ。
- ユーザーがいる場所に中央承認をルーティング。
- すべてのリモートトラフィックに TLS 暗号化 + オプションのピニング。
- 最小限のコード重複。
- 単一のマシンは 1 回だけ表示される（UI/ノードの重複エントリなし）。

## 非目標（明示的）

- 機能分離の削除（最小権限は引き続き必要）。
- スコープチェックなしで完全な Gateway コントロールプレーンを公開。
- 認証を人間のラベルに依存させる（スラッグはセキュリティとは無関係）。

# 現在の状態（as-is）

## 2 つのプロトコル

### 1) Gateway WebSocket（コントロールプレーン）

- 完全な API サーフェス: コンフィグ、チャンネル、モデル、セッション、エージェント実行、ログ、ノードなど。
- デフォルトバインド: ループバック。SSH/Tailscale 経由のリモートアクセス。
- 認証: `connect` 経由のトークン/パスワード。
- TLS ピニングなし（ループバック/トンネルに依存）。
- コード:
  - `src/gateway/server/ws-connection/message-handler.ts`
  - `src/gateway/client.ts`
  - `docs/gateway/protocol.md`

### 2) Bridge（ノードトランスポート）

- 狭いアローリストサーフェス、ノードアイデンティティ + ペアリング。
- TCP 上の JSONL; オプションの TLS + 証明書フィンガープリントピニング。
- TLS は発見 TXT でフィンガープリントをアドバタイズ。
- コード:
  - `src/infra/bridge/server/connection.ts`
  - `src/gateway/server-bridge.ts`
  - `src/node-host/bridge-client.ts`
  - `docs/gateway/bridge-protocol.md`

## 現在のコントロールプレーンクライアント

- CLI → `callGateway` 経由の Gateway WS（`src/gateway/call.ts`）。
- macOS アプリ UI → Gateway WS（`GatewayConnection`）。
- Web コントロール UI → Gateway WS。
- ACP → Gateway WS。
- ブラウザコントロールは独自の HTTP コントロールサーバーを使用。

## 現在のノード

- ノードモードの macOS アプリは Gateway Bridge に接続（`MacNodeBridgeSession`）。
- iOS/Android アプリは Gateway Bridge に接続。
- ペアリング + ノードごとのトークンが Gateway に保存。

## 現在の承認フロー（exec）

- エージェントは Gateway 経由で `system.run` を使用。
- Gateway はブリッジ経由でノードを起動。
- ノードランタイムが承認を決定。
- UI プロンプトが Mac アプリで表示（ノード == Mac アプリの場合）。
- ノードが `invoke-res` を Gateway に返す。
- マルチホップ、UI はノードホストに結び付けられている。

## 現在のプレゼンス + アイデンティティ

- WS クライアントからの Gateway プレゼンスエントリ。
- ブリッジからのノードプレゼンスエントリ。
- Mac アプリは同じマシンの 2 つのエントリを表示する場合がある（UI + ノード）。
- ノードアイデンティティはペアリングストアに保存。UI アイデンティティは別。

---

# 問題点 / 課題

- 2 つのプロトコルスタックを維持（WS + Bridge）。
- リモートノードの承認: プロンプトはユーザーがいる場所ではなくノードホストに表示。
- TLS ピニングはブリッジにのみ存在。WS は SSH/Tailscale に依存。
- アイデンティティの重複: 同じマシンが複数のインスタンスとして表示。
- 曖昧なロール: UI + ノード + CLI の機能が明確に分離されていない。

# 提案する新しい状態（Clawnet）

## 1 つのプロトコル、2 つのロール

ロール + スコープを持つ単一の WS プロトコル。

- **ロール: ノード**（機能ホスト）
- **ロール: オペレータ**（コントロールプレーン）
- オペレータのオプション **スコープ**:
  - `operator.read`（ステータス + 閲覧）
  - `operator.write`（エージェント実行、送信）
  - `operator.admin`（コンフィグ、チャンネル、モデル）

### ロールの動作

**ノード**

- 機能（`caps`、`commands`、パーミッション）を登録できる。
- `invoke` コマンド（`system.run`、`camera.*`、`canvas.*`、`screen.record` など）を受け取れる。
- イベントを送信できる: `voice.transcript`、`agent.request`、`chat.subscribe`。
- コンフィグ/モデル/チャンネル/セッション/エージェントコントロールプレーン API は呼び出せない。

**オペレータ**

- スコープでゲートされた完全なコントロールプレーン API。
- すべての承認を受け取る。
- OS アクションを直接実行せず、ノードにルーティングする。

### 主要ルール

ロールはデバイスごとではなく接続ごと。デバイスは両方のロールを別々に開くことができます。

# 統一された認証 + ペアリング

## クライアントアイデンティティ

すべてのクライアントが以下を提供します:

- `deviceId`（安定、デバイスキーから派生）。
- `displayName`（人間の名前）。
- `role` + `scope` + `caps` + `commands`。

## ペアリングフロー（統一）

- クライアントが未認証で接続。
- Gateway がその `deviceId` の**ペアリングリクエスト**を作成。
- オペレータがプロンプトを受け取り、承認/拒否。
- Gateway が以下にバインドされたクレデンシャルを発行:
  - デバイス公開鍵
  - ロール
  - スコープ
  - 機能/コマンド
- クライアントがトークンを永続化し、認証済みで再接続。

## デバイスバウンド認証（ベアラートークンリプレーを回避）

推奨: デバイスキーペア。

- デバイスが一度キーペアを生成。
- `deviceId = fingerprint(publicKey)`。
- Gateway がノンスを送信。デバイスが署名。Gateway が検証。
- トークンは文字列ではなく公開鍵（所有証明）に発行。

代替案:

- mTLS（クライアント証明書）: 最も強力だが、運用上の複雑さが増す。
- 短命のベアラートークンのみは一時的なフェーズとして（早期にローテーション + 失効）。

## サイレント承認（SSH ヒューリスティック）

弱いリンクを避けるために正確に定義します。以下のいずれかを優先:

- **ローカルのみ**: クライアントがループバック/Unix ソケット経由で接続する場合に自動ペアリング。
- **SSH 経由のチャレンジ**: Gateway がノンスを発行。クライアントが SSH でフェッチして証明。
- **物理的プレゼンスウィンドウ**: Gateway ホスト UI でのローカル承認後、短いウィンドウ（例: 10 分）内の自動ペアリングを許可。

自動承認は常にログに記録します。

# TLS のどこでも（開発 + 本番）

## 既存の Bridge TLS の再利用

現在の TLS ランタイム + フィンガープリントピニングを使用:

- `src/infra/bridge/server/tls.ts`
- `src/node-host/bridge-client.ts` のフィンガープリント検証ロジック

## WS への適用

- WS サーバーは同じ証明書/キー + フィンガープリントで TLS をサポート。
- WS クライアントはフィンガープリントをピン留めできる（オプション）。
- 発見はすべてのエンドポイントの TLS + フィンガープリントをアドバタイズ。
  - 発見はロケーターヒントのみ。信頼アンカーではない。

## 理由

- 機密性のために SSH/Tailscale への依存を減らす。
- リモートモバイル接続をデフォルトで安全にする。

# 承認の再設計（集中化）

## 現在

承認はノードホスト（Mac アプリノードランタイム）で行われます。プロンプトはノードが実行される場所に表示。

## 提案

承認は **Gateway ホスト**で行われ、UI はオペレータクライアントに配信。

### 新しいフロー

1. Gateway がエージェントから `system.run` インテントを受け取る。
2. Gateway が承認レコードを作成: `approval.requested`。
3. オペレータ UI がプロンプトを表示。
4. 承認の決定が Gateway に送信: `approval.resolve`。
5. 承認された場合、Gateway がノードコマンドを起動。
6. ノードが実行し、`invoke-res` を返す。

### 承認セマンティクス（ハードニング）

- すべてのオペレータにブロードキャスト。アクティブな UI のみがモーダルを表示（他はトースト）。
- 最初の解決が勝ち。Gateway は後続の解決をすでに確定済みとして拒否。
- デフォルトタイムアウト: N 秒後に拒否（例: 60 秒）。理由をログに記録。
- 解決には `operator.approvals` スコープが必要。

## 利点

- ユーザーがいる場所（Mac/スマートフォン）にプロンプトが表示。
- リモートノードの一貫した承認。
- ノードランタイムはヘッドレスのまま。UI 依存なし。

# ロールの明確化の例

## iPhone アプリ

- マイク、カメラ、ボイスチャット、位置情報、プッシュ・ツー・トーク用の**ノードロール**。
- ステータスとチャット表示用のオプション **operator.read**。
- 明示的に有効化された場合のみのオプション **operator.write/admin**。

## macOS アプリ

- デフォルトでオペレータロール（コントロール UI）。
- 「Mac ノード」が有効な場合はノードロール（system.run、スクリーン、カメラ）。
- 両方の接続に同じ deviceId → マージされた UI エントリ。

## CLI

- 常にオペレータロール。
- サブコマンドによって派生したスコープ:
  - `status`、`logs` → read
  - `agent`、`message` → write
  - `config`、`channels` → admin
  - 承認 + ペアリング → `operator.approvals` / `operator.pairing`

# アイデンティティ + スラッグ

## 安定した ID

認証に必須。変更されません。
推奨:

- キーペアフィンガープリント（公開鍵ハッシュ）。

## かわいいスラッグ（ロブスターテーマ）

人間用ラベルのみ。

- 例: `scarlet-claw`、`saltwave`、`mantis-pinch`。
- Gateway レジストリに保存。編集可能。
- 衝突処理: `-2`、`-3`。

## UI グループ化

ロール間の同じ `deviceId` → 単一の「インスタンス」行:

- バッジ: `operator`、`node`。
- 機能 + 最終確認時刻を表示。

# 移行戦略

## フェーズ 0: ドキュメント化 + 整合

- このドキュメントを公開。
- すべてのプロトコル呼び出し + 承認フローを棚卸し。

## フェーズ 1: WS にロール/スコープを追加

- `connect` パラメータに `role`、`scope`、`deviceId` を追加。
- ノードロールのアローリストゲートを追加。

## フェーズ 2: Bridge 互換性

- Bridge を引き続き実行。
- WS ノードサポートを並行して追加。
- コンフィグフラグの背後に機能をゲート。

## フェーズ 3: 中央承認

- WS に承認リクエスト + 解決イベントを追加。
- プロンプト + レスポンスのために Mac アプリ UI を更新。
- ノードランタイムが UI へのプロンプトを停止。

## フェーズ 4: TLS 統一

- Bridge TLS ランタイムを使用して WS の TLS コンフィグを追加。
- クライアントにピニングを追加。

## フェーズ 5: Bridge の廃止

- iOS/Android/Mac ノードを WS に移行。
- Bridge をフォールバックとして維持。安定したら削除。

## フェーズ 6: デバイスバウンド認証

- すべての非ローカル接続にキーベースのアイデンティティを必須に。
- 失効 + ローテーション UI を追加。

# セキュリティメモ

- ロール/アローリストは Gateway 境界で強制。
- オペレータスコープなしでクライアントが「完全」API を取得しない。
- _すべての_接続にペアリングが必要。
- TLS + ピニングはモバイルの MITM リスクを低減。
- SSH サイレント承認は利便性のためのもの。依然として記録 + 失効可能。
- 発見は信頼アンカーではない。
- 機能クレームはプラットフォーム/タイプ別のサーバーアローリストで検証。

# ストリーミング + 大きなペイロード（ノードメディア）

WS コントロールプレーンは小さなメッセージには適していますが、ノードは以下も行います:

- カメラクリップ
- スクリーン録画
- オーディオストリーム

オプション:

1. WS バイナリフレーム + チャンキング + バックプレッシャールール。
2. 別のストリーミングエンドポイント（依然として TLS + 認証）。
3. メディアヘビーなコマンドのために Bridge を長く維持し、最後に移行。

ドリフトを避けるために実装前にいずれかを選択します。

# 機能 + コマンドポリシー

- ノードが報告した caps/commands は**クレーム**として扱われます。
- Gateway はプラットフォームごとのアローリストを強制。
- 新しいコマンドはオペレータの承認または明示的なアローリスト変更が必要。
- タイムスタンプ付きで変更を監査。

# 監査 + レート制限

- ログ: ペアリングリクエスト、承認/拒否、トークンの発行/ローテーション/失効。
- ペアリングスパムと承認プロンプトのレート制限。

# プロトコルの衛生

- 明示的なプロトコルバージョン + エラーコード。
- 再接続ルール + ハートビートポリシー。
- プレゼンス TTL と最終確認セマンティクス。

# 未解決の質問

1. 両方のロールを実行する単一デバイス: トークンモデル
   - ロール（ノード vs オペレータ）ごとに個別のトークンを推奨。
   - 同じ deviceId。異なるスコープ。より明確な失効。

2. オペレータスコープの細分性
   - read/write/admin + 承認 + ペアリング（最小限実行可能）。
   - 後でフィーチャーごとのスコープを検討。

3. トークンのローテーション + 失効 UX
   - ロール変更時に自動ローテーション。
   - deviceId + ロールで失効する UI。

4. 発見
   - 現在の Bonjour TXT を拡張して WS TLS フィンガープリント + ロールヒントを含める。
   - ロケーターヒントのみとして扱う。

5. クロスネットワーク承認
   - すべてのオペレータクライアントにブロードキャスト。アクティブ UI がモーダルを表示。
   - 最初のレスポンスが勝ち。Gateway がアトミシティを強制。

---

# まとめ（TL;DR）

- 現在: WS コントロールプレーン + Bridge ノードトランスポート。
- 課題: 承認 + 重複 + 2 つのスタック。
- 提案: 明示的なロール + スコープ、統一されたペアリング + TLS ピニング、Gateway ホストの承認、安定したデバイス ID + かわいいスラッグを持つ 1 つの WS プロトコル。
- 成果: よりシンプルな UX、より強固なセキュリティ、重複の削減、より良いモバイルルーティング。
