# ワークスペースメモリ v2（オフライン）: リサーチメモ

ターゲット: Clawd スタイルのワークスペース（`agents.defaults.workspace`、デフォルト `~/.openclaw/workspace`）。「メモリ」は 1 日 1 ファイルの Markdown ファイル（`memory/YYYY-MM-DD.md`）と少数の安定したファイル（例: `memory.md`、`SOUL.md`）として保存されます。

このドキュメントでは、Markdown を正規の人間が読める情報源として維持しつつ、派生インデックスを通じて**構造化されたリコール**（検索、エンティティサマリー、信頼度更新）を追加する**オフラインファースト**メモリアーキテクチャを提案します。

## なぜ変更するのか？

現在の設定（1 日 1 ファイル）は以下に優れています:

- 「追記専用」ジャーナリング
- 人間による編集
- git バックアップの耐久性 + 監査可能性
- 低摩擦のキャプチャ（「とにかく書き留める」）

以下については弱点があります:

- 高リコール検索（「X についてどんな決定をしたか？」「最後に Y を試みたのはいつ？」）
- 多数のファイルを再読することなく、エンティティ中心の回答（「Alice / The Castle / warelay について教えて」）
- 意見/好みの安定性（変化した時の根拠を含む）
- 時間制約（「2025 年 11 月中の状況は？」）と競合解決

## 設計目標

- **オフライン**: ネットワークなしで動作します。ラップトップ/Castle 上で実行可能。クラウド依存なし。
- **説明可能**: 取得したアイテムは帰属可能（ファイル + 場所）で、推論から分離可能です。
- **低セレモニー**: 日次ログは Markdown のままで、重厚なスキーマ作業なし。
- **インクリメンタル**: v1 は FTS のみで有用。セマンティック/ベクター検索とグラフはオプションのアップグレード。
- **エージェントフレンドリー**: 「トークンバジェット内でのリコール」を容易にします（小さな事実のバンドルを返す）。

## 北極星モデル（Hindsight × Letta）

融合する 2 つのピース:

1. **Letta/MemGPT スタイルのコントロールループ**

- 小さな「コア」を常にコンテキスト内に保持（ペルソナ + 主要ユーザー事実）
- それ以外はすべてコンテキスト外で、ツールを介して取得
- メモリへの書き込みは明示的なツール呼び出し（追加/置換/挿入）で永続化され、次のターンで再注入

2. **Hindsight スタイルのメモリ基盤**

- 観察されたもの vs 信じられているもの vs 要約されたものを分離
- retain/recall/reflect をサポート
- 証拠とともに進化できる信頼度付き意見
- エンティティ対応の取得 + 時間クエリ（完全な知識グラフなしでも）

## 提案するアーキテクチャ（Markdown を信頼できる情報源とした派生インデックス）

### 正規ストア（git フレンドリー）

`~/.openclaw/workspace` を正規の人間が読めるメモリとして維持します。

推奨するワークスペースレイアウト:

```
~/.openclaw/workspace/
  memory.md                    # 小: 耐久性のある事実 + 好み（コアに近い）
  memory/
    YYYY-MM-DD.md              # 日次ログ（追記; ナラティブ）
  bank/                        # 「型付き」メモリページ（安定した、レビュー可能）
    world.md                   # 世界に関する客観的事実
    experience.md              # エージェントが行ったこと（一人称）
    opinions.md                # 主観的好み/判断 + 信頼度 + 根拠ポインタ
    entities/
      Peter.md
      The-Castle.md
      warelay.md
      ...
```

メモ:

- **日次ログは日次ログのまま**。JSON に変換する必要はありません。
- `bank/` ファイルは**キュレート**されており、リフレクションジョブによって生成され、手動で編集することもできます。
- `memory.md` は「小さくてコアに近い」ままです: Clawd に毎セッション見せたいものです。

### 派生ストア（機械によるリコール）

ワークスペース配下に派生インデックスを追加します（必ずしも git 管理下でなくてもよい）:

```
~/.openclaw/workspace/.memory/index.sqlite
```

以下でバックアップします:

- 事実 + エンティティリンク + 意見メタデータのための SQLite スキーマ
- レキシカルリコール用の SQLite **FTS5**（高速、小型、オフライン）
- セマンティックリコール用のオプション埋め込みテーブル（引き続きオフライン）

インデックスは常に **Markdown から再構築可能**です。

## Retain / Recall / Reflect（運用ループ）

### Retain: 日次ログを「事実」に正規化する

Hindsight の重要なインサイト: **ナラティブで自己完結した事実**を保存し、小さなスニペットではなく。

`memory/YYYY-MM-DD.md` の実際のルール:

- 一日の終わりに（または途中で）、以下の条件を満たす 2〜5 箇条書きの `## Retain` セクションを追加します:
  - ナラティブ（クロスターンのコンテキストが保持されている）
  - 自己完結（後で単独でも意味をなす）
  - タイプ + エンティティメンションでタグ付け

例:

```
## Retain
- W @Peter: Currently in Marrakech (Nov 27–Dec 1, 2025) for Andy's birthday.
- B @warelay: I fixed the Baileys WS crash by wrapping connection.update handlers in try/catch (see memory/2025-11-27.md).
- O(c=0.95) @Peter: Prefers concise replies (&lt;1500 chars) on WhatsApp; long content goes into files.
```

最小限のパース:

- タイププレフィックス: `W`（世界）、`B`（経験/伝記）、`O`（意見）、`S`（観察/要約; 通常は生成）
- エンティティ: `@Peter`、`@warelay` など（スラッグは `bank/entities/*.md` にマッピング）
- 意見の信頼度: `O(c=0.0..1.0)` オプション

著者に考えさせたくない場合: リフレクトジョブがログの残りからこれらの箇条書きを推論できますが、明示的な `## Retain` セクションがあることが最も簡単な「品質レバー」です。

### Recall: 派生インデックスへのクエリ

リコールは以下をサポートする必要があります:

- **レキシカル**: 「正確な用語/名前/コマンドを検索」（FTS5）
- **エンティティ**: 「X について教えて」（エンティティページ + エンティティリンク事実）
- **時間**: 「11 月 27 日頃に何があったか」/「先週以降」
- **意見**: 「Peter は何を好むか？」（信頼度 + 根拠を含む）

返却フォーマットはエージェントフレンドリーで、出典を引用する必要があります:

- `kind`（`world|experience|opinion|observation`）
- `timestamp`（ソース日、または存在する場合は抽出された時間範囲）
- `entities`（`["Peter","warelay"]`）
- `content`（ナラティブ事実）
- `source`（`memory/2025-11-27.md#L12` など）

### Reflect: 安定したページの生成 + 信念の更新

リフレクションはスケジュールされたジョブ（日次またはハートビート `ultrathink`）で:

- 最近の事実から `bank/entities/*.md` を更新（エンティティサマリー）
- 強化/矛盾に基づいて `bank/opinions.md` の信頼度を更新
- オプションで `memory.md`（「コアに近い」耐久性のある事実）の編集を提案

意見の進化（シンプルで説明可能）:

- 各意見は以下を持ちます:
  - ステートメント
  - 信頼度 `c ∈ [0,1]`
  - last_updated
  - 根拠リンク（支持 + 矛盾する事実 ID）
- 新しい事実が届いたとき:
  - エンティティの重複 + 類似性で候補意見を検索（最初 FTS、後でエンベディング）
  - 小さなデルタで信頼度を更新。大きな変化には強力な矛盾 + 繰り返しの根拠が必要

## CLI 統合: スタンドアロン vs 深統合

推奨: **OpenClaw への深統合**、ただし分離可能なコアライブラリを維持します。

### なぜ OpenClaw に統合するのか？

- OpenClaw はすでに以下を知っています:
  - ワークスペースパス（`agents.defaults.workspace`）
  - セッションモデル + ハートビート
  - ログ + トラブルシューティングパターン
- エージェント自体がツールを呼び出せるようにしたい:
  - `openclaw memory recall "…" --k 25 --since 30d`
  - `openclaw memory reflect --since 7d`

### なぜライブラリを分離するのか？

- Gateway/ランタイムなしでメモリロジックをテスト可能に保つ
- 他のコンテキスト（ローカルスクリプト、将来のデスクトップアプリなど）から再利用

形状:
メモリツールは小さな CLI + ライブラリレイヤーとして意図されていますが、これは探索的なものです。

## 「S-Collide」/ SuCo: いつ使用するか（リサーチ）

「S-Collide」が **SuCo（Subspace Collision）** を指す場合: サブスペース内の学習/構造化された衝突を使用して、強力なリコール/レイテンシのトレードオフをターゲットにした ANN 検索アプローチです（論文: arXiv 2411.14754、2024）。

`~/.openclaw/workspace` に対する実際的な見解:

- **SuCo から始めない**。
- SQLite FTS +（オプション）シンプルなエンベディングから始める。ほとんどの UX 向上はすぐに得られます。
- SuCo/HNSW/ScaNN クラスのソリューションは以下の場合にのみ検討します:
  - コーパスが大きい（数万〜数十万チャンク）
  - ブルートフォースエンベディング検索が遅すぎる
  - リコール品質がレキシカル検索によって意味のある形でボトルネックになっている

オフラインフレンドリーな代替案（複雑さの昇順）:

- SQLite FTS5 + メタデータフィルター（ML なし）
- エンベディング + ブルートフォース（チャンク数が少ければ驚くほど遠くまで機能します）
- HNSW インデックス（一般的で堅牢。ライブラリバインディングが必要）
- SuCo（研究グレード。埋め込める堅牢な実装があれば魅力的）

未解決の質問:

- あなたのマシン（ラップトップ + デスクトップ）で「パーソナルアシスタントメモリ」に最適な**オフライン**エンベディングモデルは何か？
  - すでに Ollama があれば: ローカルモデルでエンベディング。そうでなければ、ツールチェーンに小さなエンベディングモデルを同梱します。

## 最小限の有用なパイロット

最小限だが依然として有用なバージョンが欲しい場合:

- `bank/` エンティティページと日次ログに `## Retain` セクションを追加します。
- 引用（パス + 行番号）付きのリコールに SQLite FTS を使用します。
- リコール品質またはスケールが求める場合のみエンベディングを追加します。
