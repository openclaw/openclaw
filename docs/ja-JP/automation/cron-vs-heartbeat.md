# Cron vs ハートビート: 使い分けガイド

ハートビートと Cronジョブはどちらもスケジュールに従ってタスクを実行できます。このガイドはユースケースに応じた適切なメカニズムの選択を助けます。

## クイック判断ガイド

| ユースケース                        | 推奨                  | 理由                                       |
| ----------------------------------- | --------------------- | ------------------------------------------ |
| 30分ごとに受信トレイを確認          | ハートビート          | 他のチェックとまとめて実行、コンテキスト対応 |
| 毎朝9時ちょうどに日次レポートを送信 | Cron（アイソレーテッド）| 正確なタイミングが必要                     |
| 今後のイベントのカレンダーを監視    | ハートビート          | 定期的な認識に自然にフィット               |
| 週次の詳細分析を実行                | Cron（アイソレーテッド）| スタンドアロンタスク、異なるモデルを使用可 |
| 20分後にリマインドする              | Cron（main、`--at`）  | ワンショットで精密なタイミング             |
| バックグラウンドのプロジェクトヘルスチェック | ハートビート | 既存のサイクルに便乗                       |

## ハートビート: 定期的な認識

ハートビートは**メインセッション**で一定間隔（デフォルト: 30分）に実行されます。エージェントが状況を確認し、重要なことを表面化するように設計されています。

### ハートビートを使うとき

- **複数の定期チェック**: 受信トレイ、カレンダー、天気、通知、プロジェクトステータスを確認する5つの別々の Cronジョブの代わりに、1つのハートビートでこれらすべてをまとめて処理できます。
- **コンテキスト対応の判断**: エージェントはメインセッションの完全なコンテキストを持つため、緊急なものとそうでないものを賢く判断できます。
- **会話の継続性**: ハートビートの実行は同じセッションを共有するため、エージェントは最近の会話を覚えており、自然にフォローアップできます。
- **低オーバーヘッドの監視**: 1つのハートビートで多くの小さなポーリングタスクを置き換えます。

### ハートビートの利点

- **複数のチェックをまとめる**: 1回のエージェントターンで受信トレイ、カレンダー、通知をまとめて確認できます。
- **API コールを削減**: 1回のハートビートは5つのアイソレーテッド Cronジョブよりも安価です。
- **コンテキスト対応**: エージェントはあなたが取り組んでいることを把握し、優先順位をつけられます。
- **スマートな抑制**: 注意が必要なことがない場合、エージェントは `HEARTBEAT_OK` と返信し、メッセージは配信されません。
- **自然なタイミング**: キューの負荷に基づいてわずかにずれますが、ほとんどの監視には問題ありません。

### ハートビートの例: HEARTBEAT.md チェックリスト

```md
# Heartbeat checklist

- Check email for urgent messages
- Review calendar for events in next 2 hours
- If a background task finished, summarize results
- If idle for 8+ hours, send a brief check-in
```

エージェントは各ハートビートでこれを読み、1回のターンですべてのアイテムを処理します。

### ハートビートの設定

```json5
{
  agents: {
    defaults: {
      heartbeat: {
        every: "30m", // 間隔
        target: "last", // 明示的なアラートデリバリーターゲット（デフォルトは "none"）
        activeHours: { start: "08:00", end: "22:00" }, // オプション
      },
    },
  },
}
```

完全な設定については [ハートビート](/gateway/heartbeat) を参照してください。

## Cron: 精密なスケジューリング

Cronジョブは正確な時刻に実行され、メインコンテキストに影響を与えずにアイソレーテッドセッションで実行できます。繰り返しのトップオブアワースケジュールは、0〜5分のウィンドウで決定論的なジョブごとのオフセットにより自動的に分散されます。

### Cron を使うとき

- **正確なタイミングが必要**: 「毎週月曜日の午前9時に送信する」（「9時頃」ではなく）。
- **スタンドアロンタスク**: 会話のコンテキストを必要としないタスク。
- **異なるモデル/シンキング**: より強力なモデルが正当化される重い分析。
- **ワンショットリマインダー**: `--at` で「20分後にリマインドする」。
- **ノイジー/頻繁なタスク**: メインセッション履歴を散らかすタスク。
- **外部トリガー**: エージェントが他にアクティブかどうかに関わらず実行すべきタスク。

### Cron の利点

- **精密なタイミング**: タイムゾーンサポート付きの5フィールドまたは6フィールド（秒）の cron 式。
- **組み込みの負荷分散**: 繰り返しのトップオブアワースケジュールはデフォルトで最大5分ずらされます。
- **ジョブごとの制御**: `--stagger <duration>` でずらしをオーバーライドするか、`--exact` で正確なタイミングを強制します。
- **セッション分離**: メイン履歴を汚染せずに `cron:<jobId>` で実行します。
- **モデルオーバーライド**: ジョブごとに安価またはより強力なモデルを使用します。
- **デリバリー制御**: アイソレーテッドジョブはデフォルトで `announce`（サマリー）。必要に応じて `none` を選択します。
- **即時デリバリー**: アナウンスモードはハートビートを待たずに直接投稿します。
- **エージェントコンテキス不要**: メインセッションがアイドルまたはコンパクトでも実行されます。
- **ワンショットサポート**: 精密な将来のタイムスタンプに `--at`。

### Cron の例: 毎日の朝のブリーフィング

```bash
openclaw cron add \
  --name "Morning briefing" \
  --cron "0 7 * * *" \
  --tz "America/New_York" \
  --session isolated \
  --message "Generate today's briefing: weather, calendar, top emails, news summary." \
  --model opus \
  --announce \
  --channel whatsapp \
  --to "+15551234567"
```

これはニューヨーク時間の正確に午前7時に実行され、品質のために Opus を使用し、WhatsApp に直接サマリーをアナウンスします。

### Cron の例: ワンショットリマインダー

```bash
openclaw cron add \
  --name "Meeting reminder" \
  --at "20m" \
  --session main \
  --system-event "Reminder: standup meeting starts in 10 minutes." \
  --wake now \
  --delete-after-run
```

完全な CLI リファレンスは [Cronジョブ](/automation/cron-jobs) を参照してください。

## 判断フローチャート

```
タスクは正確な時刻に実行する必要がありますか?
  YES -> Cron を使用
  NO  -> 続けて...

タスクはメインセッションから分離する必要がありますか?
  YES -> Cron（アイソレーテッド）を使用
  NO  -> 続けて...

このタスクは他の定期チェックとまとめられますか?
  YES -> ハートビートを使用（HEARTBEAT.md に追加）
  NO  -> Cron を使用

これはワンショットリマインダーですか?
  YES -> --at で Cron を使用
  NO  -> 続けて...

異なるモデルまたはシンキングレベルが必要ですか?
  YES -> --model/--thinking で Cron（アイソレーテッド）を使用
  NO  -> ハートビートを使用
```

## 両方を組み合わせる

最も効率的なセットアップは**両方**を使用します:

1. **ハートビート**は30分ごとに1回のまとめたターンで定常的な監視（受信トレイ、カレンダー、通知）を処理します。
2. **Cron**は精密なスケジュール（日次レポート、週次レビュー）とワンショットリマインダーを処理します。

### 例: 効率的な自動化セットアップ

**HEARTBEAT.md**（30分ごとに確認）:

```md
# Heartbeat checklist

- Scan inbox for urgent emails
- Check calendar for events in next 2h
- Review any pending tasks
- Light check-in if quiet for 8+ hours
```

**Cronジョブ**（精密なタイミング）:

```bash
# 毎朝7時の朝のブリーフィング
openclaw cron add --name "Morning brief" --cron "0 7 * * *" --session isolated --message "..." --announce

# 毎週月曜日午前9時のプロジェクトレビュー
openclaw cron add --name "Weekly review" --cron "0 9 * * 1" --session isolated --message "..." --model opus

# ワンショットリマインダー
openclaw cron add --name "Call back" --at "2h" --session main --system-event "Call back the client" --wake now
```

## Lobster: 承認付きの決定論的ワークフロー

Lobster は決定論的な実行と明示的な承認が必要な**マルチステップツールパイプライン**のワークフローランタイムです。タスクが1回のエージェントターン以上で、人間のチェックポイントを持つ再開可能なワークフローが必要な場合に使用してください。

### Lobster が適しているとき

- **マルチステップの自動化**: 1回限りのプロンプトではなく、固定されたツールコールのパイプラインが必要な場合。
- **承認ゲート**: 副作用を承認するまで一時停止し、その後再開する場合。
- **再開可能な実行**: 以前のステップを再実行せずに一時停止したワークフローを続ける場合。

### ハートビートと Cron との組み合わせ方

- **ハートビート/Cron** は実行の_タイミング_を決定します。
- **Lobster** は実行が開始されると_どのステップ_が行われるかを定義します。

スケジュールされたワークフローには、cron またはハートビートを使用して Lobster を呼び出すエージェントターンをトリガーします。アドホックなワークフローには、Lobster を直接呼び出します。

### 操作上の注意（コードより）

- Lobster はツールモードで**ローカルサブプロセス**（`lobster` CLI）として実行され、**JSON エンベロープ**を返します。
- ツールが `needs_approval` を返した場合、`resumeToken` と `approve` フラグで再開します。
- ツールは**オプションのプラグイン**です。`tools.alsoAllow: ["lobster"]` で追加的に有効化してください（推奨）。
- Lobster は `lobster` CLI が `PATH` 上にあることを期待します。

完全な使用法と例については [Lobster](/tools/lobster) を参照してください。

## メインセッション vs アイソレーテッドセッション

ハートビートと Cron はどちらもメインセッションとやり取りできますが、方法が異なります:

|         | ハートビート                   | Cron（main）              | Cron（アイソレーテッド）      |
| ------- | ------------------------------ | ------------------------- | ----------------------------- |
| セッション | メイン                      | メイン（システムイベント経由） | `cron:<jobId>`             |
| 履歴    | 共有                           | 共有                      | 実行ごとに新規               |
| コンテキスト | 完全                       | 完全                      | なし（クリーンな状態で開始） |
| モデル  | メインセッションモデル         | メインセッションモデル    | オーバーライド可能           |
| 出力    | `HEARTBEAT_OK` でなければ配信 | ハートビートプロンプト + イベント | アナウンスサマリー（デフォルト）|

### メインセッション Cron を使うとき

次のような場合に `--session main` と `--system-event` を使用します:

- リマインダー/イベントをメインセッションコンテキストに表示したい場合
- エージェントに完全なコンテキストで次のハートビート時に処理させたい場合
- 別のアイソレーテッド実行が不要な場合

```bash
openclaw cron add \
  --name "Check project" \
  --every "4h" \
  --session main \
  --system-event "Time for a project health check" \
  --wake now
```

### アイソレーテッド Cron を使うとき

次のような場合に `--session isolated` を使用します:

- 以前のコンテキストなしのクリーンな状態が必要な場合
- 異なるモデルまたはシンキング設定が必要な場合
- チャンネルに直接サマリーをアナウンスしたい場合
- メインセッションを散らかさない履歴が必要な場合

```bash
openclaw cron add \
  --name "Deep analysis" \
  --cron "0 6 * * 0" \
  --session isolated \
  --message "Weekly codebase analysis..." \
  --model opus \
  --thinking high \
  --announce
```

## コスト考慮事項

| メカニズム          | コストプロファイル                                              |
| ------------------- | --------------------------------------------------------------- |
| ハートビート        | N分ごとに1ターン。HEARTBEAT.md のサイズに比例                  |
| Cron（main）        | 次のハートビートにイベントを追加（アイソレーテッドターンなし） |
| Cron（アイソレーテッド）| ジョブごとに完全なエージェントターン。安価なモデルを使用可   |

**ヒント**:

- トークンのオーバーヘッドを最小化するために `HEARTBEAT.md` を小さく保つ。
- 複数の Cronジョブの代わりに類似のチェックをハートビートにまとめる。
- 内部処理のみが必要な場合はハートビートに `target: "none"` を使用する。
- 定期タスクには安価なモデルでアイソレーテッド Cron を使用する。
