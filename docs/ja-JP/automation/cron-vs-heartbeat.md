# Cronとハートビート: 使い分けガイド

ハートビートとcronジョブはどちらもスケジュールに基づいてタスクを実行できます。このガイドでは、ユースケースに応じた適切な仕組みの選び方を説明します。

重要な違いは以下の通りです:

- **ハートビート**はスケジュールされた**メインセッションのターン**です — タスクレコードは作成されません。
- **Cron（main）**はスケジュールされた**メインセッションへのシステムイベント**です — `silent`通知ポリシーのタスクレコードが作成されます。
- **Cron（isolated）**はスケジュールされた**バックグラウンド実行**です — `openclaw tasks`で追跡されるタスクレコードが作成されます。

すべてのcronジョブ実行（mainおよびisolated）は[タスクレコード](/automation/tasks)を作成します。ハートビートのターンでは作成されません。メインセッションのcronタスクはデフォルトで`silent`通知ポリシーを使用するため、通知は生成されません。

## クイック判断ガイド

| ユースケース                         | 推奨                | 理由                                     |
| ------------------------------------ | ------------------- | ---------------------------------------- |
| 30分ごとに受信トレイをチェック       | ハートビート        | 他のチェックとバッチ処理でき、コンテキスト認識可能 |
| 毎朝9時に日次レポートを送信         | Cron（isolated）    | 正確なタイミングが必要                   |
| カレンダーの今後のイベントを監視     | ハートビート        | 定期的な状況把握に自然にフィット         |
| 週次の詳細分析を実行                 | Cron（isolated）    | 独立したタスクで、別のモデルを使用可能   |
| 20分後にリマインド                   | Cron（main、`--at`）| 正確なタイミングのワンショット           |
| バックグラウンドのプロジェクトヘルスチェック | ハートビート  | 既存のサイクルに便乗可能                 |

## ハートビート: 定期的な状況把握

ハートビートは**メインセッション**内で一定間隔（デフォルト: 30分）で実行されます。エージェントが状況を確認し、重要なことがあれば報告するために設計されています。

### ハートビートを使うべき場合

- **複数の定期チェック**: 受信トレイ、カレンダー、天気、通知、プロジェクトステータスをそれぞれ別のcronジョブで確認する代わりに、1回のハートビートでこれらすべてをバッチ処理できます。
- **コンテキスト認識による判断**: エージェントはメインセッションの完全なコンテキストを持っているため、何が緊急で何が後回しにできるかを賢く判断できます。
- **会話の継続性**: ハートビートは同じセッションで実行されるため、エージェントは最近の会話を記憶し、自然にフォローアップできます。
- **低オーバーヘッドの監視**: 1つのハートビートで多くの小さなポーリングタスクを置き換えます。

### ハートビートの利点

- **複数チェックのバッチ処理**: 1回のエージェントターンで受信トレイ、カレンダー、通知をまとめて確認できます。
- **API呼び出しの削減**: 1回のハートビートは5つのisolated cronジョブよりコストが低くなります。
- **コンテキスト認識**: エージェントはあなたが何に取り組んでいるかを知っており、それに応じて優先順位を付けられます。
- **スマートな抑制**: 注意が必要なことがなければ、エージェントは`HEARTBEAT_OK`を返し、メッセージは配信されません。
- **自然なタイミング**: キューの負荷に応じてわずかにずれますが、ほとんどの監視には問題ありません。
- **タスクレコードなし**: ハートビートのターンはメインセッション履歴に残ります（[バックグラウンドタスク](/automation/tasks)を参照）。

### ハートビートの例: HEARTBEAT.mdチェックリスト

```md
# Heartbeat checklist

- Check email for urgent messages
- Review calendar for events in next 2 hours
- If a background task finished, summarize results
- If idle for 8+ hours, send a brief check-in
```

エージェントはハートビートごとにこれを読み取り、すべての項目を1回のターンで処理します。

### ハートビートの設定

```json5
{
  agents: {
    defaults: {
      heartbeat: {
        every: "30m", // interval
        target: "last", // explicit alert delivery target (default is "none")
        activeHours: { start: "08:00", end: "22:00" }, // optional
      },
    },
  },
}
```

完全な設定については[ハートビート](/gateway/heartbeat)を参照してください。

## Cron: 正確なスケジューリング

cronジョブは正確な時刻に実行され、メインコンテキストに影響を与えずにisolatedセッションで実行できます。
定期的な毎時0分のスケジュールは、ジョブごとの決定論的オフセットにより0〜5分の範囲で自動的に分散されます。

### cronを使うべき場合

- **正確なタイミングが必要**: 「毎週月曜日の午前9:00にこれを送信」（「9時頃」ではなく）。
- **独立したタスク**: 会話のコンテキストを必要としないタスク。
- **異なるモデル/思考レベル**: より強力なモデルを使用する重い分析。
- **ワンショットリマインダー**: `--at`を使った「20分後にリマインド」。
- **頻繁な/ノイズの多いタスク**: メインセッションの履歴を散らかすタスク。
- **外部トリガー**: エージェントがアクティブかどうかに関係なく独立して実行すべきタスク。

### cronの利点

- **正確なタイミング**: タイムゾーンサポート付きの5フィールドまたは6フィールド（秒）cron式。
- **組み込みの負荷分散**: 定期的な毎時0分のスケジュールはデフォルトで最大5分まで自動的に分散されます。
- **ジョブごとの制御**: `--stagger <duration>`で分散をオーバーライドするか、`--exact`で正確なタイミングを強制できます。
- **セッション分離**: メイン履歴を汚さず`cron:<jobId>`で実行されます。
- **モデルオーバーライド**: ジョブごとにより安価またはより強力なモデルを使用できます。
- **配信制御**: isolatedジョブはデフォルトで`announce`（要約）です。必要に応じて`none`を選択できます。
- **即時配信**: announceモードはハートビートを待たずに直接投稿します。
- **エージェントコンテキスト不要**: メインセッションがアイドルまたはコンパクト化されていても実行されます。
- **ワンショットサポート**: 正確な未来のタイムスタンプ用の`--at`。
- **タスク追跡**: isolatedジョブは`openclaw tasks`および`openclaw tasks audit`で確認できる[バックグラウンドタスク](/automation/tasks)レコードを作成します。

### cronの例: 毎朝のブリーフィング

```bash
openclaw cron add \
  --name "Morning briefing" \
  --cron "0 7 * * *" \
  --tz "America/New_York" \
  --session isolated \
  --message "Generate today's briefing: weather, calendar, top emails, news summary." \
  --model opus \
  --announce \
  --channel whatsapp \
  --to "+15551234567"
```

これはニューヨーク時間の午前7:00ちょうどに実行され、品質のためにOpusを使用し、WhatsAppに直接要約をアナウンスします。

### cronの例: ワンショットリマインダー

```bash
openclaw cron add \
  --name "Meeting reminder" \
  --at "20m" \
  --session main \
  --system-event "Reminder: standup meeting starts in 10 minutes." \
  --wake now \
  --delete-after-run
```

完全なCLIリファレンスについては[cronジョブ](/automation/cron-jobs)を参照してください。

## 判断フローチャート

```
タスクは正確な時刻に実行する必要がありますか？
  はい -> cronを使用
  いいえ -> 続行...

タスクはメインセッションからの分離が必要ですか？
  はい -> Cron（isolated）を使用
  いいえ -> 続行...

このタスクは他の定期チェックとバッチ処理できますか？
  はい -> ハートビートを使用（HEARTBEAT.mdに追加）
  いいえ -> cronを使用

ワンショットリマインダーですか？
  はい -> --at付きのcronを使用
  いいえ -> 続行...

異なるモデルや思考レベルが必要ですか？
  はい -> --model/--thinking付きのCron（isolated）を使用
  いいえ -> ハートビートを使用
```

## 両方の組み合わせ

最も効率的な設定は**両方**を使うことです:

1. **ハートビート**はルーチンの監視（受信トレイ、カレンダー、通知）を30分ごとに1回のバッチターンで処理します。
2. **Cron**は正確なスケジュール（日次レポート、週次レビュー）とワンショットリマインダーを処理します。

### 例: 効率的な自動化設定

**HEARTBEAT.md**（30分ごとにチェック）:

```md
# Heartbeat checklist

- Scan inbox for urgent emails
- Check calendar for events in next 2h
- Review any pending tasks
- Light check-in if quiet for 8+ hours
```

**cronジョブ**（正確なタイミング）:

```bash
# Daily morning briefing at 7am
openclaw cron add --name "Morning brief" --cron "0 7 * * *" --session isolated --message "..." --announce

# Weekly project review on Mondays at 9am
openclaw cron add --name "Weekly review" --cron "0 9 * * 1" --session isolated --message "..." --model opus

# One-shot reminder
openclaw cron add --name "Call back" --at "2h" --session main --system-event "Call back the client" --wake now
```

## Lobster: 承認付きの決定論的ワークフロー

Lobsterは、決定論的な実行と明示的な承認を必要とする**マルチステップツールパイプライン**のためのワークフローランタイムです。
タスクが単一のエージェントターン以上のものであり、ヒューマンチェックポイント付きの再開可能なワークフローが必要な場合に使用します。

### Lobsterが適している場合

- **マルチステップ自動化**: 一度きりのプロンプトではなく、固定されたツール呼び出しパイプラインが必要な場合。
- **承認ゲート**: 副作用は承認されるまで一時停止し、承認後に再開する必要がある場合。
- **再開可能な実行**: 以前のステップを再実行せずに一時停止したワークフローを続行する場合。

### ハートビートやcronとの組み合わせ

- **ハートビート/cron**は実行の_タイミング_を決定します。
- **Lobster**は実行開始後に_どのステップ_を実行するかを定義します。

スケジュールされたワークフローの場合、cronまたはハートビートを使用してLobsterを呼び出すエージェントターンをトリガーします。
アドホックなワークフローの場合、Lobsterを直接呼び出します。

### 運用上の注意（コードより）

- Lobsterは**ローカルサブプロセス**（`lobster` CLI）としてツールモードで実行され、**JSONエンベロープ**を返します。
- ツールが`needs_approval`を返した場合、`resumeToken`と`approve`フラグを使って再開します。
- このツールは**オプションのプラグイン**です。`tools.alsoAllow: ["lobster"]`で追加的に有効化してください（推奨）。
- Lobsterは`lobster` CLIが`PATH`上で利用可能であることを期待します。

完全な使い方と例については[Lobster](/tools/lobster)を参照してください。

## メインセッションとisolatedセッション

ハートビートとcronはどちらもメインセッションとやり取りできますが、方法が異なります:

|                            | ハートビート                    | Cron（main）             | Cron（isolated）                                |
| -------------------------- | ------------------------------- | ------------------------ | ----------------------------------------------- |
| セッション                 | メイン                          | メイン（システムイベント経由） | `cron:<jobId>`またはカスタムセッション           |
| 履歴                       | 共有                            | 共有                     | 実行ごとに新規（isolated）/ 永続（カスタム）    |
| コンテキスト               | 完全                            | 完全                     | なし（isolated）/ 累積（カスタム）              |
| モデル                     | メインセッションのモデル        | メインセッションのモデル | オーバーライド可能                              |
| 出力                       | `HEARTBEAT_OK`でなければ配信    | ハートビートプロンプト＋イベント | アナウンス要約（デフォルト）                    |
| [タスク](/automation/tasks) | タスクレコードなし              | タスクレコード（silent） | タスクレコード（`openclaw tasks`で確認可能）     |

### メインセッションcronを使うべき場合

以下を望む場合は`--session main`と`--system-event`を使用します:

- リマインダー/イベントをメインセッションのコンテキストに表示させたい
- エージェントが次のハートビートで完全なコンテキストを持って処理するようにしたい
- 別のisolated実行を行わない

```bash
openclaw cron add \
  --name "Check project" \
  --every "4h" \
  --session main \
  --system-event "Time for a project health check" \
  --wake now
```

### isolated cronを使うべき場合

以下を望む場合は`--session isolated`を使用します:

- 事前のコンテキストなしのクリーンな状態
- 異なるモデルや思考設定
- チャネルへの直接アナウンス要約
- メインセッションを散らかさない履歴

```bash
openclaw cron add \
  --name "Deep analysis" \
  --cron "0 6 * * 0" \
  --session isolated \
  --message "Weekly codebase analysis..." \
  --model opus \
  --thinking high \
  --announce
```

## コストに関する考慮事項

| 仕組み          | コストプロファイル                                      |
| --------------- | ------------------------------------------------------- |
| ハートビート    | N分ごとに1ターン。HEARTBEAT.mdのサイズに比例            |
| Cron（main）    | 次のハートビートにイベントを追加（isolatedターンなし）   |
| Cron（isolated）| ジョブごとに完全なエージェントターン。より安価なモデルを使用可能 |

**ヒント**:

- トークンオーバーヘッドを最小化するため、`HEARTBEAT.md`は小さく保ちましょう。
- 複数のcronジョブではなく、類似のチェックをハートビートにバッチ処理しましょう。
- 内部処理のみが必要な場合は、ハートビートで`target: "none"`を使用しましょう。
- ルーチンタスクにはより安価なモデルを使ったisolated cronを使用しましょう。
