# macOSパーミッション（TCC）

macOSのパーミッション付与は脆弱である。TCCはパーミッション付与をアプリのコード署名、バンドルID、ディスク上のパスに関連付ける。これらのいずれかが変更されると、macOSはアプリを新しいものとして扱い、プロンプトが消えたり非表示になったりする可能性がある。

## 安定したパーミッションの要件

- 同一パス：固定の場所からアプリを実行する（OpenClawの場合、`dist/OpenClaw.app`）。
- 同一バンドルID：バンドルIDを変更すると新しいパーミッションIDが作成される。
- 署名済みアプリ：未署名またはアドホック署名のビルドではパーミッションが永続化されない。
- 一貫した署名：リビルド間で署名が安定するように、実際のApple DevelopmentまたはDeveloper ID証明書を使用する。

アドホック署名はビルドごとに新しいIDを生成する。macOSは以前の付与を忘れ、古いエントリがクリアされるまでプロンプトが完全に表示されなくなる可能性がある。

## プロンプトが表示されなくなった場合の復旧チェックリスト

1. アプリを終了する。
2. システム設定 → プライバシーとセキュリティでアプリのエントリを削除する。
3. 同じパスからアプリを再起動し、パーミッションを再付与する。
4. それでもプロンプトが表示されない場合は、`tccutil`でTCCエントリをリセットして再試行する。
5. 一部のパーミッションはmacOSの完全な再起動後にのみ再表示される。

リセットの例（バンドルIDは必要に応じて置き換えること）：

```bash
sudo tccutil reset Accessibility ai.openclaw.mac
sudo tccutil reset ScreenCapture ai.openclaw.mac
sudo tccutil reset AppleEvents
```

## ファイルとフォルダのパーミッション（デスクトップ/書類/ダウンロード）

macOSはターミナル/バックグラウンドプロセスに対してデスクトップ、書類、ダウンロードへのアクセスもゲートする場合がある。ファイルの読み取りやディレクトリの一覧表示がハングする場合は、ファイル操作を実行する同じプロセスコンテキスト（例：Terminal/iTerm、LaunchAgentで起動されたアプリ、SSHプロセス）にアクセスを許可する。

回避策：フォルダごとの許可を避けたい場合は、ファイルをOpenClawのワークスペース（`~/.openclaw/workspace`）に移動する。

パーミッションをテストする場合は、常に実際の証明書で署名すること。アドホックビルドは、パーミッションが不要な簡単なローカル実行にのみ許容される。
