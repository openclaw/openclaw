ほとんどの操作はゲートウェイ（`openclaw gateway`）を経由します。ゲートウェイは、チャネル接続と WebSocket コントロールプレーンを管理する単一の長時間実行プロセスです。

## 基本ルール

- ホストごとに 1 つのゲートウェイを推奨します。WhatsApp Web セッションを所有できるのはゲートウェイのみです。レスキューボットや厳密な分離が必要な場合は、プロファイルとポートを分離して複数のゲートウェイを実行してください。[複数のゲートウェイ](/gateway/multiple-gateways)を参照してください。
- ループバック優先：ゲートウェイの WS はデフォルトで `ws://127.0.0.1:18789` です。ウィザードはループバックでもデフォルトでゲートウェイトークンを生成します。Tailnet アクセスの場合は、`openclaw gateway --bind tailnet --token ...` を実行してください。非ループバックバインドにはトークンが必要です。
- ノードは必要に応じて LAN、Tailnet、または SSH 経由でゲートウェイ WS に接続します。レガシー TCP ブリッジは非推奨です。
- キャンバスホストはゲートウェイ HTTP サーバーによって、ゲートウェイと**同じポート**（デフォルト `18789`）で提供されます：
  - `/__openclaw__/canvas/`
  - `/__openclaw__/a2ui/`
    `gateway.auth` が設定されており、ゲートウェイがループバック以外にバインドされている場合、これらのルートはゲートウェイ認証によって保護されます。ノードクライアントは、アクティブな WS セッションに紐付けられたノードスコープのケイパビリティ URL を使用します。[ゲートウェイ設定](/gateway/configuration)（`canvasHost`、`gateway`）を参照してください。
- リモート利用は通常、SSH トンネルまたは Tailnet VPN を使用します。[リモートアクセス](/gateway/remote)および[ディスカバリー](/gateway/discovery)を参照してください。
