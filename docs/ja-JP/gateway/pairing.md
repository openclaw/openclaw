# Gateway ゲートウェイ所有のペアリング（オプション B）

Gateway ゲートウェイ所有のペアリングでは、**Gateway ゲートウェイ**がどのノードが参加を許可されるかの真実のソースとなります。UI（macOS アプリ、将来のクライアント）は保留中のリクエストを承認または拒否するフロントエンドに過ぎません。

**重要:** WS ノードは `connect` 時に**デバイスペアリング**（ロール `node`）を使用します。
`node.pair.*` は別のペアリングストアであり、WS ハンドシェイクを**制御しません**。
`node.pair.*` を明示的に呼び出すクライアントのみがこのフローを使用します。

## 概念

- **保留中リクエスト**: ノードが参加を要求しており、承認が必要な状態。
- **ペアリング済みノード**: 発行された認証トークンを持つ承認済みノード。
- **トランスポート**: Gateway ゲートウェイの WS エンドポイントはリクエストを転送しますが、メンバーシップを決定しません。（レガシーの TCP ブリッジサポートは非推奨/削除済み。）

## ペアリングの仕組み

1. ノードが Gateway ゲートウェイ WS に接続してペアリングをリクエストします。
2. Gateway ゲートウェイが**保留中リクエスト**を保存して `node.pair.requested` を発行します。
3. リクエストを承認または拒否します（CLI または UI）。
4. 承認時、Gateway ゲートウェイは**新しいトークン**を発行します（トークンは再ペアリング時にローテーションされます）。
5. ノードはトークンを使用して再接続し、「ペアリング済み」になります。

保留中リクエストは**5分後**に自動的に期限切れになります。

## CLI ワークフロー（ヘッドレスフレンドリー）

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` はペアリング済み/接続済みノードとそのケーパビリティを表示します。

## API サーフェス（Gateway ゲートウェイプロトコル）

イベント：

- `node.pair.requested` — 新しい保留中リクエストが作成されたときに発行。
- `node.pair.resolved` — リクエストが承認/拒否/期限切れになったときに発行。

メソッド：

- `node.pair.request` — 保留中リクエストを作成または再利用。
- `node.pair.list` — 保留中 + ペアリング済みノードを一覧表示。
- `node.pair.approve` — 保留中リクエストを承認（トークンを発行）。
- `node.pair.reject` — 保留中リクエストを拒否。
- `node.pair.verify` — `{ nodeId, token }` を検証。

注意事項：

- `node.pair.request` はノードごとに冪等です: 繰り返し呼び出すと同じ保留中リクエストが返されます。
- 同じ保留中ノードへの繰り返しリクエストは、ストアされたノードメタデータとオペレーターの可視性のための最新の許可リストで宣言されたコマンドスナップショットも更新します。
- 承認は**常に**新しいトークンを生成します; `node.pair.request` からトークンは返されません。
- リクエストは自動承認フローのヒントとして `silent: true` を含めることができます。

重要：

- ノードペアリングはトラスト/アイデンティティフローとトークン発行です。
- ノードごとのライブなノードコマンドサーフェスを**ピンしません**。
- ライブなノードコマンドは、Gateway ゲートウェイのグローバルなノードコマンドポリシー（`gateway.nodes.allowCommands` / `denyCommands`）が適用された後に、接続時にノードが宣言するものから来ます。
- ノードごとの `system.run` 許可/確認ポリシーは、ペアリングレコードではなくノード上の `exec.approvals.node.*` に存在します。

## ノードコマンドゲーティング（2026.3.31 以降）

<Warning>
**破壊的変更:** `2026.3.31` から、ノードペアリングが承認されるまでノードコマンドは無効になります。デバイスペアリングだけでは、宣言されたノードコマンドを公開するには不十分になりました。
</Warning>

ノードが初めて接続する際、ペアリングは自動的にリクエストされます。ペアリングリクエストが承認されるまで、そのノードからのすべての保留中ノードコマンドはフィルタリングされ実行されません。ペアリング承認によってトラストが確立されると、ノードの宣言されたコマンドは通常のコマンドポリシーに従って利用可能になります。

これが意味すること：

- 以前はデバイスペアリングだけでコマンドを公開していたノードは、ノードペアリングを完了させる必要があります。
- ペアリング承認前にキューに入れられたコマンドは延期されず、削除されます。

## ノードイベントのトラスト境界（2026.3.31 以降）

<Warning>
**破壊的変更:** ノード起源の実行は縮小されたトラストサーフェスに制限されます。
</Warning>

ノード起源のサマリーと関連するセッションイベントは、意図されたトラストサーフェスに制限されます。以前に広いホストやセッションのツールアクセスに依存していた通知駆動またはノードトリガーのフローは調整が必要になる場合があります。このハードニングにより、ノードイベントがノードのトラスト境界を超えたホストレベルのツールアクセスにエスカレーションできないことが保証されます。

## 自動承認（macOS アプリ）

macOS アプリは以下の条件が満たされた場合に**サイレント承認**を試みることができます：

- リクエストが `silent` とマークされている、かつ
- アプリが同じユーザーを使用して Gateway ゲートウェイホストへの SSH 接続を確認できる。

サイレント承認が失敗した場合は、通常の「承認/拒否」プロンプトにフォールバックします。

## ストレージ（ローカル、プライベート）

ペアリングステートは Gateway ゲートウェイのステートディレクトリ（デフォルト `~/.openclaw`）の下に保存されます：

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

`OPENCLAW_STATE_DIR` をオーバーライドすると、`nodes/` フォルダーも移動します。

セキュリティ上の注意：

- トークンは秘密情報です; `paired.json` を機密として扱ってください。
- トークンのローテーションには再承認が必要です（またはノードエントリを削除）。

## トランスポートの動作

- トランスポートは**ステートレス**です; メンバーシップを保存しません。
- Gateway ゲートウェイがオフラインかペアリングが無効になっている場合、ノードはペアリングできません。
- Gateway ゲートウェイがリモートモードの場合でも、ペアリングはリモート Gateway ゲートウェイのストアに対して行われます。
