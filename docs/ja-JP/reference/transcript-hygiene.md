# トランスクリプトハイジーン（プロバイダー固有の修正）

このドキュメントでは、実行前（モデルコンテキストの構築時）にトランスクリプトに適用される**プロバイダー固有の修正**について説明します。これらは厳格なプロバイダー要件を満たすために使用される**インメモリ**の調整です。これらのハイジーンステップはディスク上に保存された JSONL トランスクリプトを書き換え**ません**。ただし、別のセッションファイル修復パスが、セッションのロード前に不正な JSONL ファイルの無効な行を削除して書き換える場合があります。修復が発生した場合、元のファイルはセッションファイルの横にバックアップされます。

対象範囲:

- ツールコール ID のサニタイズ
- ツールコール入力のバリデーション
- ツール結果のペアリング修復
- ターンのバリデーション / 順序付け
- Thought シグネチャのクリーンアップ
- 画像ペイロードのサニタイズ
- ユーザー入力の出所タグ付け（セッション間ルーティングされたプロンプト向け）

トランスクリプトのストレージの詳細については、以下を参照してください:

- [/reference/session-management-compaction](/reference/session-management-compaction)

## 実行される場所

すべてのトランスクリプトハイジーンは埋め込みランナーに集約されています:

- ポリシー選択: `src/agents/transcript-policy.ts`
- サニタイズ / 修復の適用: `src/agents/pi-embedded-runner/google.ts` 内の `sanitizeSessionHistory`

ポリシーは `provider`、`modelApi`、`modelId` を使用して適用内容を決定します。

トランスクリプトハイジーンとは別に、セッションファイルはロード前に（必要に応じて）修復されます:

- `src/agents/session-file-repair.ts` 内の `repairSessionFileIfNeeded`
- `run/attempt.ts` および `compact.ts`（埋め込みランナー）から呼び出し

## グローバルルール: 画像サニタイズ

画像ペイロードは、サイズ制限によるプロバイダー側の拒否を防ぐため、常にサニタイズされます（大きすぎる base64 画像のダウンスケール / 再圧縮）。

これは、ビジョン対応モデルにおける画像由来のトークン圧迫の制御にも役立ちます。最大寸法を小さくすると一般的にトークン使用量が減少し、大きくすると詳細が維持されます。

実装:

- `src/agents/pi-embedded-helpers/images.ts` 内の `sanitizeSessionMessagesImages`
- `src/agents/tool-images.ts` 内の `sanitizeContentBlocksImages`
- 画像の最大辺は `agents.defaults.imageMaxDimensionPx` で設定可能（デフォルト: `1200`）。

---

## グローバルルール: 不正なツールコール

`input` と `arguments` の両方が欠落しているアシスタントのツールコールブロックは、モデルコンテキストの構築前に削除されます。これにより、部分的に永続化されたツールコール（例: レートリミット失敗後）によるプロバイダー拒否を防止します。

実装:

- `src/agents/session-transcript-repair.ts` 内の `sanitizeToolCallInputs`
- `src/agents/pi-embedded-runner/google.ts` 内の `sanitizeSessionHistory` で適用

---

## グローバルルール: セッション間の入力出所

エージェントが `sessions_send` を介して別のセッションにプロンプトを送信する場合（エージェント間のリプライ / アナウンスステップを含む）、OpenClaw は作成されたユーザーターンに以下を付与して永続化します:

- `message.provenance.kind = "inter_session"`

このメタデータはトランスクリプトの追記時に書き込まれ、ロール（`role: "user"` はプロバイダー互換性のため維持）は変更しません。トランスクリプトの読み取り側は、これを利用してルーティングされた内部プロンプトをエンドユーザーが作成した指示として扱うことを回避できます。

コンテキスト再構築時、OpenClaw はこれらのユーザーターンに短い `[Inter-session message]` マーカーをインメモリで先頭に付加し、モデルが外部エンドユーザーの指示と区別できるようにします。

## プロバイダーマトリクス（現在の動作）

**OpenAI / OpenAI Codex**

- 画像サニタイズのみ。
- 孤立した推論シグネチャ（後続のコンテンツブロックがない単独の推論アイテム）を OpenAI Responses/Codex トランスクリプトで削除。
- ツールコール ID のサニタイズなし。
- ツール結果のペアリング修復なし。
- ターンのバリデーションや並べ替えなし。
- 合成ツール結果なし。
- Thought シグネチャの除去なし。

**Google (Generative AI / Gemini CLI / Antigravity)**

- ツールコール ID のサニタイズ: 厳密な英数字。
- ツール結果のペアリング修復および合成ツール結果。
- ターンのバリデーション（Gemini スタイルのターン交代）。
- Google ターン順序の修正（履歴がアシスタントで始まる場合、小さなユーザーブートストラップを先頭に付加）。
- Antigravity Claude: thinking シグネチャの正規化、署名なしの thinking ブロックの削除。

**Anthropic / Minimax（Anthropic 互換）**

- ツール結果のペアリング修復および合成ツール結果。
- ターンのバリデーション（厳密な交代を満たすため、連続するユーザーターンをマージ）。

**Mistral（モデル ID ベースの検出を含む）**

- ツールコール ID のサニタイズ: strict9（英数字 9 文字）。

**OpenRouter Gemini**

- Thought シグネチャのクリーンアップ: 非 base64 の `thought_signature` 値を除去（base64 は維持）。

**その他すべて**

- 画像サニタイズのみ。

## 過去の動作（2026.1.22 以前）

2026.1.22 リリース以前、OpenClaw は複数のレイヤーのトランスクリプトハイジーンを適用していました:

- **トランスクリプトサニタイズ拡張**がすべてのコンテキスト構築時に実行され、以下が可能でした:
  - ツール use/result のペアリング修復。
  - ツールコール ID のサニタイズ（`_`/`-` を保持する非厳密モードを含む）。
- ランナーもプロバイダー固有のサニタイズを実行しており、作業が重複していました。
- プロバイダーポリシー外での追加の変更も行われていました:
  - 永続化前にアシスタントテキストから `<final>` タグを除去。
  - 空のアシスタントエラーターンの削除。
  - ツールコール後のアシスタントコンテンツのトリミング。

この複雑さはプロバイダー間のリグレッション（特に `openai-responses` の `call_id|fc_id` ペアリング）を引き起こしました。2026.1.22 のクリーンアップで拡張を削除し、ロジックをランナーに集約し、OpenAI については画像サニタイズ以外は**変更なし**としました。
