このページでは OpenClaw プラグインシステムの内部アーキテクチャについて説明します。

## パブリック機能モデル

機能は OpenClaw 内のパブリック**ネイティブプラグイン**モデルです。すべてのネイティブ OpenClaw プラグインは1つ以上の機能タイプに対して登録します：

| 機能                  | 登録メソッド                                  | プラグイン例                  |
| --------------------- | --------------------------------------------- | ------------------------- |
| テキスト推論          | `api.registerProvider(...)`                   | `openai`、`anthropic`     |
| CLI 推論バックエンド   | `api.registerCliBackend(...)`                 | `openai`、`anthropic`     |
| 音声                  | `api.registerSpeechProvider(...)`             | `elevenlabs`、`microsoft` |
| メディア理解          | `api.registerMediaUnderstandingProvider(...)` | `openai`、`google`        |
| 画像生成              | `api.registerImageGenerationProvider(...)`    | `openai`、`google`        |
| ウェブ検索            | `api.registerWebSearchProvider(...)`          | `google`                  |
| チャンネル / メッセージング | `api.registerChannel(...)`               | `msteams`、`matrix`       |

機能を登録せずにフック、ツール、またはサービスのみを提供するプラグインは**レガシーフックのみ**プラグインです。そのパターンは引き続き完全にサポートされています。

### 外部互換性の姿勢

- **既存の外部プラグイン**: フックベースのインテグレーションを機能させ続けます；これを互換性ベースラインとして扱います
- **新しいバンドル/ネイティブプラグイン**: ベンダー固有のリーチインや新しいフックのみの設計よりも明示的な機能登録を推奨します
- **機能登録を採用する外部プラグイン**: 許可されますが、ドキュメントでコントラクトが安定しているとマークされない限り、機能固有のヘルパーサーフェスは進化中として扱います

### プラグインの形状

OpenClaw は実際の登録動作（静的なメタデータだけでなく）に基づいて、ロードされたすべてのプラグインを形状に分類します：

- **plain-capability** -- 正確に1つの機能タイプを登録（例: `mistral` のようなプロバイダーのみのプラグイン）
- **hybrid-capability** -- 複数の機能タイプを登録（例: `openai` はテキスト推論、音声、メディア理解、画像生成を所有）
- **hook-only** -- フック（型付きまたはカスタム）のみを登録し、機能、ツール、コマンド、サービスなし
- **non-capability** -- ツール、コマンド、サービス、またはルートを登録するが機能なし

`openclaw plugins inspect <id>` を使用してプラグインの形状と機能の内訳を確認できます。

### 互換性シグナル

`openclaw doctor` または `openclaw plugins inspect <id>` を実行すると、以下のラベルのいずれかが表示される場合があります：

| シグナル                     | 意味                                                         |
| -------------------------- | ------------------------------------------------------------ |
| **config valid**           | 設定が正常に解析され、プラグインが解決される                   |
| **compatibility advisory** | プラグインがサポートされているが古いパターンを使用（例: `hook-only`） |
| **legacy warning**         | プラグインが非推奨の `before_agent_start` を使用              |
| **hard error**             | 設定が無効またはプラグインのロードに失敗                       |

## アーキテクチャ概要

OpenClaw のプラグインシステムには4つのレイヤーがあります：

1. **マニフェスト + ディスカバリー**
   OpenClaw は設定されたパス、ワークスペースルート、グローバル拡張ルート、バンドル拡張からプラグイン候補を検索します。
2. **有効化 + 検証**
   コアは、検出されたプラグインが有効化、無効化、ブロック、またはメモリなどの排他的スロット用に選択されるかどうかを決定します。
3. **ランタイムロード**
   ネイティブ OpenClaw プラグインは jiti 経由でインプロセスでロードされ、機能を中央レジストリに登録します。
4. **サーフェス消費**
   残りの OpenClaw はレジストリを読み取り、ツール、チャンネル、プロバイダー設定、フック、HTTP ルート、CLI コマンド、サービスを公開します。

重要な設計境界：

- ディスカバリー + 設定の検証は、プラグインコードを実行せずに**マニフェスト/スキーマメタデータ**から機能する必要があります
- ネイティブランタイムの動作はプラグインモジュールの `register(api)` パスから来ます

### チャンネルプラグインと共有メッセージツール

チャンネルプラグインは通常のチャットアクション用に別個の send/edit/react ツールを登録する必要はありません。OpenClaw はコアに1つの共有 `message` ツールを保持し、チャンネルプラグインはその背後にあるチャンネル固有のディスカバリーと実行を所有します。

現在の境界：

- コアは共有 `message` ツールホスト、プロンプト配線、セッション/スレッドのブックキーピング、実行ディスパッチを所有
- チャンネルプラグインはスコープされたアクションディスカバリー、機能ディスカバリー、チャンネル固有のスキーマフラグメントを所有

## 機能所有権モデル

OpenClaw はネイティブプラグインを無関係な統合の寄せ集めではなく、**会社**または**機能**の所有境界として扱います。

例：

- バンドルされた `openai` プラグインは OpenAI モデルプロバイダーの動作と OpenAI の音声 + メディア理解 + 画像生成の動作を所有
- バンドルされた `elevenlabs` プラグインは ElevenLabs の音声動作を所有
- `voice-call` プラグインは機能プラグインです：コールトランスポート、ツール、CLI、ルート、ランタイムを所有しますが、コア TTS/STT 機能を消費

重要な区別：

- **プラグイン** = 所有境界
- **機能** = 複数のプラグインが実装または消費できるコアコントラクト

### 機能レイヤリング

コードの配置を決定する際のメンタルモデル：

- **コア機能レイヤー**: 共有オーケストレーション、ポリシー、フォールバック、設定マージルール、配信セマンティクス、型付きコントラクト
- **ベンダープラグインレイヤー**: ベンダー固有の API、認証、モデルカタログ、音声合成、画像生成、将来のビデオバックエンド
- **チャンネル/機能プラグインレイヤー**: コア機能を消費してサーフェスに提示する Slack/Discord/voice-call などのインテグレーション

## コントラクトと強制

プラグイン API サーフェスは意図的に型付きされ、`OpenClawPluginApi` に集中化されています。

強制には2つのレイヤーがあります：

1. **ランタイム登録の強制**
   プラグインレジストリはプラグインのロード時に登録を検証します。
2. **コントラクトテスト**
   バンドルされたプラグインはテスト実行中にコントラクトレジストリにキャプチャされるため、OpenClaw は所有権を明示的にアサートできます。

## 実行モデル

ネイティブ OpenClaw プラグインは Gateway ゲートウェイと**インプロセス**で実行されます。サンドボックス化されていません。ロードされたネイティブプラグインはコアコードと同じプロセスレベルの信頼境界を持ちます。

非バンドルプラグインには許可リストと明示的なインストール/ロードパスを使用してください。ワークスペースプラグインは本番のデフォルトではなく開発時のコードとして扱ってください。

## ロードパイプライン

起動時、OpenClaw は概ねこれを行います：

1. プラグイン候補ルートを検出
2. ネイティブまたは互換バンドルマニフェストとパッケージメタデータを読み取り
3. 安全でない候補を拒否
4. プラグイン設定を正規化
5. 各候補の有効化を決定
6. jiti 経由で有効なネイティブモジュールをロード
7. ネイティブ `register(api)` フックを呼び出し、登録をプラグインレジストリに収集
8. コマンド/ランタイムサーフェスにレジストリを公開

<Note>
`activate` は `register` のレガシーエイリアスです — ローダーはどちらが存在するかを解決し（`def.register ?? def.activate`）、同じ時点で呼び出します。すべてのバンドルプラグインは `register` を使用します；新しいプラグインには `register` を推奨します。
</Note>

## レジストリモデル

ロードされたプラグインはランダムなコアグローバルを直接変更しません。それらは中央プラグインレジストリに登録します。

レジストリが追跡するもの：

- プラグインレコード（ID、ソース、オリジン、ステータス、診断）
- ツール
- レガシーフックと型付きフック
- チャンネル
- プロバイダー
- Gateway ゲートウェイ RPC ハンドラー
- HTTP ルート
- CLI レジストラー
- バックグラウンドサービス
- プラグイン所有のコマンド

コア機能はプラグインモジュールに直接話しかけるのではなく、そのレジストリから読み取ります。

## 会話バインディングコールバック

会話をバインドするプラグインは、承認が解決されたときに反応できます。

`api.onConversationBindingResolved(...)` を使用してバインドリクエストが承認または拒否された後にコールバックを受け取ります：

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        console.log(event.binding?.conversationId);
        return;
      }
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

## プロバイダーランタイムフック

プロバイダープラグインには現在2つのレイヤーがあります：

- マニフェストメタデータ: ランタイムロード前の安価な環境認証ルックアップ用の `providerAuthEnvVars`
- 設定時フック: `catalog` / レガシー `discovery`
- ランタイムフック: `resolveDynamicModel`、`prepareDynamicModel`、`normalizeResolvedModel`、`capabilities`、`prepareExtraParams`、`wrapStreamFn`、`formatApiKey`、`refreshOAuth`、`buildAuthDoctorHint`、その他

### フックの順序と使用法

| #   | フック                        | 何をするか                                                                               | 使用タイミング                                                                           |
| --- | ----------------------------- | ---------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| 1   | `catalog`                     | `models.json` 生成中にプロバイダー設定を `models.providers` に公開                        | プロバイダーがカタログまたはベース URL デフォルトを所有する場合                           |
| 2   | `resolveDynamicModel`         | まだローカルレジストリにないプロバイダー所有のモデル ID の同期フォールバック                 | プロバイダーが任意のアップストリームモデル ID を受け入れる場合                            |
| 3   | `prepareDynamicModel`         | 非同期ウォームアップ、その後 `resolveDynamicModel` が再度実行                              | プロバイダーが未知の ID を解決する前にネットワークメタデータが必要な場合                  |
| 4   | `normalizeResolvedModel`      | 組み込みランナーが解決されたモデルを使用する前の最終的な書き換え                            | プロバイダーがトランスポートの書き換えを必要とするがコアトランスポートを使用する場合        |
| 5   | `capabilities`                | 共有コアロジックで使用されるプロバイダー所有のトランスクリプト/ツールメタデータ              | プロバイダーがトランスクリプト/プロバイダーファミリーの癖を必要とする場合                  |
| 6   | `prepareExtraParams`          | 汎用ストリームオプションラッパーの前のリクエストパラメータ正規化                             | プロバイダーがデフォルトのリクエストパラメータまたはパープロバイダーのパラメータクリーンアップを必要とする場合 |
| 19  | `prepareRuntimeAuth`          | 推論の直前に設定されたクレデンシャルを実際のランタイムトークン/キーに交換                   | プロバイダーがトークン交換または短期間のリクエストクレデンシャルを必要とする場合            |

## ランタイムヘルパー

プラグインは `api.runtime` を通じて選択されたコアヘルパーにアクセスできます。TTS の場合：

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});
```

メディア理解のランタイムヘルパー：

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});
```

画像生成：

```ts
const result = await api.runtime.imageGeneration.generate({
  config: api.config,
  args: { prompt: "A friendly lobster mascot", size: "1024x1024" },
});
```

## Gateway ゲートウェイ HTTP ルート

プラグインは `api.registerHttpRoute(...)` で HTTP エンドポイントを公開できます。

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```
