# コマンドキュー（2026-01-16）

受信自動返信の実行（すべてのチャンネル）を小さなインプロセスキューで直列化して、複数のエージェント実行の衝突を防ぎながら、セッション間の安全な並列処理を可能にします。

## 理由

- 自動返信実行はコストが高く（LLM 呼び出し）、複数の受信メッセージが近接して届いた場合に衝突する可能性があります。
- 直列化により、共有リソース（セッションファイル、ログ、CLI の標準入力）の競合を避け、上流のレート制限の可能性を減らします。

## 動作の仕組み

- レーンを考慮した FIFO キューが設定可能な並行性の上限（未設定のレーンのデフォルトは 1; メインのデフォルトは 4、サブエージェントは 8）で各レーンを処理します。
- `runEmbeddedPiAgent` は**セッションキー**（レーン `session:<key>`）でエンキューして、セッションごとにアクティブな実行が 1 つだけになることを保証します。
- 各セッション実行は次に**グローバルレーン**（デフォルトは `main`）にキューイングされ、全体の並列処理が `agents.defaults.maxConcurrent` で上限設定されます。
- verbose ログが有効な場合、キューに入った実行が開始前に約 2 秒以上待機すると短い通知が発行されます。
- タイピングインジケーターはキューイング時に直ちに送信されます（チャンネルがサポートしている場合）。待機中もユーザーエクスペリエンスは変わりません。

## キューモード（チャンネルごと）

受信メッセージは現在の実行に転送したり、フォローアップターンを待機したり、またはその両方を行ったりできます。

- `steer`: 現在の実行に即座に注入します（次のツール境界後に保留中のツール呼び出しをキャンセルします）。ストリーミング中でない場合は followup にフォールバックします。
- `followup`: 現在の実行終了後の次のエージェントターン用にキューに入れます。
- `collect`: キューに入ったすべてのメッセージを**単一**のフォローアップターンに集約します（デフォルト）。メッセージが異なるチャンネル/スレッドを対象としている場合は、ルーティングを保持するために個別に処理されます。
- `steer-backlog`（`steer+backlog` とも）: 今すぐ転送**かつ**フォローアップターンのためにメッセージを保持します。
- `interrupt`（レガシー）: そのセッションのアクティブな実行を中断し、最新のメッセージを実行します。
- `queue`（レガシーエイリアス）: `steer` と同じ。

steer-backlog は転送された実行の後にフォローアップ応答を受け取れることを意味するため、ストリーミングサーフェスでは重複して見える場合があります。受信メッセージごとに 1 つの応答が必要な場合は `collect`/`steer` を使用してください。スタンドアロンコマンドとして `/queue collect` を送信（セッションごと）するか、`messages.queue.byChannel.discord: "collect"` を設定します。

デフォルト（設定が未指定の場合）:

- すべてのサーフェス → `collect`

`messages.queue` を使ってグローバルまたはチャンネルごとに設定します。

```json5
{
  messages: {
    queue: {
      mode: "collect",
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
      byChannel: { discord: "collect" },
    },
  },
}
```

## キューオプション

オプションは `followup`、`collect`、`steer-backlog`（および `steer` が followup にフォールバックする場合）に適用されます。

- `debounceMs`: フォローアップターンを開始する前に静寂を待ちます（「続けて、続けて」を防ぐ）。
- `cap`: セッションごとの最大キューメッセージ数。
- `drop`: オーバーフローポリシー（`old`、`new`、`summarize`）。

summarize は破棄されたメッセージの短い箇条書きリストを保持し、合成されたフォローアッププロンプトとして注入します。デフォルト: `debounceMs: 1000`、`cap: 20`、`drop: summarize`。

## セッションごとのオーバーライド

- スタンドアロンコマンドとして `/queue <mode>` を送信して、現在のセッションのモードを保存します。
- オプションは組み合わせ可能です: `/queue collect debounce:2s cap:25 drop:summarize`
- `/queue default` または `/queue reset` でセッションオーバーライドをクリアします。

## スコープと保証

- Gateway 返信パイプラインを使用するすべての受信チャンネルの自動返信エージェント実行に適用されます（WhatsApp Web、Telegram、Slack、Discord、Signal、iMessage、webchat など）。
- デフォルトレーン（`main`）は受信メッセージ + メインハートビートのプロセス全体です。`agents.defaults.maxConcurrent` を設定して複数のセッションを並列で許可します。
- 追加のレーン（例: `cron`、`subagent`）が存在する場合があり、バックグラウンドジョブが受信返信をブロックせずに並列実行できます。
- セッションごとのレーンにより、一度に 1 つのエージェント実行のみが特定のセッションに触れることが保証されます。
- 外部の依存関係やバックグラウンドワーカースレッドなし。純粋な TypeScript + Promise。

## トラブルシューティング

- コマンドが止まっているように見える場合は、verbose ログを有効にして「queued for …ms」行を探し、キューが処理されていることを確認します。
- キューの深さが必要な場合は、verbose ログを有効にしてキュータイミング行を監視します。
