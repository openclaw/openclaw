# Markdown フォーマット

OpenClaw は送信 Markdown を、チャンネル固有の出力をレンダリングする前に共有の中間表現（IR）に変換することでフォーマットします。IR はソーステキストをそのまま保持しながらスタイル/リンクのスパンを持ち、チャンキングとレンダリングをチャンネル間で一貫して保てます。

## 目標

- **一貫性:** 1 回のパースステップ、複数のレンダラー。
- **安全なチャンキング:** レンダリング前にテキストを分割して、インラインフォーマットがチャンク間で分断されないようにします。
- **チャンネルへの適合:** 同じ IR を Slack mrkdwn、Telegram HTML、Signal のスタイルレンジにマッピングし、Markdown を再パースしません。

## パイプライン

1. **Markdown を IR にパース**
   - IR はプレーンテキストにスタイルスパン（太字/斜体/取り消し線/コード/スポイラー）とリンクスパンを加えたものです。
   - オフセットは UTF-16 コードユニットで、Signal のスタイルレンジが API と一致します。
   - テーブルはチャンネルがテーブル変換を選択した場合のみパースされます。
2. **IR をチャンク（フォーマット優先）**
   - チャンキングはレンダリング前に IR テキスト上で行われます。
   - インラインフォーマットはチャンク間で分断されません。スパンはチャンクごとにスライスされます。
3. **チャンネルごとにレンダリング**
   - **Slack:** mrkdwn トークン（太字/斜体/取り消し線/コード）、リンクは `<url|label>` 形式。
   - **Telegram:** HTML タグ（`<b>`、`<i>`、`<s>`、`<code>`、`<pre><code>`、`<a href>`）。
   - **Signal:** プレーンテキスト + `text-style` レンジ。ラベルが URL と異なる場合はリンクが `label (url)` になります。

## IR の例

入力 Markdown:

```markdown
Hello **world** — see [docs](https://docs.openclaw.ai).
```

IR（模式的）:

```json
{
  "text": "Hello world — see docs.",
  "styles": [{ "start": 6, "end": 11, "style": "bold" }],
  "links": [{ "start": 19, "end": 23, "href": "https://docs.openclaw.ai" }]
}
```

## 使用箇所

- Slack、Telegram、Signal の送信アダプターは IR からレンダリングします。
- その他のチャンネル（WhatsApp、iMessage、MS Teams、Discord）では、プレーンテキストまたは独自のフォーマットルールを使用し、有効な場合はチャンキング前に Markdown テーブル変換を適用します。

## テーブルの処理

Markdown テーブルはチャットクライアント間で一貫してサポートされているわけではありません。`markdown.tables` を使用してチャンネルごと（およびアカウントごと）に変換を制御してください。

- `code`: テーブルをコードブロックとしてレンダリング（ほとんどのチャンネルのデフォルト）。
- `bullets`: 各行を箇条書きに変換（Signal と WhatsApp のデフォルト）。
- `off`: テーブルのパースと変換を無効にし、生のテーブルテキストをそのまま通過させます。

設定キー:

```yaml
channels:
  discord:
    markdown:
      tables: code
    accounts:
      work:
        markdown:
          tables: off
```

## チャンキングルール

- チャンク制限はチャンネルアダプター/設定から取得され、IR テキストに適用されます。
- コードフェンスは末尾の改行を含む単一ブロックとして保持されるため、チャンネルが正しくレンダリングできます。
- リストのプレフィックスとブロッククォートのプレフィックスは IR テキストの一部であるため、チャンキングがプレフィックスの途中で分断されることはありません。
- インラインスタイル（太字/斜体/取り消し線/インラインコード/スポイラー）はチャンク間で分断されません。レンダラーは各チャンク内でスタイルを再開します。

チャンネル間のチャンキング動作についての詳細は[ストリーミングとチャンキング](/concepts/streaming)を参照してください。

## リンクポリシー

- **Slack:** `[label](url)` -> `<url|label>`。ベア URL はベアのまま残ります。パース時に二重リンクを避けるため自動リンクは無効化されています。
- **Telegram:** `[label](url)` -> `<a href="url">label</a>`（HTML パースモード）。
- **Signal:** `[label](url)` -> `label (url)`（ラベルが URL と一致する場合を除く）。

## スポイラー

スポイラーマーカー（`||spoiler||`）は Signal に対してのみパースされ、SPOILER スタイルレンジにマッピングされます。その他のチャンネルではプレーンテキストとして扱われます。

## チャンネルフォーマッターの追加または更新方法

1. **1 回のパース:** チャンネルに適したオプション（自動リンク、見出しスタイル、ブロッククォートプレフィックス）で共有の `markdownToIR(...)` ヘルパーを使用します。
2. **レンダリング:** `renderMarkdownWithMarkers(...)` とスタイルマーカーマップ（または Signal スタイルレンジ）を使ってレンダラーを実装します。
3. **チャンク:** レンダリング前に `chunkMarkdownIR(...)` を呼び出し、各チャンクをレンダリングします。
4. **アダプターの接続:** 新しいチャンカーとレンダラーを使用するようにチャンネルの送信アダプターを更新します。
5. **テスト:** フォーマットテストを追加または更新し、チャンネルがチャンキングを使用する場合は送信デリバリーテストも追加します。

## よくある落とし穴

- Slack のアングルブラケットトークン（`<@U123>`、`<#C123>`、`<https://...>`）は保持する必要があります。生の HTML を安全にエスケープしてください。
- Telegram HTML はタグ外のテキストをエスケープする必要があります。そうしないとマークアップが壊れます。
- Signal のスタイルレンジは UTF-16 オフセットに依存します。コードポイントオフセットを使用しないでください。
- フェンスされたコードブロックの末尾の改行を保持し、クロージングマーカーが独自の行に来るようにしてください。
