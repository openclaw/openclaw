# Markdownフォーマット

OpenClawは送信Markdownを共有の中間表現（IR）に変換してからチャネル固有の出力をレンダリングします。IRはソーステキストをそのまま保持しながらスタイル/リンクスパンを持つため、チャンキングとレンダリングをチャネル間で一貫して保つことができます。

## 目標

- **一貫性:** 1回のパースステップ、複数のレンダラー。
- **安全なチャンキング:** レンダリング前にテキストを分割し、インラインフォーマットがチャンクをまたいで壊れないようにする。
- **チャネル適合:** 同じIRをSlack mrkdwn、Telegram HTML、Signalスタイル範囲にMarkdownの再パースなしでマッピング。

## パイプライン

1. **Markdown -> IRのパース**
   - IRはプレーンテキストにスタイルスパン（太字/イタリック/取り消し線/コード/スポイラー）とリンクスパンを加えたもの。
   - オフセットはUTF-16コードユニットで、Signalのスタイル範囲がそのAPIと一致するようにしている。
   - テーブルはチャネルがテーブル変換をオプトインした場合のみパースされる。
2. **IRのチャンキング（フォーマット優先）**
   - チャンキングはレンダリング前にIRテキスト上で行われる。
   - インラインフォーマットはチャンクをまたいで分割されない。スパンはチャンクごとにスライスされる。
3. **チャネルごとのレンダリング**
   - **Slack:** mrkdwnトークン（太字/イタリック/取り消し線/コード）、リンクは `<url|label>`。
   - **Telegram:** HTMLタグ（`<b>`、`<i>`、`<s>`、`<code>`、`<pre><code>`、`<a href>`）。
   - **Signal:** プレーンテキスト + `text-style` 範囲。ラベルが異なる場合、リンクは `label (url)` になる。

## IRの例

入力Markdown:

```markdown
Hello **world** — see [docs](https://docs.openclaw.ai).
```

IR（概要）:

```json
{
  "text": "Hello world — see docs.",
  "styles": [{ "start": 6, "end": 11, "style": "bold" }],
  "links": [{ "start": 19, "end": 23, "href": "https://docs.openclaw.ai" }]
}
```

## 使用される場所

- Slack、Telegram、Signalの送信アダプターがIRからレンダリングする。
- その他のチャネル（WhatsApp、iMessage、Microsoft Teams、Discord）はプレーンテキストまたは独自のフォーマットルールを使用し、有効な場合はチャンキング前にMarkdownテーブル変換が適用される。

## テーブル処理

Markdownテーブルはチャットクライアント間で一貫してサポートされていません。`markdown.tables` を使用してチャネルごと（およびアカウントごと）に変換を制御します。

- `code`: テーブルをコードブロックとしてレンダリング（ほとんどのチャネルのデフォルト）。
- `bullets`: 各行を箇条書きに変換（Signal + WhatsAppのデフォルト）。
- `off`: テーブルのパースと変換を無効にする。生のテーブルテキストがそのまま渡される。

設定キー:

```yaml
channels:
  discord:
    markdown:
      tables: code
    accounts:
      work:
        markdown:
          tables: off
```

## チャンキングルール

- チャンク制限はチャネルアダプター/設定から取得され、IRテキストに適用される。
- コードフェンスは末尾の改行付きの単一ブロックとして保持され、チャネルが正しくレンダリングできるようにする。
- リストプレフィックスとブロック引用プレフィックスはIRテキストの一部であるため、チャンキングがプレフィックスの途中で分割されることはない。
- インラインスタイル（太字/イタリック/取り消し線/インラインコード/スポイラー）はチャンクをまたいで分割されない。レンダラーが各チャンク内でスタイルを再開する。

チャネル間のチャンキング動作の詳細については、[ストリーミング + チャンキング](/concepts/streaming)を参照してください。

## リンクポリシー

- **Slack:** `[label](url)` -> `<url|label>`。裸のURLはそのまま。パース中に自動リンクは無効化され、二重リンクを防ぐ。
- **Telegram:** `[label](url)` -> `<a href="url">label</a>`（HTMLパースモード）。
- **Signal:** `[label](url)` -> `label (url)`（ラベルがURLと一致する場合を除く）。

## スポイラー

スポイラーマーカー（`||spoiler||`）はSignalでのみパースされ、SPOILERスタイル範囲にマッピングされます。その他のチャネルではプレーンテキストとして扱われます。

## チャネルフォーマッターの追加・更新方法

1. **1回パース:** チャネルに適したオプション（自動リンク、見出しスタイル、ブロック引用プレフィックス）で共有の `markdownToIR(...)` ヘルパーを使用する。
2. **レンダリング:** `renderMarkdownWithMarkers(...)` とスタイルマーカーマップ（またはSignalスタイル範囲）でレンダラーを実装する。
3. **チャンキング:** レンダリング前に `chunkMarkdownIR(...)` を呼び出し、各チャンクをレンダリングする。
4. **アダプターの接続:** チャネルの送信アダプターを更新して新しいチャンカーとレンダラーを使用する。
5. **テスト:** フォーマットテストを追加または更新し、チャネルがチャンキングを使用する場合は送信配信テストも追加する。

## よくある落とし穴

- Slackのアングルブラケットトークン（`<@U123>`、`<#C123>`、`<https://...>`）は保持する必要がある。生のHTMLは安全にエスケープすること。
- Telegram HTMLではタグ外のテキストをエスケープしないとマークアップが壊れる。
- Signalのスタイル範囲はUTF-16オフセットに依存する。コードポイントオフセットを使用しないこと。
- フェンスドコードブロックの末尾の改行を保持し、閉じマーカーが独立した行に配置されるようにすること。
