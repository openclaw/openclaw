# Zorg MemoryDB v1.2.51

## Bad-row quarantine and prune rule

This release documents the operator correction that broken or wrong generated memory rows should not remain in the database in a form that code can reactivate.

Required behavior:

- Wrong, broken, superseded, or bad-path generated rows are deactivated immediately.
- The deactivation reason and producing path should be recorded.
- Deactivated bad generated rows must stay out of active recall and bridge reactivation logic.
- After a 24-hour quarantine window, bad generated rows are pruned/deleted from active recall-producing tables.
- Real operator/source history remains protected. If an audit record is necessary, keep a minimal non-reactivating tombstone rather than a live recall row.

This clarifies the interaction between source-history preservation and bad generated data: preserve legitimate source history, but do not let broken generated rows persist as recallable or reactivatable memory.
