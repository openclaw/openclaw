# Zorg MemoryDB v1.2.52

## Full backup is enough for bad generated row pruning

This release corrects the v1.2.51 wording that suggested keeping minimal tombstones or extra audit rows for pruned bad generated memory rows.

Operator correction:

- A verified full backup is the necessary precaution before pruning/deleting bad generated rows.
- Do not create extra database tombstones, retained bad-row audit rows, or other database records merely as an added precaution.
- Extra retained records waste database space and can become another path for stale code to reactivate bad memory.
- Wrong, broken, superseded, or bad-path generated rows should be deactivated immediately, excluded from recall/reactivation, and pruned/deleted after a 24-hour quarantine once backup coverage is verified.

This rule targets bad generated/derived rows and broken-path artifacts. Legitimate operator/source history remains protected where it is not the bad generated row itself.
