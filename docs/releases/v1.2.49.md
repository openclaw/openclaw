# v1.2.49 - Telegram to PostgreSQL memory bridge repair

This release adds a local OpenClaw/Codex rollout bridge for Zorg installs where
Telegram direct turns are stored in the Codex session database/files but are not
automatically written into PostgreSQL memory.

## Changed

- Added `scripts/telegram_to_postgres_bridge.py`.
- Added systemd user units under `scripts/systemd/` for one-minute bridge runs.
- The bridge scans recent Codex rollout JSONL files, extracts the current
  Telegram request from the OpenClaw prompt envelope, and writes compact
  `chat_ingest_user` / `chat_ingest_assistant` rows into `public.zorg_memory`.
- The bridge repairs stale `app_write_events` and `app_activity_events`
  sequences before inserting, preventing duplicate-key failures from making
  memory write counters appear stalled.
- Existing bridge-written `chat_ingest_*` rows are updated when extraction is
  improved, so a wrapper-heavy prompt row can be corrected to the clean current
  request without deleting source history.
