# 원격 액세스(SSH, 터널 및 tailnet)

이 저장소는 전용 호스트(데스크톱/서버)에서 단일 게이트웨이(마스터)를 실행하고 이에 클라이언트를 연결함으로써 "SSH를 통한 원격"을 지원합니다.

- 을 위한 **운영자(귀하/macOS 앱)**: SSH 터널링은 보편적인 폴백입니다.
- 을 위한 **노드(iOS/Android 및 향후 장치)**: 게이트웨이에 연결 **웹소켓** (필요에 따라 LAN/tailnet 또는 SSH 터널).

## 핵심 아이디어

- 게이트웨이 WebSocket은 다음에 바인딩됩니다. **루프백** 구성된 포트에서(기본값은 18789)
- 원격 사용의 경우 해당 루프백 포트를 SSH를 통해 전달합니다(또는 tailnet/VPN 및 터널링 감소 사용).

## 일반적인 VPN/tailnet 설정(에이전트가 있는 곳)

생각해보세요 **게이트웨이 호스트** '에이전트가 사는 곳'으로 표시됩니다. 세션, 인증 프로필, 채널 및 상태를 소유합니다.
노트북/데스크톱(및 노드)이 해당 호스트에 연결됩니다.

### 1) 테일넷(VPS 또는 홈 서버)의 Always-On 게이트웨이

영구 호스트에서 게이트웨이를 실행하고 다음을 통해 연결합니다. **테일스케일** 또는 SSH.

- **최고의 UX:** 유지하다 `gateway.bind: "loopback"` 그리고 사용 **테일스케일 서브** 컨트롤 UI의 경우.
- **대체:** 액세스가 필요한 모든 시스템에서 루프백 + SSH 터널을 유지하세요.
- **예:** [exe.dev](/install/exe-dev) (쉬운 VM) 또는 [헤츠너](/install/hetzner) (프로덕션 VPS).

이는 노트북이 자주 잠자기 상태가 되지만 에이전트는 항상 켜져 있기를 원하는 경우에 이상적입니다.

### 2) 홈 데스크탑은 게이트웨이를 실행하고 노트북은 원격 제어입니다.

노트북은 그렇지 **~ 아니다** 에이전트를 실행합니다. 원격으로 연결됩니다.

- macOS 앱을 사용하세요. **SSH를 통한 원격** 모드(설정 → 일반 → "OpenClaw 실행").
- 앱이 터널을 열고 관리하므로 WebChat + 상태 확인이 "작동"합니다.

실행서: [macOS 원격 액세스](/platforms/mac/remote).

### 3) 노트북은 게이트웨이를 실행하고 다른 컴퓨터에서 원격으로 액세스합니다.

게이트웨이를 로컬로 유지하되 안전하게 노출하세요.

- 다른 컴퓨터에서 랩톱으로의 SSH 터널 또는
- Tailscale Control UI를 제공하고 게이트웨이 루프백만 유지합니다.

가이드: [테일스케일](/gateway/tailscale) 그리고 [웹 개요](/web).

## 명령 흐름(무엇이 어디에서 실행되는지)

하나의 게이트웨이 서비스는 상태 + 채널을 소유합니다. 노드는 주변 장치입니다.

흐름 예시(텔레그램 → 노드):

- 텔레그램 메시지가 도착했습니다. **게이트웨이**.
- 게이트웨이는 **대리인** 노드 도구를 호출할지 여부를 결정합니다.
- 게이트웨이는 **마디** 게이트웨이 WebSocket을 통해(`node.*` RPC).
- 노드는 결과를 반환합니다. 게이트웨이는 텔레그램으로 다시 응답합니다.

참고:

- **노드는 게이트웨이 서비스를 실행하지 않습니다.** 의도적으로 격리된 프로필을 실행하지 않는 한 호스트당 하나의 게이트웨이만 실행해야 합니다(참조: [다중 게이트웨이](/gateway/multiple-gateways)).
- macOS 앱 "노드 모드"는 Gateway WebSocket을 통한 노드 클라이언트일 뿐입니다.

## SSH 터널(CLI + 도구)

원격 게이트웨이 WS에 대한 로컬 터널을 만듭니다.

```bash
ssh -N -L 18789:127.0.0.1:18789 user@host
```

터널을 위로 올린 상태에서:

- `openclaw health` 그리고 `openclaw status --deep` 이제 다음을 통해 원격 게이트웨이에 도달하세요. `ws://127.0.0.1:18789`.
- `openclaw gateway {status,health,send,agent,call}` 다음을 통해 전달된 URL을 타겟팅할 수도 있습니다. `--url` 필요할 때.

참고: 교체 `18789` 귀하가 구성한 `gateway.port` (또는 `--port`/`OPENCLAW_GATEWAY_PORT`).
참고: 통과할 때 `--url`, CLI는 구성 또는 환경 자격 증명으로 대체되지 않습니다.
포함하다 `--token` 또는 `--password` 명시적으로. 명시적 자격 증명이 누락되면 오류가 발생합니다.

## CLI 원격 기본값

CLI 명령이 기본적으로 이를 사용하도록 원격 대상을 유지할 수 있습니다.

```json5
{
  gateway: {
    mode: "remote",
    remote: {
      url: "ws://127.0.0.1:18789",
      token: "your-token",
    },
  },
}
```

게이트웨이가 루프백 전용인 경우 URL을 다음으로 유지하세요. `ws://127.0.0.1:18789` 먼저 SSH 터널을 엽니다.

## SSH를 통한 채팅 UI

WebChat은 더 이상 별도의 HTTP 포트를 사용하지 않습니다. SwiftUI 채팅 UI는 Gateway WebSocket에 직접 연결됩니다.

- 앞으로 `18789` SSH를 통해(위 참조) 클라이언트를 연결합니다. `ws://127.0.0.1:18789`.
- macOS에서는 터널을 자동으로 관리하는 앱의 "Remote over SSH" 모드를 선호합니다.

## macOS 앱 “SSH를 통한 원격”

macOS 메뉴 표시줄 앱은 동일한 설정을 엔드투엔드(원격 상태 확인, WebChat 및 음성 깨우기 전달)로 구동할 수 있습니다.

실행서: [macOS 원격 액세스](/platforms/mac/remote).

## 보안 규칙(원격/VPN)

짧은 버전: **게이트웨이 루프백 전용으로 유지** 바인드가 필요하다고 확신하지 않는 한.

- **루프백 + SSH/Tailscale 서비스** 가장 안전한 기본값입니다(공개 노출 없음).
- **비루프백 바인드** (`lan`/`tailnet`/`custom`, 또는 `auto` 루프백을 사용할 수 없는 경우) 인증 토큰/비밀번호를 사용해야 합니다.
- `gateway.remote.token` ~이다 **오직** 원격 CLI 호출의 경우 — 그렇습니다. **~ 아니다** 로컬 인증을 활성화합니다.
- `gateway.remote.tlsFingerprint` 사용할 때 원격 TLS 인증서를 고정합니다. `wss://`.
- **테일스케일 서브** 다음과 같은 경우 ID 헤더를 통해 인증할 수 있습니다. `gateway.auth.allowTailscale: true`.
  다음으로 설정하세요 `false` 대신 토큰/비밀번호를 원한다면.
- 브라우저 제어를 운영자 액세스처럼 처리합니다. tailnet 전용 + 의도적인 노드 페어링.

심층 분석: [보안](/gateway/security).
