# 常态指令（Standing Orders）

常态指令为你的智能体授予**永久操作授权**。无需每次单独给出任务指令，你只需定义具有明确范围、触发条件和升级规则的程序，智能体即可在授权边界内自主执行。

这相当于：每周五都要说"发送周报" vs. 直接授权："周报由你负责，每周五编写并发送，有异常才上报。"

## 为什么需要常态指令？

**没有常态指令时：**
- 每个任务都需要手动提示智能体
- 智能体在请求之间处于空闲状态
- 日常工作容易被遗忘或延迟
- 你成为所有工作的瓶颈

**有常态指令时：**
- 智能体在定义边界内自主执行
- 日常工作按时执行，无需提示
- 你只需处理异常和审批
- 智能体空闲时间被高效利用

## 工作原理

常态指令定义在[智能体工作区](/zh-CN/concepts/agent-workspace)文件中。推荐做法是直接写入 `AGENTS.md`（每个会话自动注入），使智能体始终持有这些指令。对于大型配置，也可将指令放在独立文件（如 `standing-orders.md`）中，然后在 `AGENTS.md` 中引用。

每个程序需指定：

1. **范围（Scope）** — 智能体被授权执行的操作
2. **触发条件（Triggers）** — 何时执行（定时、事件或条件）
3. **审批门控（Approval gates）** — 执行前需要人工签字确认的条件
4. **升级规则（Escalation rules）** — 何时停止并寻求帮助

智能体在每个会话加载这些指令（通过工作区引导文件，见[智能体工作区](/zh-CN/concepts/agent-workspace)），并结合[定时任务](/zh-CN/automation/cron-jobs)实现时间触发的执行。

<Tip>
将常态指令写入 `AGENTS.md`，确保每个会话都被加载。工作区引导会自动注入 `AGENTS.md`、`SOUL.md`、`TOOLS.md`、`IDENTITY.md`、`USER.md`、`HEARTBEAT.md`、`BOOTSTRAP.md` 和 `MEMORY.md`，但不会自动注入子目录中的任意文件。
</Tip>

## 常态指令的结构

```markdown
## 程序：周状态报告

**授权范围：** 收集数据、生成报告、投递给利益相关方
**触发条件：** 每周五下午4点（通过定时任务强制执行）
**审批门控：** 标准报告无需审批。异常指标需人工审核。
**升级规则：** 数据源不可用或指标异常（超过历史均值2σ）时上报。

### 执行步骤

1. 从配置数据源拉取指标
2. 与上周及目标值对比
3. 生成报告至 Reports/weekly/YYYY-MM-DD.md
4. 通过配置渠道发送摘要
5. 记录完成日志至 Agent/Logs/

### 禁止事项

- 禁止向外部人员发送报告
- 禁止修改源数据
- 禁止因指标难看而跳过发送——如实报告
```

## 常态指令 + 定时任务

常态指令定义智能体**被授权做什么**。[定时任务](/zh-CN/automation/cron-jobs)定义**何时做**。两者协同工作：

```
常态指令："你负责每日收件箱分拣"
    ↓
定时任务（每天8:00）："按常态指令执行收件箱分拣"
    ↓
智能体：读取常态指令 → 执行步骤 → 汇报结果
```

定时任务提示应引用常态指令，而非重复其内容：

```bash
openclaw cron add \
  --name daily-inbox-triage \
  --cron "0 8 * * 1-5" \
  --tz Asia/Shanghai \
  --timeout-seconds 300 \
  --announce \
  --channel bluebubbles \
  --to "+86XXXXXXXXXXX" \
  --message "执行每日收件箱分拣。检查新邮件告警，解析分类并持久化，汇总后报告。未知情况上报。"
```

## 示例

### 示例一：内容运营（周循环）

```markdown
## 程序：内容与社交媒体

**授权范围：** 起草内容、安排发布、汇总互动报告
**审批门控：** 前30天所有帖子需人工审核，之后转为常态授权
**触发条件：** 周循环（周一审核 → 周中起草 → 周五简报）

### 周循环

- **周一：** 审核平台指标和受众互动数据
- **周二至周四：** 起草社交帖子，创建博客内容
- **周五：** 汇总本周营销简报 → 投递给负责人

### 内容规则

- 文风必须符合品牌调性（参见 SOUL.md 或品牌文风指南）
- 公开内容中不得以 AI 自称
- 尽可能包含数据指标
- 关注为受众创造的价值，而非自我推销
```

### 示例二：财务处理（事件触发）

```markdown
## 程序：财务处理

**授权范围：** 处理交易数据、生成报告、发送摘要
**审批门控：** 分析无需审批，建议方案需人工批准
**触发条件：** 检测到新数据文件 或 每月定时周期

### 新数据到达时

1. 检测指定输入目录中的新文件
2. 解析并分类所有交易
3. 与预算目标对比
4. 标记：异常项、超阈值项、新增周期性扣款
5. 在指定输出目录生成报告
6. 通过配置渠道向负责人发送摘要

### 升级规则

- 单笔 > ¥5000：立即告警
- 某类超预算 20%：报告中标记
- 无法识别的交易：请求负责人分类
- 重试2次后处理失败：报告失败，不自行猜测
```

### 示例三：系统监控（持续运行）

```markdown
## 程序：系统监控

**授权范围：** 检查系统健康状况、重启服务、发送告警
**审批门控：** 自动重启服务，重启失败2次后升级
**触发条件：** 每个心跳周期

### 检查项

- 服务健康端点响应状态
- 磁盘空间是否超过阈值
- 待处理任务是否超时（>24小时）
- 投递渠道是否正常运行

### 响应矩阵

| 情况 | 操作 | 是否升级 |
| ---- | ---- | -------- |
| 服务宕机 | 自动重启 | 重启2次失败后升级 |
| 磁盘空间 < 10% | 通知负责人 | 是 |
| 任务超时 > 24h | 提醒负责人 | 否 |
| 渠道离线 | 记录并下一周期重试 | 离线 > 2小时则升级 |
```

## 执行-验证-汇报 模式

常态指令配合严格的执行纪律效果最佳。每个任务步骤都应遵循此循环：

1. **执行（Execute）** — 完成实际工作（不是仅仅确认收到指令）
2. **验证（Verify）** — 确认结果正确（文件存在、消息发送、数据解析成功）
3. **汇报（Report）** — 告知负责人完成了什么以及验证结果

```markdown
### 执行规则

- 每个任务都遵循 执行-验证-汇报。无例外。
- "好的我来做" 不是执行。做完了再汇报。
- 没有验证的"已完成"不可接受。必须证明。
- 执行失败：调整方式重试一次。
- 仍然失败：汇报失败并附上诊断结果。不得静默失败。
- 不得无限重试——最多3次，然后升级。
```

此模式防止了智能体最常见的失效模式：确认任务但不完成。

## 多程序架构

对于管理多个关注领域的智能体，将常态指令组织为独立的程序，并保持清晰的边界：

```markdown
# 常态指令

## 程序一：[领域A]（每周）

...

## 程序二：[领域B]（每月 + 按需）

...

## 程序三：[领域C]（视需要）

...

## 升级规则（所有程序通用）

- [通用升级标准]
- [跨程序适用的审批门控]
```

每个程序应具有：

- 独立的**触发节奏**（每周、每月、事件驱动、持续）
- 独立的**审批门控**（某些程序需要更多监督）
- 清晰的**边界**（智能体应知道一个程序何时结束、另一个何时开始）

## 最佳实践

### ✅ 应该做

- 从窄范围授权开始，随着信任建立逐步扩展
- 对高风险操作定义明确的审批门控
- 包含"禁止事项"章节——边界与权限同样重要
- 结合定时任务以保证基于时间的可靠执行
- 每周审核智能体日志，确认常态指令被遵循
- 随着需求变化更新常态指令——它们是活的文档

### ❌ 避免做

- 第一天就授予宽泛授权（"做你认为合适的任何事"）
- 跳过升级规则——每个程序都需要"何时停止并询问"的条款
- 认为智能体会记住口头指令——所有内容必须写入文件
- 将不同关注领域混在一个程序里——不同领域用不同程序
- 忘记配合定时任务执行——没有触发条件的常态指令只是建议
