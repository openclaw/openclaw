# 常设命令

常设命令为定义的程序授予智能体**永久操作权限**。你不需要每次都给出单独的任务指令，而是定义具有清晰范围、触发器和升级规则的程序 —— 智能体在这些边界内自主执行。

这就像每个周五告诉助手"发送周报"与授予常设权限的区别："你负责周报。每周五编译、发送，只有在出现问题时才升级。"

## 为什么需要常设命令？

**没有常设命令：**

- 你必须为每个任务提示智能体
- 智能体在请求之间空闲
- 例行工作被遗忘或延迟
- 你成为瓶颈

**有常设命令：**

- 智能体在定义边界内自主执行
- 例行工作按计划进行，无需提示
- 你只在异常和审批时参与
- 智能体高效利用空闲时间

## 工作原理

常设命令在 [智能体工作区](/concepts/agent-workspace) 文件中定义。推荐的方法是直接将它们包含在 `AGENTS.md` 中（每个会话自动注入），这样智能体始终在上下文中拥有它们。对于较大的配置，你也可以将它们放在专用文件如 `standing-orders.md` 中，并从 `AGENTS.md` 引用它。

每个程序指定：

1. **范围** — 智能体被授权做什么
2. **触发器** — 何时执行（计划、事件或条件）
3. **审批门** — 在行动前需要人工签署的内容
4. **升级规则** — 何时停止并请求帮助

智能体通过工作区引导文件在每个会话中加载这些指令（请参阅 [智能体工作区](/concepts/agent-workspace) 以获取自动注入文件的完整列表），并结合 [定时任务](/automation/cron-jobs) 进行基于时间的强制执行。

<Tip>
将常设命令放在 `AGENTS.md` 中以确保它们在每个会话中加载。工作区引导会自动注入 `AGENTS.md`、`SOUL.md`、`TOOLS.md`、`IDENTITY.md`、`USER.md`、`HEARTBEAT.md`、`BOOTSTRAP.md` 和 `MEMORY.md` —— 但不会注入子目录中的任意文件。
</Tip>

## 常设命令的结构

```markdown
## 程序：周状态报告

**权限：** 编译数据、生成报告、传递给相关方
**触发器：** 每周五下午 4 点（通过定时任务强制执行）
**审批门：** 标准报告无需审批。标记异常供人工审查。
**升级：** 如果数据源不可用或指标看起来异常（>2σ 偏离正常值）

### 执行步骤

1. 从配置的源拉取指标
2. 与上周和目标比较
3. 在 Reports/weekly/YYYY-MM-DD.md 生成报告
4. 通过配置的频道传递摘要
5. 记录完成到 Agent/Logs/

### 禁止事项

- 不要向外部方发送报告
- 不要修改源数据
- 不要因为指标不好而跳过传递 —— 准确报告
```

## 常设命令 + 定时任务

常设命令定义智能体**被授权做什么**。[定时任务](/automation/cron-jobs) 定义**何时发生**。它们协同工作：

```
常设命令："你负责每日收件箱分类"
    ↓
定时任务（每天上午 8 点）："根据常设命令执行收件箱分类"
    ↓
智能体：读取常设命令 → 执行步骤 → 报告结果
```

定时任务提示应引用常设命令而不是重复它：

```bash
openclaw cron add \
  --name daily-inbox-triage \
  --cron "0 8 * * 1-5" \
  --tz America/New_York \
  --timeout-seconds 300 \
  --announce \
  --channel bluebubbles \
  --to "+1XXXXXXXXXX" \
  --message "根据常设命令执行每日收件箱分类。检查邮件中的新警报。解析、分类并持久化每个项目。向所有者报告摘要。升级未知项。"
```

## 示例

### 示例 1：内容和社交媒体（周循环）

```markdown
## 程序：内容和社交媒体

**权限：** 起草内容、安排帖子、编译参与报告
**审批门：** 前 30 天所有帖子需要所有者审查，之后常设批准
**触发器：** 周循环（周一审查 → 周中起草 → 周五简报）

### 周循环

- **周一：** 审查平台指标和受众参与
- **周二–周四：** 起草社交帖子、创建博客内容
- **周五：** 编译周营销简报 → 传递给所有者

### 内容规则

- 语气必须匹配品牌（请参阅 SOUL.md 或品牌语气指南）
- 永不在面向公众的内容中表明 AI 身份
- 尽可能包含指标
- 关注对受众的价值，而非自我推广
```

### 示例 2：财务运营（事件触发）

```markdown
## 程序：财务处理

**权限：** 处理交易数据、生成报告、发送摘要
**审批门：** 分析无需审批。建议需要所有者批准。
**触发器：** 检测到新数据文件 或 计划的月度循环

### 当新数据到达时

1. 在指定输入目录检测新文件
2. 解析并分类所有交易
3. 与预算目标比较
4. 标记：异常项目、阈值违规、新的周期性费用
5. 在指定输出目录生成报告
6. 通过配置的频道向所有者传递摘要

### 升级规则

- 单项 > $500：立即警报
- 类别 > 预算 20%：在报告中标记
- 无法识别的交易：请求所有者分类
- 重试 2 次后处理失败：报告失败，不要猜测
```

### 示例 3：监控和警报（连续）

```markdown
## 程序：系统监控

**权限：** 检查系统健康、重启服务、发送警报
**审批门：** 自动重启服务。如果重启失败两次则升级。
**触发器：** 每个心跳周期

### 检查

- 服务健康端点响应
- 磁盘空间高于阈值
- 待处理任务不陈旧（>24 小时）
- 传递频道可操作

### 响应矩阵

| 条件            | 操作                   | 升级？                  |
| --------------- | ---------------------- | ----------------------- |
| 服务宕机        | 自动重启              | 仅当重启失败 2 次       |
| 磁盘空间 < 10%  | 警报所有者            | 是                      |
| 陈旧任务 > 24h  | 提醒所有者            | 否                      |
| 频道离线        | 记录并下个周期重试    | 如果离线 > 2 小时       |
```

## 执行-验证-报告模式

常设命令与严格的执行纪律结合效果最佳。常设命令中的每个任务都应遵循此循环：

1. **执行** — 做实际工作（不要只是确认指令）
2. **验证** — 确认结果正确（文件存在、消息已传递、数据已解析）
3. **报告** — 告诉所有者做了什么以及验证了什么

```markdown
### 执行规则

- 每个任务遵循执行-验证-报告。无例外。
- "我会做"不是执行。做，然后报告。
- 没有验证的"完成"不可接受。证明它。
- 如果执行失败：用调整的方法重试一次。
- 如果仍然失败：报告失败及诊断。永不静默失败。
- 永不无限重试 —— 最多 3 次尝试，然后升级。
```

此模式防止最常见的智能体失败模式：确认任务但未完成。

## 多程序架构

对于管理多个关注点的智能体，将常设命令组织为具有清晰边界的独立程序：

```markdown
# 常设命令

## 程序 1：[领域 A]（每周）

...

## 程序 2：[领域 B]（每月 + 按需）

...

## 程序 3：[领域 C]（按需）

...

## 升级规则（所有程序）

- [通用升级标准]
- [跨程序适用的审批门]
```

每个程序应有：

- 自己的**触发节奏**（每周、每月、事件驱动、连续）
- 自己的**审批门**（某些程序需要更多监督）
- 清晰的**边界**（智能体应知道一个程序在哪里结束，另一个在哪里开始）

## 最佳实践

### 应该

- 从窄权限开始，随着信任建立逐步扩展
- 为高风险操作定义明确的审批门
- 包含"禁止事项"部分 —— 边界与权限同样重要
- 与定时任务结合以实现可靠的基于时间的执行
- 每周审查智能体日志以验证常设命令是否被遵循
- 随着需求演进更新常设命令 —— 它们是活文档

### 避免

- 第一天就授予广泛权限（"做你认为最好的事"）
- 跳过升级规则 —— 每个程序都需要"何时停止并询问"条款
- 假设智能体会记住口头指令 —— 把所有内容放在文件中
- 在单个程序中混合关注点 —— 为不同领域设置独立程序
- 忘记用定时任务强制执行 —— 没有触发器的常设命令变成建议
