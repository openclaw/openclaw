---\nsummary: \"通过 ACP 运行时会话使用 Pi、Claude Code、Codex、OpenCode、Gemini CLI 及其他线束 Agent\"\nread_when:\n  - 通过 ACP 运行编码线束\n  - 在支持线程的频道上设置线程绑定的 ACP 会话\n  - 将 Discord 频道或 Telegram 论坛话题绑定到持久性 ACP 会话\n  - 排除 ACP 后端和插件连接故障\n  - 在聊天中操作 /acp 命令\ntitle: \"ACP Agent\"\n---\n\n# ACP Agent\n\n[Agent Client Protocol (ACP)](https://agentclientprotocol.com/) 会话允许 OpenClaw 通过 ACP 后端插件运行外部编码线束（例如 Pi、Claude Code、Codex、OpenCode 和 Gemini CLI）。\n\n如果你用自然语言要求 OpenClaw \"在 Codex 中运行此任务\"或\"在线程中启动 Claude Code\"，OpenClaw 应将该请求路由到 ACP 运行时（而不是原生子 agent 运行时）。\n\n## 快速操作流程\n\n当你需要一个实用的 `/acp` 操作手册时，使用此流程：\n\n1. 启动会话：\n   - `/acp spawn codex --mode persistent --thread auto`\n2. 在绑定的线程中工作（或明确指定该会话键）。\n3. 检查运行时状态：\n   - `/acp status`\n4. 根据需要调整运行时选项：\n   - `/acp model <provider/model>`\n   - `/acp permissions <profile>`\n   - `/acp timeout <seconds>`\n5. 提醒活跃会话而不替换上下文：\n   - `/acp steer tighten logging and continue`\n6. 停止工作：\n   - `/acp cancel`（取消当前轮次），或\n   - `/acp close`（关闭会话 + 移除绑定）\n\n## 面向人类的快速入门\n\n自然语言请求示例：\n\n- \"在此线程中启动一个持久性 Codex 会话，并使其保持专注。\"\n- \"将此作为一次性 Claude Code ACP 会话运行并总结结果。\"\n- \"在此线程中使用 Gemini CLI 执行此任务，然后在此线程中继续后续操作。\"\n\nOpenClaw 应该怎么做：\n\n1. 选择 `runtime: \"acp\"`。\n2. 解析请求的线束目标（`agentId`，例如 `codex`）。\n3. 如果请求了线程绑定且当前频道支持，则将 ACP 会话绑定到该线程。\n4. 将后续线程消息路由到同一 ACP 会话，直到取消聚焦/关闭/过期。\n\n## ACP 与子 agent 对比\n\n当你需要外部线束运行时时使用 ACP。当你需要 OpenClaw 原生委托运行时时使用子 agent。\n\n| 区域          | ACP 会话                           | 子 agent 运行                      |\n| ------------- \| ---------------------------------- \| ---------------------------------- |\n| 运行时        | ACP 后端插件（例如 acpx）          | OpenClaw 原生子 agent 运行时       |\n| 会话键        | `agent:<agentId>:acp:<uuid>`       | `agent:<agentId>:subagent:<uuid>`  |\n| 主要命令      | `/acp ...`                         | `/subagents ...`                   |\\n| 启动工具      | `sessions_spawn` 配合 `runtime:\"acp\"` | `sessions_spawn`（默认运行时）     |\n\n另请参阅 [子 agent](/tools/subagents)。\n\n## 线程绑定的会话（渠道无关）\n\n当为渠道适配器启用线程绑定时，ACP 会话可以绑定到线程：\n\n- OpenClaw 将线程绑定到目标 ACP 会话。\n- 该线程中的后续消息路由到绑定的 ACP 会话。\n- ACP 输出返回到同一线程。\n- 取消聚焦/关闭/存档/空闲超时或达到最大年龄会移除绑定。\n\n线程绑定支持是适配器特定的。如果活跃的渠道适配器不支持线程绑定，OpenClaw 会返回明确的不支持/不可用消息。\n\n线程绑定 ACP 的必需功能标志：\n\n- `acp.enabled=true`\n- `acp.dispatch.enabled` 默认开启（设置为 `false` 可暂停 ACP 分派）\n- 渠道适配器 ACP 线程启动标志启用（适配器特定）\n  - Discord: `channels.discord.threadBindings.spawnAcpSessions=true`\n  - Telegram: `channels.telegram.threadBindings.spawnAcpSessions=true`\n\n### 支持线程的频道\n\n- 任何暴露会话/线程绑定能力的渠道适配器。\n- 当前内置支持：\n  - Discord 线程/频道\n  - Telegram 话题（群组/超级群组中的论坛话题和私信话题）\n- 插件频道可以通过相同的绑定接口添加支持。\n\n## 渠道特定设置\n\n对于非临时性工作流，在顶层 `bindings[]` 条目中配置持久性 ACP 绑定。\n\n### 绑定模型\n\n- `bindings[].type=\"acp\"` 标记一个持久性 ACP 对话绑定。\n- `bindings[].match` 识别目标对话：\n  - Discord 频道或线程: `match.channel=\"discord\"` + `match.peer.id=\"<channelOrThreadId>\"`\n  - Telegram 论坛话题: `match.channel=\"telegram\"` + `match.peer.id=\"<chatId>:topic:<topicId>\"`\n- `bindings[].agentId` 是所属的 OpenClaw agent ID。\n- 可选的 ACP 覆盖配置在 `bindings[].acp` 下：\n  - `mode`（`persistent` 或 `oneshot`）\n  - `label`\n  - `cwd`\n  - `backend`\n\n### 每个 agent 的运行时默认值\n\n使用 `agents.list[].runtime` 为每个 agent 定义 ACP 默认值：\n\n- `agents.list[].runtime.type=\"acp\"`\n- `agents.list[].runtime.acp.agent`（线束 ID，例如 `codex` 或 `claude`）\n- `agents.list[].runtime.acp.backend`\n- `agents.list[].runtime.acp.mode`\n- `agents.list[].runtime.acp.cwd`\n\nACP 绑定会话的覆盖优先级：\n\n1. `bindings[].acp.*`\n2. `agents.list[].runtime.acp.*`\n3. 全局 ACP 默认值（例如 `acp.backend`）\n\n示例：\n\n```json5\n{\n  agents: {\n    list: [\n      {\n        id: \"codex\",\n        runtime: {\n          type: \"acp\",\n          acp: {\n            agent: \"codex\",\n            backend: \"acpx\",\n            mode: \"persistent\",\n            cwd: \"/workspace/openclaw\",\n          },\n        },\n      },\n      {\n        id: \"claude\",\n        runtime: {\n          type: \"acp\",\n          acp: { agent: \"claude\", backend: \"acpx\", mode: \"persistent\" },\n        },\n      },\n    ],\n  },\n  bindings: [\n    {\n      type: \"acp\",\n      agentId: \"codex\",\n      match: {\n        channel: \"discord\",\n        accountId: \"default\",\n        peer: { kind: \"channel\", id: \"222222222222222222\" },\n      },\n      acp: { label: \"codex-main\" },\n    },\n    {\n      type: \"acp\",\n      agentId: \"claude\",\n      match: {\n        channel: \"telegram\",\n        accountId: \"default\",\n        peer: { kind: \"group\", id: \"-1001234567890:topic:42\" },\n      },\n      acp: { cwd: \"/workspace/repo-b\" },\n    },\n    {\n      type: \"route\",\n      agentId: \"main\",\n      match: { channel: \"discord\", accountId: \"default\" },\n    },\n    {\n      type: \"route\",\n      agentId: \"main\",\n      match: { channel: \"telegram\", accountId: \"default\" },\n    },\n  ],\n  channels: {\n    discord: {\n      guilds: {\n        \"111111111111111111\": {\n          channels: {\n            \"222222222222222222\": { requireMention: false },\n          },\n        },\n      },\n    },\n    telegram: {\n      groups: {\n        \"-1001234567890\": {\n          topics: { \"42\": { requireMention: false } },\n        },\n      },\n    },\n  },\n}\n```\n\n行为：\n\n- OpenClaw 确保配置的 ACP 会话在使用前存在。\n- 该频道或话题中的消息路由到配置的 ACP 会话。\n- 在绑定对话中，`/new` 和 `/reset` 会原地重置相同的 ACP 会话键。\n- 临时运行时绑定（例如由线程聚焦流程创建的）在存在时仍然适用。\n\n## 启动 ACP 会话（接口）\n\n### 从 `sessions_spawn`\n\n使用 `runtime: \"acp\"` 从 agent 轮次或工具调用启动 ACP 会话。\n\n```json\n{\n  \"task\": \"打开 repo 并总结失败的测试\",\n  \"runtime\": \"acp\",\n  \"agentId\": \"codex\",\n  \"thread\": true,\n  \"mode\": \"session\"\n}\n```\n\n说明：\n\n- `runtime` 默认为 `subagent`，因此请为 ACP 会话明确设置 `runtime: \"acp\"`。\n- 如果省略 `agentId`，OpenClaw 会在配置 `acp.defaultAgent` 时使用它。\n- `mode: \"session\"` 需要 `thread: true` 来保持持久绑定的对话。\n\n接口详情：\n\n- `task`（必需）：发送到 ACP 会话的初始提示。\n- `runtime`（ACP 必需）：必须是 `\"acp\"`。\n- `agentId`（可选）：ACP 目标线束 ID。如果设置了 `acp.defaultAgent`，则回退到它。\n- `thread`（可选，默认 `false`）：在支持的情况下请求线程绑定流程。\n- `mode`（可选）：`run`（一次性）或 `session`（持久性）。\n  - 默认是 `run`\n  - 如果 `thread: true` 且省略了 mode，OpenClaw 可能会根据运行时路径默认为持久性行为\n  - `mode: \"session\"` 需要 `thread: true`\n- `cwd`（可选）：请求的运行时工作目录（由后端/运行时策略验证）。\n- `label`（可选）：在会话/横幅文本中显示给操作员的标签。\n- `resumeSessionId`（可选）：恢复现有 ACP 会话而不是创建新会话。Agent 通过 `session/load` 重放其对话历史。需要 `runtime: \"acp\"`。\n- `streamTo`（可选）：`\"parent\"` 将初始 ACP 运行进度摘要作为系统事件流回请求者会话。\n  - 如果可用，接受的响应包括指向会话范围 JSONL 日志 (`<sessionId>.acp-stream.jsonl`) 的 `streamLogPath`，你可以 tail 以获取完整的 relay 历史记录。\n\n### 恢复现有会话\n\n使用 `resumeSessionId` 继续以前的 ACP 会话，而不是重新开始。Agent 通过 `session/load` 重放其对话历史，因此它会从之前完整的上下文中继续工作。\n\n```json\n{\n  \"task\": \"继续我们上次中断的地方 — 修复剩余的测试失败\",\n  \"runtime\": \"acp\",\n  \"agentId\": \"codex\",\n  \"resumeSessionId\": \"<previous-session-id>\"\n}\n```\n\n常见用例：\n\n- 将 Codex 会话从笔记本电脑切换到手机 — 告诉你的 agent 从上次中断的地方继续\n- 继续你在 CLI 中交互式启动的编码会话，现在通过 agent 以无头模式进行\n- 恢复因 Gateway 网关重启或空闲超时而中断的工作\n\n说明：\n\n- `resumeSessionId` 需要 `runtime: \"acp\"` — 如果与子 agent 运行时一起使用，会返回错误。\n- `resumeSessionId` 恢复上游 ACP 对话历史；`thread` 和 `mode` 仍然正常应用于你正在创建的新 OpenClaw 会话，因此 `mode: \"session\"` 仍然需要 `thread: true`。\n- 目标 agent 必须支持 `session/load`（Codex 和 Claude Code 支持）。\n- 如果未找到会话 ID，启动会话会失败并返回明确的错误 — 不会静默回退到新会话。\n\n### 操作员冒烟测试\n\n在 Gateway 网关部署后，如果你想快速实时检查 ACP 启动是否端到端正常工作，而不仅仅是通过单元测试，可以使用此测试。\n\n推荐的门槛：\n\n1. 验证目标主机上部署的 Gateway 网关版本/提交。\n2. 确认部署的源包含 `src/gateway/sessions-patch.ts` 中对 ACP lineage 的接受（`subagent:* 或 acp:* 会话`）。\n3. 打开一个临时 ACPX 桥接会话到活跃 agent（例如 `jpclawhq` 上的 `razor(main)`）。\n4. 要求该 agent 调用 `sessions_spawn`，参数为：\n   - `runtime: \"acp\"`\n   - `agentId: \"codex\"`\n   - `mode: \"run\"`\n   - task: `准确回复 LIVE-ACP-SPAWN-OK`\n5. 验证 agent 报告：\n   - `accepted=yes`\n   - 真实的 `childSessionKey`\n   - 没有验证器错误\n6. 清理临时 ACPX 桥接会话。\n\n给活跃 agent 的示例提示：\n\n```text\n立即使用 sessions_spawn 工具，runtime: \"acp\"，agentId: \"codex\"，mode: \"run\"。\n将任务设置为：\"准确回复 LIVE-ACP-SPAWN-OK\"。\n然后仅报告：accepted=<yes/no>; childSessionKey=<值或 none>; error=<确切文本或 none>。\n```\n\n说明：\n\n- 除非你特意测试线程绑定的持久性 ACP 会话，否则此冒烟测试请保持 `mode: \"run\"`。\n- 基本门槛不需要 `streamTo: \"parent\"`。该路径取决于请求者/会话能力，是单独的集成检查。\n- 将线程绑定的 `mode: \"session\"` 测试作为第二个、更丰富的集成测试，从真实的 Discord 线程或 Telegram 话题进行。\n\n## 沙箱兼容性\n\nACP 会话目前在主机运行时上运行，而不是在 OpenClaw 沙箱内部。\n\n当前限制：\n\n- 如果请求者会话是沙箱隔离的，ACP 启动会被阻止，无论是 `sessions_spawn({ runtime: \"acp\" })` 还是 `/acp spawn`。\n  - 错误：`沙箱隔离的会话无法启动 ACP 会话，因为 runtime=\"acp\" 在主机上运行。请从沙箱隔离的会话中使用 runtime=\"subagent\"。`\n- `sessions_spawn` 配合 `runtime: \"acp\"` 不支持 `sandbox: \"require\"`。\n  - 错误：`sessions_spawn sandbox=\"require\" 对于 runtime=\"acp\" 不支持，因为 ACP 会话在沙箱外部运行。请使用 runtime=\"subagent\" 或 sandbox=\"inherit\"。`\n\n当你需要沙箱强制执行时，请使用 `runtime: \"subagent\"`。\n\n### 从 `/acp` 命令\n\n需要时，使用 `/acp spawn` 从聊天中进行显式操作员控制。\n\n```text\n/acp spawn codex --mode persistent --thread auto\n/acp spawn codex --mode oneshot --thread off\n/acp spawn codex --thread here\n```\n\n关键标志：\n\n- `--mode persistent|oneshot`\n- `--thread auto|here|off`\n- `--cwd <absolute-path>`\n- `--label <name>`\n\n参见 [斜杠命令](/tools/slash-commands)。\n\n## 会话目标解析\n\n大多数 `/acp` 操作接受一个可选的会话目标（`session-key`、`session-id` 或 `session-label`）。\n\n解析顺序：\n\n1. 显式目标参数（或 `/acp steer` 的 `--session`）\n   - 尝试 key\n   - 然后是 UUID 格式的 session ID\n   - 然后是 label\n2. 当前线程绑定（如果此对话/线程绑定到 ACP 会话）\n3. 当前请求者会话回退\n\n如果无法解析目标，OpenClaw 会返回明确的错误（`无法解析会话目标：...`）。\n\n## 启动线程模式\n\n`/acp spawn` 支持 `--thread auto|here|off`。\n\n| 模式   | 行为                                                                                              | 备注                                                  |\n| ------ \| ------------------------------------------------------------------------------------------------- \| ----------------------------------------------------- |\n| `auto` | 在活跃线程中：绑定该线程。在线程外：在支持时创建/绑定子线程。                                     | 默认行为，在支持线程的频道中首选创建/绑定子线程。     |\n| `here` | 要求当前活跃线程；如果不在，则失败。                                                              | 仅在需要严格的线程上下文时使用。                      |\n| `off`  | 不绑定。会话启动时未绑定。                                                                        | 适用于不需要线程隔离的临时任务。                      |\n\n说明：\n\n- 在非线程绑定界面上，默认行为实际上是 `off`。\n- 线程绑定启动需要渠道策略支持：\n  - Discord: `channels.discord.threadBindings.spawnAcpSessions=true`\n  - Telegram: `channels.telegram.threadBindings.spawnAcpSessions=true`\n\n## ACP 控制\n\n可用命令系列：\n\n- `/acp spawn`\n- `/acp cancel`\n- `/acp steer`\n- `/acp close`\n- `/acp status`\n- `/acp set-mode`\n- `/acp set`\n- `/acp cwd`\n- `/acp permissions`\n- `/acp timeout`\n- `/acp model`\n- `/acp reset-options`\n- `/acp sessions`\n- `/acp doctor`\n- `/acp install`\n\n`/acp status` 显示有效的运行时选项，以及（如果可用）运行时级别和后端级别的会话标识符。\n\n某些控制取决于后端能力。如果后端不支持某个控制，OpenClaw 会返回明确的不支持控制错误。\n\n## ACP 命令手册\n\n| 命令              | 功能 `/acp model <id>` 映射到运行时配置键 `model`。\n- `/acp permissions <profile>` 映射到运行时配置键 `approval_policy`。\n- `/acp timeout <seconds>` 映射到运行时配置键 `timeout`。\n- `/acp cwd <path>` 直接更新运行时 cwd 覆盖。\n- `/acp set <key> <value>` 是通用路径。\n  - 特殊情况：`key=cwd` 使用 cwd 覆盖路径。\n- `/acp reset-options` 清除目标会话的所有运行时覆盖。\n\n## acpx 线束支持（当前）\n\n当前 acpx 内置线束别名：\n\n- `pi`\n- `claude`\n- `codex`\n- `opencode`\n- `gemini`\n- `kimi`\n\n当 OpenClaw 使用 acpx 后端时，除非你的 acpx 配置定义了自定义 agent 别名，否则请优先使用这些 `agentId` 值。\n\n直接的 acpx CLI 用法也可以通过 `--agent <command>` 定位任意适配器，但该原始逃生舱是 acpx CLI 的功能（不是正常的 OpenClaw `agentId` 路径）。\n\n## 必需配置\n\n核心 ACP 基线：\n\n```json5\n{\n  acp: {\n    enabled: true,\n    // 可选。默认 true；设置为 false 可暂停 ACP 分派，同时保留 /acp 控制。\n    dispatch: { enabled: true },\n    backend: \"acpx\",\n    defaultAgent: \"codex\",\n    allowedAgents: [\"pi\", \"claude\", \"codex\", \"opencode\", \"gemini\", \"kimi\"],\n    maxConcurrentSessions: 8,\n    stream: {\n      coalesceIdleMs: 300,\n      maxChunkChars: 权限配置\n\nACP 会话以非交互模式运行 — 没有 TTY 来批准或拒绝文件写入和 shell 执行权限提示。acpx 插件提供了两个配置键来控制如何处理权限：\n\n### `permissionMode`\n\n控制线束 agent 可以执行哪些操作而无需提示。\n\n| 值           | 行为                                                  |\n| ------------ \| ----------------------------------------------------- |\n| `approve-all` | 自动批准所有文件写入和 shell 命令。                  |\n| `approve-reads` | 仅自动批准读取；写入和执行需要提示。                 |\n| `deny-all`   | 拒绝所有权限提示。                                    |\n\n### `nonInteractivePermissions`\n\n控制当权限提示被显示但没有交互式 TTY 可用时（ACP 会话总是如此）会发生什么。\n\n| 值   | 行为                                                               |\n| ---- \| ------------------------------------------------------------------ |\n| `fail` | 使用 `AcpRuntimeError` 中止会话。**（默认）**                      |\n| `deny` | 静默拒绝权限并继续（优雅降级）。                                   |\n\n### 配置\n\n通过插件配置：\n\n```bash\nopenclaw config set plugins.entries.acpx.config.permissionMode approve-all\nopenclaw config set plugins.entries.acpx.config.nonInteractivePermissions fail\n```\n\n更改这些值后重启 Gateway 网关。\n\n> **重要提示：** OpenClaw 目前默认 `permissionMode=approve-reads` 和 `nonInteractivePermissions=fail`。在非交互式 ACP 会话中，任何触发权限提示的写入或执行都可能失败并出现 `AcpRuntimeError: Permission prompt unavailable in non-interactive mode` 错误。\n>\n> 如果你需要限制权限，请将 `nonInteractivePermissions` 设置为 `deny`，以便会话优雅降级而不是崩溃。\n\n## 故障排除\n\n| 症状
