# BTW 附带问题

`/btw` 让您可以对**当前会话**提出快速的附带问题，而不会将该问题变成正常对话历史。

它以 Claude Code 的 `/btw` 行为为模型，但适配了 OpenClaw 的 Gateway 和多频道架构。

## 它做什么

当您发送：

```text
/btw what changed?
```

OpenClaw：

1. 快照当前会话上下文，
2. 运行一个单独的**无工具**模型调用，
3. 仅回答附带问题，
4. 让主运行保持不变，
5. **不会**将 BTW 问题或答案写入会话历史，
6. 将答案作为**实时附带结果**发出，而不是正常的助手消息。

重要的心智模型是：

- 相同的会话上下文
- 独立的单次附带查询
- 无工具调用
- 无未来上下文污染
- 无转录持久化

## 它不做什么

`/btw` **不会**：

- 创建新的持久会话，
- 继续未完成的主任务，
- 运行工具或代理工具循环，
- 将 BTW 问题/答案数据写入转录历史，
- 出现在 `chat.history` 中，
- 在重新加载后存活。

它是有意的**临时性**。

## 上下文如何工作

BTW 仅将当前会话用作**后台上下文**。

如果主运行当前处于活动状态，OpenClaw 会快照当前消息状态，并将进行中的主提示作为后台上下文包含，同时明确告诉模型：

- 仅回答附带问题，
- 不恢复或完成未完成的主任务，
- 不发出工具调用或伪工具调用。

这使 BTW 与主运行隔离，同时仍使其知道会话的内容。

## 传递模型

BTW **不是**作为正常助手转录消息传递的。

在 Gateway 协议级别：

- 正常助手聊天使用 `chat` 事件
- BTW 使用 `chat.side_result` 事件

这种分离是有意的。如果 BTW 重用正常的 `chat` 事件路径，客户端会将其视为正常对话历史。

因为 BTW 使用单独的实时事件且不会从 `chat.history` 重放，所以在重新加载后会消失。

## 表面行为

### TUI

在 TUI 中，BTW 在当前会话视图中以内联方式呈现，但它仍然是临时的：

- 与正常助手回复明显不同
- 可用 `Enter` 或 `Esc` 关闭
- 在重新加载时不重放

### 外部频道

在 Telegram、WhatsApp 和 Discord 等频道上，BTW 作为明确标记的一次性回复传递，因为这些平台没有本地临时覆盖概念。

答案仍然被视为附带结果，而不是正常会话历史。

### Control UI / Web

Gateway 正确地将 BTW 作为 `chat.side_result` 发出，并且 BTW 不包含在 `chat.history` 中，因此持久化合同已经对 Web 正确。

当前的 Control UI 仍然需要一个专用的 `chat.side_result` 消费者来在浏览器中呈现 BTW 实时状态。在该客户端支持落地之前，BTW 是一个具有完整 TUI 和外部频道行为的 Gateway 级别功能，但还不是完整的浏览器 UX。

## 何时使用 BTW

在以下情况下使用 `/btw`：

- 想快速澄清当前工作，
- 在长时间运行仍在进行时获取事实性附带答案，
- 想获得不应成为未来会话上下文一部分的临时答案。

示例：

```text
/btw what file are we editing?
/btw what does this error mean?
/btw summarize the current task in one sentence
/btw what is 17 * 19?
```

## 何时不使用 BTW

当您希望答案成为会话未来工作上下文的一部分时，不要使用 `/btw`。

在这种情况下，在主会话中正常提问，而不是使用 BTW。
