# BTW 边问题

`/btw` 让你可以就**当前会话**问一个快速的边问题，而不会将该问题变成正常的对话历史。

它模仿了 Claude Code 的 `/btw` 行为，但适应了 OpenClaw 的 Gateway 和多通道架构。

## 它的作用

当你发送：

```text
/btw what changed?
```

OpenClaw：

1. 快照当前会话上下文，
2. 运行单独的**无工具**模型调用，
3. 仅回答边问题，
4. 不干扰主运行，
5. **不**将 BTW 问题或答案写入会话历史，
6. 将答案作为**实时边结果**而不是正常的助手消息发出。

重要的心智模型是：

- 相同的会话上下文
- 单独的一次性边查询
- 无工具调用
- 无未来上下文污染
- 无转录持久化

## 它不做什么

`/btw` **不**：

- 创建新的持久会话，
- 继续未完成的主要任务，
- 运行工具或代理工具循环，
- 将 BTW 问题/答案数据写入转录历史，
- 出现在 `chat.history` 中，
- 在重新加载后保留。

它有意设计为**临时的**。

## 上下文如何工作

BTW 仅将当前会话用作**背景上下文**。

如果主运行当前处于活动状态，OpenClaw 会快照当前消息状态并将进行中的主提示作为背景上下文包含在内，同时明确告诉模型：

- 仅回答边问题，
- 不要恢复或完成未完成的主要任务，
- 不要发出工具调用或伪工具调用。

这使得 BTW 与主运行隔离，同时仍然了解会话的内容。

## 交付模型

BTW **不是**作为正常的助手转录消息交付的。

在 Gateway 协议级别：

- 正常助手聊天使用 `chat` 事件
- BTW 使用 `chat.side_result` 事件

这种分离是有意的。如果 BTW 重用正常的 `chat` 事件路径，客户端会将其视为常规对话历史。

因为 BTW 使用单独的实时事件且不会从 `chat.history` 重播，所以它在重新加载后会消失。

## 表面行为

### TUI

在 TUI 中，BTW 在内联显示在当前会话视图中，但它仍然是临时的：

- 与正常助手回复明显不同
- 可通过 `Enter` 或 `Esc`  dismiss
- 重新加载时不会重播

### 外部通道

在 Telegram、WhatsApp 和 Discord 等通道上，BTW 作为明确标记的一次性回复交付，因为这些表面没有本地临时覆盖概念。

答案仍然被视为边结果，而不是正常的会话历史。

### Control UI / 网页

Gateway 正确地将 BTW 作为 `chat.side_result` 发出，并且 BTW 不包含在 `chat.history` 中，因此网页的持久化契约已经正确。

当前的 Control UI 仍然需要一个专用的 `chat.side_result` 消费者来在浏览器中实时渲染 BTW。在客户端支持落地之前，BTW 是一个 Gateway 级别的功能，具有完整的 TUI 和外部通道行为，但尚未成为完整的浏览器用户体验。

## 何时使用 BTW

当你想要以下内容时使用 `/btw`：

- 关于当前工作的快速澄清，
- 长时间运行仍在进行时的事实边答案，
- 不应成为未来会话上下文一部分的临时答案。

例子：

```text
/btw what file are we editing?
/btw what does this error mean?
/btw summarize the current task in one sentence
/btw what is 17 * 19?
```

## 何时不使用 BTW

当你希望答案成为会话未来工作上下文的一部分时，不要使用 `/btw`。

在这种情况下，在主会话中正常提问，而不是使用 BTW。
