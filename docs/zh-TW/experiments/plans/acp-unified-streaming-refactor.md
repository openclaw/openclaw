# 統一執行時串流重構計畫

## 目標

提供一個共享的串流管道給 `main`、`subagent` 和 `acp`，以確保所有執行時獲得相同的合併、分塊、交付順序和崩潰恢復行為。

## 為什麼會有這個存在

- 當前行為分散在多個特定於執行時的形狀路徑中。
- 格式化/合併的錯誤可以在一條路徑中修正，但在其他路徑中仍然存在。
- 交付一致性、重複抑制和恢復語義更難以推理。

## 目標架構

單一管道，特定於執行時的適配器：

1. 執行時適配器僅發出標準事件。
2. 共享串流組合器合併並最終確定文本/工具/狀態事件。
3. 共享通道投影器僅應用一次通道特定的分塊/格式化。
4. 共享交付帳本強制執行冪等發送/重播語義。
5. 出站通道適配器執行發送並記錄交付檢查點。

## Workstreams

### 1) 典範串流合約

- 定義嚴格的事件架構 + 在核心中進行驗證。
- 添加適配器合約測試，以保證每個執行時發出相容的事件。
- 及早拒絕格式不正確的執行時事件並顯示結構化的診斷資訊。

### 2) 共享串流處理器

- 將特定於執行時的合併器/投影器邏輯替換為一個處理器。
- 處理器擁有文本增量緩衝、閒置刷新、最大區塊拆分和完成刷新。
- 將 ACP/main/subagent 設定解析移入一個輔助函數，以防止漂移。

### 3) 共享頻道投影

- 保持通道適配器簡單：接受已完成的區塊並發送。
- 將 Discord 特定的分塊特性僅移至通道投影器。
- 在投影之前保持管道與通道無關。

### 4) 交付帳本 + 重播

- 添加每回合/每區塊的交付 ID。
- 在實際發送前後記錄檢查點。
- 在重新啟動時，重播待處理的區塊，確保操作的冪等性並避免重複。

### 5) 遷移與切換

- 階段 1：影子模式（新管道計算輸出但舊路徑發送；進行比較）。
- 階段 2：逐個執行時切換 (`acp`，然後 `subagent`，然後 `main` 或根據風險反向切換)。
- 階段 3：刪除舊有執行時特定的串流程式碼。

## 非目標

- 此次重構不會更改 ACP 政策/權限模型。
- 除了投影相容性修正外，不會擴充特定於通道的功能。
- 不會重新設計傳輸/後端（acpx 插件合約保持不變，除非需要為事件一致性進行調整）。

## 風險與緩解措施

- 風險：現有主/子代理路徑中的行為回歸。
  緩解措施：影子模式差異比較 + 適配器合約測試 + 通道端到端測試。
- 風險：崩潰恢復期間的重複發送。
  緩解措施：持久交付 ID + 在交付適配器中的冪等重放。
- 風險：執行時適配器再次發散。
  緩解措施：所有適配器所需的共享合約測試套件。

## 接受標準

- 所有執行環境都通過共享串流合約測試。
- Discord ACP/main/subagent 對於微小的增量產生等效的間距/分塊行為。
- 崩潰/重啟重播不會對相同的交付 ID 發送重複的分塊。
- 移除了舊版 ACP 投影儀/合併器路徑。
- 串流設定解析是共享的，且與執行環境無關。
