# OpenClaw 繁體中文文件自動化翻譯：開發與進化歷程全紀錄

## 📋 專案概述

為了解決 OpenClaw 專案 353 份技術文檔的在地化維護問題，我們開發了一套具備「工業級韌性」與「自我修復能力」的自動化翻譯管線。本文檔紀錄了開發過程中的重大技術挑戰與核心決策。

---

## 🛠️ 技術進化里程碑

### 1. 初始挑戰：AI 幻覺與格式破壞

**問題**：AI 經常錯誤翻譯程式碼、變數，並自行添加反引號 (` `)。  
**決策**：導入 **「物理性遮罩 (Mechanical Masking)」**。  
**效果**：將技術內容替換為 `[[BLOCK_N]]` 標籤，AI 只能看到純文字，從根本上杜絕了對程式碼的竄改。

### 2. 語法適應：順序錯位問題

**問題**：中英詞序不同，AI 調整標籤順序導致嚴格的正則驗證失敗。  
**決策**：開發 **「順序無關的集合驗證 (Order-Agnostic Set Validation)」**。  
**效果**：只要段落內的標籤內容集合正確，即允許語意化的順序調整，提升了翻譯的自然度。

### 3. 長文處理：上下文漂移

**問題**：文件過長時，AI 會產生 Context 混亂，開始胡亂分配標籤。  
**決策**：實作 **「原子化段落翻譯 (Atomic Paragraph Translation)」**。  
**效果**：將 Markdown 拆分為獨立段落處理，確保 AI 每次只處理極短的上下文，精準度提升至近 100%。

### 4. 規模化瓶頸：API 配額與 RPD 限制

**問題**：原子化請求導致極速撞上 OpenAI 每日 10,000 次請求的限制 (429 RPD)。  
**決策**：開發 **「批次聚合與模型感應 (Batch Partitioning)」**。  
**效果**：每 10-15 段聚合為一次 API 呼叫，消耗配額降低 80% 以上。

### 5. 穩定性巔峰：不死重試模式 (Self-healing)

**問題**：批次翻譯中若一段損毀，會導致整批失敗。  
**決策**：實作 **「逐級降級重試機制」**。  
**效果**：當批次驗證失敗，系統自動將該批次拆解為單段模式逐一攻克。這讓系統具備了「自動修復」能力，實現了真正的無人值守。

## 💡 核心工程哲學

**「不試圖訓練 AI 變聰明，而是透過系統工程建立防線。」**

我們將這套哲學總結為五大支柱：

1.  **Masking (遮罩)**：資料完整性。
2.  **Atomic (原子化)**：處理精確度。
3.  **Sanitization (清洗)**：排除格式干擾。
4.  **Batching (聚合)**：效能與配額管理。
5.  **Self-healing (自癒)**：任務達成保障。
