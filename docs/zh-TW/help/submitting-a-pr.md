好的 PR 易於審核：審核者應迅速了解其意圖、驗證行為並安全地提交變更。本指南涵蓋針對人工和 LLM 審核的簡潔、高訊號提交。

## 什麼是好的 PR

- [ ] 解釋問題、重要性以及所做的變更。
- [ ] 保持變更聚焦。避免大規模重構。
- [ ] 總結使用者可見/設定/預設的變更。
- [ ] 列出測試覆蓋率、跳過項目及其原因。
- [ ] 添加證據：日誌、螢幕截圖或錄影（UI/UX）。
- [ ] 通關密語：如果您閱讀了本指南，請在 PR 說明中加入「lobster-biscuit」。
- [ ] 在建立 PR 之前執行/修復相關的 `pnpm` 命令。
- [ ] 搜尋程式碼庫和 GitHub 以尋找相關功能/問題/修復。
- [ ] 根據證據或觀察提出主張。
- [ ] 好的標題：動詞 + 範圍 + 結果（例如 `Docs: add PR and issue templates`）。

請保持簡潔；簡潔的審核 > 文法。省略任何不適用的部分。

### 基本驗證命令（針對您的變更執行/修復失敗）

- `pnpm lint`
- `pnpm check`
- `pnpm build`
- `pnpm test`
- 協定變更：`pnpm protocol:check`

## 漸進式揭露

- 頂部：摘要/意圖
- 接下來：變更/風險
- 接下來：測試/驗證
- 最後：實作/證據

## 常見 PR 類型：具體說明

- [ ] 修復：添加重現步驟、根本原因、驗證。
- [ ] 功能：添加使用案例、行為/演示/螢幕截圖（UI）。
- [ ] 重構：說明「無行為變更」，列出移動/簡化的內容。
- [ ] 雜項：說明原因（例如，建置時間、CI、依賴項）。
- [ ] 文件：更新前/更新後情境，連結更新的頁面，執行 `pnpm format`。
- [ ] 測試：涵蓋了哪些空白；如何防止回歸。
- [ ] 效能：添加前後指標以及測量方式。
- [ ] UX/UI：螢幕截圖/影片，注意輔助功能影響。
- [ ] 基礎設施/建置：環境/驗證。
- [ ] 安全性：總結風險、重現步驟、驗證，不含敏感資料。僅限有根據的主張。

## 檢查清單

- [ ] 問題/意圖清晰
- [ ] 範圍聚焦
- [ ] 列出行為變更
- [ ] 測試列表和結果
- [ ] 手動測試步驟（如適用）
- [ ] 無機密/私人資料
- [ ] 有證據支持
