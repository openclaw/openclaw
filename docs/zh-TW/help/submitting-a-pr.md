好的 PR 易於審閱：審閱者應能快速了解意圖、驗證行為，並安全地合併變更。本指南涵蓋了適合人類與 LLM 審閱的簡潔、高品質提交方式。

## 什麼是好的 PR

- [ ] 解釋問題、其重要性以及變更內容。
- [ ] 保持變更集中。避免大規模重構。
- [ ] 摘要使用者可見的變更、Skills 設定或預設值變更。
- [ ] 列出測試覆蓋範圍、跳過的項目及其原因。
- [ ] 增加證據：日誌、螢幕截圖或錄影 (UI/UX)。
- [ ] 暗號：如果你閱讀了本指南，請在 PR 說明中加入「lobster-biscuit」。
- [ ] 在建立 PR 前執行/修復相關的 `pnpm` 指令。
- [ ] 搜尋程式碼庫和 GitHub 以尋找相關功能、問題或修復。
- [ ] 基於證據或觀察提出主張。
- [ ] 好的標題：動詞 + 範圍 + 結果 (例如：`Docs: add PR and issue templates`)。

請保持簡潔；簡潔的審閱比語法更重要。省略任何不適用的章節。

### 基準驗證指令 (針對你的變更執行/修復失敗項目)

- `pnpm lint`
- `pnpm check`
- `pnpm build`
- `pnpm test`
- 通訊協定變更：`pnpm protocol:check`

## 漸進式呈現

- 頂部：摘要/意圖
- 次之：變更/風險
- 再次：測試/驗證
- 最後：實作/證據

## 常見 PR 類型：細節

- [ ] Fix (修復)：加入重現步驟、根本原因、驗證。
- [ ] Feature (功能)：加入使用場景、行為/示範/螢幕截圖 (UI)。
- [ ] Refactor (重構)：聲明「無行為變更」，列出移動或簡化的部分。
- [ ] Chore (例行任務)：說明原因 (例如：建置時間、CI、依賴項目)。
- [ ] Docs (文件)：前後文對比、連結更新頁面、執行 `pnpm format`。
- [ ] Test (測試)：補足了什麼缺口；如何防止迴歸。
- [ ] Perf (效能)：加入前後指標對比，以及衡量方式。
- [ ] UX/UI：螢幕截圖/影片，註記對無障礙環境的影響。
- [ ] Infra/Build (基礎架構/建置)：環境/驗證。
- [ ] Security (安全)：摘要風險、重現步驟、驗證、無敏感資料。僅提出有根據的主張。

## 核對表

- [ ] 清晰的問題/意圖
- [ ] 專注的範圍
- [ ] 列出行為變更
- [ ] 測試列表與結果
- [ ] 手動測試步驟 (適用時)
- [ ] 無秘密/私有資料
- [ ] 基於證據
