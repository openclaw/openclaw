# OpenClaw 仓库部件与多 Agent 链路全景指南

这是一份面向**第一次接触 OpenClaw 的开发者/运维者**的导览文档。目标是让你能够从仓库结构、核心模块职责、请求在系统中的流转、到部署落地步骤，建立一条完整的心智链路。

> 读法建议：先看“整体架构一图流”和“端到端链路”，再回到“模块与关键文件索引”按需深入。

## 一、OpenClaw 在做什么（先建立产品边界）

OpenClaw 是一个本地优先（local-first）的多通道 AI 助手系统：

- 你通过 Telegram/Slack/Discord/WhatsApp 等外部渠道或本地入口发消息。
- Gateway 作为中枢接收消息、做路由与策略决策。
- Agent 运行时负责会话上下文、模型调用、工具调用、多 Agent 委派。
- Plugins/Channels 把不同外部生态接进统一的运行时契约。
- UI / Apps / Docs / Scripts 负责可视化、辅助调试与工程化交付。

## 二、整体架构一图流（按职责分层）

### 1) 入口与协议层

- `src/gateway/`：Gateway 服务主逻辑、生命周期、管理与路由。
- `src/gateway/protocol/`：网关协议与 RPC 契约，定义“外部如何调用 Gateway”。
- `src/channels/`：各消息渠道的统一接入实现（收发、身份、权限边界）。

### 2) Agent 编排层

- `src/agents/`：多 Agent 的核心执行与编排。
- `src/agents/tools/`：Agent 可调用工具的实现与工具面定义。
- `src/plugins/`：插件发现、注册、加载与运行时桥接。
- `src/plugin-sdk/`：提供给插件作者的公共 SDK 边界（避免插件直接耦合 core 内部）。

### 3) 插件与扩展层

- `extensions/`：内置/官方插件目录（如 Telegram 等扩展）。
- `packages/`：可复用包、底层库与共享能力。

### 4) 表层体验与交付层

- `ui/`：前端界面相关能力。
- `apps/`：平台应用（如桌面/移动配套）。
- `docs/`：产品与开发文档。
- `scripts/`：构建、验证、自动化脚本。

## 三、模块与关键文件索引（按“你要改什么”组织）

> 说明：OpenClaw 仓库规模大，不建议按“从上到下读完整个仓库”。下面按模块给出建议入口，帮助快速定位。

### A. 仓库根与工程入口

- `README.md`：产品定位、快速启动、主要能力地图。
- `package.json`：workspace 顶层脚本与开发命令入口。
- `pnpm-workspace.yaml`：多包工作区定义。
- `tsconfig*.json`：TypeScript 工程分层配置。
- `vitest.config.ts`：测试框架总配置。
- `openclaw.mjs`：CLI/runtime 入口桥接文件。

### B. Gateway（控制平面）

- `src/gateway/`：
  - 负责启动、配置装载、路由入口、与 Agent/Channel/Plugin 交互。
  - 这里是“消息进入系统后的第一站”以及“运维控制接口”汇聚点。
- `src/gateway/protocol/`：
  - 定义 RPC 与消息协议，保证 CLI、UI、外部客户端和 Gateway 的契约一致。
- `src/gateway/server-methods/`：
  - Gateway 对外提供的方法集合（可理解为“服务端 API 实现层”）。

### C. Channels（外部消息通道接入）

- `src/channels/`：
  - 渠道适配抽象：把 Telegram/Slack/Discord 等的消息模型归一。
  - 处理入站消息格式化、鉴权策略、回执发送、状态同步。

### D. Agents（多 Agent 运行时与编排）

- `src/agents/`：
  - Agent 生命周期、会话态、上下文拼装、模型调用编排。
  - 多 Agent 场景中，主 Agent 与子 Agent 的职责拆分、委托和结果汇总在此完成。
- `src/agents/tools/`：
  - 工具定义与执行层（例如 shell、文件、外部动作等）。
  - Agent 的“行动能力”在这里被暴露为可调用工具接口。

### E. Plugins 与 SDK（扩展机制核心）

- `src/plugins/`：
  - 插件注册表、发现/加载、运行时注入。
  - 保障 core 与具体插件实现解耦。
- `src/plugin-sdk/`：
  - 插件开发者使用的稳定边界与类型契约。
  - 新能力优先通过 SDK 暴露，避免插件绕过边界调用 core 私有细节。
- `extensions/`：
  - 实际插件实现（包括 channel 相关插件）。
  - 每个插件通常包含自身配置、能力实现、测试与文档。

### F. UI / Apps / 配套

- `ui/`：
  - 管理界面、状态展示、交互控制。
- `apps/`：
  - 平台端应用工程，承载设备能力、可视控制与人机交互入口。

### G. 工程辅助与质量体系

- `scripts/`：
  - 测试包装器、CI 辅助、构建与发布脚本。
- `test/`：
  - 集成测试与测试辅助。
- `docs/`：
  - 公开文档源，部署到 `docs.openclaw.ai`。

## 四、从“收到一条消息”到“多 Agent 完成任务”的完整链路

下面按时间顺序描述一次典型请求生命周期（以聊天渠道消息为例）：

1. **用户消息进入渠道插件/通道适配层**  
   Channel 适配器监听到入站事件，完成来源校验、必要清洗、字段归一。

2. **Gateway 接收标准化事件并做路由决策**  
   Gateway 根据配置（账号、channel、路由规则、会话策略）将请求分配给目标 Agent。

3. **Agent 会话构建与上下文装配**  
   Agent runtime 拉取会话历史、系统提示词、策略开关、可用工具集，形成本轮执行上下文。

4. **主 Agent 规划（Plan）与工具/子 Agent 选择**  
   - 若任务简单：主 Agent 直接调用模型与必要工具。
   - 若任务复杂：主 Agent 会把可并行、边界清晰的子任务分派给子 Agent（delegate/spawn）。

5. **子 Agent 执行并回传结构化结果**  
   子 Agent 在自己的上下文里完成局部任务，返回可整合结果（结论、补丁、证据）。

6. **主 Agent 汇总、校验、形成最终答复**  
   主 Agent 合并子结果，做冲突消解与质量检查，产出最终响应与可追踪执行痕迹。

7. **Gateway 将结果回传到原渠道**  
   Gateway 通过对应 channel adapter 发送回复；必要时附带状态事件、日志或工具执行信息。

8. **会话与审计信息落盘/缓存**  
   便于下一轮连续对话、问题复盘、线上排障与运营治理。

## 五、多 Agent 工作流的关键机制（你需要真正理解的 6 点）

1. **上下文隔离**：主/子 Agent 有各自任务边界，避免互相污染上下文。  
2. **工具边界**：不同 Agent 可暴露不同工具权限，减少误操作风险。  
3. **并行收益**：信息检索、静态分析、独立代码块改动可并发执行。  
4. **主控合流**：最终一致性由主 Agent 负责，不把“整体验证”外包给子 Agent。  
5. **失败可恢复**：子任务失败可重试/替换策略，不必回滚整条流水线。  
6. **可观测性**：每步都应有可检查证据（日志、测试、命令输出、变更文件）。

## 六、部署步骤要点（面向第一次落地）

> 这里给的是“可靠首跑路径”，不是所有高级场景配置全集。

### 1) 运行前提

- Node 版本：推荐 Node 24（至少满足仓库要求）。
- 包管理：使用仓库默认的 `pnpm` 工作流。

### 2) 安装依赖

```bash
pnpm install
```

若遇到依赖构建/平台库问题，优先按仓库文档与脚本提示排查，不要随意替换包管理器或手改锁文件策略。

### 3) 本地开发启动

```bash
pnpm dev
# 或根据场景使用 pnpm openclaw ... 子命令
```

### 4) 首次配置与连接

- 使用 onboard/onboarding 流程完成基础配置（账户、通道、模型提供商认证等）。
- 最小化先接一个渠道做端到端验证，再逐步扩展更多渠道与工具。

### 5) 健康检查与验证

- 小范围改动：先跑针对性测试/检查。
- 变更跨模块时：增加构建与更广测试验证。
- 重点确认：消息入站、路由、Agent 执行、回复出站四段都能打通。

### 6) 生产化注意点

- 明确 DM/allowlist/pairing 等安全策略默认值。
- 对非主会话开启 sandbox 策略，避免高风险工具暴露。
- 保证日志与监控可追踪“某条消息到底经过了哪些模块和决策”。

## 七、给新同学的“阅读顺序建议”（1~2 天入门）

### Day 1：先建立宏观地图

1. `README.md`（产品、能力、快速入口）
2. `docs/` 中 architecture / gateway / channels / tools 相关主题
3. `package.json` + 根配置文件（理解命令与工程结构）

### Day 2：按链路解剖

1. `src/gateway/` 与 `src/gateway/protocol/`
2. `src/channels/`
3. `src/agents/` 与 `src/agents/tools/`
4. `src/plugins/` + `src/plugin-sdk/` + `extensions/`

完成后，建议手动做一次“从某渠道发消息 -> 到收到回复”的完整 trace。

## 八、常见误区（避免踩坑）

- 把 core 与插件边界打穿：会破坏可维护性与升级路径。
- 直接在多个层重复做同一种路由/解析：会导致行为漂移。
- 没有区分“主 Agent 负责合流”与“子 Agent 负责局部任务”：会让结果不可控。
- 只看单元测试，不做端到端验证：上线后最容易出现链路断点。
