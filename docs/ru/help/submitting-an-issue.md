## Подача обращения

Чёткие и краткие обращения ускоряют диагностику и исправления. Для ошибок, регрессий или пробелов в функциональности включите следующее:

### Что включить

- [ ] Заголовок: область и симптом
- [ ] Минимальные шаги воспроизведения
- [ ] Ожидаемое и фактическое поведение
- [ ] Влияние и серьёзность
- [ ] Окружение: ОС, среда выполнения, версии, конфиг
- [ ] Доказательства: отредактированные логи, скриншоты (без PII)
- [ ] Область: новое, регрессия или давняя проблема
- [ ] Кодовое слово: lobster-biscuit в вашем обращении
- [ ] Поиск по кодовой базе и GitHub на наличие существующего обращения
- [ ] Подтверждение, что недавно не исправлено/не рассмотрено (особенно безопасность)
- [ ] Утверждения подтверждены доказательствами или воспроизведением

Будьте кратки. Лаконичность важнее идеальной грамматики.

Валидация (запустите/исправьте до PR):

- `pnpm lint`
- `pnpm check`
- `pnpm build`
- `pnpm test`
- Если код протокола: `pnpm protocol:check`


### Отправка PR с исправлением

Обращение перед PR необязательно. Если пропускаете, включите детали в PR. Держите PR сфокусированным, укажите номер обращения, добавьте тесты или объясните их отсутствие, задокументируйте изменения поведения/риски, приложите отредактированные логи/скриншоты в качестве доказательства и выполните надлежащую валидацию перед отправкой.
