# A2A Dual-Channel Relay Plan (Human-Chat UX)

Status: proposed (implementation-ready)
Owner: dev-openclaw
Scope: make inter-agent `sessions_send` conversations visible in Slack as if two humans are chatting across channels.

## Goal

When agent A talks to agent B via `sessions_send`, both agent channels should see a readable conversation transcript in near real time:

- A channel sees A->B request and B->A reply turns.
- B channel sees A->B request and B->A reply turns.
- Messages look human-chat-like (speaker + text), not raw internal tool JSON.

## Current behavior (baseline)

- `ingressEcho` can post the **initial request** to the target channel before run.
- Round-1 reply returns to caller tool result (`reply`) but is not auto-posted to source channel.
- A2A announce step posts only to target channel.
- Ping-pong steps run with `deliver:false` and are not mirrored to channels.

So current UX is target-side visibility, not bilateral conversation visibility.

## Definition of success

For one A2A handoff (`handoff.id`):

1. Every conversational turn (request + replies) is mirrored to both source and target Slack channels.
2. Messages include clear speaker identity (agent id/name) and direction.
3. Ordering is preserved and deterministic.
4. Existing installs remain unchanged unless feature is enabled.
5. Loop/cost guardrails still apply (`maxPingPongTurns`, nested relay guard).

## Non-goals (first iteration)

- Cross-provider parity polish for every chat platform.
- Rich UI blocks/attachments beyond plain text.
- Replacing native `sessions_send` execution with channel transport.

## Proposed design

## 1) Add relay config under `session.agentToAgent`

```yaml
session:
  agentToAgent:
    relay:
      enabled: false
      mode: target-only # target-only | dual-channel
      mirrorTurns: all # round1 | all
      requireDelivery: false
```

Semantics:

- `enabled=false`: existing behavior.
- `mode=target-only`: close to today (safe default).
- `mode=dual-channel`: mirror turns to both requester+target channels.
- `mirrorTurns=round1`: only initial request + first reply.
- `mirrorTurns=all`: include ping-pong turns.
- `requireDelivery=true`: strict mode can block/abort when required relay delivery fails.

## 2) Resolve both announce targets up front

In `sessions_send` flow, resolve:

- target channel target (already exists), and
- source channel target (from requester session key/channel context).

Persist these in a relay context object for the entire run.

## 3) Relay concrete turns, not synthetic summaries

Mirror actual turn texts:

- Turn 0: requester message (`A -> B`)
- Turn 1+: model replies generated by each agent in ping-pong flow

Do **not** depend on the current post-run “announce prompt” text generation for primary transcript. Keep announce step optional/compat mode.

## 4) Standard relay message format

Use deterministic plain text:

```text
[A2A handoff:<id>] gpod -> dev-googleworkspacecli
<message body>
```

and

```text
[A2A handoff:<id>] dev-googleworkspacecli -> gpod
<reply body>
```

This is human-readable while still traceable.

## 5) Delivery + idempotency

- Each mirrored send gets idempotency key derived from `<runId>:<turn>:<target>`.
- Best-effort by default; strict behavior only if configured.
- Never mirror if target/source channel cannot be resolved and strict mode is off; record status in result metadata.

## Test plan

## Unit + integration

1. **Dual relay round1 success**
   - request mirrored to both channels
   - first reply mirrored to both channels
2. **Dual relay all turns**
   - ping-pong replies mirrored in order
3. **Best-effort delivery failure**
   - one mirror send fails, run still completes
   - status metadata reflects partial relay failure
4. **Strict delivery failure**
   - required mirror failure blocks/aborts as configured
5. **Backward compatibility**
   - relay disabled => existing ingress/announce behavior unchanged
6. **No source target resolvable**
   - target relay still works in best-effort mode

## Rollout plan

1. Land code with `relay.enabled=false` default.
2. Enable for controlled pair (`gpod` <-> `dev-googleworkspacecli`) in staging config.
3. Validate in live Slack with bounded ping-pong turns.
4. Expand to other agent pairs after confirming no duplication/loop regressions.

## Open questions (for implementation PR)

- Should existing target-side announce step remain enabled in dual-channel mode, or be disabled to avoid duplicate target posts?
- Should source-channel relays always post in the original thread when requester session is thread-bound?
- Do we want per-agent/channel relay allowlists in config for gradual rollout?

## Recommended first PR scope

- Config schema (`relay.*`)
- Dual target resolution (source + target)
- Turn-by-turn mirror posting for request + round1 reply
- Tests + docs

Second PR can add full ping-pong `mirrorTurns=all` and announce-step de-duplication behavior.
