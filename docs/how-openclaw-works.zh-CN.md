# OpenClaw 工作原理 — 技术概览

本文分为两个层面。第一部分（下方）是面向所有希望了解 OpenClaw 功能及其整体架构的读者的高层概览。第二至第六部分则深入探讨：部署与安装、功能与特性、内部架构、性能与可靠性，以及与其他方案的对比。

## 什么是 OpenClaw

OpenClaw 是一个运行在本地机器上的个人 AI 助手，以一个名为 Gateway 的后台常驻进程运行。它连接你日常使用的消息渠道——WhatsApp、Telegram、Slack、Discord、Signal、iMessage、Feishu/飞书等——以及 macOS、iOS 和 Android 上的配套应用。你可以像给朋友发消息一样与它交流：提问、分配任务或进行对话。在幕后，它使用大语言模型（LLM）进行思考，通过工具在你的电脑上执行操作，然后通过你发送消息的同一渠道回复你。由于 Gateway 运行在你自己的硬件上，所有数据——对话记录、记忆文件和工具输出——都保留在你的机器上。

## Agent 工作流

当你从任意渠道发送消息时，Gateway 接收并将其路由到对应的会话。会话是串行化的，同一时间只处理一个轮次，避免竞争条件或操作冲突。Agent 运行时随后加载你的工作区上下文（人格设定、操作指令、记忆）以及会话历史，并将所有内容交给配置的语言模型。模型生成文本、工具调用或两者兼有。工具调用在你的电脑上本地执行，执行结果反馈给模型以进行进一步推理。这个循环不断重复，直到模型生成最终的文本回复，回复通过 Gateway 流式传输回渠道并送达你手中。

Gateway 是一个长期运行的守护进程，管理所有渠道连接和会话状态。它在后台持续运行——通常以 macOS 菜单栏应用或 Linux 系统服务的形式——因此无论你打开哪个消息应用，助手都随时可达。

```
  你
   |
   v
消息渠道 (WhatsApp, Telegram, Slack, ...)
   |
   v
Gateway (路由 + 串行化)
   |
   v
Agent 运行时
   |
   v
LLM 推理  <──>  工具执行
   |                (shell, 文件,
   v                 浏览器, ...)
回复流式返回
   |
   v
消息渠道 --> 你
```

## 记忆与上下文

OpenClaw 的记忆系统建立在三个层次之上。第一层是**引导文件**——Agent 在每次会话开始时从工作区加载的纯 Markdown 文件。包括人格与语气文件（SOUL.md）、操作指令（AGENTS.md）、用户档案（USER.md）以及经过整理的长期记忆（MEMORY.md）。这些文件在对话开始之前，就赋予了 Agent 身份认知、用户偏好和积累的知识。

第二层是**会话历史**。每次对话都以文本记录的形式存储在磁盘上。当会话内容超出模型的上下文窗口大小时，OpenClaw 会自动压缩——但在压缩之前会先提示 Agent 将重要笔记保存到记忆文件中，确保关键信息不会丢失。第三层是**语义搜索**。工作区文件通过向量嵌入建立索引，支持跨会话的混合关键词与向量检索。这意味着 Agent 的记忆远超单次对话所能容纳的范围；它可以在回答新问题时调取数周甚至数月前的相关笔记。

## LLM 驱动逻辑

Agent 运行时从引导文件、工作区上下文和对话历史中组装系统提示词，然后发送给配置的语言模型。模型决定下一步动作：回复文本、调用工具或两者兼有。当模型调用工具时，执行结果被反馈回模型以继续推理，这个循环不断重复直到生成最终的文本回复。OpenClaw 支持多个模型提供商——包括 Anthropic、OpenAI、Google、Mistral、DeepSeek 和 Qwen——并在某个提供商返回错误时自动切换到备选模型。思考深度和推理级别可以按需调整。

## 本地计算机交互

OpenClaw 内置了一组工具，使 Agent 能够与你的计算机交互。它可以运行 shell 命令、读写文件、编辑文档，并通过任何已连接的渠道发送消息。默认工作目录是 Agent 的工作区文件夹，确保文件操作井然有序。除基础功能外，扩展能力还包括浏览器自动化、网页搜索、媒体理解（图片、PDF 和视频）、通过 Cron 执行定时任务，以及接收外部触发器的 Webhook。

安全性通过工具策略来保障，控制每个渠道或会话允许执行的操作。例如，群聊可以被限制为只读工具，而私聊则拥有完整访问权限。可选的沙箱机制可以进一步隔离 Agent 的文件系统访问范围。Agent 始终在你配置的边界内运行——它无法自行提升权限。

# 第二部分：部署

## 云端模型 API 配置

OpenClaw 开箱即用地支持多种云端模型提供商：Anthropic (Claude)、OpenAI (GPT)、Google (Gemini)、Mistral、DeepSeek、Qwen/通义千问和 OpenRouter。要开始使用，运行交互式配置向导（`openclaw configure`），它会引导你选择提供商并输入凭证。对于大多数提供商，只需粘贴一个 API 密钥；Google 还支持基于 OAuth 的认证。如果你愿意，也可以跳过向导直接编辑配置文件——效果完全相同。

为了提高可靠性，你可以为同一个提供商配置多个账号或 API 密钥。如果某个密钥触发了速率限制或返回错误，OpenClaw 会自动轮换到下一个可用密钥后再重试。这就像同一个场馆有多个入口——如果某个入口排队太长，你就走到另一个。一旦密钥配置就绪，无需额外操作即可让你的助手更加稳健。

## 本地模型部署

如果你希望完全在自有硬件上运行模型——出于隐私、成本或离线访问的考虑——OpenClaw 支持多种本地推理引擎。推荐从 Ollama 开始，因为它配置简单直观；此外也支持 LM Studio、vLLM 和 LiteLLM。理解方式很简单：安装本地推理引擎，拉取模型，OpenClaw 会自动发现它。

实际操作流程如下：安装 Ollama，用它拉取一个模型（例如 qwen3:8b），设置一个环境变量告知 OpenClaw 启用本地提供商，然后重启 Gateway。启动时，OpenClaw 会查询每个已配置的本地引擎获取其可用模型列表，并将它们添加到你的模型菜单中——无需手动注册或逐个配置模型。如果你之后在 Ollama 中拉取了更多模型，它们会在 Gateway 下次启动时自动出现在 OpenClaw 中。

## 混合模式（云端 + 本地）

最灵活的配置是在同一套设置中组合云端和本地模型——你可以把它想象成配备一个主驾驶员和一个备用副驾驶员。指定一个模型作为主力（比如一个推理能力强的云端模型），另一个或多个作为后备（比如一个在云端不可达时使用的快速本地模型）。如果主提供商出错——网络波动、速率限制、服务中断——OpenClaw 会自动切换到后备链中的下一个模型，无缝衔接，即使在对话进行中也不中断。聊天另一端的用户完全不需要知道发生了切换。

恢复过程同样是全自动的。当某个提供商故障时，它会进入冷却期，期间 OpenClaw 停止向其发送请求。在后台，Gateway 会定期用轻量级探测请求测试该提供商。一旦提供商恢复响应，OpenClaw 就将其重新激活并按照原有优先级继续使用。整个流程——检测、降级、冷却、探测、恢复——全部无需人工干预。

## 优势与局限

**优势。** OpenClaw 最突出的优势在于它始终在线地驻留在你日常使用的消息应用中——你不需要打开特殊界面或记住某个 URL。原生支持 12+ 种渠道，包括飞书等中国生态渠道，意味着它在你习惯的地方等着你。所有数据留在本机，让你对对话和记忆拥有完全的自主权。三层记忆系统（引导文件、会话历史、语义搜索）提供了超越单次对话的连续性。广泛的模型提供商支持——包括 Qwen 和 DeepSeek 等中国区提供商——让你可以根据实际情况选择最合适的模型，而插件架构使得添加新渠道、工具或行为变得简单直接，无需修改核心代码。

**局限。** OpenClaw 在设计上是单用户的：它是个人助手，而非团队或企业平台。Gateway 必须持续运行在本地机器（或你控制的服务器）上，这意味着你需要一台保持开机且联网的硬件。初始配置——选择模型提供商、连接消息渠道、配置记忆系统——相比注册即用的托管方案有更陡的学习曲线。而且如果你选择运行本地模型，响应质量直接取决于你可用的硬件；配置较低的机器只能运行较小、能力较弱的模型。

# 第四部分：技术深入

## Agent 架构

核心模型：**"思考-行动-观察循环"**

当你发送消息时，Agent 并不是简单地发出一个请求然后返回一个固定回答。它会进入一个循环。首先，它组装上下文——收集其人格设定（"我是谁"）、记忆笔记（"我知道什么"）和对话历史（"之前发生了什么"），整合成一个连贯的信息包。这整个信息包被发送给语言模型。模型读取所有内容后决定下一步：它可能直接用文本回复，也可能请求执行一个操作——例如"运行这条 shell 命令"或"读取这个文件"。如果模型请求了一个操作，Gateway 就在你的机器上本地执行，观察执行结果，并将结果反馈给模型。模型读取新信息后再次决策：回复，还是执行另一个操作。这个循环不断重复——有时只循环一次，有时十几次——直到模型认为已经有了完整的答案，此时它生成最终的文本回复并流式传输回给你。就像一位厨师逐步按食谱操作——尝味、调整、再尝味——直到菜品完成。

```
消息输入
    |
    v
[组装上下文]
    |
    v
[发送至 LLM] ──> 回复？ ──是──> 流式输出回复
    |
    否
    v
[执行工具]
    |
    v
[观察结果] ──> 返回 [发送至 LLM]
```

**开发者注：** 事件循环按照明确的顺序执行：工作区解析、模型解析、认证配置选择、会话锁获取、系统提示词组装，然后是 LLM 推理与工具执行循环，最后是会话持久化。流式传输与此循环并发进行——助手文本增量、工具调用事件和生命周期事件全部通过同一事件流多路复用，使调用方可以实时看到部分进展。子 Agent（由父 Agent 衍生出来处理子任务的子进程）通过注册表进行跟踪，具备通知/重试语义和指数退避机制，确保失败的子 Agent 不会静默消失，而是被重试或上报给父 Agent。

## 调度系统

核心模型：**"每个对话一个队列，门口一个保镖"**

每个对话（会话）拥有自己的队列，队列中的消息严格按顺序逐条处理——不允许插队。如果你在 Agent 还在思考上一条消息时发送了新消息，新消息会排队等候。这种严格的顺序保证了 Agent 不会在同一对话中同时执行两个相互矛盾的操作——想象一下同时发出"删除那个文件"和"不对，保留它"两条指令并行执行的后果。队列确保它们按顺序依次处理。

Gateway 还强制执行全局并发限制——即"保镖"。所有渠道中只有有限数量的对话可以同时运行。如果 WhatsApp、Telegram 和 Slack 同时有消息到达，保镖决定有多少可以同时进入 Agent 运行时，其余的在各自队列中等待。当你在 Agent 完成当前轮次之前连续快速发送多条消息时，它们会被合并成一次交付而非创建多个独立轮次，以保持效率。

有三种队列行为控制新消息到达时（Agent 正在工作时）的处理方式：

- **收集**（默认模式）：将所有待处理消息收集起来，作为一个合并轮次交付。这是常规行为——你的消息会累积起来，Agent 准备好时一次性全部看到。
- **引导**：将新消息注入当前正在执行的轮次——就像在别人工作时从门缝里递一张纸条。Agent 在思考过程中看到你的新指令，可以中途调整方向，无需完成旧轮次。
- **后续**：耐心等待当前轮次完成，然后将新消息作为下一个轮次交付。适用于你想确保 Agent 完成当前任务后再处理下一件事的场景。

**开发者注：** 会话串行化使用基于文件的排他锁（POSIX `wx` 创建排他模式）并带有 PID 跟踪和 30 分钟过期锁检测，确保崩溃的进程不会永久阻塞会话。通道（Lane）抽象提供两级并发控制：每会话通道（串行，确保每个对话同一时间只有一个轮次）和全局通道（所有会话之间可配置的并行度）。生成令牌（Generation Token）防止进程内重启后过期的任务完成结果造成干扰——每一代 Agent 运行时获得一个唯一令牌，来自上一代的完成结果会被丢弃。防抖系统使用按键分组的缓冲区和可配置超时来合并快速连续的消息，在它们到达队列之前进行聚合。

## 记忆管理

核心模型：**"一本笔记本、一个文件柜和一位图书管理员"**

**笔记本。** 每次 Agent 开始对话时，它都会打开自己的笔记本——存储在工作区中的一组纯 Markdown 文件。这些文件包含 Agent 是谁（人格和语气）、应该如何行事（操作指令）、关于你的了解（你的档案和偏好），以及随时间积累的知识（记忆笔记）。你可以自己打开这些文件阅读或编辑，就像一本真正的笔记本。如果你纠正了 Agent 对某事的理解——"我其实更喜欢简洁的回答"——它会更新笔记。把这想象成 Agent 开始工作前的晨间汇报：它在开口之前先通读笔记本，回忆一切。

**文件柜。** 每次对话都以文本记录的形式归档到磁盘。随着对话的累积，有些会变得很长——长到超出模型的上下文窗口，即语言模型一次能处理的最大文本量。当这种情况发生时，Agent 执行一次"记忆刷新"：它重新阅读笔记本，识别对话中尚未保存的重要内容，将这些见解写入记忆笔记，然后将旧的对话记录压缩成摘要。完整的记录会被归档，但活跃的对话窗口会被裁剪以适应限制。这确保了旧对话被压缩时不会丢失关键信息——重要内容已经转移到了笔记本中。

**图书管理员。** 为了回忆数周或数月前的信息，Agent 依赖于一个搜索系统，它使用向量嵌入对工作区中的所有内容建立索引——这是一种基于语义的数学指纹。与简单的关键词搜索不同，这种方法理解概念。查询"部署步骤"能找到标题为"安装说明"的笔记，即使用词不同，因为底层含义相似。搜索将关键词匹配（权重 30%）与语义相似度（权重 70%）混合使用以获得最佳结果。这意味着 Agent 可以利用其全部的笔记和对话历史，而不仅限于当前上下文窗口所能容纳的内容。

**开发者注：** 时间衰减对较旧的笔记应用指数递减评分，可配置半衰期（默认 30 天），使近期信息在搜索结果中自然排名更高，无需手动维护相关性。MMR（最大边际相关性）使用 Jaccard 相似度对结果进行重排序，防止冗余或近似重复的片段占据前列。嵌入提供商包括云端选项（OpenAI、Gemini、Voyage、Mistral）和通过 node-llama-cpp 实现的完全本地选项，适用于不希望任何数据离开本机的用户。存储使用 SQLite 配合 sqlite-vec 扩展进行高效的余弦相似度向量搜索，将整个索引保存在单个可移植的数据库文件中。

# 第五部分：性能测试

## 测试环境

所有基准测试均在配备 32 GB 内存的 Mac mini M4（Apple Silicon）上进行，使用标准家庭宽带连接。云端模型为 Claude Sonnet 4.6，通过 Anthropic API 经由 OpenClaw Gateway 访问——与真实用户消息走过的路径完全相同。本地模型为 qwen3:8b（80 亿参数），运行在 Ollama 上，通过其本地 API 直接调用，而非经过 OpenClaw 管道。这些测试并非实验室级别的基准测试，而是旨在让你直观感受日常使用体验的实际测量。

## 用例 1：简单问答

第一个测试是一个无需工具的简单事实性问题——纯语言模型推理。提示要求模型用各一句话解释热力学三大定律。这就是你在阅读文章时可能从消息应用发给助手的那种快速查询问题。

| 指标 | 云端 (Claude Sonnet) | 本地 (qwen3:8b) |
|------|---------------------|-----------------|
| 总响应时间 | ~9 秒 | ~57 秒 |
| 响应质量 | 优秀 (5/5)——准确、简洁，附带额外背景信息 | 良好 (4/5)——准确、清晰，略显冗长 |
| 生成 Token 数 | ~150 | ~949（包含内部推理过程） |

云端模型在不到十秒内返回了一个精炼、简洁的回答，还额外补充了热力学第零定律。本地模型同样给出了准确清晰的回答，但耗时约为前者的六倍，生成的 Token 数也明显更多——其中大部分用于得出最终答案前的内部思维链推理。

## 用例 2：文件操作

第二个测试涉及读取一个 20 行的文本文件，为每行添加行号，然后将结果写回——这个任务不仅考验推理能力，还考验工具调用（文件读取和文件写入）。此测试仅通过 OpenClaw Gateway 使用云端模型进行端到端测试，因为本地模型未配置为 OpenClaw 模型提供商，因此无法参与工具调用管道。

| 指标 | 云端 (Claude Sonnet) | 本地 (qwen3:8b) |
|------|---------------------|-----------------|
| 总耗时 | ~12.5 秒 | ~60-70 秒（估算） |
| 工具调用次数 | 2 次（读取 + 写入） | 预计相似 |
| 准确性 | 完美——全部 20 行正确编号 | 未进行端到端测试 |

云端模型在约 12.5 秒内完成了整个操作，进行了两次工具调用（一次读取文件，一次写回编号后的版本），结果完全正确。本地模型的估算值是根据问答测试中观察到的约 6 倍推理速度比推算的。要端到端测试本地模型需要将 Ollama 配置为 OpenClaw 模型提供商，这是支持的但不在此次基准测试范围内。

## 分析

云端模型的响应速度明显更快——在本次测试中，简单查询的速度约为本地模型的六倍。质量差异虽然存在，但更为细微：两个模型都回答正确，但云端模型生成了更精练的输出，并附带了展示更深层理解的上下文补充。对于需要速度或复杂推理的任务——多步骤工具链、细腻的写作，或任何你在聊天另一端等待回复的场景——云端模型是明确的首选。简单事实问题 9 秒、包含两次工具调用的文件操作 12.5 秒的响应时间，都足够快到让人感觉像在对话。

本地模型在不同场景下大放异彩。当隐私至上——确保没有任何数据离开你的机器——本地推理是唯一能真正做到的选项。同样适用于离线环境、受限网络，或者你只是想避免按 Token 计费的 API 成本（本地推理在初始硬件投入后完全免费）。像 qwen3:8b 这样的 80 亿参数模型能很好地处理简单的事实和推理任务，而 700 亿参数级别的更大本地模型则能显著缩小与云端模型的质量差距。对许多用户来说，最佳实践是混合方案：将高要求任务（复杂工具链、创意写作、细致分析）路由到云端模型，将日常查询（快速查找、简单文件操作、闲聊）交给本地处理。OpenClaw 的降级系统使这一切无缝衔接——你只需配置一次首选模型顺序，Gateway 就会自动路由。

# 第六部分：替代方案与如何选择

## 简要介绍

**Auto-GPT。** 专为自主多步骤任务执行而设计。启动"任务"并运行至完成，无需持续交互。最适合一次性的复杂自动化；缺乏始终在线的能力和多渠道消息支持。

**Open Interpreter。** 一个轻量级的 CLI Agent，可以通过终端运行代码并与计算机交互。仅限会话范围，运行之间没有持久化记忆。最适合快速的开发者任务和命令行脚本编写。

**LangChain Agent。** 一个用于构建自定义 AI Agent 的框架，而非即用型产品。极其灵活但需要大量开发工作才能创建一个可用的 Agent。最适合构建定制化 Agent 驱动应用的团队。

**Claude Code。** Anthropic 面向开发者的 CLI 和 IDE 工具，专注于软件工程任务。深度编辑器集成和代码库感知。最适合希望在开发环境中拥有 AI 结对编程伙伴的开发者。

## 详细对比

有关所有五个项目的功能并排对比，请参阅[第三部分的对比表](#与其他-agent-框架的对比)。

## 如何选择

如果你想要一个始终在线、可通过消息应用访问、具备持久化记忆和本地数据主权的个人助手，选择 OpenClaw。如果你是开发者且希望将 AI 融入编程工作流，选择 Claude Code。如果需要在终端中快速完成一次性任务，选择 Open Interpreter。如果你正在构建定制化的 Agent 驱动产品且需要最大灵活性，选择 LangChain。如果需要无需交互即可运行的自主多步骤任务，选择 Auto-GPT。
