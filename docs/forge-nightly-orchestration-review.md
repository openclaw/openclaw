# Forge nightly orchestration review

Use this when running the nightly Forge self-improvement pass.

## Evidence sweep

- Read the latest memory note for prior failure modes and unfinished reliability items.
- Sample at least one current dated follow-up in `docs/follow-ups/` when one exists. Keep generic heartbeat polls as pure no-tool sentinels; stale tracker checks belong in nightly orchestration/maintenance unless a runner performs the gate outside the agent turn. A sampled clean repo list is not sufficient when recent memory or a self-owned tracker names open follow-up work. If a recent manager wake sampled only convenient clean repos while a known self-owned tracker stayed open, count it as missed proactive-action drift and tighten the nightly/follow-up maintenance path rather than adding tool work to generic heartbeat turns.
- Check Forge workspace guidance in `AGENTS.md` and `HEARTBEAT.md`.
- Inspect key repos for:
  - branch vs remote state
  - local-only commits
  - untracked operational debris
  - whether multiple workers touched the same repo/branch/PR and, if so, which SHA/remote state is canonical
  - whether a same-worktree delegation lock should have been used from `docs/worker-delegation-branch-lock.md` before the second lane entered the dirty branch
  - whether standalone/source-package extraction work used the package addendum before mixing demo/frontend and package-boundary/backend lanes in the same internal app worktree
  - whether redo/iteration requests stacked overlapping workers on one branch/worktree instead of draining, steering, cancelling, or forking cleanly
  - work happening in the wrong repo
  - whether the claimed repo path actually exists before using it as evidence
  - multi-worktree reality: do not require `.git` to be a directory; use `git -C <path> rev-parse` and `git worktree list --porcelain` before declaring a worktree missing
  - repo-sweep implementation: do not build nightly sweeps with only `[ -d "$repo/.git" ]`; that misses linked worktrees whose `.git` is a file. Enumerate likely paths, then verify each candidate with `git -C "$path" rev-parse --show-toplevel`, or start from canonical repo `git worktree list --porcelain` output.
  - stale upstream reality: when `git status --branch` shows `[gone]`, do not run ahead/behind comparisons that assume `@{u}` still resolves; classify it as stale worktree hygiene and decide whether to switch/fast-forward, archive, or carry a cleanup follow-up
- Inspect recent cron or automation snapshots and do not treat top-level `error` status as sufficient evidence by itself. Check whether useful work still landed, and explicitly correct stale failure notes when the underlying condition has already recovered.
- Reconcile sibling-nightly contradictions before repeating a standing blocker. If one same-night report says an install/auth/cron/repo blocker exists and another says it recovered, verify the live state with the smallest direct check, then record the reconciliation in the current report/memory instead of letting stale blocker language bounce between nightlies.
- For scheduled jobs with user-facing output, inspect `openclaw cron runs --id <job-id-or-uuid> --limit N` delivery metadata. `cron runs` already emits JSON and does not currently accept `--json`; if the CLI rejects an option, run `openclaw cron runs --help` and update the workflow instead of carrying stale syntax. Prefer UUIDs from `openclaw cron list --json`; friendly-name lookups can return empty `entries` even when UUID lookups show delivered runs. A run can be `status=ok` while `deliveryStatus` is `not-delivered` or `not-requested`, so delivery proof is separate from execution proof.
- If a user-facing cron run is `status=ok` but `deliveryStatus=not-delivered`, do not stop at reporting the anomaly. Capture the missed run/report, resend or queue the summary to the intended channel when appropriate, or record the exact delivery blocker and owner in the nightly note.
- If a prior nightly delivered its final summary but live `lastRunStatus=error`, inspect the exact `lastError` and fix the failing operation in the next run. For doc/memory/report edits inside the OpenClaw workspace, prefer exact `edit`/`write` operations and verify the changed files instead of relying on a failed patch attempt plus a successful final message.
- Bounded nightly pass rule: once the evidence sweep identifies a real improvement, make that improvement, write the report/memory note, verify the changed files, and exit cleanly. Do not let self-improvement crons become open-ended investigations; broader findings should become explicit follow-ups with owner/blocker instead of stretching the cron into another idle timeout.
- Nightly final-write gate: write the report and memory note before any optional deep dive. If one concrete improvement has landed and verification passes, stop; do not add more evidence gathering that can only improve confidence marginally while risking another idle-timeout final.
- Evidence retrieval budget: avoid broad `sessions_history(..., includeTools:true)` on Discord/channel sessions during nightly crons unless the tool transcript itself is the exact evidence needed. Prefer `sessions_list` previews, targeted `message.search/read`, or one small no-tools history sample. If tool evidence is required, narrow by session/channel and limit sharply, then stop gathering once a concrete improvement is identified.
- Compare live `openclaw cron list --json` output with `reports/cron-before.json`, `reports/cron-after.json`, and `reports/cron-changed.txt` so intentional scope narrowing, disabled non-engineering channels, prompt/claim filters, or stale snapshots are not misreported as current automation health. In current `cron list` output, the root is an object with `jobs[]`; last/next/delivery/running values live under each job's `state.*`. Current observed keys are flat (`state.lastRunStatus`, `state.lastDeliveryStatus`, `state.lastDelivered`, `state.lastRunAtMs`, `state.lastDurationMs`, `state.runningAtMs`, `state.nextRunAtMs`), not nested `lastRun`/`currentRun` objects. Do not assume the root is a raw array, and do not read top-level `lastRunStatus`/`deliveryStatus` projections unless the JSON actually contains them. If `state.runningAtMs` is present for the same job currently executing, treat `lastRunStatus`/`lastDeliveryStatus` as the previous completed run, not proof that the in-flight recovery has already failed or succeeded; do not enqueue another recovery while the current run is active.
- When parsing `openclaw cron runs` output, read the top-level `entries` array first and fall back to `runs` only if present. If a non-empty JSON file yields no parsed rows, treat that as a parser bug and rerun before drawing a cron-health conclusion.
- When parsing CLI JSON during the review, avoid `command | python3 - <<'PY'` because the heredoc supplies Python's stdin and discards the pipe. Capture JSON to a temp file or use `python3 -c`; if a parser fails, rerun with safe capture before turning the failure into a cron/API health conclusion.
- For Mission Control reporting failures, record the exact endpoint + HTTP status/body and retry once with the customer API schema before classifying the issue as bad auth, bad key, or service outage. Heartbeat `metadata.current_task` must be a short string, not an object; HTTP 400 on that field is schema drift.
- Mission Control heartbeat success should include the requested returned status. Customer API responses are wrapped as `{ ok, data }`, so verify `data.status` / `data.request_id` rather than guessed top-level fields. If HTTP 200 echoes an earlier state for the same `request_id`, use a phase-unique request id such as `<run-id>:end` and verify the status changed before counting the end heartbeat complete.
- Inspect any recent config edits that affect runtime behavior, routing, install flow, channel scope, or publish/sync coverage.
- Inspect recent running or aborted sessions for live config, policy, routing, install, or publish-path edits that never reached a clean verification closeout.
- Inspect external install API work separately for read discovery vs write probes. Unknown create/update tests in live customer accounts need explicit disposable naming and same-run rollback/delete proof; if safe write endpoints are unavailable, carry the API/manual-UI blocker rather than implying install capability from read endpoints.
- Inspect recent credential/env handoffs. If Steven sent keys for a named repo/service, verify Forge applied them to a gitignored env/secret path while visible, or converted the redaction blocker into one exact secure-delivery next step. Never copy secret fragments into reports, memory, issues, or closeouts.
- For reusable cross-app/team credentials, verify the handoff produced a durable tracker for remaining app/env/live-smoke work and the post-smoke rotate-or-keep decision. A chat closeout that only says the key was stored/wired is incomplete if any install proof remains pending.
- Inspect the execution path for secret leakage through tool-call arguments (`exec.env`, command strings, heredocs, issue bodies, reports, logs). If a key touched transcript-visible tool input, record only a sanitized rotation follow-up after smoke/prod verification and update workflow so future secret delivery uses a `.env` attachment, native secret store, or host-side placement.
- Inspect recent background/yielded verification commands for deploy/live/smoke claims. If the command completed after the final closeout, count that as premature verification and reconstruct the actual evidence before treating the task as clean.
- Inspect for stale local UI smoke servers (`next dev`, `next start`, orphaned `next-server`, `vite --host`, `vite preview`, preview servers) left from prior worker validations. Stop them after proof is captured unless an explicit active handoff still needs the running process, and note repeated leaks as workflow drift. Treat multi-day `0.0.0.0` dev/preview listeners as exposure risk to clear before routine repo hygiene. If Canvas/screenshot tooling fails during source-channel proof, record the missing visual proof in the closeout instead of generating standalone channel noise.
- Inspect claimed image/video/frame evidence against the tool results that produced it. A failed inspection tool, missing dependency, or unavailable analyzer is not proof; either rerun with a working path or record that the closeout overstated the evidence.
- Inspect delegated runs for child-session completion events, async process completions, and delivery events that landed after the final closeout. If any late event could affect owner lane, validation, runtime proof, or user-facing delivery, count the closeout as prematurely drained and carry the corrected release story forward.
- Inspect worker timeouts/aborts. If Forge recovered locally afterward, verify the closeout explicitly named the local-recovery exception and reason; otherwise count it as lane-accountability drift. Prefer resume/re-delegate/split-scope before treating Forge-local implementation as the recovery path.
- Inspect source-channel worker timeout leaks. If a child/subagent timeout auto-posted raw `Partial progress` or an incomplete worker result into the channel, verify Forge immediately followed with a manager recovery path and later closed with real proof; otherwise count it as noisy handoff drift even if the code eventually shipped.
- Inspect yielded/background deploy runs for raw CLI/process output or stale `process still running` messages in source channels. If later worker/GitHub/runtime proof completed the work but the source channel never received the final manager closeout, recover it once and tighten the handoff so process logs stay internal.
- For Discord/channel sessions that end with an empty final assistant text, inspect recent tool calls and source-channel messages before classifying it as a missed response. Empty final text is acceptable only when a `message.send` result or delivery mirror proves the substantive closeout already landed in the source channel; otherwise reopen the follow-up.
- For delegated/source-channel updates, check for duplicate closeout risk: if a worker result or `message.send` already posted the same substantive update, suppress the second delivery rather than letting manual+automatic paths double-post. In live Discord channel sessions, prefer one normal final answer; use `message.send` only for different-target delivery, recovery sends, or deliberate side effects, and then keep the final empty/NO_REPLY. Same-channel `message.send` closeouts should be counted as delivery-owner drift unless there was a specific recovery/side-effect reason; verify the source-channel message before treating the empty final as clean. Do not repair old duplicate bot messages by deleting them unless Steven/Jarvis asks.
- Review the delivery-owner choice for recent source-channel delegations. If both the worker and Forge posted full closeouts, mark it as a workflow miss; if only the worker posted but lacked manager labels, reconstruct the manager proof in memory/reporting and tighten the next delegation prompt.
- For source-channel worker auto-delivery, check whether the worker handoff included `docs/worker-auto-closeout-template.md` or an equivalent literal label block. If Forge returned `NO_REPLY` after a worker posted without owner/repo/branch/SHA/validation/runtime/tracker labels, count that as release-story debt even when the technical work was useful.
- If a worker auto-announcement contains `Runtime verification needed`, `missing smoke`, or another self-owned proof gap, verify that Forge later completed the proof, created/updated a durable tracker, or sent one manager follow-up with the blocker, owner, and unblock trigger. Otherwise count the handoff as still open even if the worker message looked like a closeout.
- For yield-heavy cron reviews, verify that the parent cron run produced an intended final delivery after child agents finished. If `deliveryStatus=not-delivered`, recover the integrated summary or explicitly carry the delivery blocker; child subagent announcements alone do not prove the scheduled report delivered. Until the parent-final delivery bug is fixed, run `nightly-review-previous-day` synchronously instead of using `sessions_yield` for child audits. Use `docs/cron-yield-recovery-checklist.md` as the literal pre-yield/recovery gate for user-facing nightlies. If a later synchronous/no-yield run delivers via fallback, record that as workaround health only; do not call the yielded parent-final path repaired until a yielded run delivers the parent final without fallback.
- For any user-facing “I’m fixing it” or “I’m changing config” claim in an aborted/still-running session, reconstruct the change before closing the review: sanitized config diff/state, runtime/gateway status, and the cheapest downstream proof of behavior.
- When channel scope, tool exposure, or security posture is part of the question, run `openclaw security audit` and correct stale config assumptions against the current audit plus the actual `~/.openclaw/openclaw.json`.
- If a prior note claims Discord allowlists or channel restrictions, verify them from the current file directly before repeating the claim.
- Sample at least one recent real execution thread or session, not just static docs, when reviewing delegation quality, closeout discipline, or proactive heartbeat quality, and compare any child-session ownership to the final closeout if delegation was used.
- If a sampled manager wake or nightly run returned `HEARTBEAT_OK` or no visible output while recent memory or a self-owned tracker named open work, count it as missed proactive-action drift. Keep generic heartbeat turns as strict no-tool sentinels; tighten the nightly maintenance path or convert the latent blocker into a tracker instead of letting OK become a reflex.
- Compare the advertised lane roster with the live security audit. If a lane description says to use Bolt/Reviewer/Architect for repo/code/GitHub work while the audit still flags small-model agents unsafe, tighten the lane rule instead of relying on scattered caveats.
- For lane-scoped work, compare the actual execution path to the expected lane. An `Owner: Spark` closeout is not enough if Forge did the repo work locally; if Forge executed locally, the owner field must say Forge/local execution and can separately name the recommended lane.
- For “make it live” / “ship it” / “merge it” requests, separate manager release action from implementation. If Forge only verified a lane-built PR, merged/deployed it, fetched canonical state, and smoked live, that can be manager release work; if Forge resolved conflicts, edited code, envs, or implementation without delegation, require a named Forge/local recovery exception. In either case, the sampled closeout must include Owner, Repo/path, PR or merge SHA, validation, live/runtime proof, and proof-scaffolding status.
- Treat standing direct-to-main / fast-iteration exceptions as branch policy only. They do not override lane ownership. If Forge used the branch exception as implicit permission to implement locally, count it as a delegation miss unless a separate local-execution exception was named and justified.
- For delegated work, compare the worker proof against Forge's follow-up. Forge should do integration-level verification or close a specific evidence gap, not redo the worker lane by default.
- While the live security audit flags small-model agents as unsafe with sandbox=off/web tools enabled, treat Reviewer/Architect/Bolt host-exec repo sweeps as a finding, not the default review pattern; use a hardened/capable lane or local inspection until that risk is fixed.
- Before any nightly subagent spawn for host exec, repo sweeps, GitHub writes, code/env work, config checks, or security-sensitive inspection, check the live agent list/security posture and confirm the selected lane is safe for that surface. If not, keep the check local or use Anvil/Spark/Steel and record the skipped lane in the nightly note.
- For cross-agent worker spawns, verify Forge explicitly used `context:"isolated"` unless same-agent fork context was required. A fork/context spawn error followed by `sessions_yield` is a delegation failure; the review should confirm the task was re-spawned successfully before treating the lane as owner.
- For worker/subagent spawns, verify the child session actually accepted work. Immediate `failed` status, empty transcript, or no acceptance/proof event is a failed delegation; reconstruct whether Forge re-delegated, split scope, or named a Forge/local recovery exception before counting the closeout as clean.
- If two or more same-family worker spawns fail immediately with empty transcripts in the prior day, classify it as lane-runtime maintenance debt. Verify `agents_list`, route active work to a safe capable fallback, and record the failed session keys plus chosen fallback in the nightly memory/report instead of only saying the work was recovered.
- If a worker result for product/repo work reports `Repo/path` inside `/home/forge/.openclaw/workspace/agents/<lane>` or says the full repo is unavailable while a real repo should exist, count it as repo-context drift. Reconstruct whether Forge re-resolved the real repo/path and re-delegated with explicit `cwd` before relying on the worker output.
- If the day included work in a project-specific Discord channel, verify the repo/path stayed aligned with that project context instead of drifting to another convenient local repo.
- If the day included a task from `#engineering` or a bare mention/follow-up, verify Forge did not treat the coordination channel itself as repo scope; the target repo/source channel must be explicit before edits begin.
- If Jarvis routed work through a coordination channel but named a source channel, verify Forge stayed anchored to that source channel for repo choice and final verification.
- If a bare/naked source-channel mention and a richer Jarvis coordination handoff are likely the same task, verify Forge chose one canonical execution path before additional probing. The noncanonical session is not clean just because it timed out; it is clean only if the canonical path produced the requested closeout and the duplicate session made no external side effects that need recovery.
- If the source channel has a no-thread preference, verify delegation did not create a Discord thread or thread-bound worker session there; use isolated workers plus one in-channel manager relay instead.
- Sample at least one recent direct Jarvis handoff when reviewing handoff discipline, and check whether any self-owned blocker in the reply was converted into a tracked follow-up or just left floating.
- If a sampled Jarvis handoff was routed through `#engineering` or another coordination channel while Jarvis told Steven/the source channel that Forge would post the result back "here," verify the source-channel completion/blocker actually landed there. If the only closeout went to the coordination channel, recover once with a concise source-channel blocker/result and record the relay miss.
- For Jarvis stand-down messages (`disregard`, `standing down`, `no action needed`, or Jarvis taking the work directly), verify Forge did not add a noisy acknowledgment or Mission Control churn unless there was a real cleanup/recovery action. A stand-down is resolved by stopping work and ensuring no worker/repo side effect remains, not by posting `copy`.
- For Jarvis-local-path handoffs, verify Forge did not turn `/Users/jarvis/...`, `~/clawd`, `~/.openclaw`, local logs/tmp/config, or security-cleanup-only work into Forge lane execution. The accepted inputs are a cloneable branch/PR, Drive bundle, signed tarball/log bundle, or sanitized remote artifact; otherwise the correct action is blocked/released, not scanning analogous Forge-local paths.
- For Jarvis-local cron/wrapper handoffs, treat `mc-update` stdout leaks, local `jq` failures, local OpenClaw cron output, or Jarvis "my machine" wrapper symptoms as local-Jarvis work even if the bug names a repo or Mission Control concept. Verify a real Forge-accessible repo file/branch owns the failing path before opening a PR; otherwise record `blocked/released to Jarvis/Mechanic local` and the exact artifact Forge needs. If a repo PR is still opened as optional hardening around an unproven Jarvis-local failure path, verify the closeout labels it as optional hardening only and names the promotion condition; do not let it read as the canonical fix.
- If Steven/Jarvis sent a new operating default or tooling rule, verify it landed in durable memory and reconcile any conflicting execution docs. A memory-only update is not enough when `AGENTS.md`, `HEARTBEAT.md`, or this workflow would still drive the old behavior.
- If a closeout left a self-owned missing live check or smoke test, treat it as open work and look for the follow-up artifact or verification action.
- For merged/deployed install, onboarding, or integration work with smoke deferred on env/auth/account/credits, verify the missing proof is captured in a GitHub issue or equivalent durable tracker. If only chat/report text exists and Forge owns the gap, create/update the tracker during the review before the report calls the pass clean.
- If a PR closeout explicitly deferred smoke behind preview auth, Vercel protection, a real account/session, or production deploy, check whether the PR has since merged or deployed. Once unblocked, run the deferred smoke; if still blocked, name the unblock trigger in the nightly note.
- If an install/onboarding/integration PR closeout lists env/auth/account/credits needed for live hookup, verify the closeout names a durable tracker URL/ID plus owner lane and unblock trigger. If it only lists env vars or says "after merge" in chat, create/update the follow-up issue/task during the review.
- If a publish or sync job was claimed healthy, verify the real source and target paths, not just the cron line or `PUBLISH_MODE` label. Treat `SOURCE_ROOT == TARGET_REPO` as suspect unless the task explicitly intended a self-sync, and confirm from logs that the intended content classes actually moved. Repeated publish commits with an unchanged final tree are still unresolved job churn; update the existing tracker with commit counts/tree proof instead of calling the job healthy just because canonical content currently matches.
- Sample at least one shipped closeout, not just the repo state, and check whether it preserved owner, repo/path, branch/PR or explicit direct-to-main exception, commit SHA/local-only state, and validation as explicit labels rather than implicit clues from a PR URL.
- If a sampled shipped closeout is a narrative `Done` / PR summary without literal manager labels, count it as release-story drift even when the technical work passed. Reconstruct the missing fields in the nightly note and tighten the template/checklist instead of treating the PR link as enough.
- If a sampled merge/deploy closeout names a production deploy as `running`, `pending`, `building`, or otherwise not complete, count runtime proof as deferred rather than verified. The closeout needs either a later successful live smoke or a labeled `Runtime verification / missing step` plus owner/unblock trigger.
- If a sampled batch closeout merges/closes multiple PRs or cards, require the batch release story: canonical repo/main SHA, each PR/branch/lane/purpose/final SHA, validation, live/runtime proof, proof scaffolding, and remaining trackers. Treat "all remaining cards shipped" as release-story drift unless those fields are visible.
- If a closeout says “same PR,” “merged,” or “pushed” without the manager labels, reconstruct the release story in the nightly note and tighten the workflow instead of treating the PR link as sufficient.
- If there was a same-day default-branch push, sample at least one of those closeouts and confirm it explicitly named the Steven exception or explicitly called out the policy deviation. Standing project exceptions still need the literal `Branch/PR or exception` label; `pushed to main` alone is release-story debt. Treat “be on main,” “main page,” or “update the main page” as target/base wording, not default-branch push permission, unless Steven explicitly said push/merge/commit to `main`.
- For delegated default-branch pushes, inspect the worker handoff as well as the closeout. If the handoff did not explicitly state `branch + PR required` or name a direct-to-main exception, count it as prevention drift even when the final closeout later tracked the policy deviation; tighten the worker handoff template/branch lock rather than relying only on after-the-fact recovery.
- If a default-branch push was accidental and recovered, verify all four recovery artifacts: bad SHA, revert/uncommit SHA on `main`, recreated branch/PR, and validation on the PR branch. If any artifact is missing, carry the recovery as open work instead of calling it fixed.
- If install, onboarding, lead-capture, or integration work shipped, sample at least one closeout and verify it named the exercised entrypoint, visual/preview/live smoke proof for UI funnel changes, the downstream hop or effect that was checked, and the exact missing smoke test if runtime proof was not gathered. Use `docs/install-reliability-closeout-checklist.md` as the literal checklist when the sampled work touched live hookups, customer paths, webhooks, CRM objects, Retell/voice, package installs, or onboarding state.
- For mobile app closeouts, real device/camera/API/account proof is part of downstream smoke. If it is deferred behind simulator/device access, deployed API URL, account state, env, or credits, verify the closeout has a durable tracker with owner lane and unblock trigger; otherwise backfill or carry it as open debt.
- If visual/browser smoke was deferred because Canvas/CDP/Browser Use/remote Chrome tooling was missing, verify whether that is a tool-install blocker or an auth/session blocker. If Forge owns the missing tool install, make one concrete move: install it through an intentional manager-maintenance path or create/carry a durable follow-up with owner and unblock trigger.
- If a UI fix responded to a user screenshot, breakpoint, or device-specific visual complaint, verify the proof includes that same viewport/device class. Do not accept wider tablet/desktop screenshots as proof for a mobile complaint unless the original mobile case was also checked.
- If a UI/content fix responded to a named remaining artifact, image, or section, verify the exact asset/reference and rendered section changed. Broad repo text scans are not enough when the visible complaint names a concrete thing still on the page.
- If yesterday included a verbatim or exact-copy request, compare at least one shipped artifact directly against the source text before treating the task as clean.
- If Jarvis/Steven says “reply with the EXACT …” for DNS records, env vars, verification strings, commands, registrar values, or another relay-critical payload, verify the final answer had one complete copyable relay block and did not split required values across messages or leave them as implicit provider defaults.
- If an agent/package deliverable shipped, verify it started from the named blueprint or canonical package schema. For Agent Installs-style packages, require blueprint-first structure, per-agent identity/ops files, memory/crons/docs/tests/skills/source materials, QA/handoff folders, zip integrity, required-file scan, markdown/memory/credential checks, and a verified Drive handoff before treating it as complete.
- If any file/package/deck deliverable is blocked by Drive/gog auth, verify a concrete dated follow-up exists with artifact path, checksum, owner, and one unblock action. Clean local build leftovers and do not count the task as delivered from a local path or rejected chat upload.

- If a client-facing deck, one-pager, package PDF, or sales collateral shipped, verify design/offer QA separately from PDF QA: audience, buying decision, concrete workflows installed, client deliverables, selected skill stack, final generation path, screenshots, text extraction, Drive handoff, and no fake proof claims.

## Review rubric

### 1. Delegation quality

- Was Forge doing worker-lane execution that should have gone to Anvil, Spark, Steel, or Bolt?
- Was there a clear owner for each active task?
- Did at least one recent multi-step task show the right lane ownership in execution or closeout?
- If child sessions were used, did the final closeout keep the owner lane visible instead of only showing the technical result?
- Did Forge avoid duplicating worker-lane execution after delegation unless the worker's proof was missing or suspicious?
- If child sessions were used, were their completion events drained or intentionally superseded before Forge sent the final closeout?
- Did the final closeout also preserve repo/branch/validation context instead of only the technical result?
- Did manager output stay tighter than worker output?

### 2. Jarvis handoff discipline

- Did direct Jarvis asks get action instead of acknowledgment?
- Did Forge avoid replying to status-only Jarvis posts?
- Were closeouts sent only on completion, blocker, or required decision?
- Was any handoff closed before runtime/config enforcement was actually verified?
- If Jarvis or Steven asked for exact/verbatim/match-source work, was that exactness checked before the closeout claim?
- If a live session aborted mid-change, was that treated as an unresolved handoff instead of a finished task?
- Does any live config or policy edit still sit in a running or aborted session without a clean verification closeout?
- Did any user-facing “I’m fixing it” config/routing claim get a real closeout, or was it only inferred later from behavior?
- For install/onboarding/integration asks, did the closeout prove the real entrypoint and downstream effect, instead of stopping at build/lint success?
- If Jarvis or Steven was told about a blocker that Forge itself owns, was that blocker turned into a concrete follow-up artifact, or was it just restated back to them?
- If Steven provided credentials/env values for a named repo/service, did Forge apply them immediately or ask one precise secure-delivery question before redaction made the values unrecoverable?
- Did Forge avoid putting raw secrets into tool-call JSON, shell commands, heredocs, issue bodies, or other transcript/log surfaces? If not, is there a sanitized rotation tracker and a safer next delivery path?

### 3. Install and release reliability

- Was the real target repo identified before editing?
- Are claims like `in main`, `pushed`, and `live` backed by repo, SHA, remote, and deploy verification?
- Did multiple workers touch the same branch/PR without serialization? If so, reconstruct the canonical SHA, remote state, and any lost/local-only drift before calling the release story clean.
- Did any deploy/live/smoke claim depend on an async process whose result arrived after the closeout? If so, record it as premature and add a pending-proof or reconstructed-proof follow-up.
- Did any delegated closeout get followed by late child-session/process/delivery events that should have been incorporated before the final answer?
- Did shipped repo work bypass branch -> PR flow without an explicit Steven exception?
- Did any branch-ready closeout only provide a hypothetical `/pull/new/...` URL or “PR link if you want one” instead of an actual PR? If so, open the PR when the branch is pushed/clean and meant for review, or carry the exact blocker.
- If a child lane or ACP run was blocked from lint/build/git/push, did Forge finish that work in a capable runtime or report the missing step plainly?
- Did config and routing changes get restart/reload verification, or is the missing step called out clearly?
- For install/onboarding/integration changes, did the closeout show entrypoint plus smoke-test evidence, or at least carry the exact missing smoke test forward?
- For UI fixes, did the visual proof match the user-reported screenshot/viewport/device class, especially mobile-vs-tablet breakpoints?
- For named visible artifacts/images/sections, did the closeout prove the exact asset/reference and rendered section changed, rather than relying only on broad text scans?
- For agent/package deliverables, did the package follow the named blueprint/schema and include manifest, required files, QA checks, source materials, and Drive handoff proof?

- For client-facing decks/PDFs/offer collateral, did the artifact prove the concrete offer and installed workflows, or did it pass technical rendering while still reading generic?

- If smoke was deferred because it needs merge/prod/auth/account state, is there a concrete trigger for the next manager pass to verify it?
- If that missing smoke depends on env/auth/account/credits after merge/deploy, is it tracked in a durable issue or equivalent artifact instead of only a chat caveat?

### 4. Manager-level proactive coverage

- Are key cron or polling loops still enabled where Forge depends on them?
