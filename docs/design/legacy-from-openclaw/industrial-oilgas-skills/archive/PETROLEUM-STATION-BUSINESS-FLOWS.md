## 二、ClawTwin 产品框架（描述工作流时用这一张表）

| 产品 / 触点                  | 在油气场站故事里的角色                                                                                                                                      |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **飞书（客户既有）**         | 员工主入口：IM 对话、群卡片、OA 审批；**不替换** IMS/SCADA。                                                                                                |
| **ClawTwin Studio**          | PC 工作台：场站/设备孪生大屏、告警与工单工作台、管理员与 KB、**触发 AI Job**（`POST /v1/ai/jobs`）。                                                        |
| **ClawTwin Mobile**          | **非独立 App**：体验 = 飞书 Bot + 卡片 + 小程序式审批（产品文档中的「移动端」触点）。                                                                       |
| **OpenClaw + ClawTwin Sage** | 飞书对话背后的 **Agent Runtime + Skills**；通过 **Bearer `openclaw-service-token`** 调 **`GET/POST /mcp`** 拉孪生上下文、查 KB、开工单草稿等（LOCK §1.7）。 |
| **ClawTwin Nexus**           | **工业语义与执行中枢**：本体与设备读数、告警 FSM、工单 HITL、KB、SSE、AI Job 编排、Webhook（飞书卡片）等。                                                  |
| **ClawTwin Connect**         | **连接器套件（随 Nexus）**：工单与 SAP PM / CMMS **SoT + 回填**；Historian/MES/OA 等配置化接入。                                                            |
| **ClawTwin Bridge**          | **DMZ OT 采集桥**：订阅 OPC-UA，**仅向 Kafka** 生产 `ot.telemetry` / `ot.events`，**不调 Nexus HTTP**（LOCK §四）。                                         |

**客户既有、不划入 ClawTwin 产品**：OPC-UA Server / SCADA、SAP PM / Maximo / 用友等 IMS —— 在流程里写成 **外部系统**，由 **Bridge** 或 **Connect** 对接。

### 4.4 Studio 触发的离线 AI 诊断 / 分析报告（并行于飞书会话）

站上工程师在 **Studio** 点「AI 诊断」时，**主导方是 Nexus**（见 `ARCHITECTURE-FINAL-REVIEW.md` §3.3）：

Studio → **`POST /v1/ai/jobs`** → 队列 **`platform.ai-jobs`** → Worker 驱动 OpenClaw/Sage → MCP 拉上下文 → **`POST /v1/ai/jobs/{id}/result`**；Studio **`GET /v1/sse/ai-jobs/{job_id}`** 订阅进度。  
与同场站 **`4.3` 的差别**：入口是 Studio 而非飞书会话，但下层仍通过 **同一套 MCP + Nexus 领域模型**。
