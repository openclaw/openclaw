## 一、核心心智模型：工业版 JARVIS

在开始任何设计讨论之前，先对齐一个心智模型：

```
钢铁侠的战斗方式：
  Tony Stark 不读传感器数据表格。
  JARVIS 持续监控战甲系统，主动推送关键信息：
  "右侧排气口过热，建议切换弧形推力" → Stark 拇指扫过确认 → 执行
  "检测到 5 点钟方向能量武器，已自动激活护盾" → Stark 无需操作
  "核心能量剩余 18%，建议优先击败目标后撤退" → Stark 决策

ClawTwin 的工业版本：
  操作员不盯着 200 个传感器仪表。
  Sage 持续监控所有设备，主动推送：
  "C-001 轴承振动异常，预计 4 小时后达告警阈值，已准备诊断报告" → 操作员确认 → 工单创建
  "站场入口压力超标 P1，已自动通知值班工程师" → 操作员无需操作
  "今日 C-001/C-002/C-003 综合健康度下降 12%，建议本周安排预防性检修" → 管理层决策
```

这就是目标用户体验：**AI 主动告诉你该做什么，你只负责最终确认权**。

## 三、新增核心引擎：Pulse Engine（站场脉搏引擎）

这是之前架构中缺失的关键组件，也是"游戏感"的技术基础。

### 3.1 什么是 Pulse Engine

```
传统系统：人主动查询 → 系统被动展示
Pulse Engine：系统主动感知 → 推送关键信号给人

Pulse Engine 每 30 秒完成一次全站"心跳"：
  1. 读取所有活跃设备的最新状态（Redis）
  2. 计算每台设备的健康分（0-100）
  3. 计算全站综合脉搏指数（Station Pulse Score）
  4. 与上次心跳对比，识别"变化中的信号"
  5. 按变化重要性排序，生成"注意力清单"
  6. 推送 SSE 给 Studio / 推送飞书卡片给关键人
```

### 3.2 Pulse Score 算法

```python
# engines/pulse/pulse_engine.py

def compute_equipment_health(equipment_id: str, readings: dict,
                              anomaly_score: float, active_alarms: list) -> EquipmentHealth:
    """
    设备健康分计算（0-100）

    分数组成：
      - 传感器基线分（40分）：关键指标在正常范围内
      - 趋势分（30分）：MOIRAI 预测未来 4h 的趋势
      - 告警扣分（30分）：P1 扣 30，P2 扣 15，P3 扣 5
    """
    sensor_score = 40
    for metric, value in readings.items():
        threshold = get_threshold(equipment_id, metric)
        if value > threshold.alarm:
            sensor_score -= 20
        elif value > threshold.warn:
            sensor_score -= 10

    trend_score = 30 * (1 - anomaly_score)  # anomaly_score 0-1，越高越差

    alarm_deduction = sum(ALARM_WEIGHTS[a.level] for a in active_alarms)

    health = max(0, sensor_score + trend_score - alarm_deduction)

    # 状态映射
    status = "critical" if health < 30 else \
             "warning"  if health < 60 else \
             "good"     if health < 85 else "excellent"

    return EquipmentHealth(score=health, status=status,
                           trend="declining" if anomaly_score > 0.5 else "stable")


def compute_station_pulse(station_id: str, equipment_healths: list[EquipmentHealth]) -> StationPulse:
    """
    全站脉搏指数：取关键设备加权平均
    关键设备（critical_path=True）权重 3 倍
    """
    weighted_score = compute_weighted_average(equipment_healths)

    # 找出"最需要关注"的信号
    attention_signals = [
        e for e in equipment_healths
        if e.status in ("critical", "warning") or e.trend == "declining"
    ]
    attention_signals.sort(key=lambda e: e.score)  # 最差的排最前

    return StationPulse(
        station_id=station_id,
        pulse_score=weighted_score,
        pulse_status=compute_status(weighted_score),
        top_attention=attention_signals[:3],  # 最多推送 3 个信号
        timestamp=utcnow()
    )
```

### 3.3 Pulse Engine 触发链

```
每30秒
  Pulse Engine 计算全站健康
         │
         ├─ 健康分变化 > 5 分 ──► SSE: PULSE_UPDATE → Studio 更新设备列表排序
         │
         ├─ 任何设备进入 "critical" ──► 事件总线: CriticalStateEntered
         │                              └─ 飞书推送：P1 级紧急卡片
         │                              └─ SSE: 全屏 InvestigationBanner
         │
         ├─ 趋势恶化（anomaly > 0.8）──► 事件总线: TrendDegrading
         │                              └─ Sage 触发预诊断（不等用户请求）
         │                              └─ 飞书: 预警卡片（附 AI 初步分析）
         │
         └─ 全站脉搏 < 60 持续 2h ──► 事件总线: StationUnderStress
                                       └─ Sage: 触发站场综合分析
                                       └─ 飞书: 推送管理层汇报卡片
```

### 4.1 Level 3 的技术实现：Action Policy Engine

```python
# engines/action/policy_engine.py

"""
行动策略引擎：定义哪些行动 AI 可以自主执行（无需人工审批）
这是 Level 2 → Level 3 跃升的关键组件
"""

@dataclass
class ActionPolicy:
    """一条自动化策略"""
    policy_id: str
    action_type: str            # 行动类型，对应 EQUIPMENT_TYPE_REGISTRY.actions
    conditions: list[Condition] # 触发条件（设备类型+指标范围）
    risk_level: str             # "low" | "medium"（高风险不允许自动执行）
    auto_execute: bool          # 是否自动执行（True = Level 3）
    notify_on_execute: list[str]  # 执行后通知哪些角色
    max_daily_executions: int   # 防止策略失控的安全上限
    requires_validation_after: int  # 执行后多少分钟需人工确认结果

# 示例策略配置（存储在 DB，Admin 可配置）
EXAMPLE_POLICIES = [
    ActionPolicy(
        policy_id="auto_kb_ingest",
        action_type="ingest_document",
        conditions=[],            # 无条件
        risk_level="low",
        auto_execute=True,        # 文档上传后自动向量化，不需要人审
        notify_on_execute=[],     # 静默执行
        max_daily_executions=100,
        requires_validation_after=0
    ),
    ActionPolicy(
        policy_id="auto_l3_knowledge",
        action_type="capture_workorder_experience",
        conditions=[Condition(field="wo.state", op="eq", value="done")],
        risk_level="low",
        auto_execute=True,        # 工单完成后自动提炼经验
        notify_on_execute=[],
        max_daily_executions=50,
        requires_validation_after=0
    ),
    ActionPolicy(
        policy_id="auto_minor_alarm_ack",
        action_type="acknowledge_alarm",
        conditions=[
            Condition(field="alarm.level", op="eq", value="P3"),
            Condition(field="alarm.metric", op="in", value=["filter_dp"]),  # 过滤器压差，例行
            Condition(field="alarm.duration_min", op="lt", value=5),
        ],
        risk_level="low",
        auto_execute=False,       # Phase B 开始可以设为 True
        notify_on_execute=["operator"],
        max_daily_executions=10,
        requires_validation_after=30
    ),
]
```

### 5.2 多巴胺触发点设计

```
触发点 1：异常发现（tension 张力）
  视觉：设备卡片从绿色→橙色→红色 渐变动画（2秒过渡）
  声音：微弱的提示音（不惊吓，引起注意）
  通知：右上角弹出告警卡片（可以不打断当前工作）
  感受：「有事情发生了，我要去看看」

触发点 2：AI 分析完成（insight 洞察）
  视觉：AI 卡片从"分析中..." → 展开显示诊断结论（渐现动画）
  内容：「轴承磨损 · 可能原因：润滑油路堵塞 · 置信度：高 · 建议：立即更换」
  操作：两个大按钮「采纳建议（创建工单）」「修改后采纳」
  感受：「AI 帮我分析完了，我只需要拍板」

触发点 3：审批通过（control 掌控）
  视觉：工单卡片状态条更新，绿色勾动画
  通知：飞书「工单 WO-2024-0512 已获批准，维修人员将在 30 分钟内到达」
  感受：「问题处理中，在我掌控之中」

触发点 4：工单完成（resolution 解决）
  视觉：设备健康环从橙色慢慢恢复绿色（2秒动画）
  通知：「C-001 振动值恢复正常，健康分 +23」
  成就：「+1 经验已录入知识库」（浮现然后消失）
  感受：「问题解决了，系统更聪明了」

触发点 5：预测命中（trust 信任）
  定期推送：「上周 Sage 预测 C-001 轴承问题，实际证实准确率 92%」
  可视化：AI 预测 vs 实际发生的时间轴对比
  感受：「AI 在变得更可信」→ 逐渐愿意扩大自动化权限
```

### 6.2 Fleet Intelligence 的 Nexus 实现

```python
# engines/fleet/fleet_intelligence.py
# 注：Fleet Intelligence 是 Nexus 的一个新引擎，聚合多站场数据

class FleetIntelligenceEngine:
    """
    跨站场智能聚合。
    当一个站场的问题在另一个站场也出现时，自动发现共性。
    """

    async def find_cross_station_patterns(self) -> list[FleetPattern]:
        """
        发现跨站场的相同故障模式。
        例：「3个站场的同型号 C-001 都在同月出现轴承问题」
        → 推送给所有站场的 Sage：「该型号轴承有批次质量问题，建议全面检查」
        """
        recent_workorders = await self.get_all_station_workorders(days=30)
        patterns = self.cluster_by_equipment_type_and_symptom(recent_workorders)
        return [p for p in patterns if p.station_count >= 2]

    async def benchmark_stations(self) -> StationBenchmark:
        """
        站场间 KPI 对标。
        让每个站长知道自己站场的绩效在舰队中处于什么位置。
        不是羞辱，是激励。
        """
        all_kpis = await self.get_all_station_kpis()
        return StationBenchmark(
            top_performer=all_kpis[0],
            fleet_average=compute_average(all_kpis),
            improvement_opportunities=identify_laggards(all_kpis)
        )
```

## 七、"AI 机器人执行"的完整闭环设计

### 7.1 从"建议"到"执行"的路径进化

```
当前（Level 2）：AI 建议 → 人审批 → 人执行 → 人记录
目标（Level 3）：AI 发现 → AI 分析 → AI 提案 → 人1-Click批准 → AI 追踪 → AI 记录

关键变化：
  · 发现：从人主动查询 → Pulse Engine 主动推送
  · 分析：从人工分析 → AI 实时生成报告（用户进入设备页面时，报告已经准备好）
  · 提案：AI 不只说"有问题"，而是给出"应该这么做"的具体工单草稿
  · 批准：从填写表单 → 1-Click（所有信息 AI 已填好，人只需要确认）
  · 追踪：AI 监控维修进度，如果进度滞后主动提醒
  · 记录：AI 自动提炼经验，写入 L3 知识库
```

### 7.2 "预备好的工单"设计（Prepared WorkOrder）

```python
# 当 Pulse Engine 检测到设备需要关注时，不等用户请求
# 提前触发 Sage 分析，准备好工单草稿

# engines/pulse/proactive_intelligence.py

class ProactiveIntelligence:
    """
    主动智能：在用户打开设备页面之前，工单草稿已经准备好
    """

    async def on_anomaly_predicted(self, event: AnomalyPredicted):
        """当 MOIRAI 预测到异常时，立即触发 Sage 分析（不等用户）"""

        if event.anomaly_score < 0.6:
            return  # 分数不够，不提前分析

        # 检查是否已有正在进行的诊断
        existing_job = await self.ai_job_repo.find_recent(
            equipment_id=event.equipment_id,
            job_type="diagnose",
            within_hours=4
        )
        if existing_job:
            return  # 最近已经分析过了，不重复

        # 创建"预备型"AI 任务（用户未请求，系统主动发起）
        job = await self.ai_job_service.create_job(
            job_type="diagnose",
            equipment_id=event.equipment_id,
            requested_by="pulse_engine",   # 系统发起，不是用户
            priority="background",         # 低优先级，不占用用户请求的 GPU 配额
            metadata={"anomaly_score": event.anomaly_score, "proactive": True}
        )

        # 当 AI 分析完成，工单草稿会在设备页面自动显示
        # 用户打开设备页面时看到的是"AI 已为您准备好分析报告"
        # 而不是"请点击'请求分析'等待..."
```

### 7.3 Sage 主动推送协议

```
目前：用户发消息 → Sage 响应（被动）
目标：Sage 主动找用户（主动）

主动推送触发条件（优先级排序）：
  P0（立即推送，任何时间）：
    · P1 告警产生
    · 设备健康分骤降 > 20 分（10分钟内）
    · Sage 发现当前异常与历史重大故障高度相似（相似度 > 85%）

  P1（推送到飞书，工作时间）：
    · P2 告警产生且未被确认 > 30 分钟
    · 预测异常分数 > 0.8 且无进行中工单
    · 跨站场发现同类故障模式

  P2（每日摘要）：
    · 晨报（07:00）：昨日概况 + 今日需关注
    · 交班前（班次结束前15分钟）：本班次摘要 + 下班需关注

  P3（周报/月报，发给管理层）：
    · 设备健康趋势分析
    · AI 准确率报告
    · KPI 对标报告
```
