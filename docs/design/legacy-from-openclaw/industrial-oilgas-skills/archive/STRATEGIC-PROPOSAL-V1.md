# 第一章　战略视野：从数字孪生到智能决策指挥

## 1.1 时代背景：工业的第四次跃迁

工业史上经历了三次重大跃迁：机械化（蒸汽）、电气化（电力）、自动化（计算机）。  
我们正处于第四次跃迁的起点：**智能化——由 AI 驱动、由数据定义的工业新范式**。

领先指标已经出现：

- Palantir AIP 让军事/情报决策指挥从"地图+电话"升级到"实时数据+AI 推荐+一键行动"
- NVIDIA 将物理仿真引擎（Omniverse/PhysX）与大模型融合，生成"能理解物理规律的世界模型"
- Siemens/ABB 的数字孪生开始接受 LLM 作为推理层，而非只做可视化
- 中国工业现场：操作员仍在用纸质巡检表、微信群传图、Excel 统计异常

**机会窗口**：AI 能力已经可用，工业场景的数据主权要求私有化，两者叠加创造了独特的战略空间。

## 1.2 参照对象：Palantir 的决策指挥体系

Palantir 最重要的贡献不是「数据库」，而是**决策指挥的新范式**：

```
传统范式：
  数据 → 报表 → 人分析 → 人决策 → 人执行
  问题：慢、孤岛、无法闭环、AI 无法介入

Palantir 范式：
  实时数据融合（Foundry）
    → 语义层（Ontology）：所有数据挂载到业务对象
      → AI 推理（AIP）：基于对象提供建议与行动草稿
        → 人在回路（HITL）：人审批 AI 草稿，高风险动作人执行
          → 闭环记录：每次动作写回对象，持续学习
```

**对工业场站的启示**：

- 压缩机不只是一行数据表，而是一个**有历史、有知识、有预测、可行动**的对象
- 操作员面对的不是仪表盘数字，而是**会说话的设备**（AI 解读、预警、建议）
- 异常不是等报警铃响才知道，而是**AI 预判**、在阈值触发前提示
- 审批不是电话沟通，而是**飞书卡片一键**、带上下文、可追溯

# 第二章　四层 AI 模型架构：工业世界的数字映射

这是 ClawTwin 区别于普通"IoT 平台+ChatBot"的核心技术架构。  
我们建立四个相互支撑的 AI 模型层，形成**从物理现实到智能决策的完整映射链**。

```
现实世界                     数字世界（ClawTwin）
─────────────────────────────────────────────────────────────
天然气压缩机       ←→      [Layer 4] 世界模型 / 数字孪生
  运转、振动                  3D 实体 + 实时状态 + 空间关系

设备内部物理过程   ←→      [Layer 3] 物理模型
  流量、压差、热场            简化物理仿真 + 过程约束

时序传感器数据     ←→      [Layer 2] 时序 AI 模型
  振动频谱、压力曲线          MOIRAI 异常检测 + 趋势预测

操作员知识经验     ←→      [Layer 1] 语言大模型（LLM）
  规程、案例、判断            Qwen3 推理 + 知识检索 + 行动规划
```

## 2.1 Layer 1：语言层——LLM 与知识体系

**技术**：Qwen3-235B-A22B INT4（vLLM，私有化 GPU 推理）+ BAAI/bge-m3（向量检索）

**知识分层**（L0 → L3，越来越专）：

| 层级 | 内容                                | 来源               | 用途                   |
| ---- | ----------------------------------- | ------------------ | ---------------------- |
| L0   | 通用行业标准（GB/T、ISO、API 标准） | 公开文献、厂商手册 | 基础问答，合规校验     |
| L1   | 行业深度知识（故障模式、维修方法）  | ClawTwin 知识包    | 异常诊断，方案推荐     |
| L2   | 企业内部规程（客户操作手册、预案）  | 客户上传           | 操作指引，安全规程     |
| L3   | 本站实战经验（历史工单、维修记录）  | 工单完成自动沉淀   | 「这台机上次怎么修的」 |

**关键能力**：

- **Grounding**：所有 AI 回答必须 citations 到具体文档/工单/读数时间戳，不允许无来源幻觉
- **上下文注入**：提问时自动附加设备当前状态，AI 基于实时数据推理，不是泛泛而谈
- **分级检索**：先搜 L3 本站经验 → L2 企业规程 → L1 行业知识 → L0 标准，越具体越优先

## 2.2 Layer 2：时序层——MOIRAI 预测与异常检测

**技术**：Salesforce MOIRAI 2.0 Large（零样本时序基础模型）

**为什么时序模型独立于 LLM**：

```
阈值告警（传统）：  振动 > 5.0 mm/s → 触发告警
  问题：只能发现已知问题；告警风暴；事后响应

MOIRAI 预测（我们）：
  输入：过去 7 天的振动/温度/压力时序
  输出：未来 24 小时的预测区间 + 偏离置信度
  价值：在问题发生前 6-12 小时预警
       不需要标注数据（零样本）
       自动适应每台设备的「正常基线」
```

**告警分级**：

| 级别          | 条件                            | 响应                       |
| ------------- | ------------------------------- | -------------------------- |
| P1（ALARM）   | MOIRAI 置信度 > 0.85 或阈值触发 | 飞书即时推送 + Studio 红色 |
| P2（WARNING） | MOIRAI 置信度 0.6-0.85          | 飞书卡片 + Studio 黄色     |
| P3（WATCH）   | 趋势偏离基线但未达预警          | Studio 蓝色，不推送        |

## 2.3 Layer 3：物理层——设备行为约束模型

**技术**：简化物理方程 + 厂商参数 + Phase C 接 OpenFOAM/Ansys API（可选）

**Phase A（规则约束）**：

```python
# 基于物理规律的约束检查（不需要全量仿真）
# 例：压缩机出口压力 vs 进口压力 vs 功率的物理关系
def check_physical_consistency(readings: EquipmentReadings) -> list[str]:
    violations = []
    # 多变指数 n = ln(P2/P1) / ln(V1/V2)，正常范围 1.2-1.4
    if readings.compression_ratio > 0 and readings.power_kw > 0:
        theoretical_power = calculate_theoretical_power(readings)
        actual_efficiency = theoretical_power / readings.power_kw
        if actual_efficiency < 0.7:
            violations.append(f"压缩效率异常低 ({actual_efficiency:.1%})")
    return violations
```

**Phase B（数字孪生仿真）**：

- Eclipse Ditto 存储设备实时状态（孪生运行时）
- 物理参数（摩擦系数、热容）挂载到设备 Ontology 对象
- 孪生状态驱动 3D 材质和动画（运转中/停机/异常）

**Phase C（高保真仿真）**：

- 接 OpenFOAM/Ansys 计算流体动力学（按需，单次仿真任务）
- 仿真结果入 Milvus 知识库，未来类似场景直接检索（不重复计算）

## 2.4 Layer 4：世界层——数字孪生 3D 实体

**技术**：Babylon.js 8 + WebGPU（浏览器级光追）

**从点到世界的三个层次**：

```
Level A（Phase A，当前可交付）：
  设备抽象表示：压缩机=红/黄/绿色方块
  空间布局：按真实坐标摆放（工程图坐标导入）
  状态驱动：实时颜色 + 悬浮数值标注

Level B（Phase B，6-12 个月）：
  CAD 模型轻量化导入（FBX/glTF）
  PBR 材质（金属、管道、外壳的真实渲染）
  管道系统：粒子流动动画（压力可视化）
  异常部位高亮（3D 模型上标注问题位置）

Level C（Phase C，12-24 个月）：
  HDRI 环境光（天气/时间真实感渲染）
  设备内部可视化（X 射线模式：看轴承磨损位置）
  物理仿真可视化（流场颜色图、温度分布）
  AR 眼镜支持（WebXR 协议）
```

**Palantir 地图范式在 3D 孪生中的应用**：

```
Palantir Gotham：地图作为「所有信息的空间锚点」
  → 事件发生在地图上某个位置
  → 点击位置看到完整情报链
  → 在地图上标记行动

ClawTwin TwinSurface：3D 孪生作为「所有信息的空间锚点」
  → 告警发生在 3D 空间中某台设备上
  → 点击设备看到完整数据链（实时+历史+知识+工单）
  → 在 3D 场景中发起工单草稿
```

## 2.5 四层模型的协同推理链

四层模型不是孤立的，在每次 AI 响应中形成完整的**证据链**：

```
用户提问："2号压缩机今天振动偏高，是什么问题？"
                        │
              ┌─────────▼──────────┐
              │  Layer 4 世界模型   │  → 定位 2号压缩机的 3D 位置和实时状态
              │  （空间理解）        │    当前振动 4.8 mm/s，环境温度 32°C
              └─────────┬──────────┘
                        │提供实体上下文
              ┌─────────▼──────────┐
              │  Layer 2 时序模型   │  → MOIRAI 分析过去 7 天振动趋势
              │  （时序理解）        │    发现：从 3 天前开始持续爬升
              └─────────┬──────────┘    预测：6 小时内可能超 P1 阈值
                        │提供异常证据
              ┌─────────▼──────────┐
              │  Layer 3 物理模型   │  → 振动-温度-功率三者关系校验
              │  （因果约束）        │    排除：温度影响（在正常范围）
              └─────────┬──────────┘    指向：轴承磨损或对中偏差
                        │提供物理约束
              ┌─────────▼──────────┐
              │  Layer 1 语言模型   │  → 搜索 L3 本站知识："2号机历史工单"
              │  （知识推理）        │    找到：2024-03 曾更换 6203 轴承
              └─────────┬──────────┘    检索 L1 知识：轴承磨损早期特征
                        │
              ┌─────────▼──────────────────────────────────────────┐
              │  AI 回答（含完整 citations）：                        │
              │  "2号压缩机振动从3天前持续上升（数据：4.2→4.8mm/s）  │
              │   物理分析排除温度因素，结合轴承磨损特征，建议检查    │
              │   6203轴承状态（参考：2024-03 工单 WO-031，同台设备  │
              │   同类故障）。预计6小时内超P1阈值，建议提前安排检修。" │
              └────────────────────────────────────────────────────┘
                  → Studio 一键草稿工单 → 飞书推送主管审批
```

**关键价值**：AI 的每个结论都有四个层次的证据支撑，不是"模型感觉"，而是可溯源的工程判断。

## 4.2 AI 交互的五个嵌入节点（不是聊天框）

AI 不是旁边一个聊天窗口，而是渗透到每一个操作节点：

```
节点 1：设备选中 → AI 自动分析（无需触发）
  用户行为：点击设备

  用时：2-4 秒

节点 2：工单草稿 → AI 自动填写
  用户行为：点击「建工单」
  AI 行为：填写问题描述、处理方案、所需备件、安全注意事项
  用时：3-6 秒（可同时编辑）

节点 3：审批时 → AI 附上风险评估
  用户行为：主管打开待审批工单
  AI 行为：自动附加「延迟处理的历史案例」和「预计影响」
  用时：2-3 秒

节点 4：完工后 → AI 提炼 L3 知识
  用户行为：操作员标记工单 DONE，填写实际操作
  AI 行为：提炼结构化经验 → 写入 L3 知识库
  用时：后台，不影响操作员

节点 5：晨报/交班 → AI 生成摘要
  触发：定时（早 8 点）或操作员主动请求
  AI 行为：生成「昨日告警数 / 处置情况 / 今日关注点」
  交付：飞书消息卡片
```

# 附录 B：给 IT 部门的问答

**Q: 这套系统会替代我们现有的 SCADA/ERP 吗？**  
A: 不会。ClawTwin **读取** SCADA/ERP 的数据，做 AI 分析和 HITL 工单管理，不替代任何现有系统。现有系统继续作为主系统运行。

**Q: 数据安全怎么保证？数据会上传到云吗？**  
A: 全部私有化部署在客户服务器，数据不离开客户网络。AI 推理也在客户 GPU 服务器上运行，不调用任何外部 AI 接口。

**Q: 接入 OT 网络（SCADA/DCS）安全吗？**  
A: opcua-bridge 部署在 DMZ 隔离区，单向数据推送（只从 OT 读，不向 OT 写），符合工业安全规范。如果不接 OPC-UA，系统 Phase A 完全在 IT 网运行，对 OT 网无影响。

**Q: 飞书审批通过后，系统会自动操作设备吗？**  
A: 不会。审批通过后，系统发送飞书通知给执行人，由人去现场执行。系统不发控制指令给 DCS/PLC。

**Q: 如果 AI 给出错误建议怎么办？**  
A: AI 的所有建议都有 citations（来源标注），审批人可以验证依据。高风险动作必须主管审批（HITL），AI 不能自主执行。系统设计上假设 AI 可能出错，由人来把关。

**Q: 以后如果换模型/换数据库，迁移成本高吗？**  
A: LLM 换模型只需修改配置，兼容 OpenAI API 格式的模型都能用。数据库替换成本较高，但 Platform API 对外接口不变，上层应用不受影响。

**Q: 我们公司已经有 HiAgent（华为企业智能体平台），和这个有冲突吗？**  
A: 没有冲突，可以共存。HiAgent 和 OpenClaw 都通过 Platform 的 Tool API 访问工业数据，Platform 用 Service Token 管理两个系统的权限和场站访问范围，互不干扰。HiAgent 可以作为工作流引擎处理复杂流程，OpenClaw 处理实时 AI 对话，两者分工互补。

**Q: 我们公司有 OA 审批系统，工单审批要走 OA 流程，怎么对接？**  
A: Platform 支持 OA 回调接口。工单提交后，Platform 调用 OA 系统创建审批任务；OA 审批完成后，OA 系统回调 Platform 的 `/v1/hitl/workorders/{id}/oa-callback` 接口，Platform 更新工单状态并通知飞书。OA 系统不需要改造，只需配置一个 Webhook 回调地址。

**Q: 我们已有飞书私有化，版本可能不是最新的，有兼容性问题吗？**  
A: Platform 使用飞书的 Bot 消息推送和 Webhook 功能，这些是飞书私有化版本的核心功能，兼容性一般没有问题。具体需要确认私有化飞书的 Bot API 版本，提前做兼容性测试。如有问题，告警通知可以退化为邮件/企业微信。

**Q: 这个系统最少需要几台服务器、最少要多少 GPU？**  
A: Phase A（demo 阶段）：1 台服务器（32 核 64G）即可运行全部服务（不含 GPU，用 API 接外部模型）。生产最低配置：2 台服务器 + 1 张 A10/RTX4090 GPU。GPU 不是硬性要求，无 GPU 时接硅基流动/阿里云百炼 API 也可以运行，只是数据出厂问题需要单独评估。

**Q: 维护人员需要什么技能水平？**  
A: 日常维护（看日志、重启服务、知识库上传）：Linux 基础操作即可。知识库管理（PDF 上传审核）：业务人员通过 Studio Admin UI 操作，不需要技术背景。版本升级/配置变更：需要能操作 Docker Compose 的运维人员或我们远程支持。
