## 一、哲学层：我们在做什么？

### 1.1 人类知识的三次工业化

```
第一次：文字的发明
  知识从「只能口耳相传」变成「可以书写存储」
  效果：知识跨越了时间，可以积累

第二次：印刷术的发明
  知识从「只能手工复制」变成「可以大规模传播」
  效果：知识跨越了空间，可以共享

第三次：AI 的崛起（现在）
  知识从「人读人懂才能使用」变成「机器可以推理执行」
  效果：知识跨越了认知门槛，可以自动化
```

**ClawTwin 正在做的是第三次工业化的工业领域版本**：

把油气场站几十年积累的**操作经验、设备规律、故障模式、最佳实践**——这些过去只存在于老工程师脑子里的知识——转化为**AI 可以推理、可以调用、可以传递的计算形式**。

这不是「做一个 BI 系统」，不是「加了 AI 的 SCADA」。  
这是**让工业知识具备了计算属性**。

### 1.2 三个问题的终极答案

```
问题 1：我们在做什么产品？
答：工业知识的「可计算化」基础设施

问题 2：为什么这件事有价值？
答：石油天然气行业每年因设备异常损失数百亿，
    根因是「知识无法及时传递到需要它的人和系统」
    我们解决这个根因，而不是症状

问题 3：为什么是现在？
答：LLM 的出现让「非结构化知识的推理」第一次成为可能
    OT/IT 融合趋势让「数据获取」门槛降低
    工业老龄化危机让「知识传承」紧迫性史无前例
```

### 1.3 用户的真实需求（不是他们说的那个）

用户说的需求：「我需要一个数字孪生系统」  
用户真实的需求：「我不想因为老工程师退休就失去几十年经验」

用户说的需求：「我需要实时监控设备状态」  
用户真实的需求：「我不想在异常变成事故之前还不知道」

用户说的需求：「我需要一个 AI 助手」  
用户真实的需求：「我不想让操作员因为信息不足做出错误决策」

**哲学家的洞见**：人类追求的不是工具，而是**确定性**（Certainty）。  
工业场站最大的焦虑是：「我不知道下一刻会发生什么」。  
ClawTwin 卖的是**工业确定性**——不是监控，是预知；不是数据，是判断。

## 二、产品层：应该是什么结构？

### 2.1 核心洞见：三层分离

所有成功的平台产品都有一个共同结构：

```
Salesforce =  Schema（对象定义）  + Runtime（数据存储+流程）+ Intelligence（AI/规则）
Shopify    =  Schema（商品/订单）  + Runtime（库存/支付）  + Intelligence（推荐/分析）
Palantir   =  Schema（Ontology）  + Runtime（Foundry）   + Intelligence（AIP）
Tesla      =  Schema（车辆模型）  + Runtime（OTA/车机）  + Intelligence（Autopilot）

ClawTwin 应该是：
             Schema（工业本体）  + Runtime（孪生运行时）+ Intelligence（工业技能）
```

**这三层有本质差异**，必须清晰分离：

```
Schema 层（工业世界的语言）
  定义「什么是什么」
  ·  设备有哪些类型？每种类型有哪些指标？
  ·  正常范围是什么？异常如何分级？
  ·  设备之间有什么关系？
  ·  可以对设备执行哪些操作？
  特点：相对稳定，更新慢，可跨场站共享，可标准化
  价值：为 Runtime 和 Intelligence 提供「语义地基」

Runtime 层（工业世界的实时镜像）
  映射「正在发生什么」
  ·  每台设备现在的真实状态是什么？
  ·  最近发生了哪些事件？
  ·  有哪些未处理的工单和告警？
  ·  历史趋势是怎样的？
  特点：高频更新，站场私有，不可跨站共享，时序特性强
  价值：让 Intelligence 的推理有真实根基

Intelligence 层（工业世界的推理者）
  决策「应该做什么」
  ·  这个读数代表什么问题？
  ·  应该采取什么行动？
  ·  预计多久后会发生故障？
  ·  知识库里有没有处理先例？
  特点：调用 LLM，依赖 Schema + Runtime 的数据，高价值
  价值：把「数据」变成「决策建议」
```

### 2.3 用户界面的正确定位

用户界面不是产品，而是产品的**呈现层**，服务于不同的用户角色：

```
Studio（PC 工作台）         → 工程师 / 调度员 / 管理层（深度操作）
飞书 Bot + 卡片            → 操作员 / 班长（高频轻量交互）
飞书 OA 审批               → 管理层（HITL 审批，在已有习惯里）
API / MCP 接口             → 第三方系统 / 合作伙伴开发者
大屏 Command 视图          → 指挥中心 / 汇报展示

关键原则：用户在哪里工作，AI 就出现在哪里
          而不是：「来我们的系统里用 AI」
```

## 三、架构层：如何实现这个结构？

### 3.1 组件与产品的映射关系

```
ClawTwin Schema（本体定义）
  实现：Platform 中的 /v1/ontology/* API
        PostgreSQL 中的 equipment_types / metrics / actions 表
        Studio Admin 中的本体管理界面
  未来：独立微服务，可被多个 Runtime 实例共享

ClawTwin Runtime（运行平台）
  实现：Platform 核心服务（FastAPI）
        TimescaleDB（时序数据）
        Milvus（知识向量）
        Redis（实时状态缓存）
        Kafka（事件总线）
        connectors/（企业系统连接器）
  未来：多租户运营，Schema 共享，Runtime 隔离

ClawTwin Intelligence（AI 技能包）
  实现：OpenClaw 部署 + Industry Skills
        GPU Server（vLLM + bge-m3 + MOIRAI）
        Prompt 模板库（版本化管理）
  未来：Skills 可独立更新/升级，不影响 Runtime
```

### 3.2 数据流的三个方向

```
方向 A：物理世界 → 数字世界（「感知」）
  OPC-UA → opcua-bridge → Kafka → Runtime
  目的：让系统知道「现在发生了什么」

方向 B：知识世界 → 推理空间（「理解」）
  文档/规程 → Knowledge Pipeline → Milvus
  目的：让 AI 有「可信的依据」

方向 C：AI 推理 → 行动执行（「决策」）
  AI 分析 → 工单草稿 → HITL 审批 → 企业系统执行
  目的：把「洞察」变成「改变」

三个方向形成闭环：
  行动执行的结果（工单历史/维修记录）
  → 自动回流到知识世界（L3 知识积累）
  → 下次 AI 推理更准确
  → 「越用越聪明」的数据飞轮
```

### 3.3 关键设计原则：可替换性

每个组件都应该可被替换，而不影响其他层：

```
替换场景                    影响范围               保护机制
──────────────────────────────────────────────────────────────
换 LLM（Qwen → Claude）    只影响 Intelligence    Skill 接口不变
换向量库（Milvus → 其他）  只影响 Runtime 存储    KB 搜索 API 不变
换 DCS（Honeywell → ABB）  只影响 OT 连接器       IMS Adapter SDK
换 CMMS（Maximo → 普联）   只影响企业连接器       Connector SDK
换审批流（飞书 → 钉钉）    只影响通知渠道         Channel 抽象层
增加新设备类型             只影响 Schema         本体 API 扩展
增加新行业（化工）         增加 Schema + Pack     不改 Runtime 核心
```

## 四、商业层：如何构建护城河？

### 4.1 数据飞轮：真正的竞争壁垒

```
传统软件的竞争壁垒：功能多、价格低、服务好
AI 时代的竞争壁垒：数据飞轮 + 知识积累

ClawTwin 的数据飞轮：

客户 A 部署 → 积累工单/经验/告警处理数据
            ↓
          知识提炼（脱敏后的模式：「这类振动 → 这类故障」）
            ↓
          Intelligence Pack 质量提升
            ↓
客户 B 部署 → 起点更高（冷启动问题消失）
            ↓
          积累更多数据，Pack 继续提升
            ↓
更多客户部署 → 单个 Skill 的准确率接近「行业专家级别」
            ↓
          竞争对手很难追上（因为他们没有这些历史数据）
```

关键：数据飞轮需要 **客户信任 + 脱敏协议 + 行业验证**。  
不是偷数据，而是「用你的数据提升你（和行业）的 AI 能力」。

### 4.2 三层商业变现

```
Layer 1：Runtime 部署（项目收入，一次性 + 维护）
  部署 ClawTwin Runtime 到客户站场
  包含：实施服务、培训、年度维护
  特点：收入稳定，但需要团队扩张
  目标客户：大中型石油公司生产部门

Layer 2：Intelligence Pack 订阅（产品收入，经常性）
  按行业/场景打包 AI 技能
  包含：最新 Prompt 模板、知识更新、模型升级
  特点：边际成本低，随客户数量扩张收入
  目标客户：所有 Runtime 部署客户（自动转化）

Layer 3：Schema 标准化（生态收入，平台费用）
  成为行业本体标准的制定者
  包含：标准 Schema 库、认证、互操作性
  特点：高利润，建立行业地位
  目标客户：行业协会、设备商、软件集成商

最优先获取的是 Layer 1（活下去）
最值钱的是 Layer 2 的数量（估值驱动）
最有战略价值的是 Layer 3（行业话语权）
```

### 4.3 关键项目战略：如何获得关键数据

```
「通过关键产品获得关键项目、关键数据、关键发展机会」

关键项目 = 行业标杆客户（如中石油/中石化一个地区公司）
           价值：背书、数据、技术打磨

如何获得关键项目：
  1. 先做「AI 诊断试用」（低门槛，0 部署成本）
     → 用客户真实数据运行 Intelligence Pack，给出诊断报告
     → 证明 AI 有实际价值，再谈完整部署

  2. 从「痛点最急的场景」切入（不从全系统开始）
     → 聚焦「设备异常发现」这一个场景
     → 让客户两周内看到价值
     → 再扩展到工单/知识库/分析

  3. 定价策略：「结果付费」（对早期客户）
     → 按「AI 发现的有效异常数量」收费
     → 客户没风险，我们有压力打磨质量
     → 同时建立「AI 诊断准确率」的行业标准
```

```

### 6.3 架构调整优先级（以下是对现有设计的修正方向）

```
立即（Phase A，影响交付定位）：
  · 将 Schema 相关 API 从 Platform 中明确标记为独立层
    /v1/ontology/* 是 Schema 层，/v1/objects/* 是 Runtime 层

  · Intelligence Pack 可以独立版本化（Skill 更新不需要重新部署 Platform）
  · Connector SDK 对外开放文档（合作伙伴可以自己写连接器）

中期（Phase C，影响商业模式）：
  · Schema 可以跨多个 Runtime 实例共享（标准 Schema 库）
  · Intelligence Pack 可以独立销售和更新
  · 数据飞轮机制建立（匿名经验提炼，优化 Intelligence Pack）

## 七、简洁产品定义（给不同受众）

### 7.1 给客户 CIO（30 秒版）

> ClawTwin 把你现有的设备数据、操作规程和工程经验  
> 转化为 AI 可以推理的知识，  
> 让操作员用自然语言管理设备，  
> 让 AI 在异常变成事故之前就提出处理建议，  
> 并通过现有的飞书审批流执行——  
> 不替换你的任何系统，不把数据送出你的内网。

### 7.2 给工厂经理（30 秒版）

> 你的压缩机振动超标，过去要靠老师傅凭经验判断。  
> 现在 AI 分析历史数据 + 设备手册 + 维修记录，
