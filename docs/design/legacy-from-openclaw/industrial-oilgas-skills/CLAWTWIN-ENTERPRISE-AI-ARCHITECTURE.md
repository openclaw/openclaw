# ClawTwin 企业 AI 架构（客户向）

**地位**: 🟢 核心 / 产品定位权威  
**版本**: v1.0.0 (2026-05-13)  
**读者**: 客户决策层、IT 架构师、解决方案工程师  
**关键原则**: 充分利用已有资源，不重复建设，与现有 IT 自洽

---

## 一、一句话架构

> 在您现有的 IT/OT 系统上，加一层**运营语义平台**（ClawTwin Platform），为运营人员提供**数字化工作台**（ClawTwin Studio），并接入**AI 智能体**（OpenClaw 或其他 Agent），三者协作形成完整的企业 AI 运营体系。

## 二、与 Palantir 产品家族的对照（客户熟悉的参照系）

```
Palantir 产品家族                    ClawTwin 产品家族
────────────────────────────────     ────────────────────────────────
Gotham       用户智能应用层      ←→  ClawTwin Studio    运营工作台
Foundry      数据语义平台       ─┐
AIP          AI 行动平台       ─┼→  ClawTwin Platform  运营 AI 平台
Apollo       部署运维平台       ─┘
（AIP Assist 对话 AI）          ←→  OpenClaw（外部 AI 智能体）
```

**关键差异**：Palantir 的 AIP Assist（对话 AI）是 Palantir 自己提供的闭源产品；ClawTwin 的 AI 智能体是**开放接入**（OpenClaw / Coze / Dify / 任何支持 MCP 的 Agent），客户可以选择最适合自己的智能体，不被锁定。

按 Gotham / Foundry / AIP / Apollo **对外介绍口径**展开的叙事版（演讲稿友好、无大图依赖）见 **`CLAWTWIN-PALANTIR-POSITIONING.md`**。

## 四、Platform 的 AI 分工（最重要的设计澄清）

**Platform 不是 AI 智能体。** Platform 做两件明确不同的 AI 相关事情：

### 4.1 FunctionType 单次 AI 调用（Platform 自己做）

```
触发：Playbook 步骤 / API 请求 / 定时任务
  ↓
FunctionExecutor.execute("DiagnoseEquipment", params={equipment_id: "C-001"})
  ↓
单次 LLM 调用（结构化输入 → 结构化输出，< 8 秒）
  ↓
返回：{summary: "振动偏高，建议检查轴承", confidence: 0.82, actions: [...]}
```

**类比**：这不是在"和 AI 对话"，而是"查询一个 AI 数据库"——像 SQL 查询一样确定性，只是底层用了 LLM。自动化 Playbook 3 点运行不需要人，不需要对话，只需要这个。

### 4.2 复杂推理委托给外部 Agent（OpenClaw 做）

```
触发：置信度 < 阈值 / Playbook 标记 "需要复杂推理" / 用户在飞书发问
  ↓
AgentRuntime.send_task(AgentTask(kind="reasoning", context=业务上下文))
  ↓
OpenClaw（或 Coze/Dify）处理：多步推理 · 工具调用循环 · 会话记忆
  ↓
返回结构化结论 → Platform 执行下一步 Playbook 步骤
```

**Platform 给 Agent 提供什么（通过 MCP Server）：**

- 设备当前状态和历史数据
- 相似案例（CBR 推荐）
- 可执行的操作列表（ActionType）
- 知识库片段（相关规程/经验）

**Agent 做什么**：推理、对话、规划——Platform 不做这些。

### 4.3 分工图

```
                   AI 能力分工
─────────────────────────────────────────────────────
问题类型          谁来做          为什么
─────────────────────────────────────────────────────
诊断函数调用      Platform 直接   确定性，< 8秒，自动化可靠
  (单次结构化)    (FunctionType)

告警推送通知      Platform 直接   Outbox 可靠投递，不需要 AI
  (Channel)      (EventDispatcher)

复杂推理分析      OpenClaw       多步、上下文、工具调用循环
  (多步、对话)    (AgentRuntime)

自然语言交互      OpenClaw       对话记忆、意图理解、NL生成
  (飞书/钉钉)    (AgentRuntime)

知识生成/总结     OpenClaw       长文本生成，需要会话上下文
  (KB 草稿)      (委托后回写)
─────────────────────────────────────────────────────
```

## 五、与现有 IT 系统的集成层次

客户的现有系统**一个都不需要替换**，ClawTwin 是叠加层：

```
集成方式         方向          说明
─────────────────────────────────────────────────────────────
OPC-UA/Modbus   OT → Platform  实时设备数据流入（只读）
ERP Webhook     ERP → Platform  工单状态回写（双向可选）
MES API         Platform → MES  生产指令下发（按需）
飞书/钉钉        双向            告警推送 + 用户指令接收
OpenClaw MCP    双向            AI 工具调用 + 推理结果接收
Grafana         Platform → BI   数据导出（只读）
─────────────────────────────────────────────────────────────
```

**原则**：OT 系统（SCADA/DCS）永远是**只读来源**，ClawTwin 不写回控制系统——这是工业安全边界。写回只发生在 L3/L4 管理系统（ERP/MES），且需要人工审批或高置信度门控。

## 七、模块逐步交付路径（与现有 IT 自洽）

企业 AI 化不是替换现有系统，而是**分层叠加**，每层都有独立价值：

```
阶段 0：存量盘点（1-2 周）
  ─────────────────────────────────────────────
  现有系统摸底 → 本体映射（把现有数据描述清楚）
  不部署任何新系统，只是用 ClawTwin Ontology YAML 把现有资产描述清楚

阶段 1：数字化底座（2-4 周）
  ─────────────────────────────────────────────
  部署：Platform Core + Studio（只读模式）
  接入：OPC-UA → equipment_readings（只读，不改 OT）
  价值：所有设备状态可见，告警统一管理，工单数字化
  替代：Excel 台账 + 纸质工单

阶段 2：AI 辅助（4-8 周）
  ─────────────────────────────────────────────
  部署：Intelligent SKU（+ai +kb +recommendations）
  接入：OpenClaw（通过 MCP）
  价值：工程师在飞书问"C-001 最近怎么了"，AI 基于真实数据回答
  替代：人工经验判断 + 电话汇报

阶段 3：工作流自动化（2-3 月）
  ─────────────────────────────────────────────
  部署：Autonomous SKU（+playbook +outcome_tracking）
  配置：Playbook YAML（告警→诊断→推荐→HITL 审批）
  价值：80% 常规告警自动处理，工程师只需审批异常
  替代：人工排班轮值 + 人工经验决策

阶段 4：深度集成（3-6 月）
  ─────────────────────────────────────────────
  对接：ERP/MES（双向工单同步）
  开启：飞轮学习（OutcomeEvent → 推荐越来越准）
  价值：运营数据进入企业管理层，AI 推荐持续优化

阶段 5：自治运营（6 月+）
  ─────────────────────────────────────────────
  实现：L3 高置信度自动执行，L4 规则化全自动
  价值：从"AI 辅助人"到"AI 主导+人审批异常"
```

## 八、客户常见顾虑与解答

| 顾虑                             | 解答                                                                                   |
| -------------------------------- | -------------------------------------------------------------------------------------- |
| "又要买新系统，现有系统怎么办？" | ClawTwin 是叠加层，现有 SCADA/ERP 全部保留，只在上面加一层语义理解和 AI 能力           |
| "AI 会控制我们的设备吗？"        | 不会。OT 系统只读接入。写回控制系统有硬件隔离要求；写 ERP/MES 需要人工审批或明确授权   |
| "AI 出错了怎么办？"              | HITL（人机协同）门控，置信度低于阈值时强制人工审批。飞轮学习持续提升准确率             |
| "AI 供应商锁定怎么办？"          | 客户自带 OpenAI/Anthropic/Ollama key；也可以换用 Coze/Dify/本地模型，Platform 代码不变 |
| "数据安全怎么保障？"             | 私有部署，数据不出厂；AI 调用本地 Ollama 可完全离线                                    |
| "运维团队够用吗？"               | Doctor 自检 + CLI 一键诊断 + ReloadPlan 热重载，单人运维一个站点没问题                 |
| "这和 Palantir 什么区别？"       | 同样的架构思想，但：私有部署、开放扩展、客户自带 AI、没有天价合同                      |
