## 执行摘要（结论先行）

1. **建设内容**：在**不替换** SCADA / IMS / 企业 IM 的前提下，落地**工业语义层 + 指挥中心 + IM 协同**，形成告警—规程—工单—审计的**同源口径**。
2. **客户价值**：一线「问得到、点得开依据」；管理「看得见、批得动、追得清」；资产随时间沉淀为**可治理对象与知识**。
3. **交付形态**：分阶段交付可演示闭环与可验收接口；**目标架构**以设计权威（Foundry / LOCK）为准，**当期实现范围**以合同与阶段\_EXIT 清单为准。

---

## 情境：行业常态

- 系统已多：**IMS、历史学家、MES/CMMS、SCADA** 与 **企业 IM** 并存。
- AI 已进入现场：**企业级 Agent / 网关**常与飞书同源，而非再装孤岛 App。
- 真实痛点不在「缺一个 AI」，而在 **口径分散、规程难举证、闭环难追溯**。

---

## 张力：若不建协同层会怎样

| 现象            | 业务风险                               |
| --------------- | -------------------------------------- |
| 多入口各自表述  | 值站/设备/安监对「同一告警」叙述不一致 |
| 规程在 PDF/口头 | 处置难对齐标准，复盘难取证             |
| 流程跨系统断裂  | 审批与现场执行脱节，责任界面模糊       |

→ 需要一层**可治理的语义与协同中枢**，而非再堆一个「大屏」。

---

## 命题：我们卖的是什么

**不是**替代 ERP / SCADA / 飞书 / 企业 AI 平台。

**是**工业 **Foundry 式**能力：**对象化语义 + 可执行动作 + 可审计协同**，把已有系统与 IM 缝成**同一张业务图**。

---

## 建设方案总览（三根支柱）

| 支柱           | 对外表述                                    | 对内落点（设计权威）                     |
| -------------- | ------------------------------------------- | ---------------------------------------- |
| **同源对象**   | 设备 / 告警 / 工单 / 知识在同一套口径下管理 | Industrial Ontology + ObjectStore        |
| **双入口体验** | 一线 IM · 调度/工程师 Web                   | `apps/feishu` · `apps/http`（JWT + SSE） |
| **可控集成**   | OT 侧**只读汇聚**；写操作走审批与权限       | IMS/OT Connector + Marking / HITL FSM    |

## 产品架构：七层模块化（对齐研发结构）

自上而下，每层**一句话**：做什么、不负责什么。

| 层（`platform-api/`） | 负责                              | 不负责                       |
| --------------------- | --------------------------------- | ---------------------------- |
| **apps**              | HTTP、飞书卡片/事件适配、SSE 出口 | SCADA **下控**、聊天消息总线 |
| **aip**               | MCP、`/mcp`、多 AgentRuntime 适配 | 替代网关内会话编排           |
| **ontology**          | 对象 / 动作 / 函数声明            | 业务硬编码散落在各路由器     |
| **core**              | 对象持久化、Action 执行、领域 FSM | 绕过本体直写库的「野接口」   |
| **connectors**        | IMS · OT · KB 标准化接入包        | 客户核心系统选型             |
| **providers**         | 模型、向量、通道通知等可插拔      | 绑死单一供应商               |
| **infra + workers**   | 身份·Marking·审计·定时·流消费     | 忽视合规与可追溯             |

## 三种「对话」的路径（沟通必讲）

| #     | 场景                         | 路径（模块语义）                                          | 备注                |
| ----- | ---------------------------- | --------------------------------------------------------- | ------------------- |
| **A** | IM 里 @机器人问工况 / 规程   | 飞书 → **OpenClaw** → **MCP `/mcp`** → **core**，再回会话 | IM **不是**数据源   |
| **B** | 卡片上点「同意」「提交巡检」 | 飞书 → **`POST /v1/feishu/events`** → **ActionExecutor**  | 与 A **分流**       |
| **C** | 指挥中心操作台               | 浏览器 JWT → **`/v1/*` + SSE**                            | 与 A 并行，同源对象 |

→ 避免画出「聊天进 events」的箭头（与设计终态 LOCK 相悖）。

## 数据与安全边界（信任状）

| 边界            | 原则                                                        |
| --------------- | ----------------------------------------------------------- |
| **OT ↔ 平台**   | 采集与健康分析 **只读汇聚**为主；控制风险的操作走工单与授权 |
| **飞书 ↔ 平台** | 协同与身份；不把 IM 当作工业时序仓库                        |
| **身份**        | JWT + Marking（场站/角色）；可与企业帐号体系对接            |
