# OpenClaw — 技术概述

## 1. 技术架构

OpenClaw 是一款个人 AI 助手，以单一后台守护进程（Gateway）运行在用户自有硬件上。Gateway 统一管理所有通道连接、会话状态和工具执行，将来自 12+ 个通道（WhatsApp、Telegram、Slack、Discord、Signal、iMessage、Feishu 及其他平台）的消息汇入统一的 Agent 运行时，并对每个对话进行串行化处理，确保同一时刻只有一个对话回合在执行。

Agent 遵循"思考—行动—观察"循环：

```
Message In → [Assemble Context] → [Send to LLM]
                                       |
                              Reply? ──Yes──→ Stream reply out
                                       |
                                      No
                                       ↓
                              [Execute Tool] → [Observe Result] → back to LLM
```

模型读取组装好的上下文——角色设定、记忆、对话历史——然后决定直接回复还是执行操作。工具执行结果会反馈给模型继续推理，直至最终回复流式输出。

**模型灵活性。** OpenClaw 支持 10+ 家云端提供商（Anthropic、OpenAI、Google、DeepSeek、Qwen、Mistral、OpenRouter）以及本地推理引擎（Ollama、vLLM、LM Studio）。混合模式可指定主模型并自动回退——冷却和探测恢复机制无需人工干预。

**记忆系统。** 三层架构保障对话连贯性：（1）引导文件——会话启动时加载的 Markdown 文件，包含角色设定、指令、用户画像和积累的笔记；（2）会话历史——对话记录，自动压缩时会将重要信息写入长期记忆后再裁剪；（3）语义检索——基于工作区文件的向量嵌入，通过关键词（30%）+ 向量相似度（70%）的混合搜索实现跨会话回忆。

**调度机制。** 每个会话拥有独立的串行队列，确保同一对话一次只处理一个回合。全局并发限制控制并行会话数量。快速连续的消息会被防抖合并为单次投递。

**工具系统。** Shell 命令、文件操作、浏览器自动化、媒体理解、定时任务（cron）和 webhook——全部在沙盒中运行，通过按通道配置的策略控制每个对话的权限范围。

## 2. 与同类方案的对比

每个产品的一句话定位：

- **Auto-GPT** — 自主任务执行器，执行一次性任务。
- **LangChain Agent** — 面向开发者的自定义 Agent 构建工具包。
- **Open Interpreter** — 智能终端，提供会话范围内的 CLI 交互。
- **Claude Code** — IDE 和命令行下的 AI 结对编程工具。
- **OpenClaw** — 生活在你消息应用中的个人助手。

| 维度 | OpenClaw | Auto-GPT | Open Interpreter | LangChain Agent | Claude Code |
|------|----------|----------|-----------------|-----------------|-------------|
| 定位 | 常驻个人助手 | 自主任务执行器 | 智能终端 | 开发者框架 | AI 结对编程 |
| 多通道 | 12+ 原生通道 | 无 | 无 | 需自行构建 | 无 |
| 常驻运行 | 是（Gateway 守护进程） | 否（运行即结束） | 否（会话制） | 取决于实现 | 否（会话制） |
| 持久记忆 | 三层记忆系统 | 短期 + 向量 | 仅会话内 | 需自行构建 | 会话 + 项目 |
| 本地模型 | 原生支持（Ollama、vLLM、LM Studio） | 有限支持 | 支持 | 通过封装层 | 不支持 |
| 数据本地化 | 是——所有数据存于本机 | 部分 | 是 | 取决于实现 | 是 |

**优势。** 12+ 个通道的常驻接入意味着你可以通过日常使用的应用直接交互。原生支持中国生态通道（Feishu）和模型提供商（Qwen、DeepSeek），满足国内用户需求。本地优先架构确保所有数据留在你的硬件上。三层记忆系统实现跨会话的上下文延续。插件架构使得添加通道、工具或行为扩展非常便捷。

**局限。** 单用户设计——这是个人助手，不是团队平台。需要你控制的常驻硬件。配置（模型提供商、通道、记忆）的学习曲线比托管方案更陡。本地模型质量取决于可用硬件性能。

## 3. 我们的本地化开发工作

### Feishu 扩展

我们为 Feishu 通道扩展贡献了 bug 修复和新功能——修复了白名单系统中的消息路由问题，并新增了文档权限管理能力（查询和修改分享设置），同时补充了测试覆盖。这项工作使 OpenClaw 能够在中国企业生态中运行。目前已配置并运行四个 Feishu 应用账号。

### Peekaboo（macOS UI 自动化）

Peekaboo 是一款视觉自动化工具，能够截取屏幕、读取 UI 元素、执行点击和文本输入——让 Agent 无需 API 即可操控 macOS 应用。实际使用中，我们用它来进行 Ctrip（携程旅行）机票预订、Photos 图库浏览、Calendar 日程管理，以及与任意 macOS 应用的交互，所有操作均通过消息通道的自然语言指令驱动。

### 已部署技能

十项技能运行在配备 32 GB 内存的 Mac mini M4 上：

| 技能 | 功能 |
|------|------|
| Feishu Doc | 读写飞书文档 |
| Feishu Drive | 云端存储文件管理 |
| Feishu Wiki | 知识库导航 |
| Feishu Perm | 文档权限管理 |
| Peekaboo | macOS 屏幕截取与 UI 自动化 |
| Himalaya | 通过 IMAP/SMTP 管理邮件 |
| Coding Agent | 将任务委派给 Codex/Claude Code Agent |
| Browser | 通过 Chrome 扩展进行网页浏览和搜索 |
| Weather | 实时天气与预报查询 |
| Healthcheck | 主机安全加固与风险评估 |

### 实际应用场景

从任何已连接的消息通道，用户可以撰写飞书文档、协调会议、通过 Peekaboo 自动化操作 macOS 应用、管理邮件、浏览网页、委派编程任务——全部通过自然语言完成。Gateway 将每个请求路由到对应的技能，在本地执行工具调用，并将结果通过用户发送消息的同一通道流式返回。

## 4. 性能测试结果

### 测试环境

所有测量均在配备 32 GB 内存的 Mac mini M4 上通过普通家庭宽带完成。云端模型：Claude Sonnet 4.6，通过 Anthropic API 经 OpenClaw Gateway 调用。本地模型：qwen3:8b（80 亿参数），运行于 Ollama。

### 简单问答

一个不需要工具调用的事实性问题——纯语言模型推理（用一句话分别解释热力学三大定律）：

| 指标 | 云端（Claude Sonnet） | 本地（qwen3:8b） |
|------|----------------------|-------------------|
| 响应时间 | 约 9 秒 | 约 57 秒 |
| 回答质量 | 优秀（5/5）——准确、简洁，额外补充了背景信息 | 良好（4/5）——准确，略显冗长 |
| 生成 token 数 | 约 150 | 约 949（含内部推理过程） |

云端模型在十秒内返回了精炼的回答，并主动补充了第零定律。本地模型回答准确，但耗时约为云端的 6 倍，大部分 token 用于内部思维链推理。

### 文件操作

读取一个 20 行的文件，添加行号后写回——除推理外还测试了工具调用能力（读取 + 写入）。仅对云端模型进行了工具调用基准测试。

| 指标 | 云端（Claude Sonnet） |
|------|----------------------|
| 总耗时 | 约 12.5 秒 |
| 工具调用次数 | 2 次（读取 + 写入） |
| 准确性 | 完全正确 |

云端模型在 12.5 秒内完成操作，执行了两次干净的工具调用，结果完全正确。

### 分析总结

云端模型的响应速度约为本地模型的 6 倍，输出质量也更精炼。对于追求速度、复杂推理或多步工具链的场景，云端是明确的首选——9 秒的问答和 12.5 秒的文件操作已接近对话般的流畅体验。

本地模型在隐私保护（数据不离开本机）、离线可用性和零边际成本方面胜出。实际最佳方案是混合模式：将高要求任务路由到云端，日常查询交给本地处理。OpenClaw 原生支持这一模式——只需配置一次模型优先级，Gateway 即可自动路由。
