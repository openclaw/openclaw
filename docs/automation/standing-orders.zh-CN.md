# 常规指令

常规指令授予你的代理对已定义程序的**永久操作权限**。你不再需要每次都给出单独的任务指令，而是定义具有明确范围、触发器和升级规则的程序 —— 代理会在这些边界内自主执行。

这就像区别于每周五告诉你的助手"发送周报"，而是授予常规权限："你负责周报。每周五编译它，发送它，只有在看起来有问题时才升级。"

## 为什么需要常规指令？

**没有常规指令：**

- 你必须为每个任务提示代理
- 代理在请求之间处于空闲状态
- 常规工作被遗忘或延迟
- 你成为瓶颈

**有了常规指令：**

- 代理在定义的边界内自主执行
- 常规工作按计划进行，无需提示
- 你只参与异常和批准
- 代理有意义地填补空闲时间

## 它们如何工作

常规指令在你的[代理工作区](/concepts/agent-workspace)文件中定义。推荐的方法是直接在 `AGENTS.md` 中包含它们（每个会话都会自动注入），这样代理在上下文中始终拥有它们。对于较大的配置，你也可以将它们放在专用文件（如 `standing-orders.md`）中，并从 `AGENTS.md` 引用它。

每个程序指定：

1. **范围** — 代理被授权执行的操作
2. **触发器** — 执行时间（计划、事件或条件）
3. **批准门** — 什么需要人工签字才能行动
4. **升级规则** — 何时停止并寻求帮助

代理通过工作区引导文件（请参阅[代理工作区](/concepts/agent-workspace)获取自动注入文件的完整列表）在每个会话中加载这些指令，并与[定时任务](/automation/cron-jobs)结合执行，以进行基于时间的强制执行。

<Tip>
将常规指令放在 `AGENTS.md` 中，以保证它们在每个会话中都被加载。工作区引导会自动注入 `AGENTS.md`、`SOUL.md`、`TOOLS.md`、`IDENTITY.md`、`USER.md`、`HEARTBEAT.md`、`BOOTSTRAP.md` 和 `MEMORY.md` —— 但不会注入子目录中的任意文件。
</Tip>

## 常规指令的结构

```markdown
## 程序：每周状态报告

**权限：** 编译数据，生成报告，交付给利益相关者
**触发器：** 每周五下午4点（通过定时任务强制执行）
**批准门：** 标准报告无需批准。标记异常以供人工审查。
**升级：** 如果数据源不可用或指标看起来异常（与正常值偏差>2σ）

### 执行步骤

1. 从配置的来源获取指标
2. 与前一周和目标进行比较
3. 在 Reports/weekly/YYYY-MM-DD.md 中生成报告
4. 通过配置的渠道传递摘要
5. 将完成情况记录到 Agent/Logs/

### 不要做什么

- 不要向外部各方发送报告
- 不要修改源数据
- 不要在指标看起来不好时跳过交付 —— 准确报告
```

## 常规指令 + 定时任务

常规指令定义代理**被授权做什么**。[定时任务](/automation/cron-jobs)定义**何时**执行。它们一起工作：

```
常规指令："你负责每日收件箱分类"
    ↓
定时任务（每天上午8点）："按照常规指令执行收件箱分类"
    ↓
代理：阅读常规指令 → 执行步骤 → 报告结果
```

定时任务提示应该引用常规指令，而不是重复它：

```bash
openclaw cron add \
  --name daily-inbox-triage \
  --cron "0 8 * * 1-5" \
  --tz America/New_York \
  --timeout-seconds 300 \
  --announce \
  --channel bluebubbles \
  --to "+1XXXXXXXXXX" \
  --message "按照常规指令执行每日收件箱分类。检查邮件是否有新警报。解析、分类并持久化每个项目。向所有者报告摘要。升级未知项。"
```

## 示例

### 示例 1：内容和社交媒体（每周周期）

```markdown
## 程序：内容和社交媒体

**权限：** 起草内容，安排帖子，编译参与度报告
**批准门：** 所有帖子在头30天需要所有者审查，然后是常规批准
**触发器：** 每周周期（周一审查 → 周中起草 → 周五简报）

### 每周周期

- **周一：** 审查平台指标和受众参与度
- **周二至周四：** 起草社交媒体帖子，创建博客内容
- **周五：** 编译每周营销简报 → 交付给所有者

### 内容规则

- 语气必须与品牌匹配（见 SOUL.md 或品牌语气指南）
- 永远不要在面向公众的内容中自称为 AI
- 尽可能包含指标
- 关注对受众的价值，而非自我推广
```

### 示例 2：财务运营（事件触发）

```markdown
## 程序：财务处理

**权限：** 处理交易数据，生成报告，发送摘要
**批准门：** 分析无需批准。建议需要所有者批准。
**触发器：** 检测到新数据文件或计划的月度周期

### 当新数据到达时

1. 检测指定输入目录中的新文件
2. 解析并分类所有交易
3. 与预算目标进行比较
4. 标记：异常项目、阈值突破、新的经常性费用
5. 在指定输出目录中生成报告
6. 通过配置的渠道向所有者交付摘要

### 升级规则

- 单项 > $500：立即警报
- 类别 > 预算 20%：在报告中标记
- 无法识别的交易：询问所有者分类
- 处理失败 2 次后：报告失败，不要猜测
```

### 示例 3：监控和警报（持续）

```markdown
## 程序：系统监控

**权限：** 检查系统健康状况，重启服务，发送警报
**批准门：** 自动重启服务。如果重启失败两次则升级。
**触发器：** 每个心跳周期

### 检查

- 服务健康端点响应
- 磁盘空间高于阈值
- 待处理任务不过期（>24小时）
- 交付渠道运行正常

### 响应矩阵

| 条件           | 行动                   | 升级？              |
| -------------- | ---------------------- | ------------------- |
| 服务宕机       | 自动重启               | 仅当重启失败 2 次时 |
| 磁盘空间 < 10% | 警报所有者             | 是                  |
| 过期任务 > 24h | 提醒所有者             | 否                  |
| 渠道离线       | 记录并在下一个周期重试 | 如果离线 > 2 小时   |
```

## 执行-验证-报告模式

当与严格的执行纪律结合时，常规指令效果最佳。常规指令中的每个任务都应遵循此循环：

1. **执行** — 实际完成工作（不要只是确认指令）
2. **验证** — 确认结果正确（文件存在，消息已传递，数据已解析）
3. **报告** — 告诉所有者做了什么以及验证了什么

```markdown
### 执行规则

- 每个任务都遵循执行-验证-报告。无例外。
- "我会做那件事"不是执行。去做，然后报告。
- 没有验证的"完成"是不可接受的。证明它。
- 如果执行失败：用调整的方法重试一次。
- 如果仍然失败：报告失败并提供诊断。永远不要静默失败。
- 永远不要无限重试 — 最多 3 次尝试，然后升级。
```

此模式可防止最常见的代理失败模式：确认任务而不完成它。

## 多程序架构

对于管理多个关注点的代理，将常规指令组织为具有明确边界的单独程序：

```markdown
# 常规指令

## 程序 1：[领域 A]（每周）

...

## 程序 2：[领域 B]（每月 + 按需）

...

## 程序 3：[领域 C]（按需）

...

## 升级规则（所有程序）

- [通用升级标准]
- [适用于所有程序的批准门]
```

每个程序应该有：

- 自己的**触发节奏**（每周、每月、事件驱动、持续）
- 自己的**批准门**（有些程序比其他程序需要更多监督）
- 明确的**边界**（代理应该知道一个程序在哪里结束，另一个程序在哪里开始）

## 最佳实践

### 应该做

- 从狭窄的权限开始，随着信任的建立而扩展
- 为高风险操作定义明确的批准门
- 包括"不要做什么"部分 —— 边界与权限同样重要
- 与定时任务结合以实现可靠的基于时间的执行
- 每周审查代理日志，以验证是否遵循了常规指令
- 随着需求的发展更新常规指令 —— 它们是活文档

### 避免

- 在第一天授予广泛的权限（"做你认为最好的事情"）
- 跳过升级规则 —— 每个程序都需要一个"何时停止并询问"的条款
- 假设代理会记住口头指令 —— 将所有内容放在文件中
- 在单个程序中混合关注点 —— 为不同领域使用不同程序
- 忘记使用定时任务强制执行 —— 没有触发器的常规指令变成了建议
