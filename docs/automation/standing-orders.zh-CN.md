# 长期指令

长期指令授予您的代理对定义程序的**永久运行权限**。您不再需要每次给出单独的任务指令，而是定义具有明确范围、触发器和升级规则的程序 —— 代理会在这些边界内自主执行。

这相当于区别于每周五告诉助手"发送周报"，而是授予长期权限："你负责周报。每周五编制，发送，只有在看起来有问题时才升级。"

## 为什么需要长期指令？

**没有长期指令：**

- 您必须为每个任务提示代理
- 代理在请求之间处于空闲状态
- 常规工作被遗忘或延迟
- 您成为瓶颈

**有了长期指令：**

- 代理在定义的边界内自主执行
- 常规工作按计划进行，无需提示
- 您只参与异常和审批
- 代理有效利用空闲时间

## 它们如何工作

长期指令在您的[代理工作区](/concepts/agent-workspace)文件中定义。推荐的方法是直接在 `AGENTS.md` 中包含它们（每个会话都会自动注入），这样代理始终将它们置于上下文中。对于较大的配置，您也可以将它们放在专门的文件（如 `standing-orders.md`）中，并从 `AGENTS.md` 引用它。

每个程序指定：

1. **范围** — 代理被授权做什么
2. **触发器** — 何时执行（时间表、事件或条件）
3. **审批门** — 什么需要在行动前获得人工签字
4. **升级规则** — 何时停止并寻求帮助

代理通过工作区引导文件（请参阅[代理工作区](/concepts/agent-workspace)获取自动注入文件的完整列表）加载这些指令，并结合[定时任务](/automation/cron-jobs)进行基于时间的执行。

<Tip>
将长期指令放在 `AGENTS.md` 中，以保证它们在每个会话中都被加载。工作区引导会自动注入 `AGENTS.md`、`SOUL.md`、`TOOLS.md`、`IDENTITY.md`、`USER.md`、`HEARTBEAT.md`、`BOOTSTRAP.md` 和 `MEMORY.md` —— 但不会注入子目录中的任意文件。
</Tip>

## 长期指令的结构

```markdown
## 程序：每周状态报告

**权限：** 编译数据，生成报告，交付给利益相关者
**触发器：** 每周五下午 4 点（通过定时任务强制执行）
**审批门：** 标准报告无需审批。标记异常以供人工审查。
**升级：** 如果数据源不可用或指标看起来异常（与标准偏差 >2σ）

### 执行步骤

1. 从配置的来源拉取指标
2. 与上周和目标进行比较
3. 在 Reports/weekly/YYYY-MM-DD.md 中生成报告
4. 通过配置的渠道传递摘要
5. 将完成情况记录到 Agent/Logs/

### 不要做什么

- 不要向外部方发送报告
- 不要修改源数据
- 不要在指标看起来不好时跳过交付 —— 准确报告
```

## 长期指令 + 定时任务

长期指令定义**代理被授权做什么**。[定时任务](/automation/cron-jobs)定义**何时发生**。它们协同工作：

```
长期指令："你负责每日收件箱分类"
    ↓
定时任务（每天早上 8 点）："按照长期指令执行收件箱分类"
    ↓
代理：读取长期指令 → 执行步骤 → 报告结果
```

定时任务提示应引用长期指令，而不是复制它：

```bash
openclaw cron add \
  --name daily-inbox-triage \
  --cron "0 8 * * 1-5" \
  --tz America/New_York \
  --timeout-seconds 300 \
  --announce \
  --channel bluebubbles \
  --to "+1XXXXXXXXXX" \
  --message "按照长期指令执行每日收件箱分类。检查邮件中的新警报。解析、分类并持久化每个项目。向所有者报告摘要。升级未知项。"
```

## 示例

### 示例 1：内容和社交媒体（每周周期）

```markdown
## 程序：内容和社交媒体

**权限：** 起草内容，安排帖子，编制参与度报告
**审批门：** 所有帖子在前 30 天需要所有者审查，然后获得长期批准
**触发器：** 每周周期（周一审查 → 周中草稿 → 周五简报）

### 每周周期

- **周一：** 审查平台指标和受众参与度
- **周二至周四：** 起草社交帖子，创建博客内容
- **周五：** 编制每周营销简报 → 交付给所有者

### 内容规则

- 语气必须与品牌一致（请参阅 SOUL.md 或品牌语气指南）
- 在面向公众的内容中永远不将自己标识为 AI
- 尽可能包含指标
- 关注对受众的价值，而不是自我推广
```

### 示例 2：财务运营（事件触发）

```markdown
## 程序：财务处理

**权限：** 处理交易数据，生成报告，发送摘要
**审批门：** 分析无需审批。建议需要所有者批准。
**触发器：** 检测到新数据文件 OR 计划的月度周期

### 当新数据到达时

1. 检测指定输入目录中的新文件
2. 解析并分类所有交易
3. 与预算目标进行比较
4. 标记：异常项目、阈值突破、新的经常性费用
5. 在指定的输出目录中生成报告
6. 通过配置的渠道向所有者传递摘要

### 升级规则

- 单项 > $500：立即警报
- 类别 > 预算 20%：在报告中标记
- 无法识别的交易：请所有者进行分类
- 处理失败后 2 次重试：报告失败，不要猜测
```

### 示例 3：监控和警报（持续）

```markdown
## 程序：系统监控

**权限：** 检查系统健康状况，重启服务，发送警报
**审批门：** 自动重启服务。如果重启失败两次则升级。
**触发器：** 每个心跳周期

### 检查

- 服务健康端点响应
- 磁盘空间高于阈值
- 待处理任务不过时（>24 小时）
- 交付渠道正常运行

### 响应矩阵

| 条件           | 操作                   | 升级？              |
| -------------- | ---------------------- | ------------------- |
| 服务宕机       | 自动重启               | 仅当重启失败 2 次时 |
| 磁盘空间 < 10% | 提醒所有者             | 是                  |
| 过时任务 > 24h | 提醒所有者             | 否                  |
| 渠道离线       | 记录并在下一个周期重试 | 如果离线 > 2 小时   |
```

## 执行-验证-报告模式

长期指令与严格的执行纪律结合使用时效果最佳。长期指令中的每项任务都应遵循此循环：

1. **执行** — 实际完成工作（不要只是确认指令）
2. **验证** — 确认结果正确（文件存在，消息已传递，数据已解析）
3. **报告** — 告诉所有者完成了什么以及验证了什么

```markdown
### 执行规则

- 每项任务都遵循执行-验证-报告。无例外。
- "我会做的"不是执行。去做，然后报告。
- 没有验证的"完成"是不可接受的。证明它。
- 如果执行失败：用调整后的方法重试一次。
- 如果仍然失败：报告失败并提供诊断。永远不要静默失败。
- 永远不要无限重试 — 最多 3 次尝试，然后升级。
```

此模式可防止最常见的代理失败模式：确认任务但未完成。

## 多程序架构

对于管理多个关注点的代理，将长期指令组织为具有明确边界的单独程序：

```markdown
# 长期指令

## 程序 1：[领域 A]（每周）

...

## 程序 2：[领域 B]（每月 + 按需）

...

## 程序 3：[领域 C]（按需）

...

## 升级规则（所有程序）

- [通用升级标准]
- [适用于所有程序的审批门]
```

每个程序应具有：

- 自己的**触发节奏**（每周、每月、事件驱动、持续）
- 自己的**审批门**（有些程序需要比其他程序更多的监督）
- 明确的**边界**（代理应该知道一个程序在哪里结束，另一个程序在哪里开始）

## 最佳实践

### 应该做

- 从狭窄的权限开始，随着信任的建立而扩展
- 为高风险操作定义明确的审批门
- 包括"不要做什么"部分 — 边界与权限同样重要
- 与定时任务结合使用，以确保可靠的基于时间的执行
- 每周审查代理日志，以验证长期指令是否被遵循
- 随着您的需求演变更新长期指令 — 它们是活文档

### 避免

- 在第一天授予广泛的权限（"做你认为最好的事情"）
- 跳过升级规则 — 每个程序都需要"何时停止并询问"的条款
- 假设代理会记住口头指令 — 将所有内容放在文件中
- 在单个程序中混合关注点 — 为不同领域使用单独的程序
- 忘记用定时任务强制执行 — 没有触发器的长期指令变成了建议
