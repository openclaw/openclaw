---\nsummary: \"Evolution of OpenClaw Heartbeat: from passive checks to proactive monitoring and self-iteration\"\ntitle: Proactive Heartbeat: Best Practices\nread_when:\n  - You want to make your agent more proactive\n  - You are optimizing heartbeat configurations for cost and effectiveness\n  - You are integrating heartbeat with other automation patterns\n---\n\n# Proactive Heartbeat: Best Practices\n\nThe OpenClaw `heartbeat` is a powerful mechanism for agent proactivity. It evolves beyond simple health checks into a comprehensive system for monitoring, task nudging, and self-improvement.\n\nThis guide outlines a 4-stage evolution, drawing from real-world deployments to demonstrate how to maximize your agent's proactive capabilities while optimizing token usage.\n\n## The 4-Stage Evolution of Heartbeat\n\nMost agent deployments start with a basic heartbeat and expand its responsibilities as the agent matures.\n\n### Stage 1: Passive Recovery (Basic Setup)\n\nThis is the initial state for many agents. The heartbeat's primary role is to ensure continuity after interruptions.\n\n**Purpose**: To check if any background tasks crashed or if the agent needs to resume work.\n\n**HEARTBEAT.md content (minimal)**:\n\n```md\n# Heartbeat Checklist - Stage 1: Passive Recovery\n\n- Check `~/.openclaw/recovery/restart-signal.json` for unrecovered restarts.\n- Review `active_tasks.json` for `in_progress` tasks with dead processes.\n- Check `memory/YYYY-MM-DD.md` for `pending_confirmation` and `PENDING_TASKS`.\n```\n\n**Characteristics**:\n\n- **Reactive**: Only responds if a problem is detected.\n- **Low token usage**: Minimal checks, small `HEARTBEAT.md`.\n- **Limited proactivity**: Does not actively seek out new issues or prompt the user.\n\n### Stage 2: Active System Monitoring (Infrastructure Awareness)\n\nAt this stage, the heartbeat expands to monitor the agent's operational environment and critical services.\n\n**Purpose**: To ensure the underlying infrastructure is healthy and to detect potential issues before they impact operations.\n\n**HEARTBEAT.md content (extended)**:\n\n```md\n# Heartbeat Checklist - Stage 2: Active System Monitoring\n\n## 1. Recovery Checks (Inherited from Stage 1)\\n- Check `~/.openclaw/recovery/restart-signal.json` for unrecovered restarts.\n- Review `active_tasks.json` for `in_progress` tasks with dead processes.\n- Check `memory/YYYY-MM-DD.md` for `pending_confirmation` and `PENDING_TASKS`.\n\n## 2. System Patrol (New)\\n- **Tailscale**: Check `/Applications/Tailscale.app/Contents/MacOS/Tailscale status`.\n  - *Action*: If stopped, attempt auto-restart.\n- **Disk Space**: Monitor `df -h /Volumes/OPENCLAWSSD` and `df -h /`.\n  - *Threshold*: Alert if any partition >85% usage.\n- **Podcast Pipeline Health**:\n  - Count `FAILED` `stage_runs` in last 24h.\n  - Check `MAX(created_at)` for `deliver` `DONE` in last 48h.\n  - *Action*: Report failures/stalls.\n- **OpenClaw Gateway**: Verify `openclaw gateway status`.\n  - *Action*: Report `warning`/`error` messages (e.g., outdated config).\n- **Key Process (caffeinate)**: Ensure `pgrep -l caffeinate` is running.\n  - *Action*: Report if missing (prevents unintended sleep).\n```\n\n**Characteristics**:\n\n- **Proactive detection**: Identifies infrastructure issues without user intervention.\n- **Self-healing**: Attempts to fix simple issues (e.g., restart Tailscale).\n- **Early warning**: Alerts user to non-critical but important issues.\n\n### Stage 3: Task Nudging (Project Management)\n\nIntegrating basic project management. The heartbeat helps ensure tasks don't fall through the cracks.\n\n**Purpose**: To keep ongoing tasks visible and to prompt action on stalled items.\n\n**HEARTBEAT.md content (further extended)**:\n\n```md\n# Heartbeat Checklist - Stage 3: Task Nudging\n\n## 1. Recovery Checks (Inherited)\\n## 2. System Patrol (Inherited)\\n\n## 3. Task Nudge (New)\\n- **Pending Tasks**: Review `PENDING_TASKS` from recent daily memories (`memory/YYYY-MM-DD.md`).\n  - *Action*: If a task is `todo` or `blocked` for >3 days, send a reminder.\n  - *Action*: If a `blocked` task's condition is resolved, prompt for next steps.\n- **Lessons Learned (action_required)**: Check `memory/lessons_learned.md` for `action_required` items.\n  - *Action*: Remind if unaddressed.\n- **Cron Job Errors**: Monitor `cron` job list for `consecutiveErrors > 0`.\n  - *Action*: Report failed cron jobs.\n```\n\n**Characteristics**:\n\n- **Active reminder**: Prevents tasks from being forgotten.\n- **Improved workflow**: Helps manage multi-step processes and follow up on dependencies.\n- **User engagement**: Gentle prompts to keep projects moving.\n\n### Stage 4: Self-Iteration (Agent Self-Improvement)\n\nThe most advanced stage, where the agent actively participates in its own development and optimization.\n\n**Purpose**: To formalize learning from experience, adapt strategies, and optimize its own operation.\n\n**HEARTBEAT.md content (full example)**:\n\n```md\n# Heartbeat Checklist - Stage 4: Self-Iteration\n\n## 1. Recovery Checks (Inherited)\\n## 2. System Patrol (Inherited)\\n## 3. Task Nudge (Inherited)\\n\n## 4. Self-Review Trigger (New)\\n- **Weekly Review**: Check `memory/last_weekly_review.md` date.\n  - *Action*: If >7 days since last review, perform the self-review process:\n    1. Review past 7 days of `memory/YYYY-MM-DD.md`.\n    2. Review `memory/lessons_learned.md` for `action_required` items.\n    3. Extract patterns (successes, recurring issues, user feedback).\n    4. Update `AGENTS.md`, `SOUL.md`, `TOOLS.md` as needed.\n    5. Record review date + summary to `memory/last_weekly_review.md`.\n    6. Announce completion to user.\n```\n\n**Characteristics**:\n\n- **Continuous learning**: Systematically integrates lessons learned.\n- **Adaptive behavior**: Evolves its own rules and persona based on experience.\n- **Improved efficiency**: Identifies and automates recurring tasks, optimizes configurations.\n\n## Optimizing Heartbeat for Cost and Effectiveness\n\n### Use `activeHours` to save tokens\n\nConfigure `activeHours` to prevent heartbeats from running during inactive periods (e.g., overnight). This significantly reduces token consumption without impacting responsiveness when needed.\n\n```json5\n{\n  agents: {\n    defaults: {\n      heartbeat: {\n        every: \"1h\",\n        activeHours: { start: \"08:00\", end: \"24:00\" }, // Only run between 8 AM and midnight\n      },\n    },\n  },\n}\n```\n\n### Keep HEARTBEAT.md concise\n\nEvery line in `HEARTBEAT.md` contributes to the token count of each heartbeat turn. Move infrequent or heavy tasks to isolated cron jobs.\n\n### Suppress `HEARTBEAT_OK` responses\n\nConfigure `channels.<channel>.heartbeat.showOk: false` (default) to prevent your agent from flooding chat with `HEARTBEAT_OK` messages when nothing needs attention.\n\n## Integrating Heartbeat with Other Patterns\n\n### Heartbeat + Cron for balanced automation\n\n- **Heartbeat**: For frequent, lightweight checks and context-aware nudges.\n- **Cron**: For precise scheduling of heavier, isolated tasks (e.g., weekly deep dives, monthly reports).\n\nThis hybrid approach maximizes efficiency while keeping token costs in check.\n\n### Heartbeat + Multi-Agent Coordination\n\n- A coordinator agent (Pattern 1) uses heartbeat to monitor the overall system and nudge specialist agents as needed via `sessions_send`.\n- Specialists only run heartbeats if they have autonomous background responsibilities.\n\n## Related\n\n- [Cron vs Heartbeat](/automation/cron-vs-heartbeat) — detailed comparison for scheduling decisions\n- [Memory](/concepts/memory) — how agents persist state and knowledge\n- [Multi-Agent Orchestration Patterns](/concepts/multi-agent-patterns) — advanced team structures\nDOCEOF\n
