# Model Config (Exploration)

ဤစာရွက်စာတမ်းသည် အနာဂတ် မော်ဒယ် ဖွဲ့စည်းပြင်ဆင်မှုအတွက် **အယူအဆများ** ကို မှတ်တမ်းတင်ထားသည်။ ၎င်းသည် ထုတ်လုပ်ပြီး အသုံးပြုရန် ပြီးပြည့်စုံသော စံသတ်မှတ်ချက် မဟုတ်ပါ။ လက်ရှိ အပြုအမူများအတွက် အောက်ပါတို့ကို ကြည့်ပါ—

- [Models](/concepts/models)
- [Model failover](/concepts/model-failover)
- [OAuth + profiles](/concepts/oauth)

## Motivation

အော်ပရေတာများအနေဖြင့် လိုလားသည့်အချက်များ—

- provider တစ်ခုစီအတွက် အထောက်အထားအတည်ပြု ပရိုဖိုင် အများအပြား (ကိုယ်ရေးကိုယ်တာ vs အလုပ်)။
- ကြိုတင်ခန့်မှန်းနိုင်သော fallback များပါရှိသည့် ရိုးရှင်းသော `/model` ရွေးချယ်မှု။
- စာသား မော်ဒယ်များနှင့် ရုပ်ပုံ ထောက်ပံ့နိုင်သော မော်ဒယ်များအကြား ရှင်းလင်းသော ခွဲခြားမှု။

## Possible direction (high level)

- မော်ဒယ် ရွေးချယ်မှုကို ရိုးရှင်းစွာ ထိန်းသိမ်းထားရန်— optional aliases ပါရှိသည့် `provider/model`။
- provider များတွင် အထောက်အထားအတည်ပြု ပရိုဖိုင် အများအပြားရှိစေရန်နှင့် အစီအစဉ်အလိုက် အတိအလင်း သတ်မှတ်ရန်။
- ဆက်ရှင်အားလုံးတွင် တစ်ပြိုင်နက်တည်း fail over ဖြစ်စေရန် အတွက် အပြည်ပြည်ဆိုင်ရာ fallback စာရင်းကို အသုံးပြုရန်။
- အတိအလင်း ဖွဲ့စည်းပြင်ဆင်ထားသောအခါသာ ရုပ်ပုံ လမ်းကြောင်းသတ်မှတ်ခြင်းကို override ပြုလုပ်ရန်။

## Open questions

- ပရိုဖိုင် လှည့်ပြောင်းအသုံးပြုခြင်းကို provider အလိုက် လုပ်သင့်သလား၊ မော်ဒယ် အလိုက် လုပ်သင့်သလား။
- UI တွင် ဆက်ရှင်တစ်ခုအတွက် ပရိုဖိုင် ရွေးချယ်မှုကို မည်သို့ ဖော်ပြသင့်သနည်း။
- အဟောင်း config keys များမှ ပြောင်းရွှေ့ရန် အလုံခြုံဆုံး လမ်းကြောင်းက ဘာလဲ။
