# Dispatch interceptor

The `DispatchInterceptorPlugin` API lets a plugin intercept every inbound message
**before the main Agent starts**. When the interceptor handles a request, the
Agent is never invoked — no LLM tokens are consumed and no tools are executed.

Interceptors run on every surface: all messaging channels (Telegram, Discord,
Slack, Signal, iMessage, WhatsApp, and extension channels), the Control UI
(WebSocket/webchat), and the HTTP API endpoints (`/v1/chat/completions` and
`/v1/responses`).

## Use cases

### Content moderation

Send the user message to an external classification service. Return a block
message immediately if the content policy is violated; pass through transparently
if it is clean. Because the Agent never starts on blocked requests, there is no
risk of tool side-effects and no LLM cost.

### Rate limiting and quota control

Track request volume per `sessionKey` or `channelId` inside the interceptor.
Return a throttle message when a user exceeds their quota without forwarding the
request downstream.

### Tenant access control

In multi-tenant deployments, resolve the `sessionKey` to a tenant record and
check entitlements before each request. Block or redirect unauthorized sessions
at the dispatch layer, before any agent-level logic runs.

### Message routing and A/B testing

Inspect the message text or context and decide which downstream path to take —
for example, forward high-priority messages to a different agent profile or return
a canned response for known FAQ patterns.

## How it works

```
inbound message
  └─► interceptor chain (plugin order)
        ├─ intercepted: true  →  plugin sends reply  →  done (Agent never starts)
        └─ intercepted: false →  next interceptor (or Agent if none remain)
```

Each interceptor in the chain receives:

- **`text`** — the user message string
- **`context`** — `{ sessionKey, channelId, userId }`
- **`output`** — an `InterceptorOutputHandler` for sending a response

The chain short-circuits at the first interceptor that returns
`{ intercepted: true }`. If all interceptors return `{ intercepted: false }`,
the Agent runs normally.

## Registering an interceptor

In your plugin entry file, call `api.registerDispatchInterceptor`:

```ts
// index.ts
export default function (api) {
  api.registerDispatchInterceptor({
    async intercept(text, context, output) {
      const decision = await checkPolicy(text, context);

      if (decision.blocked) {
        output.sendBlock(decision.message ?? "Request not allowed.");
        return { intercepted: true };
      }

      return { intercepted: false };
    },
  });
}
```

### `InterceptorOutputHandler` methods

| Method                  | Description                                                         |
| ----------------------- | ------------------------------------------------------------------- |
| `sendBlock(message)`    | Set a static reply text. Call before returning `intercepted: true`. |
| `sendStreamChunk(text)` | Append a chunk to a streaming reply (for progressive output).       |
| `sendStreamDone()`      | Signal end of a streaming reply.                                    |

If you return `intercepted: true` without calling any output method, a default
`"Request intercepted."` message is returned to the user.

## Streaming reply

For richer responses — for example when the replacement content is itself
generated by another model — you can stream content chunk by chunk:

```ts
async intercept(text, context, output) {
  const stream = await myModel.streamResponse(text);

  for await (const chunk of stream) {
    output.sendStreamChunk(chunk);
  }
  output.sendStreamDone();

  return { intercepted: true };
}
```

The streamed content is delivered through the same pipeline as normal Agent
responses, so channel-specific formatting and coalescing rules apply
automatically.

## Plugin config (`openclaw.json`)

Enable the plugin under `plugins.entries`:

```json
{
  "plugins": {
    "entries": {
      "my-interceptor-plugin": {
        "enabled": true,
        "config": {
          "apiUrl": "https://policy-service.example.com",
          "timeoutMs": 3000,
          "fallbackOnError": "pass"
        }
      }
    }
  }
}
```

To disable without removing: set `"enabled": false` and restart the gateway.

## Error handling and fallback

When the interceptor throws an error or an external dependency is unavailable,
the `fallbackOnError` setting determines behavior:

| Value     | Behavior                                                                   |
| --------- | -------------------------------------------------------------------------- |
| `"pass"`  | Silently pass through — Agent runs normally (high-availability default)    |
| `"block"` | Return the configured `blockMessage` — Agent does not run (strict default) |

Always set an explicit `timeoutMs` on any external call inside the interceptor.
An unresponsive external service will stall every incoming message until the
timeout fires.

```ts
async intercept(text, context, output) {
  try {
    const result = await fetchWithTimeout(policyUrl, text, 3000);
    if (result.blocked) {
      output.sendBlock(result.message);
      return { intercepted: true };
    }
  } catch {
    // External service unreachable — apply fallback policy
    if (config.fallbackOnError === "block") {
      output.sendBlock(config.blockMessage ?? "Service unavailable.");
      return { intercepted: true };
    }
  }
  return { intercepted: false };
}
```

## Multiple interceptors

When more than one plugin registers an interceptor, they run in plugin
registration order. The chain stops at the first `intercepted: true`:

```
Plugin A → intercepted: false
Plugin B → intercepted: true  ← chain stops here; Plugin C never runs
Plugin C → (skipped)
```

If no interceptor fires, the Agent runs as normal.

## Minimal plugin example

A complete plugin that blocks messages containing a keyword:

```
my-filter/
├── openclaw.plugin.json
└── index.ts
```

**`openclaw.plugin.json`**

```json
{
  "id": "my-filter",
  "configSchema": {
    "type": "object",
    "additionalProperties": false,
    "properties": {
      "blocklist": { "type": "array", "items": { "type": "string" } },
      "blockMessage": { "type": "string" }
    }
  }
}
```

**`index.ts`**

```ts
export default function (api) {
  const config = api.getConfig();
  const blocklist: string[] = config.blocklist ?? [];
  const blockMessage: string = config.blockMessage ?? "Message not allowed.";

  api.registerDispatchInterceptor({
    async intercept(text, _context, output) {
      const blocked = blocklist.some((word) => text.toLowerCase().includes(word.toLowerCase()));

      if (blocked) {
        output.sendBlock(blockMessage);
        return { intercepted: true };
      }

      return { intercepted: false };
    },
  });
}
```

Enable it in `openclaw.json`:

```json
{
  "plugins": {
    "entries": {
      "my-filter": {
        "enabled": true,
        "config": {
          "blocklist": ["spam", "prohibited"],
          "blockMessage": "Your message was blocked by the content filter."
        }
      }
    }
  }
}
```
