# 보안 🔒

> [!WARNING]
> **개인 어시스턴트 신뢰 모델:** 이 지침은 게이트웨이당 하나의 신뢰할 수 있는 운영자 경계(단일 사용자/개인 어시스턴트 모델)를 가정합니다.
> OpenClaw는 하나의 게이트웨이를 공유하는 여러 적대적 사용자를 위한 적대 다중 테넌트 보안 경계가 **아닙니다**.
> 혼합 신뢰 또는 적대 사용자 운영이 필요하면 신뢰 경계를 분할합니다(별도의 게이트웨이 + 자격 증명, 이상적으로 별도의 OS 사용자/호스트).

## 범위 먼저: 개인 어시스턴트 보안 모델

OpenClaw 보안 지침은 **개인 어시스턴트** 배포를 가정합니다: 하나의 신뢰할 수 있는 운영자 경계, 잠재적으로 많은 에이전트.

- **지원되는 보안 태세**: 호스트당 하나의 사용자/신뢰 경계(호스트/VPS당 하나의 OS 사용자를 선호합니다).
- **지원되지 않는 보안 경계**: 상호 신뢰할 수 없거나 적대적 사용자가 공유하는 하나의 게이트웨이/에이전트.
- **적대 사용자 격리가 필요한 경우**: 신뢰 경계별로 분할합니다(별도의 게이트웨이 + 자격 증명, 이상적으로 별도의 OS 사용자/호스트).
- **여러 신뢰할 수 없는 사용자가 하나의 도구 지원 에이전트에 메시지를 보낼 수 있는 경우**: 이를 동일한 위임 도구 권한을 공유하는 것으로 취급합니다.

이 페이지는 **해당 모델 내에서** 강화하는 방법을 설명합니다. 하나의 공유 게이트웨이에서 적대 다중 테넌트 격리를 청구하지 않습니다.

## 빠른 검사: `openclaw security audit`

정기적으로 실행하세요(특히 설정 변경 후 또는 네트워크 표면 노출 후):

```bash
openclaw security audit
openclaw security audit --deep
openclaw security audit --fix
openclaw security audit --json
```

일반적인 footgun(게이트웨이 인증 노출, 브라우저 제어 노출, 승격 허용 목록, 파일 시스템 권한)을 표시합니다.

OpenClaw는 제품이자 실험입니다. 국경 모델 동작을 실제 메시징 표면 및 실제 도구에 연결하고 있습니다. **완벽하게 안전한 설정은 없습니다.** 목표는 의도적이어야 합니다:

- 봇과 대화할 수 있는 사람
- 봇이 행동할 수 있는 곳
- 봇이 만질 수 있는 것

가장 작은 액세스로 시작합니다. 자신감이 증가하면 확대합니다.

## 배포 가정(중요)

OpenClaw는 호스트 및 설정 경계가 신뢰할 수 있다고 가정합니다:

- 게이트웨이 호스트 상태/설정(`~/.openclaw`, `openclaw.json` 포함)을 수정할 수 있는 사람은 신뢰할 수 있는 운영자로 취급합니다.
- 여러 상호 신뢰할 수 없는/적대적 운영자가 하나의 게이트웨이를 실행하는 것은 **권장 설정이 아닙니다**.
- 혼합 신뢰 팀의 경우 별도의 게이트웨이(또는 최소한 별도의 OS 사용자/호스트)로 신뢰 경계를 분할합니다.
- OpenClaw는 한 머신에서 여러 게이트웨이 인스턴스를 실행할 수 있지만 권장 운영은 깔끔한 신뢰 경계 분리를 선호합니다.
- 권장 기본값: 머신/호스트당 하나의 사용자(또는 VPS), 해당 사용자를 위한 하나의 게이트웨이, 그리고 해당 게이트웨이의 하나 또는 여러 에이전트.
- 여러 사용자가 OpenClaw를 원하면 사용자당 하나의 VPS/호스트를 사용합니다.

### 실제 결과(운영자 신뢰 경계)

하나의 게이트웨이 인스턴스 내에서 인증된 운영자 액세스는 사용자별 테넌트 역할이 아닌 신뢰할 수 있는 제어 평면 역할입니다.

- 읽기/제어 평면 액세스가 있는 운영자는 설계상 게이트웨이 세션 메타데이터/이력을 검사할 수 있습니다.
- 세션 식별자(`sessionKey`, 세션 ID, 레이블)는 권한 부여 토큰이 아닌 라우팅 선택기입니다.
- 예: `sessions.list`, `sessions.preview` 또는 `chat.history`와 같은 메서드에 대한 운영자별 격리를 기대하는 것은 이 모델 밖입니다.
- 적대 사용자 격리가 필요하면 신뢰 경계당 별도의 게이트웨이를 실행합니다.
- 한 머신에서 여러 게이트웨이는 기술적으로 가능하지만 다중 사용자 격리를 위한 권장 기본선이 아닙니다.

## 위협 모델

AI 어시스턴트는:

- 임의의 셸 명령 실행
- 파일 읽기/쓰기
- 네트워크 서비스 액세스
- 메시지를 누군가에게 전송(WhatsApp 액세스 권한이 있는 경우)

당신에게 메시지를 보낼 수 있는 사람들:

- AI를 나쁜 것을 하도록 속이려는 시도
- 데이터 액세스를 위한 사회 공학
- 인프라 세부 정보 프로빙

## 핵심 개념: 지능 이전에 액세스 제어

여기서 대부분의 실패는 멋진 익스플로잇이 아닙니다. "누군가가 봇에 메시지를 보냈고 봇이 요청한 것을 수행했습니다."

OpenClaw의 입장:

- **신원 먼저:** 봇과 대화할 수 있는 사람 결정(DM 페어링 / 허용 목록 / 명시적 "열기").
- **범위 다음:** 봇이 행동할 수 있는 위치 결정(그룹 허용 목록 + 언급 게이팅, 도구, 샌드박싱, 디바이스 권한).
- **모델 마지막:** 모델을 조작할 수 있다고 가정합니다. 조작 blast 반경이 제한되도록 설계하세요.

## 명령 권한 부여 모델

슬래시 명령과 지시문은 **인증된 발신자**에게만 적용됩니다. 권한 부여는 채널 허용 목록/페어링 + `commands.useAccessGroups`(`[Configuration](/gateway/configuration)` 및 `[Slash commands](/tools/slash-commands)` 참조)에서 파생됩니다. 채널 허용 목록이 비어 있거나 `"*"`을 포함하면 명령이 해당 채널에 대해 효과적으로 열립니다.

`/exec`는 인증된 운영자를 위한 세션 전용 편의입니다. 설정을 작성하거나 다른 세션을 변경하지 않습니다.

## 보안 감사 체크리스트

감사가 발견을 인쇄할 때 우선 순위 순서로 취급합니다:

1. **"열기" + 도구 활성화됨**: DM/그룹을 먼저 잠급니다(페어링/허용 목록), 그런 다음 도구 정책/샌드박싱을 강화합니다.
2. **공개 네트워크 노출**(LAN 바인드, Funnel, 인증 누락): 즉시 수정.
3. **브라우저 제어 원격 노출**: 운영자 액세스처럼 취급합니다(tailnet만, 의도적 노드 페어링, 공개 노출 피하기).
4. **권한**: 상태/설정/자격 증명/인증이 그룹/세계 읽기 가능하지 않은지 확인합니다.
5. **플러그인/확장**: 명시적으로 신뢰하는 것만 로드합니다.
6. **모델 선택**: 도구가 있는 봇의 경우 최신 지침 강화 모델을 선호합니다.

[원본 /gateway/security](/gateway/security)를 참조하여 모든 체크와 수정 경로를 확인합니다.
