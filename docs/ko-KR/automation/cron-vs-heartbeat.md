# Cron과 하트비트: 각각을 사용할 때

하트비트와 cron 작업 모두 일정에 따라 작업을 실행할 수 있습니다. 이 가이드는 사용 사례에 맞는 메커니즘을 선택하는 데 도움이 됩니다.

## 빠른 결정 가이드

| 사용 사례                        | 권장                | 이유                                  |
| -------------------------------- | ------------------- | ------------------------------------- |
| 30분마다 받은편지함 확인         | 하트비트            | 다른 검사와 배치, 상황 인식           |
| 매일 9am에 일일 보고서 전송      | Cron (격리)         | 정확한 타이밍 필요                    |
| 달력에서 곧 오는 이벤트 모니터링 | 하트비트            | 주기적 인식의 자연스러운 적합         |
| 주간 깊은 분석 실행              | Cron (격리)         | 독립 실행형 작업, 다른 모델 사용 가능 |
| 20분 후에 알려주기               | Cron (메인, `--at`) | 정확한 타이밍이 있는 일회성           |
| 배경 프로젝트 상태 확인          | 하트비트            | 기존 주기에 편승                      |

## 하트비트: 주기적 인식

하트비트는 정기적 간격 (기본값: 30분)에서 **메인 세션**에서 실행됩니다. 에이전트가 사항을 확인하고 모든 중요한 것을 표시하도록 설계되었습니다.

### 하트비트 사용 시기

- **여러 주기적 검사**: 받은편지함, 달력, 날씨, 알림 및 프로젝트 상태를 확인하는 5개의 별도 cron 작업 대신 단일 하트비트가 모두 배치할 수 있습니다.
- **상황 인식 결정**: 에이전트는 전체 메인 세션 컨텍스트를 가지므로 긴급 vs 대기할 수 있는 항목에 대해 똑똑한 결정을 내릴 수 있습니다.
- **대화 연속성**: 하트비트는 같은 세션을 공유하므로 에이전트는 최근 대화를 기억하고 자연스럽게 따를 수 있습니다.
- **낮은 오버헤드 모니터링**: 하나의 하트비트는 많은 작은 폴링 작업을 대체합니다.

### 하트비트 장점

- **여러 검사 배치**: 하나의 에이전트 턴은 받은편지함, 달력 및 알림을 함께 검토할 수 있습니다.
- **API 호출 감소**: 5개의 격리 cron 작업보다 단일 하트비트가 더 저렴합니다.
- **상황 인식**: 에이전트는 당신이 무엇을 작업 중인지 알고 있으며 우선 순위를 정할 수 있습니다.
- **스마트 억제**: 아무것도 관심이 필요하지 않으면 에이전트가 `HEARTBEAT_OK`에 응답하고 메시지가 전달되지 않습니다.
- **자연스러운 타이밍**: 대기열 로드를 기반으로 약간 드리프트하며, 이는 대부분의 모니터링에 좋습니다.

### 하트비트 예: HEARTBEAT.md 체크리스트

```md
# 하트비트 체크리스트

- 긴급 메시지에 대해 이메일 확인
- 다음 2시간 내 이벤트에 대한 달력 검토
- 배경 작업이 완료된 경우 결과 요약
- 8시간 이상 유휴인 경우 간단한 체크인 전송
```

에이전트는 각 하트비트에서 이를 읽고 한 턴에서 모든 항목을 처리합니다.

### 하트비트 구성

```json5
{
  agents: {
    defaults: {
      heartbeat: {
        every: "30m", // interval
        target: "last", // explicit alert delivery target (default is "none")
        activeHours: { start: "08:00", end: "22:00" }, // optional
      },
    },
  },
}
```

전체 구성은 [하트비트](/gateway/heartbeat)를 참조하세요.

## Cron: 정확한 예약

Cron 작업은 정확한 시간에 실행되며 메인 컨텍스트에 영향을 주지 않고 격리된 세션에서 실행될 수 있습니다.
반복 시간 상단 일정은 자동으로 0-5분 윈도우에 결정론적 오프셋으로 분산됩니다.

### Cron 사용 시기

- **정확한 타이밍 필요**: "매주 월요일 9:00 AM에 이것을 전송" (9시 근처가 아님).
- **독립 실행형 작업**: 대화 컨텍스트가 필요하지 않은 작업.
- **다른 모델/사고**: 더 강력한 모델을 보장하는 무거운 분석.
- **일회성 알림**: "20분 후에 알려주기" with `--at`.
- **시끄러운/빈번한 작업**: 메인 세션 히스토리를 복잡하게 할 작업.
- **외부 트리거**: 에이전트가 다른 활동 중일지와 무관하게 실행되어야 하는 작업.

### Cron 장점

- **정확한 타이밍**: 5필드 또는 6필드 (초) cron 표현식 및 시간대 지원.
- **기본 제공 로드 분산**: 반복 시간 상단 일정이 기본적으로 최대 5분까지 스태거됩니다.
- **작업당 제어**: 스태거를 `--stagger <duration>`으로 오버라이드하거나 `--exact`로 정확한 타이밍을 강제합니다.
- **세션 격리**: `cron:<jobId>`에서 메인 히스토리를 오염시키지 않고 실행합니다.
- **모델 오버라이드**: 작업당 저렴하거나 더 강력한 모델을 사용합니다.
- **배달 제어**: 격리 작업은 기본값 `announce` (요약); 필요에 따라 `none`을 선택합니다.
- **즉시 배달**: 하트비트를 기다리지 않고 직접 공지합니다.
- **에이전트 컨텍스트 불필요**: 메인 세션이 유휴 상태이거나 압축되어도 실행합니다.
- **일회성 지원**: `--at`을 사용한 정확한 미래 타임스탬프.

### Cron 예: 매일 아침 브리핑

```bash
openclaw cron add \
  --name "Morning briefing" \
  --cron "0 7 * * *" \
  --tz "America/New_York" \
  --session isolated \
  --message "Generate today's briefing: weather, calendar, top emails, news summary." \
  --model opus \
  --announce \
  --channel whatsapp \
  --to "+15551234567"
```

이것은 정확히 뉴욕 시간 오전 7:00에 실행되고 품질을 위해 Opus를 사용하며 WhatsApp에 요약을 직접 공지합니다.

### Cron 예: 일회성 알림

```bash
openclaw cron add \
  --name "Meeting reminder" \
  --at "20m" \
  --session main \
  --system-event "Reminder: standup meeting starts in 10 minutes." \
  --wake now \
  --delete-after-run
```

전체 CLI 참조는 [Cron 작업](/automation/cron-jobs)을 참조하세요.

## 둘 다 결합

가장 효율적인 설정은 **둘 다**를 사용합니다:

1. **하트비트**는 일상적인 모니터링 (받은편지함, 달력, 알림)을 30분마다 한 번의 배치 턴으로 처리합니다.
2. **Cron**은 정확한 일정 (매일 보고서, 주간 검토) 및 일회성 알림을 처리합니다.

## 비용 고려 사항

| 메커니즘    | 비용 프로필                                           |
| ----------- | ----------------------------------------------------- |
| 하트비트    | N분마다 한 턴; HEARTBEAT.md 크기에 따라 확장          |
| Cron (메인) | 다음 하트비트에 이벤트 추가 (격리 턴 없음)            |
| Cron (격리) | 작업당 전체 에이전트 턴; 저렴한 모델을 사용할 수 있음 |

**팁**:

- HEARTBEAT.md를 작게 유지하여 토큰 오버헤드를 최소화합니다.
- 여러 cron 작업 대신 하트비트에 유사한 검사를 배치합니다.
- 하트비트만 내부 처리를 원할 경우 `target: "none"`을 사용합니다.
- 격리 cron을 저렴한 모델과 함께 사용하여 정상 작업.
