# Cron vs 하트비트: 각각을 사용할 시점

하트비트와 cron 작업 모두 스케줄에 따라 작업을 실행할 수 있습니다. 이 가이드는 사용 사례에 적합한 메커니즘을 선택하는 데 도움을 줍니다.

## 빠른 결정 가이드

| 사용 사례                            | 권장                | 이유                                 |
| ------------------------------------ | ------------------- | ------------------------------------ |
| 30 분마다 받은편지함 확인            | 하트비트            | 다른 확인과 일괄 처리, 컨텍스트 인식 |
| 매일 오전 9 시 정각에 보고서 전송    | Cron (격리)         | 정확한 타이밍 필요                   |
| 다가오는 이벤트를 위한 캘린더 모니터 | 하트비트            | 주기적 인식에 자연스러운 적합        |
| 주간 심층 분석 실행                  | Cron (격리)         | 독립 작업, 다른 모델 사용 가능       |
| 20 분 후 알림                        | Cron (main, `--at`) | 정확한 타이밍의 원샷                 |
| 백그라운드 프로젝트 상태 확인        | 하트비트            | 기존 주기에 편승                     |

## 하트비트: 주기적 인식

하트비트는 정기 간격 (기본값: 30 분) 으로 **메인 세션**에서 실행됩니다. 에이전트가 상황을 확인하고 중요한 것을 표면화하도록 설계되었습니다.

### 하트비트를 사용할 때

- **여러 주기적 확인**: 받은편지함, 캘린더, 날씨, 알림, 프로젝트 상태를 확인하는 5 개의 별도 cron 작업 대신 단일 하트비트로 모두 일괄 처리할 수 있습니다.
- **컨텍스트 인식 결정**: 에이전트가 전체 메인 세션 컨텍스트를 가지므로 긴급한 것과 기다릴 수 있는 것에 대한 스마트한 결정을 내릴 수 있습니다.
- **대화 연속성**: 하트비트 실행이 동일한 세션을 공유하므로 에이전트가 최근 대화를 기억하고 자연스럽게 후속 조치를 할 수 있습니다.
- **저오버헤드 모니터링**: 하나의 하트비트로 많은 소규모 폴링 작업을 대체합니다.

### 하트비트 구성

```json5
{
  agents: {
    defaults: {
      heartbeat: {
        every: "30m",
        target: "last",
        activeHours: { start: "08:00", end: "22:00" },
      },
    },
  },
}
```

전체 구성은 [하트비트](/gateway/heartbeat)를 참조하세요.

## Cron: 정밀 스케줄링

cron 작업은 정확한 시간에 실행되며 메인 컨텍스트에 영향을 주지 않고 격리된 세션에서 실행될 수 있습니다.

### cron 을 사용할 때

- **정확한 타이밍 필요**: "매주 월요일 오전 9 시에 이것을 보내라" ("대략 9 시쯤"이 아님).
- **독립 작업**: 대화 컨텍스트가 필요하지 않은 작업.
- **다른 모델/사고**: 더 강력한 모델이 필요한 심층 분석.
- **원샷 알림**: `--at`를 사용한 "20 분 후 알려줘".
- **노이즈가 많은/빈번한 작업**: 메인 세션 히스토리를 어지럽힐 작업.
- **외부 트리거**: 에이전트가 달리 활성 상태인지 여부에 관계없이 독립적으로 실행해야 하는 작업.

### Cron 예시: 매일 아침 브리핑

```bash
openclaw cron add \
  --name "Morning briefing" \
  --cron "0 7 * * *" \
  --tz "America/New_York" \
  --session isolated \
  --message "Generate today's briefing: weather, calendar, top emails, news summary." \
  --model opus \
  --announce \
  --channel whatsapp \
  --to "+15551234567"
```

전체 CLI 참조는 [Cron 작업](/automation/cron-jobs)을 참조하세요.

## 결정 플로차트

```
작업이 정확한 시간에 실행되어야 하나요?
  예 -> Cron 사용
  아니오 -> 계속...

작업이 메인 세션으로부터 격리가 필요한가요?
  예 -> Cron (격리) 사용
  아니오 -> 계속...

이 작업을 다른 주기적 확인과 일괄 처리할 수 있나요?
  예 -> 하트비트 사용 (HEARTBEAT.md 에 추가)
  아니오 -> Cron 사용

이것이 원샷 알림인가요?
  예 -> --at 을 사용한 Cron
  아니오 -> 계속...

다른 모델이나 사고 수준이 필요한가요?
  예 -> --model/--thinking 을 사용한 Cron (격리)
  아니오 -> 하트비트 사용
```

## 둘 다 결합하기

가장 효율적인 설정은 **둘 다** 사용합니다:

1. **하트비트**는 30 분마다 하나의 일괄 턴으로 일상적인 모니터링 (받은편지함, 캘린더, 알림) 을 처리합니다.
2. **Cron**은 정밀 스케줄 (일일 보고서, 주간 리뷰) 과 원샷 알림을 처리합니다.

## 비용 고려 사항

| 메커니즘    | 비용 프로필                                       |
| ----------- | ------------------------------------------------- |
| 하트비트    | N 분마다 한 턴; HEARTBEAT.md 크기에 따라 확장     |
| Cron (main) | 다음 하트비트에 이벤트 추가 (격리된 턴 없음)      |
| Cron (격리) | 작업당 전체 에이전트 턴; 더 저렴한 모델 사용 가능 |

**팁**:

- `HEARTBEAT.md`를 작게 유지하여 토큰 오버헤드를 최소화합니다.
- 여러 cron 작업 대신 유사한 확인을 하트비트에 일괄 처리합니다.
- 내부 처리만 원하면 하트비트에 `target: "none"`을 사용합니다.
- 일상 작업에는 더 저렴한 모델을 사용한 격리 cron 을 사용합니다.
