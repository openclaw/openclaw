# رسمی تصدیق (سکیورٹی ماڈلز)

یہ صفحہ OpenClaw کے **رسمی سکیورٹی ماڈلز** کی نگرانی کرتا ہے (آج TLA+/TLC؛ ضرورت کے مطابق مزید)۔

> نوٹ: کچھ پرانے لنکس پراجیکٹ کے سابقہ نام کا حوالہ دے سکتے ہیں۔

**ہدف (شمالی ستارہ):** واضح مفروضات کے تحت یہ ثابت کرنے کے لیے مشین سے جانچا گیا استدلال فراہم کرنا کہ OpenClaw اپنی
منصوبہ بند سکیورٹی پالیسی نافذ کرتا ہے (اجازت، سیشن تنہائی، اوزار کی گیٹنگ، اور
غلط کنفیگریشن سے حفاظت)۔

**یہ کیا ہے (آج):** ایک قابلِ اجرا، حملہ آور پر مبنی **سکیورٹی ریگریشن سوئٹ**:

- ہر دعوے کے لیے محدود اسٹیٹ اسپیس پر قابلِ اجرا ماڈل-چیک موجود ہے۔
- بہت سے دعووں کے ساتھ ایک مربوط **منفی ماڈل** بھی ہوتا ہے جو حقیقی بگ کلاس کے لیے کاؤنٹرایگزامپل ٹریس پیدا کرتا ہے۔

**یہ کیا نہیں ہے (ابھی):** یہ اس بات کا ثبوت نہیں کہ “OpenClaw ہر پہلو سے محفوظ ہے” یا یہ کہ مکمل TypeScript نفاذ درست ہے۔

## ماڈلز کہاں موجود ہیں

ماڈلز ایک علیحدہ ریپو میں برقرار رکھے جاتے ہیں: [vignesh07/openclaw-formal-models](https://github.com/vignesh07/openclaw-formal-models)۔

## اہم انتباہات

- یہ **ماڈلز** ہیں، مکمل TypeScript نفاذ نہیں۔ ماڈل اور کوڈ کے درمیان فرق ممکن ہے۔
- نتائج TLC کے ذریعے تلاش کی گئی اسٹیٹ اسپیس کی حدود تک محدود ہیں؛ “سبز” ہونے کا مطلب ماڈل کیے گئے مفروضات اور حدود سے آگے سکیورٹی نہیں۔
- کچھ دعوے واضح ماحولیاتی مفروضات پر انحصار کرتے ہیں (مثلاً درست تعیناتی، درست کنفیگریشن ان پٹس)۔

## نتائج کی دوبارہ تیاری

آج، نتائج ماڈلز ریپو کو مقامی طور پر کلون کر کے اور TLC چلانے سے دوبارہ تیار کیے جاتے ہیں (نیچے دیکھیں)۔ مستقبل کی ایک تکرار یہ پیش کر سکتی ہے:

- CI کے ذریعے چلائے گئے ماڈلز جن کے عوامی آرٹیفیکٹس ہوں (کاؤنٹرایگزامپل ٹریسز، رن لاگز)
- چھوٹے، محدود چیکس کے لیے ایک میزبانی شدہ “اس ماڈل کو چلائیں” ورک فلو

ابتدائی رہنمائی:

```bash
git clone https://github.com/vignesh07/openclaw-formal-models
cd openclaw-formal-models

# Java 11+ required (TLC runs on the JVM).
# The repo vendors a pinned `tla2tools.jar` (TLA+ tools) and provides `bin/tlc` + Make targets.

make <target>
```

### Gateway ایکسپوژر اور اوپن گیٹ وے کی غلط کنفیگریشن

**دعویٰ:** لوپ بیک سے آگے بغیر تصدیق کے بائنڈنگ ریموٹ کمپرومائز کو ممکن بنا سکتی ہے / ایکسپوژر بڑھاتی ہے؛ ٹوکن/پاس ورڈ غیر مجاز حملہ آوروں کو روکتے ہیں (ماڈل کے مفروضات کے مطابق)۔

- سبز رنز:
  - `make gateway-exposure-v2`
  - `make gateway-exposure-v2-protected`
- سرخ (متوقع):
  - `make gateway-exposure-v2-negative`

مزید دیکھیں: ماڈلز ریپو میں `docs/gateway-exposure-matrix.md`۔

### Nodes.run پائپ لائن (اعلیٰ ترین خطرے والی صلاحیت)

**دعویٰ:** `nodes.run` کے لیے (a) نوڈ کمانڈ اجازت فہرست اور اعلان کردہ کمانڈز، اور (b) کنفیگر ہونے پر لائیو منظوری درکار ہے؛ منظوریوں کو ری پلے سے بچانے کے لیے ٹوکنائز کیا جاتا ہے (ماڈل میں)۔

- سبز رنز:
  - `make nodes-pipeline`
  - `make approvals-token`
- سرخ (متوقع):
  - `make nodes-pipeline-negative`
  - `make approvals-token-negative`

## v1++: اضافی محدود ماڈلز (ہم زمانی، ری ٹرائز، ٹریس درستی)

یہ بعد کے ماڈلز ہیں جو حقیقی دنیا کے ناکامی کے اندازوں (غیر اٹامک اپ ڈیٹس، ری ٹرائز، اور پیغام فین آؤٹ) کے گرد فِڈیلٹی کو سخت کرتے ہیں۔

### پیئرنگ اسٹور ہم زمانی / آئڈیمپو ٹینسی

**دعویٰ:** پیئرنگ اسٹور کو ہم زمانی انٹرلیونگز کے باوجود `MaxPending` اور آئڈیمپو ٹینسی نافذ کرنی چاہیے (یعنی “چیک-پھر-لکھیں” اٹامک/لاک ہونا چاہیے؛ ریفریش سے نقول نہیں بننی چاہئیں)۔

اس کا مطلب:

- ہم زمانی درخواستوں کے تحت، آپ کسی چینل کے لیے `MaxPending` سے تجاوز نہیں کر سکتے۔
- ایک ہی `(channel, sender)` کے لیے بار بار درخواستیں/ریفریشز زندہ زیرِ التواء قطاروں کی نقول پیدا نہیں کرنی چاہئیں۔

- سبز رنز:
  - `make pairing-race` (اٹامک/لاکڈ کیپ چیک)
  - `make pairing-idempotency`
  - `make pairing-refresh`
  - `make pairing-refresh-race`
- سرخ (متوقع):
  - `make pairing-race-negative` (غیر اٹامک begin/commit کیپ ریس)
  - `make pairing-idempotency-negative`
  - `make pairing-refresh-negative`
  - `make pairing-refresh-race-negative`

### انگریس ٹریس کوریلیشن / آئڈیمپو ٹینسی

**دعویٰ:** اِن جیشن کو فین آؤٹ کے دوران ٹریس کوریلیشن برقرار رکھنی چاہیے اور فراہم کنندہ کی ری ٹرائز کے تحت آئڈیمپو ٹینٹ ہونا چاہیے۔

اس کا مطلب:

- جب ایک بیرونی ایونٹ متعدد اندرونی پیغامات بنے، تو ہر حصے میں وہی ٹریس/ایونٹ شناخت برقرار رہے۔
- ری ٹرائز سے ڈبل پروسیسنگ نہ ہو۔
- اگر فراہم کنندہ کے ایونٹ IDs موجود نہ ہوں، تو ڈی ڈیوپ محفوظ کلید (مثلاً ٹریس ID) پر واپس آئے تاکہ مختلف ایونٹس کو غلطی سے نہ گرایا جائے۔

- سبز:
  - `make ingress-trace`
  - `make ingress-trace2`
  - `make ingress-idempotency`
  - `make ingress-dedupe-fallback`
- سرخ (متوقع):
  - `make ingress-trace-negative`
  - `make ingress-trace2-negative`
  - `make ingress-idempotency-negative`
  - `make ingress-dedupe-fallback-negative`

### روٹنگ dmScope ترجیح + identityLinks

**دعویٰ:** روٹنگ کو بطورِ طے شدہ DM سیشنز کو الگ رکھنا چاہیے، اور صرف اس وقت سیشنز کو ضم کرنا چاہیے جب صراحتاً کنفیگر کیا گیا ہو (چینل ترجیح + شناختی لنکس)۔

اس کا مطلب:

- چینل مخصوص dmScope اوور رائیڈز کو عالمی ڈیفالٹس پر فوقیت حاصل ہونی چاہیے۔
- identityLinks کو صرف صراحتاً لنک کیے گئے گروپس کے اندر ضم کرنا چاہیے، غیر متعلقہ پیئرز کے درمیان نہیں۔

- سبز:
  - `make routing-precedence`
  - `make routing-identitylinks`
- سرخ (متوقع):
  - `make routing-precedence-negative`
  - `make routing-identitylinks-negative`
