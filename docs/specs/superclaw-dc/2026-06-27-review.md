# Review: Superclaw DC Decision Map and RFC-00

Date: 2026-06-27

Reviewed:

- `reports/superclaw-dc/2026-06-27-decision-map.md`
- `reports/superclaw-dc/rfc-00-control-spine.md`

Verdict: request changes.

The direction is right: start with one control spine before child RFCs. But the current drafts are not yet implementable or security-reviewable. They still leave hard-boundary semantics open while using those same semantics as enforcement inputs.

## Blockers

### 1. Tenant boundary is still undefined

`reports/superclaw-dc/rfc-00-control-spine.md:608-609` leaves open whether `tenant_id` means customer tenant, isolation cell, or both. But the RFC already uses `tenant_id` as a hard enforcement key in the request context and checks:

- `reports/superclaw-dc/rfc-00-control-spine.md:66-69`
- `reports/superclaw-dc/rfc-00-control-spine.md:293`
- `reports/superclaw-dc/rfc-00-control-spine.md:310`
- `reports/superclaw-dc/rfc-00-control-spine.md:337-338`

Fix: split the model before accepting RFC-00. Suggested fields:

- `org_id`
- `customer_tenant_id`
- `isolation_cell_id`
- `runtime_pool_id`
- `namespace_id`
- `data_residency_region`

Then define legal mappings and cross-org/MSP cases.

### 2. Capability-token ownership and first-mint order conflict

L4 owns capability token minting at `reports/superclaw-dc/rfc-00-control-spine.md:188-190`, but the inbound flow mints the initial token only at `:257`, while gateway checks `capability token tenant` earlier in enforcement at `:295`. The same RFC keeps issuer ownership open at `:612`.

Fix: define:

- who can mint the initial token
- what arrives at gateway before a token exists
- whether gateway creates a pre-token authorization context
- which service owns issuer keys
- how token audience, issuer, nonce, revocation epoch, and SVID binding work

### 3. Gateway/L5 trust boundary is blurred

The flow resolves alias in L5 before gateway builds context (`reports/superclaw-dc/rfc-00-control-spine.md:250-254`), but gateway is also responsible for rejecting invalid principal/channel bindings (`:289-297`). The decision map says gateway rejects impossible tenant/principal/channel bindings (`reports/superclaw-dc/2026-06-27-decision-map.md:71-75`) without defining whether channel adapters are inside or outside the trusted gateway boundary.

Fix: choose one:

- channel adapters are inside gateway trust boundary; or
- channel adapters are outside and must present signed alias-resolution proof to gateway.

Do not let unsigned inbound channel events create canonical principal context.

### 4. Cross-tenant delegation has too broad an escape hatch

`reports/superclaw-dc/2026-06-27-decision-map.md:527` and `reports/superclaw-dc/rfc-00-control-spine.md:383` allow cross-tenant delegation by explicit org policy, but no mandatory broker, approval model, isolation-cell constraint, data-class ceiling, or audit requirement is defined.

Fix: make cross-tenant delegation a separate exceptional flow. Require brokered delegation, explicit data-class ceilings, receipt/audit proof, and no default inheritance from ordinary org policy.

## High Findings

### 5. Memory write atomicity is not implementable as written

`reports/superclaw-dc/rfc-00-control-spine.md:280-285` requires Mem0 write plus provenance/dependency edges in the same transaction boundary. It does not say whether Mem0 and org graph share a datastore, use 2PC, or use an outbox/saga.

Fix: pick one transaction model:

- single datastore transaction
- 2PC with failure handling
- recoverable outbox plus reconciliation invariants

Add stale-read and partial-write recovery rules.

### 6. PDP/cache contract is missing

Policy fail modes exist at `reports/superclaw-dc/rfc-00-control-spine.md:318-331` and tiering at `:385-416`, but the API contract is absent: decision cache key, TTL, context hash, revocation epoch, policy epoch, data-class exclusion, principal/delegation binding, and stale-decision invalidation.

Fix: add PolicyEvaluator API and cache model before using cached allow paths.

### 7. Data-class lifecycle is under-specified

The drafts list declared/inferred/effective classes at `reports/superclaw-dc/rfc-00-control-spine.md:113-119` and sources at `reports/superclaw-dc/2026-06-27-decision-map.md:118-124`, but omit precedence, classifier confidence, override audit, reclassification, and purge/retention effects.

Fix: define a lattice and lifecycle:

- declared vs inferred conflict resolution
- confidence threshold
- override authority
- reclassification event
- data-class downgrade prohibition
- purge/retention consequences

### 8. Capability attenuation lacks lattice semantics and replay controls

The claim vocabulary is listed at `reports/superclaw-dc/rfc-00-control-spine.md:349-383`, and attenuation says "remove or narrow" at `reports/superclaw-dc/2026-06-27-decision-map.md:522-527`. Missing pieces:

- action/resource hierarchy
- data-class partial order
- deny precedence
- caveat merge rules
- token audience
- nonce or idempotency binding
- revocation epoch
- SVID-bound use
- replay prevention

Fix: make RFC-02 a blocker for accepting any delegation design.

### 9. Receipts are not tamper-evident enough

Receipt fields are listed at `reports/superclaw-dc/rfc-00-control-spine.md:417-444`, and receipt gates at `reports/superclaw-dc/2026-06-27-decision-map.md:811-819`. Missing:

- signed receipt or append-only log anchor
- policy input hash
- redaction-before-export rule
- retention class
- ordering/clock model
- verifier API
- immutable trace-to-audit boundary

Fix: add receipt integrity and trace/audit schema to RFC-00 or RFC-01. Do not defer all observability to later metrics work.

### 10. SPIRE root-CA and crypto-erasure are too deferred

The decision map lists SPIRE ops at `reports/superclaw-dc/2026-06-27-decision-map.md:642-655` and erasure at `:656-665`; RFC-00 only gives CA rotation fail mode at `reports/superclaw-dc/rfc-00-control-spine.md:485-491` and acceptance mentions erasure at `:602`.

Fix: RFC-00 does not need full runbooks, but it must set minimum acceptance fields:

- root anchor custody
- intermediate rotation cadence
- SVID TTL
- revocation propagation
- key hierarchy
- crypto-shred semantics
- legal hold conflicts
- purge verification receipt

## Medium Findings

### 11. Temporal/runtime checkpoint model lacks compatibility rules

`reports/superclaw-dc/rfc-00-control-spine.md:447-461` and `reports/superclaw-dc/2026-06-27-decision-map.md:402-425` separate durable and runtime state, but omit checkpoint schema versioning, migration, retention, encryption, and invalidation owner.

Fix: add checkpoint compatibility and invalidation contract.

### 12. Policy tiering overfits implementation language

`reports/superclaw-dc/rfc-00-control-spine.md:387-402` names Rust sidecar vs Cedar-Go before the invariant/routine contract is nailed down.

Fix: make language non-normative. First specify isolation, latency, failure semantics, and proof obligations.

### 13. Org memory lifecycle lacks object states

Memory ownership is stated at `reports/superclaw-dc/rfc-00-control-spine.md:130-138` and write flow at `:276-285`, but missing states:

- candidate
- promoted
- active
- deprecated
- tombstoned
- crypto-shredded
- legal-hold
- purged

Fix: RFC-03 needs state machine and stale-read behavior after purge.

## Missing Gates

Add these gates before the RFC is considered acceptable:

- Tenant model gate: `org_id`, customer tenant, isolation cell, namespace/pool, region, MSP/cross-org cases.
- Gateway/L5 proof gate: inbound alias resolution must be signed or inside gateway trust boundary.
- Capability lifecycle gate: first mint, attenuation, expiry, revocation, replay prevention, parent expiry, child-runtime reattach.
- PDP cache gate: cache key, TTL, policy epoch, revocation epoch, context hash, data-class exclusion.
- Memory atomicity gate: single transaction, 2PC, or recoverable outbox with reconciliation invariants.
- Data-class lifecycle gate: precedence, confidence, override audit, reclassification, purge/retention impact.
- Receipt integrity gate: required events/spans, redaction, immutability, ordering, verifier API.
- CA/erasure minimum gate: SVID TTL/root custody/revocation and key hierarchy/shred verification.

## Synthesis

The plans are directionally good, but they are not ready to become accepted RFCs. The main repair is not adding more breadth. It is tightening the first 4 invariants:

1. tenant/isolation model
2. gateway/L5 trust boundary
3. capability token issuer/lifecycle
4. policy+receipt evidence integrity

After those are fixed, child RFC order still looks right.
