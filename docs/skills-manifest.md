# Skills Manifest

The canonical skill registry provides a machine-readable manifest of every bundled skill in the
gateway image. It powers the dashboard's "Install from Canonical" install path and the
wallet-level `rain-create` gate (Phase 1 prerequisite — see
[plan](./plans/canonical-skill-registry.md)).

## Schema

### Committed manifest (`skills/manifest.json`)

Source-side metadata only. Generated by `pnpm skills:manifest` from `skills/*/SKILL.md`
frontmatter. No image-tag-derived version field (image metadata is runtime-attached).

```json
[
  {
    "contentHash": "sha256:<64-char hex>",
    "description": "One-sentence skill description",
    "emoji": "🌧️",
    "name": "rain",
    "path": "skills/rain/SKILL.md",
    "requires": {
      "env": ["AGENTGLOB_RUNTIME_URL", "AGENTGLOB_RUNTIME_TOKEN"]
    }
  }
]
```

Fields:

| Field         | Type     | Description                                                                                                |
| ------------- | -------- | ---------------------------------------------------------------------------------------------------------- |
| `contentHash` | `string` | `sha256:<hex>` of the full normalized SKILL.md (frontmatter + body). This is the drift-detection identity. |
| `description` | `string` | From the `description` frontmatter field.                                                                  |
| `emoji`       | `string` | From `metadata.openclaw.emoji` in frontmatter.                                                             |
| `name`        | `string` | From the `name` frontmatter field. Must match the directory name.                                          |
| `path`        | `string` | Repo-relative path to the SKILL.md file (`skills/<name>/SKILL.md`).                                        |
| `requires`    | `object` | From `metadata.openclaw.requires`. May include `env` (string[]) and/or `bins` (string[]) arrays.           |

### Runtime envelope (`GET /skills/bundled`)

The gateway endpoint wraps the committed manifest with image/runtime metadata:

```json
{
  "generatedAt": "2026-05-24T08:00:00.000Z",
  "imageTag": "v2026.05.24.1",
  "imageSha": "sha256:710942989f24...",
  "skills": [
    /* committed manifest entries */
  ],
  "sourceSha": "f84991e3a"
}
```

Additional fields:

| Field         | Type     | Description                                                                                            |
| ------------- | -------- | ------------------------------------------------------------------------------------------------------ |
| `generatedAt` | `string` | ISO 8601 timestamp when this response was generated.                                                   |
| `imageTag`    | `string` | Value of the `OPENCLAW_IMAGE_TAG` build-arg baked into the image. `"unknown"` if not set.              |
| `imageSha`    | `string` | Value of the `OPENCLAW_IMAGE_SHA` build-arg. `"unknown"` if not set.                                   |
| `sourceSha`   | `string` | Value of the `OPENCLAW_SOURCE_SHA` build-arg (git SHA at manifest generation). `"unknown"` if not set. |

## Normalization

`contentHash` is sha256 of the SKILL.md after normalization:

1. Convert line endings to LF (`\r\n` → `\n`, `\r` → `\n`).
2. Strip trailing whitespace from each line.
3. Remove all trailing blank lines, then append exactly one trailing newline (`\n`).

The generator applies this normalization to the source files on disk so the committed hash
matches the image's baked-in content.

## API endpoints

Both endpoints require gateway bearer auth (`Authorization: Bearer <token>`).

### `GET /skills/bundled`

Returns the runtime envelope (see above). The `skills` array is read from
`/app/skills/manifest.json` (in the container) or the repo's `skills/manifest.json`
(in development).

### `GET /skills/bundled/:name`

Returns the raw SKILL.md body for one skill. `Content-Type: text/markdown`.

Errors:

- `404` if the skill name is not in the manifest or the file is missing.
- `401` if the bearer token is invalid.

## Tooling

| Command                      | What it does                                                                |
| ---------------------------- | --------------------------------------------------------------------------- |
| `pnpm skills:manifest`       | Walk all `skills/*/SKILL.md`, normalize files, write `skills/manifest.json` |
| `pnpm skills:manifest:check` | Fail if the committed manifest is out of sync with the SKILL.md files       |

## SKILL.md frontmatter requirements

For a skill to appear in the manifest, its `SKILL.md` must have a valid YAML frontmatter
block with `name`, `description`, and `metadata.openclaw.emoji`:

```markdown
---
name: my-skill
description: "One-sentence description of what the skill does."
metadata: { "openclaw": { "emoji": "✨", "requires": { "env": ["MY_ENV_VAR"] } } }
---

# My skill body...
```

Skills without a valid `metadata.openclaw` block are skipped (logged as warnings) and do
not appear in the manifest.

## Pre-commit hook

The pre-commit hook (`git-hooks/pre-commit`) automatically:

1. Regenerates `skills/manifest.json` if any `skills/*/SKILL.md` file is staged.
2. Normalizes the staged SKILL.md files (LF endings, trimmed trailing whitespace).
3. Fails the commit if the manifest is still out of sync after regeneration.

## CI

The `skills-manifest-check` CI job runs `pnpm skills:manifest:check` on every push and
pull request. It fails if the committed `skills/manifest.json` does not match what the
generator would produce from the current SKILL.md files.

## Provenance (Phase 1+)

When the dashboard installs a skill from the canonical endpoint, it writes a
`.openclaw-source.json` sibling file alongside the SKILL.md in the agent workspace:

```json
{
  "source": "canonical",
  "name": "rain",
  "contentHash": "sha256:abcd...",
  "installedAt": "2026-05-24T08:00:00.000Z",
  "installedFromImageTag": "v2026.05.24.1",
  "installedFromImageSha": "sha256:710942..."
}
```

The wallet-level `rain-create` gate (Phase 1 follow-up) checks:

1. `.openclaw-source.json` exists with `source: "canonical"` and `name: "rain-create"`.
2. The recorded `contentHash` matches a trusted canonical hash accepted by current policy.
3. The installed SKILL.md on disk re-hashes to that same `contentHash`.

See [canonical-skill-registry plan](./plans/canonical-skill-registry.md) §12 for details.
