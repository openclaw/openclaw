# OpenClaw 事件响应计划

## 1. 检测和分类

我们监控来自以下来源的安全信号：

- GitHub 安全公告（GHSA）和私有漏洞报告。
- 当报告不敏感时的公共 GitHub 问题/讨论。
- 自动信号（例如 Dependabot、CodeQL、npm 公告和秘密扫描）。

初始分类：

1. 确认受影响的组件、版本和信任边界影响。
2. 使用存储库 `SECURITY.md` 范围和超出范围规则将其分类为安全问题 vs 强化/无操作。
3. 事件所有者做出相应响应。

## 2. 评估

严重性指南：

- **严重**：包/发布/存储库被破坏、主动利用或未认证的信任边界绕过，具有高影响控制或数据暴露。
- **高**：需要有限前提条件的已验证信任边界绕过（例如已认证但未授权的高影响操作），或 OpenClaw 拥有的敏感凭证暴露。
- **中**：具有实际影响但可利用性受限或具有实质性先决条件的重大安全弱点。
- **低**：深度防御发现、范围狭窄的拒绝服务，或没有展示的信任边界绕过的强化/奇偶性差距。

## 3. 响应

1. 向报告者确认收到（敏感时私下）。
2. 在支持的版本和最新的 `main` 上重现，然后实施并验证带有回归覆盖的补丁。
3. 对于严重/高事件，尽快准备补丁版本。
4. 对于中/低事件，在正常发布流程中补丁并记录缓解指导。

## 4. 沟通

我们通过以下方式沟通：

- 受影响存储库中的 GitHub 安全公告。
- 固定版本的发布说明/变更日志条目。
- 关于状态和解决方案的直接报告者跟进。

披露政策：

- 严重/高事件应进行协调披露，在适当情况下发布 CVE。
- 低风险强化发现可能在发布说明或公告中记录，无需 CVE，具体取决于影响和用户暴露。

## 5. 恢复和后续行动

在发布修复后：

1. 验证 CI 和发布工件中的补救措施。
2. 运行简短的事件后审查（时间线、根本原因、检测差距、预防计划）。
3. 添加后续强化/测试/文档任务并跟踪其完成。
