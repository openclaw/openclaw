**Concrete shape of (a):**

The Fleet Orchestrator is a real openclaw agent (`~/.openclaw/agents/fleet-orchestrator/`) with `IDENTITY.md` text along the lines of:

> Receive a task message in JSON form `{taskId, goal, assignedAgentId, capabilities}`. Call `sessions_spawn` exactly once with `{ agent: assignedAgentId, prompt: goal, parentTaskId: taskId }`. Wait for the result. Emit it back unchanged. Do not editorialise. Do not retry.

The extension's HTTP `POST /orchestrator/tasks` flow becomes:

1. Routing engine decides `assignedAgentId` (pure function).
2. Extension writes `<taskId>.json` (state: `assigned`).
3. Extension **enqueues a message into the Fleet Orchestrator's session inbox** with the JSON above. (The session-inbox seam is the existing `delivered-message` channel path — same one Telegram/email use to give the orchestrator something to react to. The extension already consumes inbound channel deliveries via `registerInteractiveHandler` / channel plugins.)
4. The orchestrator's prompt loop wakes, parses the JSON, calls `sessions_spawn(...)`. The model is the cheap one in `models.json`; the prompt template reduces to "translate JSON → tool call."
5. `sessions_spawn`'s lifecycle hooks (`emitLifecycleHooks: true` at `src/agents/openclaw-tools.subagents.sessions-spawn.test-harness.ts:78`) emit observable events the extension can hook via `api.registerHook(["sessions.spawn.complete", ...], handler)` — verifying this hook surface is the **one additional Unit 1 follow-up**, deferred to Open Questions if not confirmed during Unit 5.
6. Extension watches the orchestrator's session for `model.completed` (already in MC's reader) AND for `subagent_done`-shaped events (verify in Unit 5); transitions task to `awaiting_approval` or `done`.

**Risks acknowledged:**

- **R-LLM-1: Cheap model freelances** (returns commentary instead of a tool call). Mitigation: orchestrator IDENTITY.md uses an extremely tight prompt + a worked example; `models.json` picks a reliable instruction-follower (NOT `gemini-flash-lite`). Recovery: extension times out at 60s, transitions task → `failed` with `error.code: "specialist_timeout"`.
- **R-LLM-2: Cheap model splits the call into multi-turn**. Mitigation: prompt explicitly says "exactly one tool call, no preamble." Worst case: same timeout path.
- **R-LLM-3: Refusal**. Cheap-model refusals to call `sessions_spawn` are rare for non-mutating goals but plausible for politically loaded ones. Mitigation: the orchestrator's role is plumbing — it is told it is a router, not a moral agent.

## A-B3. Bearer-token issuance flow — concrete spec

**Precedent inventory:**

| Pattern in tree                                               | Reference                                                                                                                                                                                                                           |
| ------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `~/.openclaw/credentials/<name>.json` for channel credentials | `extensions/discord/src/security-audit.ts:77` (`~/.openclaw/credentials/discord-allowFrom.json`); operator-managed JSON files: `imap-shared-*.json`, `xneelo-*.json`, `telegram-*.json` (verified by `ls ~/.openclaw/credentials/`) |
| Operator-supplied env var (the dashboard-launcher pattern)    | `extensions/dashboard-launcher/src/command.ts:94`, `:124` reads `process.env.MC_AUTH_TOKEN`                                                                                                                                         |
| MC server.js bearer enforcement                               | `(MC) server.js:19-27` boots with `MC_AUTH_TOKEN`; refuses to start in public mode without ≥32 chars; `:111-117` enforces bearer on WS upgrades                                                                                     |
| Per-channel token files                                       | `~/.openclaw/credentials/telegram-pairing.json` (writeable by setup wizard)                                                                                                                                                         |

**Recommended issuance flow (mirrors what dashboard-launcher does today, extended with a generation step):**

1. **Generation:** New CLI verb `openclaw orchestrator init` (registered by the extension via `api.registerCli`). Generates a 32-byte hex token via `crypto.randomBytes(32).toString("hex")`. Writes to **two** files in a single transaction (write tmp + rename, mode `0o600`):
   - `~/.openclaw/credentials/orchestrator-bearer.json` — `{"version":1,"token":"<hex>","createdAt":"<iso8601>"}`. **Authoritative.**
   - Appends/replaces `OPENCLAW_ORCHESTRATOR_TOKEN=<hex>` in the operator's `.env.local` for MissionControl (path resolved via the same logic dashboard-launcher uses to find MC's cwd, currently `process.env.MISSION_CONTROL_DIR` or `~/openclaw.ai/MissionControl/`). If the MC dir cannot be located, prints the line and asks the operator to add it manually. **Best-effort.**

2. **Persistence precedence:** The credentials file is the source of truth. The extension reads it at `register(api)` time; if missing, the extension's HTTP routes return `503 ORCHESTRATOR_NOT_INITIALIZED` with a body explaining how to run `openclaw orchestrator init`. **The plan's `~/.openclaw/openclaw.json extensions.orchestrator.authToken` storage is rejected** — the credential should not live in the main config (which gets `cat`-printed in support flows and synced more aggressively).

3. **Drift behaviour:** If `.env.local` token does not match `credentials/orchestrator-bearer.json`, MC's API routes get 401 from the extension. MC degrades to disk-fallback reads (already specified in plan §Cross-Repo Contract §Direction MC↔openclaw). MC surfaces a banner: "Orchestrator bearer mismatch — run `openclaw orchestrator rotate-token`." (Banner copy is Unit 9 work.)

4. **Rotation:** New CLI verb `openclaw orchestrator rotate-token`. Generates a fresh token; updates both files atomically (write new MC `.env.local` line, then write new credentials file, then `kill -HUP` the openclaw process if running so the route handlers re-read — falls back to "restart `openclaw dashboard`" instructions if HUP not implemented). **In-flight tasks under `awaiting_approval` are unaffected** — they live on disk and the next approve/reject just uses the new token. **In-flight `in_progress` tasks** continue (the spawn-side has no bearer interaction); the only impact is MC's poll briefly 401s until the operator restarts MC.

5. **First-issuance guard:** The extension refuses to enter `mode: "live"` unless the credentials file is present and the file mode is `0o600`. This catches the "operator forgot to init" case before live traffic.
