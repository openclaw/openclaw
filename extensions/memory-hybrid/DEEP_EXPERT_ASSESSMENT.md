## Що це таке?

`memory-hybrid` — це плагін довгострокової пам'яті для AI-агента OpenClaw. Він дозволяє агенту **запам'ятовувати** факти про користувача між сесіями, **пригадувати** їх у потрібний момент і **забувати** на вимогу (GDPR).

Це не просто "зберегти текст і знайти по ключовому слову". Це повноцінна **когнітивна архітектура** — система, що імітує принципи роботи людської пам'яті.

## Ключові технічні рішення

### 1. 🎚️ Гібридне ранжування — 7 каналів

Це головна фішка плагіну. При пошуку спогаду результати ранжуються не лише за схожістю тексту, а за **комбінацією 7 сигналів**:

| Канал                | Вага | Для чого                                                  |
| -------------------- | ---- | --------------------------------------------------------- |
| Векторна схожість    | 42%  | Наскільки текст семантично близький до запиту             |
| Важливість           | 16%  | Пріоритет факту — "переїзд" важливіше за "обід"           |
| Свіжість             | 10%  | Нові спогади сильніші за старі (decay ~50% за 30 днів)    |
| Часова відповідність | 10%  | Подія "вчора" має перевагу над подією "рік тому"          |
| Зв'язність у графі   | 8%   | Факт, пов'язаний з багатьма сутностями, отримує бонус     |
| Підкріплення         | 8%   | Чим частіше пригадуємо — тим сильніше (нейропластичність) |
| Емоційність          | 6%   | Емоційно яскраві спогади підсилюються                     |

Формула: `Score = 0.42·V + 0.16·I + 0.10·R + 0.10·T + 0.08·G + 0.08·Rf + 0.06·E`

Ваги підібрані емпірично — семантика домінує, але інші канали значно покращують релевантність порівняно з "голим" vector search.

**Оцінка: 10/10** — рівень, який зустрічаємо лише в дослідницьких прототипах.

---

### 2. 💾 Буфер робочої пам'яті (L1 кеш)

Не кожна фраза користувача заслуговує на збереження в базі. Буфер — це "зал очікування", де факти визрівають перед промоцією:

- **По частоті**: згадали >2 рази → це важливо → зберігаємо
- **По пріоритету**: importance >0.75 → відразу зберігаємо
- **По категорії**: entity (Вова, Київ) або decision → автопромоція
- **Fuzzy дедуплікація**: Відстань Левенштейна визначає, чи дві фрази — це одне й те саме ("Я Вова" ≈ "Мене звати Вова")

Евікція захищає промотовані записи — видаляються лише непромотовані, найстаріші.

**Оцінка: 8/10** — елегантне рішення, зменшує шум у базі і кількість API-викликів. Єдине зауваження: поріг Левенштейна зашитий хардкодом, варто б винести в конфіг.

---

### 3. 🕸️ Граф знань

LLM витягує сутності та зв'язки з кожного збереженого факту:

```
Факт: "Вова працює в Києві і знає Python"
→ Ноди: Вова (Person), Київ (Place), Python (Language)
→ Ребра: Вова --[працює_в]--> Київ, Вова --[знає]--> Python
```

При пригадуванні — **2-хоповий обхід** знаходить зв'язки через проміжні ноди. Запитуєш про Python → отримуєш контекст про Django, FastAPI і того, хто з ними працює.

Зберігається у **JSONL append-only форматі** — при збереженні нових записів дописуються лише нові рядки. Є `compact()` для повного перезапису файлу при обслуговуванні. Mutex-лок запобігає гонкам при паралельних записах.

**Оцінка: 8/10** — потужна фіча. Зауваження: пошук зв'язків по `findEdgesForTexts()` використовує `includes()` (O(n²)) — на графах з 10K+ ребер може бути повільно. Для малих графів — ідеально.

---

### 4. 📸 Розумне захоплення (Smart Capture)

Два режими:

1. **Smart (LLM)**: Один API-виклик витягує з розмови: category, importance, happenedAt, validUntil, emotionalTone, emotionScore. Шість параметрів за одну ціну.

2. **Rule-based (без API)**: Regex-фільтри визначають, чи варто зберігати текст. Категорію визначають ключові слова. Fallback, коли LLM недоступний.

Є також **prompt injection protection** — `looksLikePromptInjection()` блокує маніпулятивні фрази ("забудь все", "ігноруй інструкції").

**Оцінка: 9/10** — zero-cost extraction в одному виклику — це дуже розумно з точки зору API бюджету.

---

### 5. 🔄 Самовиправна пам'ять

Перед збереженням нового факту система шукає суперечності з існуючими:

```
Стара: "Живе в Києві"
Нова:  "Переїхав до Львова"
→ LLM: СУПЕРЕЧНІСТЬ → дія: "update" → старий факт замінюється
```

Чотири можливі вердикти: `update` (заміна), `keep_both` (різні теми), `ignore_new` (дублікат), `keep_both` (уточнення).

**Оцінка: 8/10** — критично важлива фіча для довгострокової пам'яті. Без неї база поступово наповнювалася б суперечливими фактами.

---

### 6. 😴 Консолідація ("Режим сну")

CLI-команда `ltm consolidate` запускає процес:

1. Визначає cosine similarity між усіма парами фактів
2. Групує в кластери (поріг 0.85)
3. LLM зливає кожен кластер в один факт
4. Видаляє оригінали, зберігає злитий результат

**Оцінка: 7/10** — O(n²) алгоритм кластеризації працює для <1000 фактів. Для великих баз потрібен locality-sensitive hashing або інший наближений підхід.

---

### 7. 📐 Embedding-модуль

Підтримка OpenAI (`text-embedding-3-small/large`) та Google (`gemini-embedding-001`). LRU-кеш на 100 записів — повторний embed тексту не викликає API. Retry з exponential backoff (1с → 2с → 4с) при 429/503.

**Оцінка: 9/10** — LRU кеш, retry з backoff, підтримка двох провайдерів. Чисто і надійно.

---

### 8. 🗄️ Шар даних (MemoryDB)

LanceDB як vector store. Recall count зберігається в in-memory Map і записується у базу батчем при обслуговуванні, а не при кожному пригадуванні. `deleteOldUnused()` для автоматичного прунінгу — спогади, які ніхто не пригадав за 90 днів, видаляються з ймовірністю 5% на кожну сесію.

**Оцінка: 8/10** — надійне рішення. LanceDB — хороший вибір для вбудованого vector store без зовнішніх залежностей.

## 🏆 Підсумок

### Загальна оцінка: **8.0 / 10**

**memory-hybrid** — це production-ready система когнітивної пам'яті для AI-агента. Вона значно перевершує типові open-source рішення, які обмежуються простим "зберегти → знайти по вектору".

**Головні переваги перед конкурентами:**

- 7-канальне ранжування замість 1
- Буфер визрівання замість прямого запису в базу
- Граф знань з multi-hop traversal
- Самовиправна пам'ять з LLM-перевіркою
- Zero-cost feature extraction

**Де варто покращити:**

- Додати тести для scoring формули
- Винести пороги (Levenshtein, similarity) в конфіг
- Для великих баз (>1000 фактів) потрібна оптимізація консолідації
- Додати метрики (час відповіді, кількість API викликів за сесію)
